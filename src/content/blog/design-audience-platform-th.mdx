## Introduction

โพสต์นี้เป็นบทความที่ 6 ในซีรีส์ backend system design แล้วนะ สูตรเดิมที่คุ้นเคยกันดี: เริ่มจากของที่ทื่อที่สุดเท่าที่มันจะ work ได้ก่อน แล้วปล่อยให้ข้อจำกัดหน้างานค่อย ๆ ดันมันให้เข้ารูปเข้ารอย

ระบบที่จะคุยกันวันนี้ คือระบบที่สองโพสต์ก่อนหน้าพูดถึงเอาไว้ ในโพสต์ omnichannel (#3) เราทิ้งท้ายไว้ว่า `ใช้ audience ที่เตรียมไว้แล้วเป็น chunk แทนที่จะมา query เอง` ส่วนในโพสต์ icon serving (#5) เราบอกว่า eligibility `มาจาก Audience domain — rule engine ที่เขียนออกมาเป็น per-user membership store` นี่แหละคือโพสต์นั้น

หน้าที่ของ **Audience Platform** มีอยู่ข้อเดียว คือตอบคำถามว่า **ใคร?** — ใครมีสิทธิ์ ใครเข้าเกณฑ์ — แล้วส่งคำตอบนั้นให้ใครก็ตามที่ต้องการมันไป

ความท้าทายอยู่ที่คำว่า `ใครก็ตามที่ต้องการมัน` นี่แหละ เพราะ consumer สองตัวที่เรา build มาก่อนหน้านี้ อยากได้คำตอบคนละทรงกันเลย:

- **Bulk, chunked list** — ฝั่ง omnichannel backbone อยากได้ประมาณ `campaign C เล็งไป user 40 ล้านคนนี้` แล้วซอยเป็น chunk ราว ๆ 40,000–60,000 ก้อน ให้มันไล่ยิงผ่าน delivery workflow ได้ — อ่าน audience ทั้งก้อนรวดเดียว เรียงลำดับ
- **Per-user membership store** — ฝั่ง icon serving อยากได้ point lookup ตอน request มาถึง: user `10001` มีสิทธิ์กับ campaign 5 มั้ย เดี๋ยวนี้เลย

audience logic ชุดเดียวกันเป๊ะ — gold-tier user ใน TH ที่เติมเงินมาภายใน 30 วัน — แต่ออกมาเป็นสอง output shape ที่ access pattern แตกต่างกัน

> **P.S.** icon serving คือตัวอย่างที่จับต้องได้ ยกมาจากโพสต์ #5 แต่มองมันเป็นตัวแทนเฉย ๆ นะ ไม่ว่าเรากำลัง build อะไรอยู่ ถ้ามันต้องถามว่า `user คนนี้มีสิทธิ์เดี๋ยวนี้มั้ย` มันก็อ่าน membership store แบบเดียวกันนี้แหละ

campaign คือตัวอย่างหลักที่เราใช้ไล่ตลอดทั้งโพสต์ แต่คำถาม **ใครมีสิทธิ์บ้าง?** โผล่ไปไกลกว่าเรื่อง marketing เยอะเลย platform ตัวเดียวกันนี้ ไม่ต้องแก้โค้ดเลย ก็ไปรับใช้งานพวกนี้ได้ด้วย:

- **Feature rollout** — ปล่อย feature ใหม่ให้ segment เล็ก ๆ ก่อน (เช่น gold-tier ใน TH) แล้วค่อยขยายวงกว้างขึ้น
- **Experimentation** — แบ่ง user เข้ากลุ่ม A/B test แล้วรักษาให้กลุ่มนั้นนิ่งอยู่ข้าม session
- **Personalization** — ตัดสินว่า banner, recommendation shelf, หรือ offer แบบไหนที่ user แต่ละ segment ควรเห็น
- **Risk แล้วก็ compliance list** — fraud watchlist กับ opt-out suppression list ก็เป็น membership lookup แบบเดียวกันเลย แค่เอาไปใช้ **กันคนออก** แทนที่จะรับเข้า

แล้วมันก็โผล่มาในชื่ออื่นด้วยนะ ตามแต่ละวงการ — segment engine ที่อยู่ข้างใน **CDP** (Customer Data Platform), **DMP** (Data Management Platform) ในวงการ ad tech, หรือส่วน targeting ของ feature-flag service ชื่อเรียกต่างกัน แต่ design คล้าย ๆ กัน

มาลองดูคร่าว ๆ ว่าใน Blog นี้จะพูดถึงอะไรบ้าง

**Scale ต่ำ — กล่องเดียวยังพอไหว:**

1. **Inline SQL** ตอนส่ง — work จนกว่า query จะโตไป 40 ล้านแถวกลางที่ส่งอยู่
2. **Precompute ตาม schedule** — สร้าง segment ไว้ล่วงหน้า แล้วมาอ่านตอนส่ง
3. **Read replica** — บีบเอาสิ่งที่กล่องเดียวทำได้ให้สุดก่อนไปทาง distributed
4. **Analytical engine** — พอ workload มันกลายเป็น analytics เต็มตัว OLTP ก็เริ่มไม่พอ

**Scale สูง — ตอนนี้เข้าเขต distributed system เต็มตัว:**

1. **Data lake** — เอาข้อมูลจากสิบกว่าระบบมากองไว้ที่เดียว build มาเพื่อ analytical read โดยเฉพาะ
2. **Spark** — พอ SQL เริ่มพูด logic ไม่ไหว rule ก็ต้องกลายเป็น code
3. **Near-real-time** — บาง rule นั้น `nightly` มันสดไม่พอ
4. **Reusable segmentation service** — เลิก build job แยกทีละ use case

เริ่มจากของง่าย ๆ กันก่อน

## The naive version — a SQL query at send time

สมมติเราอยากส่งโปรโมชันหา gold-tier user ในไทยที่เติมเงินมาภายใน 30 วันที่ผ่านมา เวอร์ชันแรกแบบซื่อ ๆ ที่สุด ก็คือเขียนตามนั้นเป๊ะ ๆ เป็น query แล้วรันตอนที่กำลังจะส่งเลย:

```sql
SELECT user_id
FROM users
WHERE tier = 'gold'
  AND country = 'TH'
  AND last_topup_at > now() - interval '30 days';
```

ฝั่ง sender รัน query นี้ ได้ list ของ ID มา แล้วก็เริ่มยิงเลย ถ้ามีแค่หลักพันคนแบบนี้มัน work ดีมาก ๆ ship ไปได้เลย rule อยู่ที่เดียว สดตลอดเวลา (เพราะ query วิ่งบนข้อมูล live) แล้วก็ไม่มีอะไรต้องคอยดูแล

> **P.S.** ถ้านี่คือเรา หยุดจุดนี้ได้เลย ต่ำกว่าพันคนกับ rule แค่หนึ่งสองเงื่อนไข ไม่มี join ข้ามตาราง inline query เอาแค่นี้ก็พอ ถ้าเราไม่ได้ใช้ทำท่าอะไรยาก ของข้างล่างคงไม่ได้ทำให้ระบบ **ถูกต้อง** ขึ้นหรอก แค่ทำให้รอด scale กับความซับซ้อนที่เราในตอนนี้ยังไม่มีต่างหาก

อ่ะคราวนี้ ถ้ามันใหญ่ขึ้นล่ะ? ปัญหาสามอย่างก็จะโผล่มาพร้อมกัน:

- **query เริ่มหนัก** พอ row ไปแตะหลักล้านเข้า แถม `last_topup_at` ก็มาจากอีกตาราง `tier` ก็มาจากอีกตารางหนึ่งอีก มันเลยกลายเป็น multi-join full scan บน **production database** ตัวเดียวกับที่รับ live traffic อยู่นั่นแหละ การส่ง marketing ทีนึงก็เลยอาจทำแอปสะดุดได้เลย
- **การส่งไปค้างเติ่งรอมันอยู่** delivery job เริ่มงานไม่ได้จนกว่า query จะคืนค่ามา พอ query กินเวลาเป็นนาที (หรือ timeout ไปเลย) campaign ก็เลยนั่งรอเฉย ๆ แล้วพอ retry ทีก็ต้องรันของแพง ๆ ก้อนนี้ใหม่ทั้งดุ้นอีกรอบ
- **reuse ไม่ได้เลย** campaign ถัดไปก็อยาก gold-tier ใน TH เหมือนกัน แค่ topup window ต่างกันนิดเดียว ทุก campaign ก็เลยต้องมานั่งคำนวณ audience ใหม่จากศูนย์ กระหน่ำ DB ซ้ำอีกรอบ แล้วไม่มีใครตอบได้เลยว่า segment นี้ใหญ่แค่ไหน ถ้าไม่ลองรัน query ดูก่อน

จะเห็นว่าปัญหาที่มี: **การคำนวณ audience ตอนส่ง บน DB ที่ live อยู่ มันไม่ scale** ทางแก้ก็คือย้ายการคำนวณออกจาก hot path แล้วทำมันไว้ **ก่อน** ที่จะถึงเวลาส่ง

## Precompute on a schedule — build the segment, then read it

**ปัญหา:** audience query มันถูกคาดหวังว่า มันน่าจะหนักพอสมควร แล้วเราดันไปรันมันตอนที่แย่ที่สุดพอดี คือตอนกำลังจะส่งนี่แหละ ทั้งที่ input ของมันแทบไม่ขยับเลยในระดับนาที gold tier ใน TH ที่เติมเงินมา 30 วัน ก็แทบเป็น list เดียวกันไม่ว่าจะเช็คตอนบ่ายสองหรือตีสอง

**วิธีแก้:** รัน query **ล่วงหน้า** ตาม schedule เก็บผลลัพธ์ไว้ แล้วให้ตอนส่งแค่ **อ่าน** มันเฉย ๆ nightly job ตัวนึงคำนวณ member ของแต่ละ segment แล้วเขียนลงตาราง `segment_members`:

```sql
-- nightly, per segment: delete + insert in ONE transaction,
-- so a re-run or a retry rebuilds the segment instead of doubling it
BEGIN;
DELETE FROM segment_members WHERE segment_id = 'gold_th_active';
INSERT INTO segment_members (segment_id, user_id, computed_at)
SELECT 'gold_th_active', user_id, now()
FROM users
WHERE tier = 'gold' AND country = 'TH'
  AND last_topup_at > now() - interval '30 days';
COMMIT;
```

รายละเอียดที่ต้องคำนึงถึง:

- job นี้ต้อง **idempotent** — ความผิดพลาดคลาสสิกคือใช้แค่ `INSERT` เดี่ยว ๆ เพราะ retry รอบแรกหลัง crash จะเพิ่ม member ซ้ำสองเท่าทันที การ delete-แล้ว-insert อยู่ใน transaction เดียว (มี primary key บน `(segment_id, user_id)` เป็นตัวกันหลัง) ทำให้ re-run กี่ครั้งก็ปลอดภัยเสมอ
- พอ audience ถูก precompute ไว้แล้ว มันก็ data stale ไปได้แบบเงียบ ๆ เพราะงั้น **alert ให้ดูจากอายุของ `max(computed_at)` เป็นหลัก อย่าพึ่งแค่ alert ตอน job fail** เพราะ nightly job ที่ตายไปเงียบ ๆ คืนวันศุกร์ แปลว่า audience เก่าไป 3 วันแล้วตอนถึงวันจันทร์ แล้ว `job ไม่ page มาหาเรา` ก็ไม่ได้แปลว่า `ข้อมูลยังสด` เลยนะ

ตอนนี้การส่งก็เลยเป็นแค่ read ถูก ๆ: `SELECT user_id FROM segment_members WHERE segment_id = 'gold_th_active'` งานหนัก ๆ มันเกิดไปตั้งแต่เมื่อคืนแล้ว ตอนที่ไม่มีใครนั่งรออยู่

จุดนี้ได้มาเยอะเลยนะ การส่งกลายเป็นแค่ `segment_id` lookup ธรรมดา ไม่ใช่ multi-join scan อีกต่อไปแล้ว query แพง ๆ ก็ไปรันตอน off-peak แทน แล้ว segment ก็กลายเป็นของที่เรา reuse ได้ด้วย แต่มันก็แลกปัญหาเก่าไปกับปัญหาใหม่สองอันแทน:

- **Freshness.** list มันสดได้แค่เท่าที่ rebuild ล่าสุดพาไปเท่านั้น รัน batch job (cron job) วันละครั้ง list ก็เก่าได้ถึง 24 ชั่วโมง ไม่สำคัญหรอกว่า job จะรันเที่ยงคืน ตีหนึ่ง หรือตีสาม สิ่งที่กำหนด staleness คือ **ความถี่ที่มันรัน** ไม่ใช่เวลาที่รันเป๊ะ ๆ user ที่เติมเงินไปเมื่อเช้านี้ ก็จะยังไม่ `gold-tier active` จนกว่า rebuild รอบถัดไปจะมาถึง สำหรับโปรโมรายสัปดาห์นี่ไม่มีปัญหาเลย แต่ถ้าเป็น `เตือน user ที่เพิ่ง abandon cart ไป` มันใช้ไม่ได้เลยนะ จำจุดนี้ไว้ดี ๆ นี่แหละคือสิ่งที่จะบังคับให้ต้อง streaming ทีหลัง
- **มันก็ยังรันบน production DB อยู่ดี** ตัว rebuild ต้องมีที่ให้ลง ตารางใหม่ `segment_members(segment_id, user_id, computed_at)` ที่เขียนโดย job ที่ยังไปแตะ Postgres ตัวเดียวกับที่รับ live traffic อยู่ดี แล้วตารางนี้ก็ไม่เล็กด้วยนะ user คนนึงเข้าได้หลาย segment เพราะงั้น worst case มันถือได้ถึง **S × N** row (S = จำนวน segment ในองค์กร, N = จำนวน user ทั้งหมด) พอ segment เพิ่มขึ้นแล้ว rule ก็ซับซ้อนขึ้น (cross join, windowed aggregate ข้ามปีของ transaction) กล่องเดียวที่แบกทั้ง live traffic แล้วก็ batch analytical นี้ก็เริ่มช้าลงแล้วก็เริ่มพังในที่สุด

ก่อนจะไปหยิบอะไรที่หนักกว่านี้ ขอพูดถึงทางแก้ freshness แบบง่าย ๆ ก่อน นั่นคือ **แค่รัน job ให้ถี่ขึ้น** ไม่มีอะไรห้ามเราจากการรัน delete-and-insert ก้อนเดิมทุกชั่วโมงแทนที่จะรันคืนละครั้งเลยนะ production DB เดิม ตารางเดิม ไม่ต้องมี infrastructure ใหม่เลยสักตัว แค่ตั้ง interval ให้สั้นลงบน cron schedule เดิมเท่านั้นเอง แต่จุดนี้มีเงื่อนไขซ่อนอยู่นะ — มันฟรีได้ก็ต่อเมื่อ traffic กับจำนวน row ยังน้อยอยู่ แบบ user ต่ำกว่าพันคนกับ rule ง่าย ๆ ที่บอกไปตอนต้น ถ้าไม่ใช่ การรันถี่ขึ้นก็ไม่ได้หนีปัญหา load ไปไหนเลย แค่ทำให้ production DB ของเราเองต้องรับ query ก้อนนั้นบ่อยขึ้นเท่านั้นเอง

ท่านี้ใช้ได้จนกว่า full rebuild เองจะกินเวลานานเกินกว่าที่ window เหลือให้ พอคำนวณทุกอย่างจากศูนย์ทุกชั่วโมง สุดท้ายมันก็กินเวลานานกว่าหนึ่งชั่วโมงที่มีซะเอง นั่นแหละคือ trigger ของ **incremental refresh** ที่ซีรีส์นี้จะย้อนกลับมาพูดถึงอีกที ตอนที่มี data lake ให้พึ่งพิงแล้ว

ปัญหาที่สองเป็นเรื่องเชิงโครงสร้างแล้ว การ precompute อย่างเดียวไม่พอ มันต้องไปเกิดใน **ที่ที่ไม่ใช่ live transactional database** ด้วย แต่ `ที่อื่น` ไม่ได้แปลว่า cluster ตั้งแต่ตอนนี้นะ แล้ว step ถัดไป ก็เป็นแค่การบอกว่า ขอแยกจาก live traffic เฉย ๆ

## Squeeze the single box first — a read replica

โพสต์ส่วนใหญ่ชอบกระโดดจาก `Postgres ตัวเดียว` ไปที่ `Spark cluster` เลย คนอ่านก็เลยเข้าใจไปว่า distributed compute คือ step ที่สาม ทั้งที่มันไม่ใช่เลย มีท่าที่ประหยัดกว่านั้นอยู่

**เอา segment job ออกจาก primary** ตอนนี้ production กำลังทำสองงานที่ต่างกันอยู่ในตัวเดียว: รับ live transactional traffic แล้วก็ถือตาราง `segment_members` ที่ batch นี้เขียนเข้าไป แยกมันออกจากกันซะ:

- **Reads** — ชี้ query ของ batch ไปที่ **read replica**: สำเนาที่ live ของ primary คอยซิงค์ผ่าน replication รับได้แต่ read อย่างเดียว ไม่รับ write เลย เพราะมันเป็น read-only มันเลยรับ read traffic มาจากไหนก็ได้ ทั้งจากตัว app เองแล้วก็จาก batch job ตัวนี้ ส่วน write ทุกตัว ไม่ว่าจาก app หรือจาก job นี้ ก็ยังต้องวิ่งผ่าน primary อยู่ดี แทบทุก managed database มี replica ให้เปิดใช้แบบติ๊กช่องเดียวจบ แล้ว replication lag แค่ไม่กี่วินาทีก็ไม่มีผลอะไรเลย สำหรับ audience use case
- **Output** — เปลี่ยนจาก `segment_members` ใน DB เอาไปลงที่อื่นไปเลย: database เล็ก ๆ ที่ dedicate ไว้เก็บ segment โดยเฉพาะ, plain file บน object storage (S3/GCS) ที่ sender อ่านเองได้เลย, หรือ Redis ที่เก็บ segment เป็น set

ทั้งหมดนี้ยังไม่ต้องใช้ Spark หรือ warehouse เลยนะ การคำนวณก็ยังเป็น SQL query ตัวเดิมเป๊ะ แค่ย้ายที่อยู่ ให้ production ทำแค่งานเดียวที่มันควรทำคือรับ live traffic นี่คือการเปลี่ยน infrastructure ที่ถูกมาก ๆ แต่ซื้อเวลาให้หลายบริษัทได้เป็นปี ๆ เลย

- **Icon serving** — อยากได้ eligibility ของ user คนเดียวแบบ **real-time read** เป็น point lookup เดี๋ยวนี้เลย

## Move the analytics to an analytical engine — from OLTP to OLAP

ท่าเดิมกับ step ที่แล้วเป๊ะเลย คือ precompute แล้วก็อ่านจากที่ที่ไม่ใช่ hot path แค่คราวนี้ชี้ไปที่อย่างอื่นแทน แทนที่จะอ่านจาก replica ก็เปลี่ยนไปอ่านจาก **OLAP** engine แทน

เหตุผลก็คือ พอ rule กลายเป็น windowed aggregate 90 วัน กับ multi-way join บนหลักร้อยล้าน row เข้า แม้แต่ replica ก็เริ่มกินเวลาเป็นชั่วโมงแล้ว ไม่ใช่เพราะ hardware อ่อนนะ แต่เพราะนี่มันงาน analytical ของแท้แล้วต่างหาก replica ก็ยังเป็น **OLTP** engine อยู่ดี (Postgres/MySQL สร้างมาเพื่อ read/write จุดเล็ก ๆ ทีละ row) ส่วน **OLAP** engine นี่สร้างมาเพื่อทำอะไรที่สวนทางกันไปเลย คือ scan ไม่กี่ column ข้าม row เป็นล้าน ๆ ทีเดียว

- **Columnar analytical database** — ClickHouse หรือ DuckDB บนเครื่องแรง ๆ ตัวเดียว
- **Managed warehouse** — BigQuery, Snowflake, Redshift, Athena

ทั้งคู่รัน query แบบเดียวกัน แต่บน engine ที่ build มาเพื่อสิ่งนี้โดยเฉพาะ — scan เป็น column แทนที่จะฝืนให้ row store ทำงานที่มันไม่ได้ถูกออกแบบมาให้ทำ แล้วรูปแบบ `scheduled SQL query บน warehouse เขียนผลลัพธ์ลงตาราง segment` แบบนี้แหละ คือ audience platform ที่ดีพอแล้ว ต่อให้เป็นปี 2026 ก็ตาม หลายทีมใช้วิธีนี้กัน แล้วก็ไม่ต้องการอะไรมากไปกว่านี้เลย

ขอขยายความอีกนิด เพราะคำว่า `แค่ชี้ไปที่อย่างอื่น` มันน่าจะไม่พอซักเท่าไหร่ ขอเอาเทียบ read replica กับ OLAP engine ก่อน ว่ามัน**แก้คนละปัญหากัน**:

- **Replica** — engine OLTP ตัวเดิม แค่เป็นสำเนาที่ sync ไว้ แยกแค่ว่า *read เกิดที่ไหน* กับ *write เกิดที่ไหน* ข้อมูลไม่เคยหลุดออกจากโลก OLTP เลย แล้วการ sync ก็เป็น native database replication ล้วน ๆ
- **OLAP engine** — storage engine คนละตัวไปเลย build มาเก็บข้อมูลในทรงที่เหมาะกับ analytics โดยเฉพาะ มันไม่เติมตัวเองผ่าน replication นะ ต้องมีคนย้ายข้อมูลเข้าไปเอง ไม่ว่าจะ bulk copy รอบแรก, scheduled Spark job, หรือ CDC (stream write log ของ OLTP เข้า OLAP store ต่อเนื่อง)

งานเดียวกัน query แบบเดียวกัน แค่มาจาก source คนละที่ แล้วการเติมข้อมูลลง source นั้นก็กลายเป็นปัญหาใหม่ของมันเอง (SQL text เองก็มักต้องเขียนใหม่ด้วยนะ เพราะ Postgres, ClickHouse, BigQuery, Snowflake, Redshift ต่างก็พูด SQL dialect ของตัวเองทั้งนั้น) ถ้ายังไม่ได้รัน OLAP store อยู่ replica คือคำตอบที่ง่ายกว่า แต่ถ้าจะไปหา OLAP แบบเต็มตัว **ปัญหาการย้ายข้อมูลนี่แหละคือเรื่อง ingestion เต็ม ๆ** ที่หัวข้อ data lake จะพูดถึงต่อไป

**แล้วเราจะรู้ได้ยังไงว่าถึงเวลาต้องขยับ** step trigger คร่าว ๆ:

| ตอนนี้เราอยู่ที่ | ขยับขึ้นเมื่อ |
|---|---|
| Inline SQL ตอนส่ง | audience โตเกิน ~1 ล้าน user, campaign เพิ่มขึ้นเรื่อย ๆ หรือ send query เริ่มทำแอปช้าลง |
| Precompute บน primary | batch แย่ง resource กับ live traffic ให้เห็นชัด replica แค่ config เดียวเปลี่ยน คว้ามันมาใช้เลย |
| Read replica | analytical SQL กินเวลาเป็นชั่วโมงบน row store เพราะ shape มันผิด และต้องการแยกออกไปเป็นฝั่ง Analytic |
| Warehouse / columnar DB | job รันไม่จบใน refresh window, rule ต้องการ code แท้ ๆ (custom scoring, ML feature) แทนที่จะเป็น SQL หรือ input ไม่ได้อยู่ในที่เดียวกันอีกต่อไป |

อีกเรื่องนึงที่ต้องคิดดูก่อน: **ทุก step ที่ขยับขึ้นไป แบก cost แล้วก็งาน on-call เพิ่มตลอด** replica ก็แค่ติ๊กช่องเดียว, warehouse ก็คือบิลค่าใช้ ส่วนโลกของ Spark-กับ-lake ข้างหน้าคือทั้งบิล **บวกกับ** ทีมงานเลย ขยับขึ้นเพราะ trigger มันเกิดขึ้นเอง ไม่ใช่เพราะ step ถัดไปมันดูน่าสนใจในเรซูเม่ของเรา

ถ้าเราอยู่บน managed warehouse แล้วมันรับไหวอยู่ ข้ามอ่านผ่าน ๆ สองหัวข้อถัดไปได้เลย ซีรีส์นี้จะเดินตามทาง self-managed ต่อจากนี้ data lake แล้วก็ Spark ทับบนมัน เพราะนั่นแหละคือจุดที่ของมันพังแบบน่าสนใจที่ scale ใหญ่สุด แล้วก็เพราะพ้นจุดนี้ไป การเดินหน้าแบบมีประสิทธิภาพต้องมี data lake อยู่ดี ไม่ว่า engine ไหนพาเรามาถึงจุดนี้

## Land the data in one place — the data lake

**ปัญหา:** ข้อมูลที่ audience rule ต้องใช้ ตอนนี้กระจายอยู่ในสิบกว่าระบบ ทั้ง payments DB, auth service, ไฟล์ export จาก data science, event stream ถ้าจะไป query แต่ละที่แบบ live ทุกครั้งที่สร้าง segment มันก็วนกลับไปเป็นปัญหาเรื่อง load เดิมอีก คือ job analytical ตัวเดียวที่ไปกระหน่ำทุก production system พร้อมกันหมด แล้ว job พวกนี้ก็ยังต้องการ **history** ด้วยนะ (`transaction ย้อนหลัง 90 วัน`)

**วิธีแก้:** เอาสำเนาของทุกอย่างไปกองไว้ที่เดียว build มาเพื่อ analytical read โดยเฉพาะ **data lake** production ถูก query แค่ครั้งเดียว เพื่อ ingestion เท่านั้น ไม่ต้อง query ซ้ำเพื่อ compute อีกเลย

**Columnar format — Parquet** เก็บข้อมูลแยกตาม column ไม่ใช่ตาม row query ที่แตะแค่ 3 จาก 30 column ก็อ่านแค่ 3 column นั้นจาก disk แล้ว min/max stats ของ Parquet ยังช่วยให้มันข้าม chunk ที่ไม่มีทางแมตช์ไปได้เลย (**predicate pushdown**) บนตารางกว้าง ๆ นั่นแปลว่าอ่านข้อมูลจาก disk **น้อยกว่า row format ถึง 10 เท่า** บ่อยครั้งก็มากกว่านั้นอีก

**Table format — Iceberg / Delta Lake** ไฟล์ Parquet เฉย ๆ ไม่มีแนวคิดเรื่อง transaction อยู่ในตัวเลย reader อาจไปจับตารางที่เขียนค้างครึ่งเดียวได้, schema change ก็ต้องทำมือ แล้วจะถามว่า `เมื่อวานมันหน้าตายังไง` ก็ตอบไม่ได้เลยด้วย table format เลยมาเติม **ACID snapshot**, **schema evolution**, แล้วก็ **time travel** (query snapshot ย้อนหลังได้) ให้ คุ้มกับ infrastructure ที่เพิ่มขึ้น ตอนที่ output ของเราไปขับเคลื่อน campaign ที่มีเม็ดเงินอยู่ด้วย

**Partitioning** partition ด้วย **event date** (`dt=YYYY-MM-DD`) ให้ query 90 วันอ่านแค่ 90 folder ไม่ใช่ทั้ง history อย่า partition ด้วยอะไรที่ cardinality สูงอย่าง `user_id` จะได้ไฟล์เล็ก ๆ เป็นล้านไฟล์ metadata ใหญ่กว่าตัวข้อมูลเองซะอีก (**small-files problem**) รัน **compaction** เพื่อรวมมันกลับเข้าด้วยกัน

**Ingestion — เอาข้อมูลเข้ามา** ปกติใช้ทั้งคู่: **batch snapshot** (dump รายวัน ง่าย เก่าได้วันนึง) กับ **CDC** (stream write-ahead log ของ source DB ผ่าน Debezium → Kafka เก่าแค่ระดับนาทีแทนที่จะเป็นวัน แล้วก็เป็นจุดที่ streaming ทีหลังจะเริ่มต้นจากจุดนี้ด้วย)

**Orchestration** Ingest → transform → build-segment → publish แต่ละสเตจ depend กับสเตจก่อนหน้า ตัว orchestrator (Airflow, Dagster) รันตาม schedule แล้วจะไม่เริ่ม build segment จนกว่า input ของวันนั้นจะมาถึงก่อน

freshness กับ cost มันแลกกันจุดนี้แหละ: daily batch ถูกแล้วก็ง่าย แต่ตามหลังอยู่วันนึง; CDC สดกว่า แต่เป็น infrastructure เต็มรูปแบบ Debezium, Kafka, compaction, orchestrator ทุกตัวอยากได้ monitoring แล้วก็คนคอย page นี่แหละคือ step ที่ platform เริ่มมีทีมงานเป็นของตัวเอง ซึ่งเป็นเหตุผลที่หัวข้อ OLAP บอกให้รอ trigger ที่ชัดก่อน ส่วนใหญ่ platform ลงเอยอยู่ระดับกลาง ๆ: บางตารางรัน daily บางตารางรันภายในชั่วโมง

ตอนนี้เรามีที่เดียวให้อ่านแล้ว โดยไม่ต้องแตะ production อีกเลย สิ่งที่ยังพูดจุดนี้ไม่ได้คือตัว **logic** เอง นั่นคือหัวข้อถัดไป

## Rules become code — move to Spark

**ปัญหา:** ต่อให้ทุกอย่างอยู่ใน lake เดียวแล้ว step ของ warehouse ก็ยังมีเพดานอยู่ดี แล้วเพดานที่ว่านี้ไม่ใช่เรื่องขนาดด้วยนะ แต่เป็นเรื่อง **shape** targeting ที่ธุรกิจอยากได้เต็ม ๆ มันหน้าตาแบบนี้ต่างหาก: `user ที่จำนวน transaction 30 วันที่ผ่านมาสูงกว่า X, เติมเงินผ่าน QR อย่างน้อย 1 ครั้งในเดือนที่ผ่านมา, ถือ gold tier, ไม่โดน suppress, แล้วก็ join กับ churn score จาก data science` windowed aggregate กับ anti-join นี่ warehouse ทำได้ง่ายอยู่แล้ว แต่ scoring logic ที่ธุรกิจอยากได้ มันคือ **code** ไม่ใช่ `WHERE` clause พอ rule เริ่มพูดผ่าน SQL ไม่ได้แล้ว เราก็ต้องการ engine ที่ build มาเพื่อรองรับ logic แบบไหนก็ได้เป็นเรื่องปกติ บน user หลักร้อยล้านคน แล้วก็ row เป็น **พันล้าน** Back-of-envelope: แค่ user active รายวัน 1 ล้านคน จ่ายค่าอาหาร 3–5 ครั้งต่อวัน ย้อนหลังแค่ 90 วัน ก็ปาไป 270–450 ล้าน row แล้ว ยังไม่นับ user ที่เหลือทั้งหมด หรือ transaction ประเภทอื่นเลยด้วยซ้ำ นี่แหละคือเหตุผลที่คำว่า `พันล้าน` ไม่ได้เว่อร์เลย

**วิธีแก้:** ย้ายการคำนวณไปที่ distributed engine **Spark** มันประมวลผล row เป็นพันล้านแบบขนานข้าม cluster ได้ แล้ว (ส่วนที่ผมแคร์ที่สุด) มันให้เราเขียน rule แค่ครั้งเดียว ส่วน engine ไปจัดการเรื่อง distribution ให้เองหมด

**ตัว job ในแบบ DataFrame** อ่าน input มา, apply predicate, join attribute, aggregate, แล้วเขียน membership ออกมา:

```scala
val users   = spark.read.parquet("s3://lake/users/dt=2026-07-01")        // today's snapshot
val txns    = spark.read.parquet("s3://lake/transactions")               // partitioned by event_date
val churn   = spark.read.parquet("s3://lake/churn_scores/dt=2026-07-01") // today's snapshot
val suppress = spark.read.parquet("s3://lake/suppression/current")

val txn90 = txns
  .where($"event_date" >= date_sub(current_date(), 90))
  .groupBy($"user_id")
  .agg(avg($"amount").as("avg_amt"), count("*").as("txn_count"))

val audience = users
  .where($"tier" === "gold" && $"country" === "TH")
  .join(txn90, Seq("user_id"))
  .join(broadcast(churn), Seq("user_id"))           // small dim → broadcast
  .where($"avg_amt" > 5000 && $"churn_score" < 0.3)
  .join(suppress, Seq("user_id"), "left_anti")       // exclude suppressed
  .select($"user_id")
```

**Broadcast ฝั่งที่เล็กกว่า** `churn_scores` กับ `suppression` เล็กกว่า `users`/`transactions` เยอะ การ mark ว่า `broadcast` คือส่งมันไปให้ทุก executor เก็บไว้ เปลี่ยน join ที่ shuffle หนัก ๆ ให้กลายเป็น map-side join ชนะง่ายที่สุดแล้วก็ใหญ่ที่สุดในบรรดาเทคนิคทั้งหมด `autoBroadcastJoinThreshold` จับบางเคสให้อัตโนมัติอยู่แล้ว แต่ระบุชัด ๆ ไปเลยสำหรับตัวที่เรารู้อยู่แล้วว่าเล็ก

**ระวัง data skew** power user ไม่กี่คนที่มี transaction เป็นล้าน หรือประเทศเดียวที่ถือ user 80% partition เดียวแบกงานทั้งหมด ส่วนที่เหลือนั่งว่าง แล้ว job ทั้งก้อนก็ช้าเท่ากับ partition ที่ช้าที่สุดตัวเดียวนั้นแหละ ที่ช่วยได้: salt hot key, เปิด Adaptive Query Execution (`spark.sql.adaptive.enabled`), แล้วก็ pre-aggregate ก่อน join (rollup `txn90` ข้างบนนั่นแหละ ที่ช่วยย่อฝั่งใหญ่ให้เล็กลงก่อน)

## Nightly isn't fresh enough — refresh every 1–2 hours

**ปัญหา:** เก่าไปวันนึงมันเก่าเกินไปสำหรับหลาย rule เลย `เติมเงินเช้านี้ → มีสิทธิ์เห็น bonus banner ตอนเที่ยง ไม่ใช่พรุ่งนี้` nightly แปลว่าทุก segment เก่ากว่าสถานะปัจจุบันเฉลี่ย 12 ชั่วโมง

**วิธีแก้:** ยังไม่ต้องกระโดดไป streaming หรอก ไปที่ **microbatch** ก่อน refresh ทุก segment ทุก 1–2 ชั่วโมง ให้ไม่มีอะไรเก่าเกินชั่วโมงสองชั่วโมง ได้ freshness win แบบเดียวกับที่รัน cron job ให้ถี่ขึ้นใน precompute step เมื่อกี้ แค่ทำให้มันรับมือได้ที่ scale ระดับ data lake มันเป็น **แนวทาง** ไม่ใช่ technology ตัวใดตัวหนึ่งนะ จะเป็น scheduled batch job หรือ streaming engine ที่รันแบบ micro-batch mode ก็ลงเอยที่จุดเดียวกัน **เป้าหมายคือทำให้ eligibility ถูกต้องภายใน freshness target ที่ตั้งไว้** ไม่ใช่การได้ label ที่เจาะจงว่าเป็นอะไร

ที่ทำให้มันใช้งานได้ในทางปฏิบัติ:

- **Incremental ไม่ใช่ full-rebuild** การอ่านทั้งหมดใหม่ทุกรอบทำได้เต็มที่เลย แล้วสำหรับหลาย use case มันก็เป็นทางเลือกที่ถูกด้วยซ้ำ full rebuild ทุกชั่วโมงไม่มีอะไรผิดเลย มันจะเริ่มไม่คุ้มก็ต่อเมื่อตารางใหญ่ขึ้นมาก ๆ พอ row เกินหลักพันล้านเข้า full rebuild แบบเดิมก็รันไม่จบใน window แล้ว บิลค่า cloud จากการ scan ข้อมูลขนาดนั้นทุกรอบก็พองตามไปด้วย จุดนี้แหละคือจังหวะที่ incremental ได้ที่ทางของมัน: เก็บ watermark (เวลาหรือ offset ล่าสุดที่อ่านไป) ไว้ที่ไหนสักที่ที่ทนทาน แล้วอ่านแค่ส่วนที่ **เปลี่ยน** ไปจากจุดนั้น (Iceberg/Delta incremental read หรือ CDC feed) update เฉพาะ user ที่ได้รับผลกระทบ ที่เหลือปล่อยไว้ ส่วน full rebuild ที่หนักกว่าก็ยังรัน nightly ต่อไปเพื่อแก้ drift
- **Idempotent, publish แบบ snapshot** ทุกรอบที่รันจะ publish versioned snapshot ที่ consumer สลับไปใช้แบบ atomic ไม่มี segment ที่ update ค้างครึ่งทางให้เห็นเลย แล้วถ้ารอบไหน fail ก็แค่ปล่อยให้ snapshot ล่าสุดที่ดีอยู่คงอยู่ต่อไป
- **แบ่ง tier ตาม freshness** segment ส่วนใหญ่โอเคกับ 1–2 ชั่วโมงอยู่แล้ว มีบางตัว (`abandon cart ไป 10 นาทีที่แล้ว`) ที่ต้องการระดับวินาที นั่นแหละคือ opt-in ที่หายาก สำหรับ streaming ของแท้ (Flink หรือ Structured Streaming trigger ที่ตั้งไว้ถี่มาก) ไม่ใช่ default

## Stop rebuilding per campaign — a reusable segmentation service

**ปัญหา:** ทุก segment ยังเป็น Spark job ที่เขียนด้วยมือ ให้ engineer คนนึงมา ship แล้วก็ดูแลอยู่ดี พอมี segment เกินกำมือเดียว มันก็กลายเป็น bottleneck ทันที campaign platform (#7) ก็ self-serve audience เองไม่ได้ ถ้าทุก segment ต้องรอ engineer

**วิธีแก้:** เปลี่ยน hand-written job ให้กลายเป็น **platform** rule กลายเป็น **configuration** ไม่ใช่ code:

- **Rule เป็น DSL / config** segment คือ data ไม่ใช่ code predicate, time window, freshness tier `tier = gold AND country = TH AND avg_txn_90d > 5000 AND NOT suppressed` เก็บไว้เป็นข้อมูล ไม่ใช่เขียนโค้ด platform validate แล้วก็ประเมินขนาดให้ก่อนที่จะรันด้วย
- **Engine เดียว, job ที่ generate เอง** platform compile rule ให้กลายเป็น Spark plan เต็มรูปแบบ (แล้วก็ streaming topology ถ้า tag ไว้ว่าเป็น real-time) attribute ใหม่ลง register ใน catalog แค่ครั้งเดียว แล้วใช้ได้ทุกที่เลย
- **Reusability** segment ประกอบกันได้ `gold_th_active` คำนวณครั้งเดียว แล้วถูก campaign สิบตัวอ้างอิงไปใช้ แทนที่จะมานั่งคำนวณใหม่ทีละ campaign ความผิดพลาดของ v1 ที่ในที่สุดก็ถูกแก้แล้ว
- **Guardrail ก่อนเงินจะขยับ** rule ที่แค่พลิก `AND` เป็น `OR` ก็เปลี่ยน user 2 ล้านคน ให้กลายเป็น 40 ล้านคนแบบเงียบ ๆ ได้เลยนะ นั่นแปลว่า campaign กำลังจะเผา budget ก้อนโตกับคน 38 ล้านคนที่ไม่ควรได้รับ platform เลยกัก segment ที่ขนาดกระโดดออกนอกช่วงที่สมเหตุสมผล (±30%) ไว้ก่อน จนกว่าจะมีคนมายืนยัน ก่อนที่ consumer ไหนจะหยิบไปใช้ด้วยซ้ำ นี่แหละคือเช็คที่ถูกที่สุด กันความผิดพลาดที่แพงที่สุดที่ระบบนี้ทำพลาดได้เลย

นี่แหละคือ payoff ที่ทั้งโพสต์นี้ build มาถึง output ดั้งเดิมของ ETL คือ **segment result** `campaign → [users]` เขียนลง **object storage** เป็น versioned snapshot ทุก 1–2 ชั่วโมง artifact ก้อนเดียวนี้ป้อนให้ consumer ทั้งสองตัวเลย อ่านคนละทิศทางกัน:

- **Chunk read (#3 Omnichannel)** อ่านทั้ง list `campaign → [users]` รอบเดียว เรียงลำดับ แล้วซอยเป็น chunk ละ ~1,000 user สำหรับ audience 40–60 ล้านคน ก็เป็น chunk ราว ๆ 40,000–60,000 ก้อน ที่โพสต์ omnichannel ใช้ตั้ง observability ไว้พอดี read แบบ sequential รอบเดียว ไม่มี random access

**Churn คือกับดัก** ทุกรอบ refresh มีทั้ง add แล้วก็ remove member บาง user เพิ่ง **match** เข้ามาใหม่ บาง user **หลุด** ออกไป ฝั่ง add ง่าย: `SADD` campaign เข้า set ของ user นั้นไปเลย ฝั่ง remove นี่แหละที่คนมักลืม: ลืม `SREM` ไป user ก็ยังถือ eligibility ที่เขาหลุดไปแล้วอยู่ดี มีสองท่าที่ทำให้มันถูกต้อง:

- **Diff and apply** คำนวณ set ที่ **added** กับ **removed** ต่อ segment แล้ว apply ทั้งคู่ ถูกมากตอน churn มันน้อย แต่ต้องพึ่ง snapshot ก่อนหน้าแล้วก็ต้อง diff ให้ถูกด้วย ที่ scale ใหญ่ ๆ โหลด snapshot ทั้งสองก้อนเป็น **Roaring Bitmap** แล้วให้การลบ bitmap ทำเลขให้เลย `added = new.andNot(old)`, `removed = old.andNot(new)` แม่นยำแล้วก็เร็ว แม้ที่ระดับ ID หลักสิบล้าน
- **Rebuild and swap** คำนวณทุก set ใหม่หมด แล้ว swap เข้าไปแบบ atomic (shadow keyspace, flip pointer) self-correcting เอง ไม่มี drift ตกค้าง แลกกับการต้องเขียนทุกอย่างใหม่ทุกรอบ memory peak จะเพิ่มขึ้นเกือบ **เท่าตัว** ตอน swap เพราะ shadow กับ live อยู่พร้อมกัน size ให้พอรองรับจุดนั้น หรือไม่ก็ swap ทีละ shard แทน

ไม่ว่าจะเลือกทางไหน object storage คือ source of truth ส่วน Redis เป็นแค่ view ที่ทิ้งได้ rebuild ใหม่ได้ตลอด ทั้งสองกลยุทธ์มี safety net เดียวกันวางทับไว้อีกชั้น: ตั้ง TTL ของแต่ละ per-user set ให้นานกว่า refresh cadence นิดหน่อย เพื่อให้ removal ที่พลาดไป มันหมดอายุไปเองแทนที่จะค้างอยู่จนกว่าจะมีคนสังเกตเห็น banner ผิด ๆ ใน production

**เรื่อง Bloom filter สักหน่อย** มันน่าดึงดูดอยู่นะ ย่อการเช็คให้เหลือแค่ probabilistic structure เล็ก ๆ แทนที่จะเป็น set เต็ม ๆ แต่จุดที่ต้องระวังคือความหมายของคำว่า **probabilistic** เอง Bloom filter ไม่มีทาง false-negative เลย แต่มันจะ **false-positive** ในอัตราหนึ่งเสมอ แล้วสำหรับ eligibility นั่นแปลว่า user ถูกบอกว่าเข้าเกณฑ์ทั้งที่ตัวเองไม่เข้า ต่อให้ rate ต่ำแค่ไหน พอมาถึง volume ใหญ่ ๆ ก็กลายเป็น error ทางธุรกิจได้เลย eligibility ต้อง exact เท่านั้น Bloom filter คุ้มค่าใช้แค่ตอนที่ false positive มันไม่เป็นอันตรายอะไร เป็น pre-check ก่อน lookup ที่เราจะ verify ซ้ำอยู่แล้ว ไม่ใช่คำตอบสุดท้ายเด็ดขาด

## Wrapping up

เราเริ่มจาก `WHERE` clause ที่พิมพ์ตอนส่ง แล้วมาจบที่ platform ที่คำนวณ audience บน data lake แล้วก็ serve มันออกมาสองทาง เส้นทางทั้งหมด เรียงตามลำดับ:

1. **Inline SQL ตอนส่ง** โอเคจนกว่า query จะโตไป 40 ล้านแถวบน production DB กลางที่กำลังส่งอยู่
2. **Precompute ตาม schedule** สร้าง segment ไว้ล่วงหน้า (idempotent, มี alarm เรื่อง staleness) อ่านมันถูก ๆ ตอนส่ง แต่ก็ยังกระหน่ำ OLTP DB อยู่ดี
3. **Read replica** step ถูก ๆ ที่บทความส่วนใหญ่มักข้ามไป แยก read แล้วก็ output ของ segment ออกจาก primary โดยยังไม่ต้องมี engine ใหม่
4. **Analytical engine** พอ workload กลายเป็น analytics เต็มตัว (OLTP vs. OLAP) columnar DB หรือ warehouse ก็รัน SQL ตัวเดิมได้เร็วกว่าเดิมเยอะ หลายทีมหยุดจุดนี้ได้อย่างถูกต้องแล้ว
5. **Data lake** Parquet + table format, partition แล้วก็ ingest ไว้ เอาข้อมูลจากสิบกว่าระบบมากองที่เดียว
6. **Spark** สำหรับตอนที่ rule เองต้องกลายเป็น code ไม่ใช่ SQL
7. **Refresh ทุก 1–2 ชั่วโมง** incremental ETL รักษา segment result หลักพันถึงหลักล้านให้สดในหลักชั่วโมง ส่วน streaming ของแท้ยังเป็น opt-in ที่หายาก
8. **Reusable segmentation service** rule เป็น config, engine เดียว, guardrail คุมขนาด segment, segment result ก้อนเดียวอ่านได้สองทาง: chunk สำหรับ delivery, invert เป็น per-user index สำหรับ serving

ประเด็นใหญ่ก็ยังเหมือนที่เหลือของซีรีส์นี้: **ไม่มีใครเริ่มที่จุดนี้หรอก** user แค่หลักพันคน กับ rule ง่าย ๆ? พิมพ์ `WHERE` clause แล้วเดินหน้าต่อไปได้เลย ทุก layer ข้างบนได้ที่ทางของมันมาด้วยเหตุผลรองรับทั้งนั้น scale บังคับให้ precompute, load บังคับให้มี replica, analytics เต็มตัวบังคับให้มี warehouse, ข้อมูลที่กระจัดกระจายบังคับให้มี lake, rule ที่ต้องเป็น code บังคับให้มี Spark, ความสดบังคับให้มี microbatch, แล้วก็การทำซ้ำ ๆ บังคับให้มี platform แล้วทุกครั้งที่ขยับขึ้น ก็มี trigger ที่วัดออกมาเป็นตัวเลขได้เสมอ batch ที่ทำให้ live traffic ช้าลงให้เห็นชัด, job ที่รันไม่จบใน refresh window ไม่ใช่ technology ที่แค่ดูน่าสนใจ

แล้วถ้ามีอะไรสักอย่างที่อยากให้เราจำติดตัวไป งานของ audience platform มีสองส่วนเสมอ ส่วนแรกคือ **คำนวณ** ให้ได้ user ที่ถูกต้อง ส่วนที่สองคือ **serve** set นั้นออกมาในรูปแบบที่ consumer แต่ละตัวอ่านได้เอง omnichannel เอาไป stream ส่วน icon serving ก็เอาไป point-lookup ต่อให้คำนวณถูกแค่ไหน ถ้าเลือก serving shape ผิด ระบบก็ยังใช้งานไม่ได้อยู่ดี
