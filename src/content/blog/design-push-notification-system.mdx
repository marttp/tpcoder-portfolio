## Introduction

This is the first post in a backend system design series. The format is simple: I take one system, start with the most naive thing that works, and then add real-world constraints one at a time — until it grows into the kind of multi-service architecture you actually run in production.

I'm opening with push notifications because I spent a good chunk of my time in Japan on exactly this — a push pipeline that ended up sending **120M+ notifications a day**. So this is less "here's the textbook answer" and more "here's the path that got us there, and why each step was forced on us."

The important part of each version below is the **problem statement first**. None of these boxes exist because they look nice on a diagram — every one of them got added because something hurt. If you ever build this, you'll likely walk the same road, so I'll point out where the pain comes from before showing the fix.

It's a long road, so here's the map — each step forced by the one before it:

1. A **direct provider call**
2. A **provider abstraction** for iOS and web (one FCM gateway, per-platform config)
3. Move it **off the request onto a queue**
4. **Batching**
5. A **two-stage, partitioned pipeline**
6. **Lazy fan-out** for huge campaigns
7. **Dedupe**
8. **Rate limiting, delivery tracking, and bounded retry**
9. **Token cleanup**

Then the product layer that decides whether any of it was worth sending: **templating, personalization, and treating notifications by type**.

## The naive version — call FCM directly

Before any send, the backend has to *have* the tokens — and they don't come from the backend. A push token is generated **on the device** by the platform SDK (FCM). So the simplest setup is: the app reads its token from the SDK, and after the user logs in, the client registers it — a plain `POST /devices` carrying the token, platform, and app version — and the backend **upserts** it against `(user, device)` (one user can have many devices, and you never duplicate one).

<MermaidDiagram chart={`
flowchart LR
    SDK["FCM SDK\n(on device)"] -->|"generate token"| APP["App"]
    APP -->|"after login: POST /devices\n(token, platform, app version)"| BE["Backend"]
    BE -->|"upsert by (user, device)"| STORE[("Token store")]
`} />

Tokens rotate and expire too, but that's a problem for later — for now, registration is enough to have something to send to.

With the tokens on hand, the honest starting point for the *send* is: when something happens, look up that user's device tokens and call the push provider right there in the request.

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller\n(API / internal event)"] --> SVC["Notification Service"]
    SVC -->|"for each device token"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

In code it's basically a loop:

```text
on event:
  tokens = lookup_device_tokens(user_id)
  for token in tokens:
      fcm.send(token, payload)   # synchronous HTTP call
  return 200
```

And honestly? For a side project or an internal tool, this is fine. It works, it's one file, you ship it in an afternoon. Don't over-engineer before you have to. A lot of real software never leaves this box — if your volume is modest and a few hundred milliseconds of blocking hurts no one, this is a perfectly respectable place to stop.

But this assumes every user is on Android. They're not — and there's a second crack too: **the caller's request blocks on the push provider.** We'll fix the blocking next; the more basic gap comes first.

## Not every device is Android — multi-platform from the start

**The problem:** you call `fcm.send()` with one hardcoded payload, aimed at one platform. As soon as real users show up you've got iOS and web too — and while FCM *can* reach all three, "send to everyone" isn't automatic. iOS only delivers once you've uploaded your Apple **APNs auth key** into Firebase; web needs **VAPID** keys; and each platform wants a different payload shape (sound and badge on iOS, notification vs data on Android, actions on web). Hardcode a single path and you're rewriting it the day the second platform lands. And this isn't advanced territory — most real apps need iOS and Android from day one; the only question is whether you build for it cleanly.

**The fix:** keep **FCM as a single gateway** — it sends to Android directly, to iOS by relaying to APNs (that uploaded key), and to web over Web Push — and put a thin **Push Handler** in front whose only job is to build the right per-platform config from your internal notification model. You hand it a platform-neutral notification; it fills in FCM's `android` / `apns` / `webpush` blocks and makes the call. It's the Strategy pattern, but the strategy is *payload-shaping per platform*, not a different provider per platform.

<MermaidDiagram chart={`
flowchart LR
    SVC["Notification Service"] --> PH["Push Handler\n(build per-platform config)"]
    PH --> FCM["FCM"]
    FCM -->|"Android (direct)"| AND["Android"]
    FCM -->|"iOS (relayed to APNs)"| IOS["iOS"]
    FCM -->|"Web (Web Push)"| WEB["Web"]
`} />

Two things we get from this:

- **One integration, not three.** You configure each platform once (APNs key, VAPID); you don't build and operate separate APNs and Web Push clients.
- **Per-platform quirks live in one place.** The Push Handler owns the "iOS needs this, web needs that" logic and normalizes provider error codes — invalid token, rate limited, retryable 5xx — into internal codes your sender understands, which matters once we add retry.

There *is* another route: a **separate worker (or direct client) per provider** — talk to APNs and Web Push yourself instead of going through FCM. You'd reach for it to be independent of Google, or for finer control and lower latency at very high volume. It's more to build and operate, so my default is FCM-for-everything until something concrete forces the split.

Either way, you need each device's platform and current token to do this, so this is also where **device information** (token, platform, app version, last-seen) becomes a first-class lookup.

Now you reach every platform — but we still haven't touched that first crack: every send happens **inline, in the caller's request.**

## Stop blocking the caller — async worker + queue

**The problem:** FCM is a network call to someone else's system. When it's slow, *your* API is slow. When it times out, *your* caller eats the timeout. And a notification spike (say, a big event fanning out to a lot of users) means a pile of synchronous HTTP calls holding your request threads hostage. You also can't retry a failed send without making the caller wait for the retry.

**The fix:** make the API do almost nothing — validate the request, drop it on a queue, and return `202 Accepted` immediately. A separate background worker pulls from the queue and runs the Push Handler.

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller"] --> SVC["Notification Service"]
    SVC -->|"enqueue, return 202"| Q["Message Broker (Queue)"]
    Q --> WK["Delivery Worker"]
    WK --> PH["Push Handler"]
    PH -->|"send"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

Two things we get almost for free:

- **The caller is decoupled from the provider.** The API responds in milliseconds; provider latency is now the worker's problem, not the caller's.
- **Workers scale independently.** A spike fills the queue instead of melting the API — and we add workers to drain it.

This is the single highest-leverage change in the whole post. Almost everything later is a refinement of "do the work asynchronously, off a durable queue."

And for most applications, this is already the finish line. Naive, plus multi-platform config, plus a queue and a worker covers the overwhelming majority of products that will ever ship — everything below is the smaller set of cases where sheer volume or strict reliability force more structure. If that's not you, you can stop right here with a clear conscience.

But one send per token — even off a worker — falls apart the moment a single notification has to reach a lot of devices at once.

## One HTTP call per token doesn't scale — batching

**The problem:** "new feature launched, tell everyone" fans out to millions of tokens. At one HTTP request per token you pay the round-trip overhead millions of times, you burn through provider connections, and you hit rate limits fast. The send itself becomes the bottleneck.

**The fix:** stop doing it naively one at a time — but know what "batching" actually buys you. FCM's batch helper (`sendEachForMulticast`, up to 500 tokens) is convenient, but under the hood it doesn't collapse 500 tokens into one network call — it still sends each token as its own request, just concurrently over a shared HTTP client. So you really have two equivalent designs:

- **Use the SDK's batch helper** — it manages the per-token fan-out and concurrency for you.
- **Iterate and send yourself** — same outcome, more control over how you group tokens and how many you fire at once.

Both work. What actually decides whether this is *stable* is the **HTTP client connection pool**: reuse connections (keep-alive), cap max connections, set sane timeouts. Whichever route you pick, a tuned pool is what keeps you from thrashing TCP/TLS handshakes and exhausting sockets under load — get that wrong and neither the SDK nor your own loop saves you. (On the direct-to-APNs route the same principle shows up as one long-lived HTTP/2 connection with many multiplexed streams.)

So we add a **Notification Batcher**: it groups tokens — by platform, and by anything else worth grouping — and hands batches to the Push Handler, which sends them concurrently over that pooled client.

<MermaidDiagram chart={`
flowchart LR
    WK["Delivery Worker"] --> NB["Notification Batcher"]
    NB -->|"batch of ≤500 tokens"| PH["Push Handler"]
    PH -->|"per-token sends,\npooled HTTP client"| FCM["FCM"]
    FCM --> DEV["Devices"]
`} />

The non-obvious part is **partial failure**. A multicast of 500 doesn't succeed or fail as one unit — it comes back with a per-token result list: some delivered, some failed, some with an *invalid token* you should delete. So the batcher can't just check a 200; it has to fan the response back out and decide what to retry, what to drop, and which tokens to prune. Keep that in mind — it's the reason delivery tracking later isn't optional.

Two mechanics make batching actually behave:

- **Chunk, then bound the concurrency.** Split the token list into 500-token chunks for FCM (or N concurrent streams if you went direct to APNs) — but don't fire all chunks at once, or you've just moved the rate-limit problem from "per token" to "per chunk." A bounded worker pool drains chunks at a pace the provider tolerates.
- **Map every per-token result to an action.** `Unavailable` / 5xx → retry; `TooManyRequests` → back off the whole chunk; `NotRegistered` / `InvalidRegistration` → prune the token. That per-token mapping is exactly the signal that feeds token cleanup later — the batch response is where dead tokens first reveal themselves.

Batching makes a single big send efficient. The next wall is the pipeline itself: one worker doing lookups, preference checks, templating, *and* sending can't scale those stages independently — and we have no ordering or replay.

## Split the work — a two-stage Kafka pipeline

**The problem:** our single worker now does a lot — load user info, load device info, validate, check preferences, render the template, batch, and send. Those have wildly different costs. A slow DB lookup blocks a worker that could be sending. You can't scale "enrichment" separately from "delivery." And a plain queue gives you no ordering and no replay — if a worker crashes mid-batch, you're guessing what happened.

**The fix:** split into **two stages** with a durable log (Kafka) between them, and **partition by user ID**.

- **Pre-Delivery Worker** (the enrichment stage): user information, **device/token enrichment** (resolving the user to their device tokens + platform), a validator, the notification batcher, a user-preference checker, and a template parser. It turns a raw "notify user X" event into a fully-resolved, ready-to-send payload, then publishes to the next topic.
- **Delivery Worker** (the send stage): consumes ready payloads and does nothing but hand off to FCM via the Push Handler.

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller"] --> SVC["Send Notification Service"]
    SVC --> Q1["Kafka topic 1\n(partition by user_id)"]
    subgraph PRE["Pre-Delivery Worker"]
        ENR["user info\ndevice/token enrichment\n(user → device tokens)\nvalidator\npreference checker\ntemplate parser\nbatcher"]
    end
    Q1 --> PRE
    PRE --> DB[("DB")]
    PRE --> Q2["Kafka topic 2\n(ready-to-send)"]
    Q2 --> DEL["Delivery Worker"]
    DEL --> PH["Push Handler"]
    PH --> DEV["Devices"]
`} />

Why Kafka specifically, and why partition by `user_id`:

- **Independent scaling.** Enrichment is DB-bound, delivery is network-bound — now each scales on its own consumer group.
- **Ordering where it matters.** All events for one user land on the same partition, so they're processed in order (you don't want a "your order shipped" overtaking "your order is being packed").
- **Replay and durability.** A crashed consumer resumes from its offset. If a downstream bug corrupts something, you can reset the offset and reprocess — a plain queue can't do that.

One piece is worth pulling out of that enrichment stage: the **device/token enricher**. A caller says "notify user X" — but you never push to a *user*, you push to *devices*. This is the step that resolves `user_id` into the real targets: every active `(device_token, platform, locale)` the user has. It quietly does three jobs:

{/* IMAGE PROMPT (two-fan-outs.png): Clean conceptual illustration on a light background contrasting TWO kinds of fan-out, side by side. ![The two fan-outs: a campaign expands to many users, while one user expands to their devices](/assets/images/blogs/design-push-notification-system/two-fan-outs.png)

- **It's the second fan-out.** A campaign fans one message out to millions of *users*; this fans one user out to their *N devices*. "One notification" becomes "N provider sends" right here — a separate fan-out layered on top of the campaign one.
- **It attaches the platform**, which is the only thing the Push Handler can switch on downstream — no platform tag, no way to build the right iOS / Android / web payload.
- **It drops dead devices at read time** — skip tokens whose `last_seen` is ancient before you waste a send. That's the read-side companion to the token cleanup further down.

So that unassuming "device info" box is really a token enricher: the boundary where user-centric events turn into device-centric sends.

This is the shape of the real thing — and it holds up right until a single campaign tries to reach most of your user base in one shot.

## One campaign, 60M recipients — fan-out without flooding the queue

**The problem:** marketing wants a campaign sent to the whole active base — at peak that was **60–70M recipients in a single blast**. This is the *campaign* fan-out — one message to millions of users, the bigger sibling of the per-user device fan-out from the pipeline. The naive move is to expand that into 60–70M individual notification events and dump them all on the broker at once. That floods the queue, starves everything else (your 2FA codes are now stuck behind 60M promos), and if someone cancels the campaign halfway, you've got tens of millions of in-flight messages you can't easily recall.

**The fix:** don't materialize the whole audience up front. Store the **campaign definition + an audience query**, and let the pipeline **expand it lazily** — page through recipients in chunks: enqueue a chunk, let it drain, fetch the next. A dedicated fan-out step (a producer sitting in front of the pre-delivery stage) walks the audience a page at a time instead of exploding it into the broker in one shot.

{/* IMAGE PROMPT (fan-out.png): Clean, modern conceptual illustration on a light background. Left: a single labeled card "Campaign + audience query (60M users)". A small "fan-out worker" reads it with a cursor/pointer and emits a few small batches (chunks) into a queue — show only 3–4 chunks in flight, NOT millions, to convey lazy/bounded expansion. Contrast visual (faded/ghosted, crossed out) above it: the naive version where one campaign explodes directly into a massive wall of 60M tiny message dots flooding the queue. Minimal flat vector style, soft shadows, 2–3 accent colors, no photorealism. */}

![Lazy audience fan-out: a campaign expanded in bounded chunks instead of all at once](/assets/images/blogs/design-push-notification-system/fan-out.png)

What this gets us:

- **Bounded queue depth.** The broker holds a working set — a few chunks — not the entire campaign. Other traffic keeps flowing.
- **Backpressure for free.** Slow consumers just mean we page slower; the audience cursor waits instead of the broker overflowing.
- **A pause/cancel switch that actually works.** Someone spots a typo in the copy, or marketing decides mid-blast to pull the campaign — and because most of the audience hasn't been enqueued yet, stopping it is just "stop advancing the cursor," not "hunt down 60M messages already on the broker." Whatever's already gone out is gone, but you cap the blast radius instead of finishing a send you know is wrong.

This is also where *priority* starts to matter — a 60M marketing blast and a single 2FA code should not be queued as equals — but we'll come back to that. First: at this volume, every hop is at-least-once, and the same push can fire twice. At low volume you shrug; here, users notice.

## Stop the double-buzz — a dedupe processor

**The problem:** every reliable hop in this pipeline is at-least-once. Kafka redelivers after a consumer restart. A delivery worker that crashes *after* calling the provider but *before* committing its offset will reprocess and send again. Retries (next version) add even more chances. The result is the thing users hate most about push: the same notification buzzing twice.

{/* IMAGE PROMPT (double-buzz.png): Conceptual illustration on a light background — a smartphone lock screen showing the SAME push notification appearing twice (two identical banner cards stacked), conveying an annoying duplicate "double buzz". Subtle vibration/motion lines on the phone, optional small frustrated reaction. Flat vector, soft shadows, clean tech-blog style, 2–3 accent colors, no photorealism. */}

![The double-buzz problem: duplicate push notifications](/assets/images/blogs/design-push-notification-system/double-buzz.png)

True exactly-once across a network is a fantasy (it's the Two Generals Problem — same one from my [payment backend post](/blog/payment-backend-stripe-integration-en/)). The practical answer is the same: **at-least-once transport + idempotent delivery**.

**The fix:** give every notification a stable **idempotency key** (e.g. `notificationId:deviceToken`) and check it against a fast store — Redis — right before the provider call. First time through, claim the key with a TTL and send. If the key already exists, it's a duplicate: skip it.

The placement matters: the check lives **at the Push Handler**, not the Delivery Worker. The key is per *device token*, and batching happened upstream — so by the time a payload reaches delivery it may be a batch of tokens. You want the per-token claim after the batch is unpacked, at the last gate before the external send, where it catches duplicates from *every* path (Kafka redelivery, the retry loop, batch expansion) at exactly the right granularity.

<MermaidDiagram chart={`
flowchart LR
    Q2["Kafka topic 2"] --> DEL["Delivery Worker"]
    DEL --> PH["Push Handler"]
    PH <-->|"SETNX notif:device (TTL)"| REDIS["Redis (dedupe)"]
    PH -->|"first send only"| FCM["FCM"]
    FCM --> DEV["Devices\n(Android / iOS / Web)"]
`} />

A few things that matter in practice:

- **Claim before send, not after.** Use an atomic `SETNX`-style claim so two workers racing the same key can't both win.
- **TTL sized to your retry window.** The key only needs to outlive the period in which a duplicate could realistically arrive (hours, not forever) — otherwise Redis becomes your biggest, most expensive table.
- **It's a best-effort guard, not a ledger.** Dedupe trims the duplicates that come from retries and redelivery; it's not an accounting system. (For money, you'd reconcile — but a missed-by-a-hair duplicate push is not worth a distributed transaction.)

Two more things that bite if you skip them:

- **The key has to be deterministic.** `notificationId` must be assigned at ingestion and carried through every retry — generate it per attempt and dedupe does literally nothing. And scope it per device (`notificationId:deviceToken`), so the same logical notification to a user's phone *and* tablet isn't collapsed into a single buzz.
- **Claim-before-send has a sharp edge.** If you claim the key and the send then fails — or the worker dies right after claiming — the key now blocks the legitimate retry, and a duplicate you prevented becomes a notification you *lost*. The fix: only let the key permanently win on a confirmed success; on a known failure, release it so the bounded-retry path can re-claim. The TTL is the backstop — long enough that a late redelivery can't re-send something that already succeeded, short enough that Redis doesn't turn into a graveyard.

Now sends are idempotent. The next questions are the ones a 120M/day system forces on you: how do we not get throttled into oblivion, and how do we even *know* what got delivered?

## Surviving 120M+ a day — rate limits, tracking, and bounded retry

**The problem:** at this volume three things stop being optional.

1. **The provider throttles you.** FCM has its own send limits (and the APNs capacity it relays through isn't infinite either). Blast past them and you get 429s and dropped messages — so we pace ourselves, not just fan out as fast as workers allow.
2. **You're flying blind.** With millions of sends, "did it work?" can't be a log line you grep. You need delivery results as data — delivered / failed / invalid-token — to debug, to report, and to prune dead tokens.
3. **Failures need a *bounded* retry.** A transient 5xx or network blip should be retried. But retrying forever is how you build an accidental infinite loop that hammers a provider and never drains.

**The fix** — the right-hand half of the real architecture:

- **Rate limiting + sharding.** Workers respect a per-provider rate budget (token bucket), and the pipeline shards by partition so we scale horizontally without any single consumer becoming the hot spot.
- **Delivery tracking.** Every send emits a result (success/fail + reason) onto a result queue. A **Delivery Track Aggregator** consumes those and writes them to a **columnar DB** for analytics/reporting (and updates state in the operational DB). This is also where invalid-token results turn into token cleanup.
- **Retry Delivery Worker.** Failed payloads go to a retry path that re-queues them with **exponential backoff and a max-attempt counter**. Once a payload crosses the attempt threshold, it stops (and lands in a dead-letter store for inspection) instead of looping forever.

<MermaidDiagram chart={`
flowchart LR
    DEL["Delivery Worker"] -->|"rate-limited send"| PH["Push Handler"]
    PH --> DEV["Devices"]
    DEL -->|"result (success / fail)"| RQ["Result Queue"]
    RQ --> AGG["Delivery Track Aggregator"]
    AGG --> COL[("Columnar DB\n(analytics)")]
    AGG --> RDB[("DB\n(state)")]
    DEL -->|"failed payload"| RETRY["Retry Delivery Worker"]
    RETRY -->|"backoff + attempt < max"| DEL
    RETRY -->|"attempt >= max"| DLQ[("Dead letter")]
`} />

The part that's trickier than it looks is that rate limiting here is a **distributed budget**. One provider quota — FCM's, per project (or APNs' connection/stream limits if you went direct) — is shared across *every* delivery worker. Give each worker its own local token bucket and you're stuck: divide the quota by worker count and you under-use it badly, or let each worker assume the full budget and you collectively blow past it and eat 429s. So the bucket wants to be **shared** — a central token bucket in Redis the workers draw from — or a coordinator that hands each worker a fair, rebalancing slice. And make it **adaptive**: when the provider starts returning 429s with `Retry-After`, the correct response is to *lower the refill rate*, not keep hammering at whatever limit you guessed.

The detail I'd underline from real life: **the max-attempt counter is the safety valve.** Without it, one persistently-failing payload (a bad token, a broken template, a provider outage) becomes a worker that retries forever, burns budget, and clogs the partition behind it. "Retry, but with a ceiling" is the whole game.

One honest caveat on the word **"delivered."** The result a provider hands back means *the provider accepted the push* — not that it reached the device, and definitely not that anyone saw it. APNs and FCM don't promise the last mile. So what you can actually measure is a funnel, and each stage leaks:

{/* IMAGE PROMPT (delivery-funnel.png): Clean modern funnel diagram on a light background, three descending stages with drop-off shown between them. Stage 1 (widest) "Accepted by provider — measured directly". Stage 2 "Delivered to device — only if the client reports back". Stage 3 (narrowest) "Opened — needs a client callback". Show shrinking width + small "leak" arrows falling away between stages (network drops, app killed, notifications disabled). Flat vector, soft shadows, 2–3 accent colors, label each stage clearly, no photorealism. */}

![Delivery funnel: accepted by provider vs delivered to device vs opened, leaking at each stage](/assets/images/blogs/design-push-notification-system/delivery-funnel.png)

- **Accepted** — the provider took it. You measure this directly from the send result.
- **Delivered** — it reached the device. You only know this if the device/SDK reports back; silent network drops, force-quit apps, and disabled notifications all vanish here.
- **Opened** — the user tapped it. Needs a client-side callback into your tracking.

The design implication is uncomfortable but worth saying out loud: unless you add a client ack channel, your "delivery tracking" is really *acceptance* tracking. A lot of dashboards quietly report acceptance rate and call it delivery rate. For the real engagement funnel you instrument the client and feed those events back into the same aggregator.

## Tokens rot — device registry and cleanup

**The problem:** device tokens aren't stable. They rotate when the OS feels like it, they change on reinstall, and they die the moment a user uninstalls or revokes notifications — and nobody tells you proactively. A token store that only ever *adds* slowly fills with corpses. Every campaign then wastes a slice of its sends on tokens that can never deliver, and — worse — those dead tokens inflate your "sent" count and quietly poison every delivery metric you report.

**The fix:** treat the token store as a living **registry** with a feedback loop, not an append-only list.

{/* IMAGE PROMPT (token-lifecycle.png): Clean conceptual cycle/loop illustration on a light background showing a device token's lifecycle. Nodes around a loop: "Client registers token" → "Stored in registry (user + device)" → "Used to send" → "Provider returns InvalidToken / Unregistered" → "Pruned from registry" arrow back to start. Center: a database/registry icon labeled "Token registry" staying healthy. Emphasize the feedback arrow from delivery failure back to cleanup. Flat vector, soft shadows, 2–3 accent colors, clear labels, no photorealism. */}

![Device token lifecycle: registration, use, and a self-healing cleanup loop driven by delivery failures](/assets/images/blogs/design-push-notification-system/token-lifecycle.png)

- **Registration / refresh.** The client SDK hands you a token on install and a *new* one whenever it rotates; you upsert it against the user + device and keep `last_seen` / app version so stale entries can be aged out.
- **Cleanup from delivery results.** This is the payoff of delivery tracking: when FCM reports a token is dead (`UNREGISTERED` / `registration-token-not-registered`), that result rides the same result queue into the aggregator, which **prunes the dead token** from the registry. The system heals a little every time it sends.
- **Key on (user, device), not raw token.** One user has many devices, and a device can re-register under a new token before the old one expires. Keying on the device keeps you from buzzing the same phone twice from two stale rows.

The mindset shift: a token isn't a value you stored once, it's a *lease* that needs renewing — and your own delivery failures are the signal telling you which leases expired.

With dead tokens pruning themselves, the infrastructure side is genuinely done. What's left isn't infra — it's making the notifications worth sending at all.

## Make it worth sending — templating + personalization

**The problem:** a pipeline that flawlessly delivers the *same* text to everyone is still a mediocre product. Blast identical, irrelevant pushes and people mute the app or uninstall. Send at 3am local time and you lose them faster. Hit the same user five times in an hour and you've trained them to swipe you away. Reliability got the message *out*; relevance decides whether it should have gone at all.

**The fix:** this is what the **template parser** and **user-preference checker** in the Pre-Delivery Worker are really for. Before a notification is ever queued for delivery, it passes through a content-and-consent gate:

- **Preference check** — is this user opted in to this category? Are we inside their quiet hours (in *their* timezone)? If either fails, drop or defer the notification *here*, before it costs a send.
- **Template render** — resolve the template per user: locale/language, and merge fields like name, amount, or order status.
- **Personalization** — choose *which* content this user should get, which is where this pipeline plugs into an audience/segmentation service — a topic big enough to stand on its own.

<MermaidDiagram chart={`
flowchart TD
    EV["Notify user X"] --> PREF{"Opted in?\nWithin quiet hours?"}
    PREF -->|"no"| DROP["Drop / defer"]
    PREF -->|"yes"| TMPL["Render template\n(locale + merge fields)"]
    TMPL --> READY["Ready-to-send payload"]
`} />

Doing this in the **pre-delivery** stage is deliberate: a notification that shouldn't be sent should die before it touches a provider, a rate budget, or a dedupe key. Consent and relevance are cheapest to enforce at the front.

## Not all notifications are equal — type, priority, and frequency caps

**The problem:** treating a 2FA code, a "payment received", and a "50% off this weekend" the same way. If every marketing campaign goes out at full priority with no ceiling, two things break. Users mute the app — and once they do, your *transactional* notifications, the ones they actually need, die right along with the promos. And low-urgency campaigns end up competing with genuinely urgent alerts for the same delivery budget.

**The fix:** classify every notification by **type**, and let the type drive priority and limits.

- **Transactional / activity** (2FA, payment confirmation, someone messaged you) — high priority, sent right away. In FCM you set this through the per-platform blocks — `apns-priority: 10` in the `apns` headers, high priority in the `android` block. They also benefit from **collapsing**: a stale "balance updated" should be *replaced* by the newer one, not stacked — `apns-collapse-id` for iOS, `collapse_key` for Android — so the user sees the latest state, not a pile of five.
- **Marketing / campaign** — lower priority (`apns-priority: 5` in the `apns` block — battery-friendly, and it lets the OS batch it), and subject to a **frequency cap**. Over the cap → drop or defer.

On the cap: keep it a **global policy for now** — say, at most N marketing pushes per user per day, the same N for everyone. It's a cheap counter: a per-user key in Redis with a daily TTL, `INCR` and check it in the pre-delivery stage. You *can* make the cap per-user or per-segment later, but that's tuning you don't need on day one — a single global cap already removes the worst behavior (relentless spam) for almost no complexity.

<MermaidDiagram chart={`
flowchart TD
    N["Notification"] --> T{"Type?"}
    T -->|"transactional / activity"| HI["High priority\ncollapse stale"]
    T -->|"marketing"| CAP{"Under daily cap?"}
    CAP -->|"no"| DROP["Drop / defer"]
    CAP -->|"yes"| LO["Low priority"]
`} />

Note how this is a different axis from the previous section. Per-user **consent** (opt-in, quiet hours) is personalization — about *this user's* wishes. **Type-based priority and the global cap** are platform policy — about how the system treats a whole *class* of notification, regardless of who's receiving it. Both live in the pre-delivery stage, but they're answering different questions.

**Where this goes next — separate lanes.** A type *tag* on a shared pipeline is enough at first. But once marketing volume gets big — those 60–70M campaigns — a tag isn't enough, because a campaign and a 2FA code still share the same topics and workers, so the blast can head-of-line-block the code. The fix is to physically split the lanes: a **transactional lane** with its own topics and delivery workers, generously provisioned and never throttled, and a **bulk/marketing lane** with its own pool, rate-limited and frequency-capped. At real scale this is exactly what we ran — payment and transactional notifications went down a separate path from campaign blasts, so a marketing surge could back up its own queue without ever delaying the push a user is actively waiting on.

## Wrapping up

We started with a loop that called FCM inside the request, and ended with a partitioned, two-stage pipeline doing 120M+ a day. The path, in order:

1. **Reach every platform through FCM** — one gateway plus per-platform config, not a client per provider. Table stakes, not a scaling step.
2. **Go async off a durable log** — the one change everything else builds on.
3. **Batch and pool connections** so a big send runs as concurrent requests over a reused HTTP client, not a fresh handshake per token.
4. **Split into stages and partition** so enrichment and delivery scale independently, with ordering and replay.
5. **Fan out lazily** so a 60M-recipient campaign expands in chunks instead of flooding the queue.
6. **Dedupe** so at-least-once transport doesn't become double-buzz.
7. **Rate-limit, track, and retry with a ceiling** — and be honest that you measure *acceptance*, not delivery, without a client ack.
8. **Keep the token registry self-healing** by letting delivery failures prune dead tokens.
9. **Gate on consent and relevance** so the thing you reliably deliver is worth delivering.
10. **Treat notifications by type** — priority and collapse for transactional, a frequency cap for marketing, and separate lanes once volume demands it, so promos never drown the alerts that matter.

The meta-point: **nobody should build the final version on day one.** Every box earned its place by solving a problem the previous version created. If you start simple and let real constraints push you, you tend to arrive somewhere close to this — and you'll understand every piece, because you felt the pain that added it.
