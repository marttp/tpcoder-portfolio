## Introduction

This is the first post in a backend system design series. The format is simple: I take one system, start with the most naive thing that works, and then add real-world constraints one at a time — until it grows into the kind of multi-service architecture you actually run in production.

I'm opening with push notifications because I spent a good chunk of my time in Japan on exactly this — a push pipeline that ended up sending **120M+ notifications a day**. So this is less "here's the textbook answer" and more "here's the path that got us there, and why each step was forced on us."

The important part of each version below is the **problem statement first**. None of these boxes exist because they look nice on a diagram — every one of them got added because something hurt. If you ever build this, you'll likely walk the same road, so I'll point out where the pain comes from before showing the fix.

> Series — *Backend System Design (start simple → evolve)*. This is **#1**. Next up: a multi-channel Delivery System, then a Webhook Callback System.

## v1: The naive version — call FCM directly

You have to start somewhere, and the honest starting point is: when something happens, look up the user's device tokens and call the push provider right there in the request.

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller\n(API / internal event)"] --> SVC["Notification Service"]
    SVC -->|"for each device token"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

In code it's basically a loop:

```text
on event:
  tokens = lookup_device_tokens(user_id)
  for token in tokens:
      fcm.send(token, payload)   # synchronous HTTP call
  return 200
```

And honestly? For a side project or an internal tool, this is fine. It works, it's one file, you ship it in an afternoon. Don't over-engineer before you have to.

But notice what we've quietly coupled: **the caller's request now blocks on the push provider.** That's the crack everything else widens.

---

## v2: Stop blocking the caller — async worker + queue

**The problem:** FCM is a network call to someone else's system. When it's slow, *your* API is slow. When it times out, *your* caller eats the timeout. And a notification spike (say, a big event fanning out to a lot of users) means a pile of synchronous HTTP calls holding your request threads hostage. You also can't retry a failed send without making the caller wait for the retry.

**The fix:** make the API do almost nothing — validate the request, drop it on a queue, and return `202 Accepted` immediately. A separate background worker pulls from the queue and does the actual FCM call.

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller"] --> SVC["Notification Service"]
    SVC -->|"enqueue, return 202"| Q["Message Broker (Queue)"]
    Q --> WK["Delivery Worker"]
    WK -->|"send"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

Two things we get almost for free:

- **The caller is decoupled from the provider.** The API responds in milliseconds; provider latency is now the worker's problem, not the caller's.
- **Workers scale independently.** A spike fills the queue instead of melting the API — and we add workers to drain it.

This is the single highest-leverage change in the whole post. Almost everything later is a refinement of "do the work asynchronously, off a durable queue."

But now we've assumed every device speaks FCM. They don't.

---

## v3: Not every device is Android — multi-platform abstraction

**The problem:** the worker calls `fcm.send()` directly. The moment you have iOS users you need **APNs**, web users need the **Web Push** protocol, and each provider has its own auth, payload shape, and error codes. If you keep hardcoding FCM, every new platform means surgery on the worker, and the worker starts to know far too much about each provider's quirks.

**The fix:** put a **Push Handler** in front of the providers — one internal interface, one adapter per provider. The worker looks up the device's platform (from device info) and hands off to the right adapter. Classic Strategy pattern; the worker stops caring *how* a push is sent.

<MermaidDiagram chart={`
flowchart LR
    WK["Delivery Worker"] --> PH["Push Handler"]
    PH -->|"Android"| FCM["FCM adapter"]
    PH -->|"iOS"| APNS["APNs adapter"]
    PH -->|"Web"| WEB["Web Push adapter"]
    FCM --> DEV["Devices"]
    APNS --> DEV
    WEB --> DEV
`} />

Two things this buys:

- **Adding a platform is a new adapter, not a worker rewrite.** The internal contract (`send(deviceToken, renderedPayload)`) stays stable.
- **Provider-specific error mapping lives in one place.** "Invalid token", "rate limited", "retryable 5xx" get normalized to internal codes the worker understands — which matters a lot once we add retry.

To route correctly we need to know each device's platform and current token, so this is also where **device information** (token, platform, app version, last-seen) becomes a first-class lookup.

This works great until one notification has to reach a lot of devices at once — and we're back to one HTTP call per token.

---

## v4: One HTTP call per token doesn't scale — batching

**The problem:** "new feature launched, tell everyone" fans out to millions of tokens. At one HTTP request per token you pay the round-trip overhead millions of times, you burn through provider connections, and you hit rate limits fast. The send itself becomes the bottleneck.

**The fix:** stop sending one at a time. Providers give you bulk primitives — use them:

- **FCM multicast** takes up to 500 tokens per request (and topics/device groups for true fan-out).
- **APNs** keeps a long-lived HTTP/2 connection and lets you stream many pushes over it.

So we add a **Notification Batcher**: it groups tokens — by platform, and by anything else that lets a single call do more work — and hands batches to the Push Handler instead of single tokens.

<MermaidDiagram chart={`
flowchart LR
    WK["Delivery Worker"] --> NB["Notification Batcher"]
    NB -->|"batch of tokens"| PH["Push Handler"]
    PH -->|"multicast (≤500)"| FCM["FCM"]
    PH -->|"HTTP/2 stream"| APNS["APNs"]
`} />

The non-obvious part is **partial failure**. A multicast of 500 doesn't succeed or fail as one unit — it comes back with a per-token result list: some delivered, some failed, some with an *invalid token* you should delete. So the batcher can't just check a 200; it has to fan the response back out and decide what to retry, what to drop, and which tokens to prune. Keep that in mind — it's the reason delivery tracking later isn't optional.

Batching makes a single big send efficient. The next wall is the pipeline itself: one worker doing lookups, preference checks, templating, *and* sending can't scale those stages independently — and we have no ordering or replay.
