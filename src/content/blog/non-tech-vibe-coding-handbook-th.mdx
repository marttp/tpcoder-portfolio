ในโพสต์เรื่อง [Mid-Level](/blog/mid-level-2026-what-it-looks-like-and-what-it-costs-th) ผมเคยพูดถึงคนอีกกลุ่มที่ยังไม่ได้คุยกันครับ คือคนที่ไม่ได้ทำงานสาย Software Engineering แต่รู้จักหน้างานของตัวเองดีมาก แล้วตอนนี้ก็ใช้ AI สร้าง software ที่มีประโยชน์ได้ ผมทิ้งไว้ตอนนั้นว่ากลุ่มนี้ควรมีอีกโพสต์นึงแยกออกมา และนี่ก็คือโพสต์นั้นครับ

คุณอาจทำงานอยู่ใน Finance, Operations, HR, Design, Sales, การศึกษา หรือเปิดธุรกิจของตัวเอง ช่วงนี้ AI coding tool เก่งพอให้คุณอธิบายปัญหาด้วยภาษาปกติ แล้วได้ software ที่ทำงานได้กลับมา บางคนอาจสร้าง tool เล็ก ๆ, automation หรือ dashboard ไปแล้วด้วยซ้ำ เมื่อ 10 ปีก่อน ถ้าอยากได้ software ที่ทำมาเพื่องานของเรา ทางเลือกมักมีอยู่ 2 ทาง คือจ้างคนทำ หรือใช้เวลาหลายเดือนเรียนเขียน program ก่อน

AI builder อย่าง [Lovable](https://lovable.dev/) ช่วยตั้ง [Google login](https://developers.google.com/identity) หรือรับ payment ผ่าน [Stripe](https://stripe.com/) ให้ได้หลายส่วนครับ ส่วน AI coding tool อย่าง [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview) ให้คุณควบคุม code ได้มากกว่า แต่คุณต้องขอให้ tool สร้างแต่ละส่วนเอง ปัญหาคือ **ถ้าคุณไม่รู้ว่าส่วนนั้นมีอยู่ คุณก็ไม่รู้ว่าจะต้องขออะไร** ผมเห็นคำถามแบบนี้ในกลุ่มคนไทยที่สร้าง app ด้วย AI บ่อยครับ เอา app ขึ้น online ยังไง [GitHub](https://github.com/) คืออะไร แล้ว data ควรเก็บไว้ที่ไหน โพสต์นี้จะค่อย ๆ แกะทีละเรื่องครับ

ผมจะใช้คำสาย Tech อยู่บ้าง เพราะคุณจะเจอคำเหล่านี้ใน AI tool อยู่ดี แต่ทุกคำจะมีขั้นตอนง่าย ๆ และตัวอย่างจากงานมาช่วยอธิบาย เราจะคุยกันตั้งแต่ data structure, algorithm, API, architecture ไปจนถึง concurrency คุณไม่ต้องเรียนแบบนักศึกษา Computer Science ครับ แค่รู้ว่าแต่ละแนวคิดเกิดมาเพื่อแก้ปัญหาอะไร ก็ช่วยให้คุณคุยกับ AI และทีม Software ได้ดีขึ้นมากแล้ว

ถ้าหัวข้อไหนคุณรู้อยู่แล้ว ข้ามไปได้เลยนะครับ นี่คือ handbook อ่านส่วน business กับ architecture สักรอบ แล้วค่อยกลับมาหา database, security, testing, prompting หรือเครื่องมือในตลาดตอนที่ project เดินมาถึงจุดนั้นก็ได้

ขอแนะนำตัวสั้น ๆ ก่อนครับ ผมทำงานเป็น backend engineer มาประมาณ 7 ปี งานส่วนใหญ่อยู่กับ payment และ wallet เลยได้ดูแลส่วนหลังบ้านที่ user ไม่ค่อยเห็น โพสต์นี้เป็นมุมจากฝั่ง engineer เพื่อช่วยให้คนสาย business เห็นภาพเดียวกับ tech team ครับ แต่ละทีมอาจเลือก technology ต่างกันได้ ขอให้เข้าใจเหตุผลกับ risk ของตัวเลือกนั้นก็พอ

## The World Your Software Lives In — B2C, B2B, and Internal Tools

ก่อนเข้าเรื่อง Tech ผมอยากให้เริ่มจากคำถามนี้ก่อนครับ: **software ของคุณต้องการอะไรบ้าง ใครใช้บ้าง และถ้าพังแล้วเกิดอะไรขึ้น**

software ในโพสต์นี้แบ่งตามกลุ่ม user ได้คร่าว ๆ 3 แบบครับ:

- **Internal tools** — software ที่คุณกับเพื่อนร่วมงานใช้ เช่น form ลางาน, tool สร้าง report หรือ dashboard ของทีม Sales
- **B2C (business to consumer)** — software ที่เปิดให้คนทั่วไปใช้ เช่น หน้าจองโต๊ะร้านอาหาร หรือ app ที่ใครก็สมัครได้
- **B2B (business to business)** — software ที่อีกบริษัทนึงใช้และมักจ่ายเงินให้คุณ เช่น tool ที่ขายให้หลายบริษัทเอาไปใช้ทำงาน

แล้วเรื่องนี้เกี่ยวอะไรกับคุณ? Feature เดียวกันมีมาตรฐานคำว่า **ดีพอ** ไม่เท่ากันในแต่ละโลกครับ

| | Internal tool | B2C | B2B |
|---|---|---|---|
| ใครใช้ | เพื่อนร่วมงานของคุณ | คนที่เราไม่รู้จักบน Internet | พนักงานของอีกบริษัท |
| ถ้าพัง 1 ชั่วโมง | งานสะดุด และเพื่อนร่วมงานมาถามคุณได้ทันที | ลูกค้าออกจาก app และอาจไม่กลับมา | คุณอาจผิดเงื่อนไขใน contract |
| ถ้า data รั่ว | แย่ แต่ยังอยู่ในขอบเขตบริษัท | ข้อมูลส่วนตัวของคนหลุดออกไป และอาจมีปัญหาด้านกฎหมาย | ข้อมูลธุรกิจของอีกบริษัทหลุด และ deal อาจจบได้ |
| Login ต้องมีอะไร | บัญชี Google ของบริษัทอาจพอแล้ว | สมัครสมาชิก, reset password และส่วนอื่น ๆ สำหรับคนทั่วไป | ระบบ login ของบริษัทลูกค้า ซึ่งมักเจอคำว่า SSO |
| หน้าตาต้องดีแค่ไหน | ไม่ต้องเนี้ยบมากก็ทำงานได้ | ต้องดูน่าเชื่อถือ เพราะลูกค้าตัดสินจากสิ่งที่เห็น | ดูเป็นงาน professional แต่การใช้งานต้องมาก่อนความสวย |

ประเภทของ software เปลี่ยนได้ครับ Tool ทำ report อาจเริ่มจากคนใช้คนเดียว ต่อมามีเพื่อนร่วมงานใช้อีก 20 คน และหลังจากนั้นคุณอาจนำไปขายบริษัทอื่น พอขยับจาก internal ไป B2B หรือ B2C กฎเรื่อง data, login, support และ downtime ก็ต้องเพิ่มตามไปด้วย

ก่อนเขียน feature list ลองตอบคำถาม business 4 ข้อนี้ก่อนครับ:

1. **ใครใช้?** คนที่กด button คือ user
2. **ใครจ่าย?** ใน B2C คนใช้กับคนจ่ายอาจเป็นคนเดียวกัน ส่วน B2B อาจเป็น manager และ internal tool ก็ใช้ budget ของแผนก
3. **ใครเป็นเจ้าของ data?** ลูกค้าอาจเป็นคนกรอก แต่บริษัทของเขาเป็นเจ้าของตามกฎหมาย เรื่องนี้สำคัญตอนมีคนขอ export หรือลบข้อมูล
4. **งานส่วนไหนต้องพึ่ง tool นี้?** Tool บางตัวหยุดวันนี้แล้วกลับมาใช้พรุ่งนี้ได้ แต่ payroll หรือ delivery system อาจทำให้ทั้งทีมหยุดงานครับ

ข้อสุดท้ายสำคัญมากครับ พอทุกคนเลิกใช้ spreadsheet เก่า tool ของคุณก็ไม่ใช่ demo แล้ว ทีมเริ่มต้องพึ่ง tool นี้เพื่อทำงาน

ผมเห็นคนสาย Non-Tech พลาดกันอยู่ 2 ทางครับ:

- **สร้างมากเกินไป:** ใช้เวลาหลายสัปดาห์รองรับ user 1,000,000+ ทั้งที่ตอนนี้มีคนใช้ 5 คน
- **สร้างน้อยเกินไป:** เอา internal tool ที่ทำเร็ว ๆ ไปเปิด public คนนอกทีมมีโอกาสกรอกผิด, double-click, ลืม password และบางคนก็อาจลองโจมตีระบบ

ก่อนเริ่มสร้าง บอก AI ไปเลยว่า software อยู่ในโลกไหน เช่น **นี่คือ internal tool สำหรับทีมของผม** ประโยคเดียวช่วยให้ AI เลือกของได้เหมาะขึ้นเยอะครับ

สำหรับคนสาย Non-Tech หลายคน internal tool เป็น project แรกที่ดีมาก เพราะ user อยู่ใกล้คุณ มีอะไรพังก็ถามได้เลย แถม tool เล็ก ๆ ที่ช่วยลดงาน 3 ชั่วโมงทุกสัปดาห์ก็มี value วัดได้ไม่ยาก

### Start from the workflow, not the app idea

หลายคนเริ่มจากหน้าจอครับ อยากได้ dashboard, form แล้วก็ admin page ส่วน tech team มักเริ่มอีกทาง คือถามว่างานไหลจากต้นจนจบยังไง:

สมมุติว่าเราจะทำ **app อนุมัติค่าใช้จ่าย** คำว่า workflow จะช่วยได้มากกว่าชื่อ app ครับ:

- **Trigger:** พนักงานส่งรายการค่าใช้จ่าย
- **Input:** จำนวนเงิน, วันที่, หมวดหมู่, ใบเสร็จ และคำอธิบาย
- **Decision:** ต่ำกว่า 5,000 บาทส่งให้ manager ถ้าสูงกว่านั้นต้องผ่าน Finance เพิ่ม
- **Action:** อนุมัติ, ปฏิเสธ หรือส่งกลับไปขอข้อมูลเพิ่ม
- **Outcome:** พนักงานได้รับแจ้งเตือน, ระบบบันทึกผลการตัดสินใจ และส่งรายการที่ผ่านแล้วเข้า Queue รอจ่ายเงิน

Flow นี้ทำให้เราเห็นคำถามที่หน้าจอสวย ๆ ซ่อนไว้ครับ ถ้าพนักงานไม่มี manager ล่ะ? คนส่งอนุมัติรายการตัวเองได้ไหม? ถ้าส่งใบเสร็จเดิม 2 รอบจะเกิดอะไรขึ้น? ตอนคุณตอบคำถามพวกนี้ คุณกำลังทำ software design อยู่แล้ว

ด้วย AI tool วันนี้คุณทำเรื่องพวกนี้ได้แล้ว:

- **เปลี่ยนงาน spreadsheet ที่ชวนเหนื่อยให้เป็น web tool เล็ก ๆ** เช่น report ประจำสัปดาห์ที่ต้อง copy-paste 3 ชั่วโมง หรือ form ที่ทุกเดือนต้องไล่เก็บข้อมูลผ่าน email ทั้งหมดนี้คือ software project และตอนนี้คุณเริ่มทำเองได้
- **สร้าง dashboard จาก data ที่มีอยู่** ต่อ spreadsheet หรือ file export แล้วแสดงเฉพาะตัวเลขที่สำคัญ พร้อม update ให้อัตโนมัติ
- **เชื่อม tool เข้าด้วยกัน** พอมี form ใหม่ก็ส่งข้อความเข้า team chat แล้วเพิ่ม row ลง sheet
- **สร้าง prototype ที่เปิดให้ดูได้** ถ้าคุณเคยอธิบาย product idea ด้วย slide จะรู้เลยว่าการบอกให้คนจินตนาการหน้าจอ ต่างจากยื่น link ที่กดเล่นได้แค่ไหน Investor, หัวหน้า และลูกค้าเข้าใจของที่ลองใช้ได้ง่ายกว่าครับ
- **ถามเรื่องระบบเก่า** เอา error, formula หรือ macro ใส่ให้ AI แล้วขอให้อธิบายด้วยคำง่าย ๆ

แต่มีบางเรื่องที่ต้องระวังมากขึ้น:

- **รับ จ่าย หรือโอนเงินจำนวนมาก** การรับ payment ผ่าน service อย่าง Stripe ทำได้ครับ แต่ถ้าจะเขียน payment หรือ wallet logic เอง เช่น คำนวณ balance และย้ายเงินระหว่าง account ความพลาดเล็ก ๆ อาจกลายเป็นเงินหายได้ นี่คืองานประจำของผม และผมก็ใช้ AI ช่วยเขียน code เหมือนกัน แต่ผมไม่ให้ AI ตัดสินกฎเรื่องเงิน หรือเติม code ที่ผมอธิบายไม่ได้ ผมเชื่อ logic ที่อธิบายและ test ได้มากกว่า ก่อนขึ้น production ควรให้ software team review การเปลี่ยนแปลงผ่าน pull request (PR) ด้วย คุณยังไม่รู้ทุกส่วนก็ไม่เป็นไรครับ ขอแค่รู้ว่าช่วงไหนต้องขอความช่วยเหลือ และระวังสิ่งที่กำลังส่งให้ user
- **เก็บข้อมูลส่วนตัว หรือ sensitive data** เช่น ข้อมูลสุขภาพ, เลขบัตรประชาชน หรือข้อมูลการเงิน กฎหมายอย่าง PDPA อาจมีผลแม้คุณยังไม่เคยศึกษาเรื่องนี้ คุณยังสร้างได้ครับ แต่ควรดึงคนมีประสบการณ์มาช่วยตรวจตั้งแต่จุดนี้
- **ระบบ public ที่มี user จำนวนมาก** ถ้ามีคน 1,000+ คนใช้พร้อมกัน ผลกระทบตอนระบบพังก็เพิ่มตามไปด้วย

วันนี้การเขียน code ง่ายขึ้นเยอะแล้วครับ คำถามสำคัญจึงกลายเป็น **ถ้า code ผิดแล้วเกิดอะไรขึ้น** คนหรือทีมที่ deploy ระบบยังต้องรับผิดชอบผลลัพธ์อยู่ดี หัวข้อถัดไปจะช่วยให้คุณมองเห็นและลด risk เหล่านี้

### AI gives a likely answer, not a fixed answer

คุณอาจเคยเจอคำว่า **probabilistic** หมายถึง AI เลือกคำตอบจากหลายทางที่มีโอกาสเป็นไปได้ ดังนั้น input เดิมจึงไม่จำเป็นต้องได้คำตอบเหมือนเดิมทุกครั้งครับ

ลองใช้ตัวอย่างจาก Marketing ครับ คุณส่ง lead ให้ AI 10 คน แล้วขอให้เลือกคนที่พร้อมคุยตอนนี้ รอบนี้ AI ส่ง 2 คนไปให้ทีม Sales ส่วนอีก 8 คนรอรอบหน้า ถ้า run ใหม่ AI อาจเลือกคนละ 2 คน เพราะ model เลือกคำตอบที่มีความเป็นไปได้สูง ไม่ได้ทำตามกฎตายตัวแบบ `amount > 5,000`

วิธีนี้เหมาะกับ draft, score, การจัดกลุ่ม หรือ suggestion แต่เสี่ยงเกินไปถ้าจะให้ AI เป็นคนตัดสินขั้นสุดท้ายเรื่อง payment, การรับคน, สุขภาพ, กฎหมาย หรือสิทธิ์เข้า account

ใช้ pattern ง่าย ๆ แบบนี้ได้ครับ:

1. ให้ AI เสนอคำตอบหรือให้ score
2. มีกฎที่บอกชัดว่าขั้นต่อไปทำอะไร
3. ให้คนตรวจกรณีที่มี risk สูง
4. เก็บ input กับ result ไว้ จะได้ย้อนดูตอนเจอผลการตัดสินใจที่มีปัญหา

Pattern นี้เรียกว่า **human-in-the-loop** คือ AI ช่วยทำงาน แล้วให้คนตรวจผลก่อนระบบทำ action ที่มี risk สูง

**ถาม AI แบบนี้:** ส่วนไหนของระบบใช้กฎตายตัว และส่วนไหนใช้คำตอบจาก AI ที่อาจเปลี่ยนได้ อธิบายด้วยคำง่าย ๆ และเพิ่ม human review ก่อน action ที่มี risk สูงทุกจุด

## Pure Vibe Coding Can Hide the Choices

**Pure vibe coding** คือการบอกสิ่งที่คุณต้องการ รับ code ที่ AI สร้างให้ แล้วขอแก้ต่อไปเรื่อย ๆ โดยไม่รู้ว่าเครื่องมือเลือกอะไรไว้เบื้องหลัง วิธีนี้ใช้ทำ demo เร็ว ๆ ได้ครับ ปัญหาจะเริ่มเมื่อ demo กลายเป็นเครื่องมือที่คนต้องพึ่งพา

เมื่อขอให้ AI builder สร้าง web app คุณมักจะได้ [JavaScript](https://developer.mozilla.org/en-US/docs/Web/JavaScript), [TypeScript](https://www.typescriptlang.org/) หรือ [Python](https://www.python.org/about/) ภาษาเหล่านี้เป็นตัวเลือกที่ดีได้ทั้งหมด ปัญหาไม่ได้อยู่ที่ภาษา แต่อยู่ที่คำตอบแรกอาจกลายเป็นตัวเลือกสุดท้าย ก่อนคุณจะรู้ด้วยซ้ำว่ามีทางเลือกอื่น

คุณไม่ต้องเขียน code ทุกบรรทัดเอง แต่ควรรู้ว่ามีเครื่องมืออื่นอยู่ และถามว่าทำไมตัวเลือกนี้จึงเหมาะกับงานของคุณ

### What is a programming language?

**Programming language** คือชุดคำและกฎที่ใช้บอก computer ว่าต้องทำอะไร และมีผลมากกว่าหน้าตาของ code ครับ เพราะแต่ละภาษาทำให้เรื่องเหล่านี้ต่างกัน:

- app run ได้ที่ไหน
- error แบบใดจะถูกพบก่อน app เริ่มทำงาน
- app ใช้ memory อย่างไร
- มี library สำเร็จรูปอะไรให้ใช้บ้าง
- จะนำ app ขึ้น online อย่างไร
- และใครจะดูแลระบบต่อในอนาคต

**Type** บอกว่าข้อมูลแต่ละค่าเป็นแบบใด เช่น text, number หรือ date การตรวจ type ช่วยหยุดข้อมูลผิดบางแบบได้ก่อน app ขึ้น online

**Memory** คือพื้นที่ทำงานระยะสั้นของ computer ถ้าใช้ memory ผิด program อาจล่มหรือเกิดช่องโหว่ด้าน security ได้ **Memory safety** หมายถึงภาษาช่วยป้องกันการใช้พื้นที่นี้แบบไม่ปลอดภัย ตัวอย่างเช่น [Rust](https://doc.rust-lang.org/book/ch04-00-understanding-ownership.html) จะตรวจว่าส่วนใดของ program เป็นเจ้าของ memory แต่ละชิ้น จึงช่วยดูแล memory safety โดยไม่ต้องมี garbage collector ส่วน garbage collector คือระบบที่หาและคืน memory ซึ่ง program เลิกใช้แล้ว

คำว่า **Enterprise** ไม่ได้วัดจากขนาดอย่างเดียวครับ แต่ยังรวมถึงระบบที่ต้องใช้ต่อหลายปี เชื่อมกับของเดิมในบริษัท ทำตาม policy และมีหลายทีมช่วยกันดูแล [Java](https://www.oracle.com/java/technologies/) กับ [C# with .NET](https://dotnet.microsoft.com/en-us/learn/dotnet/what-is-dotnet) จึงพบได้บ่อย เพราะมี ecosystem, library, type check และระยะ support ที่รองรับงานแบบนี้ ภาษาอื่นก็ใช้สร้าง enterprise system ได้เช่นกัน

ไม่มีภาษาไหนดีที่สุดสำหรับทุก app ครับ Rust มี memory safety ก็ไม่ได้แปลว่าทุกงานควรใช้ Rust ส่วน Java หรือ C# ก็ไม่จำเป็นต้องเลือกเพียงเพราะลูกค้าเป็นบริษัทใหญ่ Python ทำได้มากกว่า script เล็ก ๆ และ Go ก็เหมาะกับ service ขนาดเล็กได้ตั้งแต่วันแรก

เริ่มจากสถานการณ์ของคุณ:

1. **ระบบต้อง run ที่ไหน?** Browser, phone, server ในบริษัท, cloud service หรืออุปกรณ์ขนาดเล็ก
2. **ถ้าพลาดแล้วเกิดอะไรขึ้น?** Report ช้า, เงินหาย, data รั่ว หรือ delay ชั่วคราว
3. **ทีมรู้อะไรอยู่แล้ว?** ภาษาที่ทีมคุ้นเคยมักปลอดภัยกว่าตัวเลือกใหม่ที่ดูฉลาด แต่ไม่มีใคร review ได้
4. **ต้องเชื่อมกับระบบใดบ้าง?** ตรวจว่าภาษานั้นมี library ที่ดูแลต่อเนื่องสำหรับระบบเหล่านั้น
5. **ปีหน้าใครจะดูแลต่อ?** Code ยังไม่จบเพียงเพราะ version แรกใช้งานได้

สมมติว่าบริษัทมี Java service อยู่ 10 ตัว และมี engineer ที่ review Java ทุกวัน ภาษาใหม่อาจเพิ่มงานมากกว่าคุณค่า แต่ถ้าคุณกำลังทำ internal tool เล็ก ๆ เพื่อทำความสะอาด data ใช้ Python ก็อาจพอ หรือ Go service ขนาดเล็กที่เริ่มจากทีมเดียวอาจโตไปใช้ทั้งบริษัทได้ การเติบโตเพียงอย่างเดียวไม่ได้แปลว่าต้องเขียนใหม่ครับ

Programming language เป็นเพียง 1 ในตัวเลือกที่ถูกซ่อนไว้ Pure vibe coding ยังซ่อนเรื่องเหล่านี้ได้:

- **Framework** กำหนดว่า code ถูกสร้างและ update อย่างไร
- **Database** กำหนดว่า record มีหน้าตาแบบไหน เชื่อมกันอย่างไร และ query แบบใดทำได้ง่าย
- **Vendor** มีผลต่อราคา ข้อจำกัด และความยากตอนย้ายออก
- **Login system** กำหนดว่าใครเข้าได้และมองเห็นอะไร
- **Library** ต้อง update ด้าน security และอาจหยุด support ในภายหลัง
- **Test, backup และ alert plan** ที่จะสำคัญหลัง demo แรกใช้งานได้

หน้าจอที่กดได้ไม่ได้บอกว่า AI เลือก technology อะไรไว้ข้างหลัง คุณไม่ต้องเลือกทุกอย่างคนเดียวครับ แต่ควรขอให้ AI แจกแจงตัวเลือก เหตุผล และทางเลือกอื่น แล้วดึงคนที่มีความรู้มาช่วยตรวจเมื่อระบบแตะเรื่องเงิน, data ส่วนตัว, public user หรืองานสำคัญของบริษัท

**ถาม AI แบบนี้:** ก่อนเขียน code ช่วยบอก technical choice สำคัญที่กำลังจะเลือก สำหรับ programming language ให้เทียบอย่างน้อย 3 ตัวเลือกจาก user, team, risk, current system และ future support ของผม อธิบายด้วยคำง่าย ๆ ว่าแต่ละตัวเลือกเหมาะหรือไม่เหมาะเพราะอะไร แนะนำ 1 ตัวเลือก และบอกด้วยว่าข้อมูลใดอาจทำให้คำแนะนำเปลี่ยน

### Frontend — what runs in the user's hands

**Frontend** คือส่วนที่ user เห็นและกดใช้งานครับ เช่น page, button, form และ table ส่วนนี้ run อยู่ใน browser จึงใช้รับ input และแสดงผลได้ แต่อย่าเก็บ secret หรือใช้ Frontend ตัดสินกฎสำคัญ เพราะ user แก้ request ที่ browser ส่งออกไปได้ครับ

สมมติว่า form เบิกค่าใช้จ่ายซ่อน button **Approve** จากพนักงานทั่วไป หน้าจอจะใช้ง่ายขึ้นครับ แต่การซ่อน button ไม่ได้ช่วยเรื่อง security เพราะคนที่รู้วิธีเรียก API ยังส่ง request เองได้ Backend จึงต้องเช็ก permission อีกรอบก่อนเปลี่ยน record

**กฎที่ควรจำ:** Frontend แสดงว่า user น่าจะทำอะไรได้ ส่วน Backend เป็นคนตัดสินว่า user มีสิทธิ์ทำหรือไม่ครับ

### Backend — the part you trust with the rules

**Backend** run อยู่บน server ที่คุณควบคุมหรือเช่าไว้ครับ หน้าที่คือรับ request, เช็กตัวตนกับ permission, บังคับใช้ business rule, บันทึกข้อมูล แล้วส่ง response กลับไป สำหรับ app เบิกค่าใช้จ่าย Backend ต้องเช็กว่าผู้จัดการมีสิทธิ์อนุมัติรายการนี้ และรายการยังอยู่ในสถานะรออนุมัติ

งานที่ใช้ secret ต้องอยู่ใน Backend เช่น เรียก AI model แบบเสียเงิน, ตัดเงินผ่าน Stripe, ส่ง email หรือใช้ private database key ถ้าใส่ key ไว้ใน Frontend ทุก browser จะ download code ชุดนั้นไป แล้ว key ก็หลุดได้ครับ

### Database — memory that outlives one request

**Database** เก็บ record ทางธุรกิจไว้ระยะยาวครับ เช่น ใครส่งรายการเบิก สถานะตอนนี้คืออะไร ใครอนุมัติ และอนุมัติเมื่อไหร่ Backend เป็นส่วนที่ทำงานตามกฎ ส่วน Database เป็นที่อ่านและเขียน record เหล่านี้

### External services — rent the solved problems

app ส่วนใหญ่ใช้ service จากบริษัทอื่นครับ เช่น Stripe รับ payment, Google หรือ [Microsoft](https://learn.microsoft.com/en-us/entra/identity-platform/) ดูแล login, [Resend](https://resend.com/) หรือ [Postmark](https://postmarkapp.com/) ส่ง email ส่วน AI provider run AI model และ map service ช่วยค้นหาสถานที่กับเส้นทาง

วิธีนี้พบได้ทั่วไปครับ คุณสร้างส่วนที่ทำให้ product มีคุณค่า แล้วใช้ service สำเร็จรูปกับงานที่มีคนทำไว้ดีอยู่แล้ว แต่ละ service ที่เพิ่มเข้ามาก็เพิ่มทั้งค่าใช้จ่ายและจุดที่อาจพังได้ ดังนั้นจดข้อมูลเหล่านี้ไว้ด้วยครับ:

- คุณส่งข้อมูลอะไรให้เขาบ้าง;
- จะเกิดอะไรขึ้นถ้าบริการนั้นใช้งานไม่ได้;
- คุณจะส่งออกหรือเปลี่ยนย้ายข้อมูลของคุณอย่างไร;
- ใครเป็นเจ้าของบัญชีและเป็นคนจ่ายบิล

### API — the agreed doorway between parts

**API (application programming interface)** คือข้อตกลงที่ให้ระบบ A เรียกใช้งานระบบ B ครับ เช่น app จองที่พักส่งที่อยู่ไปหา map API แล้วรับพิกัดกลับมาแสดงบนแผนที่

คำขอ (Request) มักจะประกอบด้วย:

- **ไปที่ไหน** — ที่อยู่ที่เรียกว่า endpoint เช่น `/expenses/123`;
- **ต้องการทำอะไร** — อ่าน, สร้าง, update หรือลบ;
- **ใครเป็นคนขอ** — ข้อมูลยืนยันตัวตน (authentication);
- **input** — เช่น ผลการอนุมัติและ comment

Response มักมีผลลัพธ์และ **status code** ครับ คุณไม่ต้องจำทุก code แค่รู้กลุ่มหลักก็พอ: `2xx` คือ request สำเร็จ, `4xx` คือข้อมูลจากฝั่งเรียกผิดหรือไม่มีสิทธิ์ และ `5xx` คือระบบฝั่งรับ request มีปัญหา

API คล้าย form ที่ทั้ง 2 ระบบตกลงใช้ร่วมกันครับ ชื่อ field, type และความหมายต้องใช้แบบเดียวกัน ถ้า `amount` เคยเป็น number แล้วเปลี่ยนเป็น text อีกระบบอาจอ่านไม่ได้ ควรเก็บตัวอย่าง request, response และรายการ field ที่จำเป็นไว้ให้ครบ

### One application is often the best architecture

คุณอาจได้ยินคำว่า monolith, microservices และ serverless สำหรับ product ตัวแรก การมี app เดียวกับ database เดียวมักพอแล้วครับ วิศวกรเรียกโครงแบบนี้ว่า **monolith** คือส่วนหลักถูก build และ deploy ไปด้วยกัน

Monolith ไม่ได้แปลว่าต้องเอา code ทุกอย่างกองไว้ใน file เดียวครับ คุณยังแบ่ง project เป็น module ชัด ๆ ได้ เช่น `expenses`, `users`, `approvals` และ `reports` จุดต่างอยู่ที่:

- **Modules** ช่วยเก็บ code ไว้เป็นกลุ่มที่ชัดเจน เช่น ค่าใช้จ่าย, user และ report
- **Many separate services** คือหลาย app ที่ต้อง deploy, monitor และแก้ปัญหาแยกกัน งานดูแลจึงเพิ่มตามจำนวน service ครับ

ขอให้ AI แบ่งเป็น module ที่ชัดเจนตั้งแต่เริ่มครับ และจะแยก app ก็ต่อเมื่อเจอปัญหาที่ทำให้ต้องแยก

**ลองถาม AI ของคุณว่า:** ช่วยวาด architecture ของ project นี้เป็น diagram โดยระบุว่าส่วนไหนทำงานใน browser ส่วนไหนทำงานบน server ข้อมูลแต่ละชนิดเก็บไว้ที่ไหน และมี external service อะไรบ้าง จากนั้นช่วยอธิบายว่าส่วนไหนเป็นคนคุมกฎทางธุรกิจแต่ละข้อ

## Naive-But-Works Is a Strategy, Not a Shortcut

Engineer ที่มีประสบการณ์มักเริ่มจาก **version ที่เรียบง่ายและใช้งานได้** แล้วค่อยเพิ่มส่วนใหม่เมื่อเจอเหตุผลที่ต้องเพิ่มครับ

**code ที่มากขึ้นหมายถึงงานที่มากขึ้น และงานที่มากขึ้นก็หมายถึงโอกาสเกิดปัญหาที่มากขึ้นครับ** ทุกส่วนที่เพิ่มเข้ามาต้องถูกอ่าน, ทดสอบ, แก้ไข, update และดูแลความปลอดภัย

โพสต์ด้าน system design ของผมก็ใช้แนวนี้ครับ เริ่มเล็ก ๆ ดูว่าข้อจำกัดโผล่ที่ไหน แล้วค่อยเพิ่มส่วนที่ช่วยแก้ปัญหานั้น

version แรกที่เรียบง่ายอาจพอใช้ไปอีกนานครับ AI ชอบเตือนว่าแบบนี้จะ scale ไม่ได้ เพราะคำแนะนำจำนวนมากมาจากปัญหาของบริษัทใหญ่ใน Silicon Valley แต่ลูกค้าของคุณอาจให้คุณค่ากับเรื่องอื่น คุณรู้เนื้องานดีกว่า ให้เริ่มจากคุณค่าของลูกค้า แล้วค่อยถามว่าอะไรจะพัง เมื่อมี user เท่าไหร่ และเราจะสังเกตอาการนั้นอย่างไร

- **จะ scale ได้ไหม? คือคำถามเกี่ยวกับอนาคตที่อาจไม่เกิดขึ้น** เครื่องมือใช้ภายในส่วนใหญ่รองรับคนไม่ถึง 100 คนไปตลอด หรือสินค้าส่วนใหญ่เลิกใช้ก่อนเรื่อง scale จะเป็นปัญหา การสร้างระบบเพื่อรองรับ user 1,000,000 คนที่คุณยังไม่มี ทำให้คุณเสียเวลาในวันนี้เพื่อประโยชน์ที่อาจไม่เกิดขึ้นครับ
- **ปัญหาการเติบโตมักเริ่มจากจุดเล็ก ๆ** เช่น page เริ่มช้า report ใช้เวลานานขึ้น ถ้าคุณเฝ้าดู app คุณสามารถแก้ปัญหาได้ก่อนที่มันจะกลายเป็นเรื่องใหญ่ครับ
- **ข้อยกเว้นเดียวคือ data ครับ** code เขียนใหม่ทีหลังได้ แต่ data ย้อนกลับไปแก้ไม่ได้ ถ้า record ของ 6 เดือนถูกเก็บมาผิด การเขียน code ใหม่ก็ช่วยเอาข้อมูลที่หายไปกลับมาไม่ได้ ดังนั้นคำถามว่า **data ของผมอยู่ที่ไหน** จึงสำคัญกว่า **framework ไหนดูดีที่สุด** ครับ

**Scalable** หมายถึง app สามารถเติบโตได้โดยไม่ต้องรื้อสร้างใหม่ทั้งหมด สิ่งที่เติบโตได้มีหลายอย่างครับ:

| สิ่งที่เติบโต | อาการแรกที่เห็น | ขั้นตอนถัดไปที่เหมาะสม |
|---|---|---|
| user | page ช้าลงในช่วงเวลาที่มีคนใช้เยอะ | วัดผล request ที่ช้า; ปรับปรุง database query หรือเพิ่ม capacity |
| จำนวน record | รายการและ report ใช้เวลานานเกินไป | เพิ่ม pagination และ database indexes; เลิกโหลดข้อมูลทุกแถว |
| งานที่ทำใน request เดียว | การ import data หรือเรียก AI ใช้เวลานานเกินไป | ย้ายงานที่รอนานไป Queue และแสดง progress |
| feature | การแก้ไขจุดใดจุดหนึ่งทำให้จุดอื่นพัง | จัดกลุ่ม code ให้ชัดเจนและเพิ่มการทดสอบ (tests) |
| ทีมงาน | 2 คนเขียนทับงานของกันและกัน | ใช้ Git branches, ตรวจสอบการเปลี่ยนแปลง และกำหนดเจ้าของงาน |
| ความต้องการทางธุรกิจ | ระบบล่มทำให้ทีมทำงานไม่ได้ | เพิ่มระบบแจ้งเตือน, ขั้นตอนการกู้คืน และระบุผู้รับผิดชอบที่ชัดเจน |
| ความต้องการของลูกค้า | ลูกค้าแต่ละรายขอเงื่อนไขพิเศษ | ย้ายกฎไปไว้ใน settings; อย่า copy app ทั้งชุดเพื่อลูกค้าแต่ละราย |

### Easy-to-change and hard-to-change choices

บางทางเลือกเปลี่ยนง่าย บางทางเลือกเปลี่ยนยากครับ

- **เปลี่ยนง่าย:** สี button, layout ของ page และคำใน report เลือกแบบที่พอใช้ได้แล้วไปต่อได้เลยครับ
- **เปลี่ยนยาก:** ความหมายของ data, วิธีที่ user login, เจ้าของ vendor account, ที่เก็บ data ส่วนตัว และสิ่งที่สัญญาไว้กับลูกค้า เรื่องเหล่านี้ควรคิดให้รอบก่อนเลือกครับ

AI ทำให้เขียน code ใหม่ได้เร็วขึ้น แต่บางเรื่องที่ตัดสินใจไปแล้วเปลี่ยนยากครับ ถ้าลูกค้ามี invoice ย้อนหลัง 6 เดือน การเปลี่ยน Database ต้องทำ migration ถ้าพนักงาน 100 คนใช้ Google login การเปลี่ยน auth service จะกระทบทั้ง 100 คน และถ้า contract ระบุ availability 99.9% คุณก็แก้คำสัญญานั้นด้วยการแก้ code อย่างเดียวไม่ได้

### Add a new part only for a clear reason

ก่อนจะยอมรับแผนงานที่ซับซ้อน ให้ถามว่า:

1. **สิ่งนี้แก้ปัญหาอะไรในตอนนี้?** page ช้า, งานพัง หรือเป็นแค่การคาดเดาถึงอนาคต?
2. **วิธีแก้ที่เรียบง่ายกว่านี้คืออะไร?** การทำ index อาจแก้ปัญหาที่เคยเสนอให้เปลี่ยน database ใหม่ได้ครับ
3. **สิ่งนี้เพิ่มงานอะไรเข้ามาบ้าง?** ระบบ queue ช่วยเรื่องงานที่รอนาน แต่ตอนนี้คุณต้องจัดการกับงานที่พัง และงานที่อาจถูก run ซ้ำ 2 ครั้งครับ

ดังนั้นเมื่อ AI เสนอสิ่งที่ฟังดูซับซ้อนจนน่าทึ่ง ลองใช้รูปแบบนี้ครับ

**ถาม AI แบบนี้:** ขอ version ที่เรียบง่ายที่สุดซึ่งใช้ได้กับ user ภายใน 20 คน บอกผมด้วยว่าต้องเปลี่ยนอะไรถ้าระบบโตขึ้น แต่ตอนนี้ยังไม่ต้องสร้างส่วนนั้น

คุณจะได้ version ที่เรียบง่ายพร้อมแผนสำหรับอนาคต ซึ่งเป็นสิ่งที่วิศวกรระดับ senior ที่ดีจะเตรียมให้คุณครับ

## How to Brief an AI — Prompting Is Specification

สำหรับงาน software วิธี brief สำคัญกว่าการหา prompt ที่ใช้ได้กับทุกงานครับ ยิ่งคุณอธิบาย context, rule และวิธีตรวจงานได้ชัด AI ก็ยิ่งเดาน้อยลง

ลองนึกถึงตอนทำห้องครัวใหม่ครับ คุณคงไม่บอกช่างแค่ว่า **ทำให้ดีนะ** แต่จะบอกว่าใครทำอาหาร ใช้บ่อยแค่ไหน มี budget เท่าไหร่ และของอะไรต้องเก็บบ้าง การ brief งาน software ก็คล้ายกันครับ

brief งาน software ที่ดีมี 4 ส่วนครับ:

**1. Context.** ใครใช้? มีกี่คน? ใช้ผ่าน phone หรือ computer? เป็น internal tool, B2C หรือ B2B? เช่น **นี่คือ internal tool สำหรับทีมบัญชี 8 คน ใช้บน computer ช่วงสิ้นเดือน**

**2. Rules that must stay true.** Data ต้องไม่หาย app ต้องรองรับภาษาไทย พนักงานห้ามเห็นเงินเดือนของคนอื่น และ user ต้อง login ด้วย Google account ของบริษัท คุณรู้ business rule เหล่านี้ดีกว่า AI ดังนั้นเขียนออกมาให้ครบครับ

**3. Ask it to explain the result.** ถามว่า **สร้างอะไรไปแล้ว มีส่วนหลักอะไรบ้าง และ data อยู่ที่ไหน ช่วยอธิบายด้วยคำง่าย ๆ** ถ้ายังไม่เข้าใจส่วนไหน ให้ถามซ้ำครับ เพราะคุณเป็นเจ้าของ tool นี้และต้องอ่านภาพรวมของระบบออก

**4. Ask what is missing.** ถามว่า **ส่วนไหนยังไม่ได้สร้าง และมีอะไรพังได้บ้าง** คำตอบอาจมีประโยชน์กว่า code ที่เพิ่ง generate ออกมาครับ

สำหรับงานขนาดใหญ่ ให้บอก AI ว่า **ก่อนเริ่มสร้าง ให้ถามข้อมูลที่ต้องรู้จากผมก่อน** คำถามเหล่านั้นจะช่วยให้คุณเห็นว่า business rule ส่วนไหนยังขาดครับ

### A prompt template for real software work

คุณไม่จำเป็นต้องมี prompt เดียวที่ใช้ได้กับทุกงาน แต่ให้ใช้ brief ที่ชัดเจนแทน คุณสามารถคัดลอกตัวอย่างนี้ไปใช้ได้ครับ:

```text
Goal
Build an internal expense approval tool for an 8-person company.

Users and permissions
- Employee: create a request and view only their own requests.
- Manager: view and decide requests from their team.
- Finance: view all approved requests and mark them paid.

Workflow
1. Employee submits amount, date, category, note, and receipt.
2. A manager approves, rejects, or returns it for changes.
3. Approved requests appear in a finance queue.
4. Every decision records who did it and when.

Rules and edge cases
- Amount must be positive and use THB.
- A user cannot approve their own request.
- Submitting twice must not create two expenses.
- A rejected or paid request cannot be edited.
- Receipt files may be PDF, JPG, or PNG, maximum 10 MB.

Must-have rules
- Use company Google accounts for login.
- Desktop-first, but the submission form must work on mobile.
- Start for fewer than 20 users.
- Use ready-made services and one simple app.
- Never put secrets in frontend code.

How I will check it
- An employee cannot open another employee's request, even by changing the URL.
- A manager can filter pending requests.
- Finance can export approved requests as CSV.
- Refreshing the page does not lose saved work.
- Tests cover access rules and status changes.

Process
Before editing code, ask questions about unclear business rules.
Then show the data tables, app map, and work plan.
Build one small working slice at a time. After each slice, run tests and
explain what changed, what remains, and what I should verify manually.
```

เนื้อหาส่วนใหญ่ใน brief เป็นเรื่องของงานครับ คุณคือคนที่รู้ว่าใครทำอะไรได้ workflow เริ่มและจบที่ไหน และต้องผ่านเงื่อนไขอะไรถึงเรียกว่า **เสร็จ**

### Give examples, especially the awkward ones

คำอย่าง **เรียบง่าย**, **ดี**, **ปลอดภัย** และ **ใช้ง่าย** เปิดให้ AI เดาได้เยอะครับ ให้ยกตัวอย่างสิ่งที่ผ่านและไม่ผ่านแทน

แทนที่จะบอกสั้น ๆ ว่า **ตรวจสอบยอดเงิน** ให้ใส่ตัวอย่างแบบนี้ครับ:

- `500` คือค่าที่ถูกต้อง;
- `0`, `-10` และค่าว่าง คือค่าที่ผิด;
- `1,250.50` ควรถูกเข้าใจว่าเป็น 1,250.50 บาท;
- ถ้าวางค่าที่มีคำว่า `THB` ติดมาด้วย ควรแสดงข้อความแนะนำการแก้ไขแทนที่จะทำให้ program ค้าง

ตัวอย่างเหล่านี้อธิบาย rule และนำไปเขียน test ต่อได้ครับ ทีม tech เรียกรายการที่ใช้ตัดสินว่างานผ่านหรือไม่ว่า **acceptance criteria**

### Separate plan, build, and review

ถ้าขอให้ AI สร้าง app ทั้งก้อนในครั้งเดียว เวลาพังจะหาต้นเหตุยากครับ ให้แบ่งเป็นชิ้นเล็ก ๆ ที่ test ได้แทน:

ทุก checkpoint ให้ AI แสดงข้อมูลเหล่านี้ครับ:

- เปลี่ยน file หรือ component ไหนบ้าง;
- assumption ที่ AI ใช้มีอะไรบ้าง;
- run test อะไรไปบ้าง และผลเป็นอย่างไร;
- ขั้นตอนที่คุณควรลองทดสอบด้วยตัวเองมีอะไรบ้าง;
- มี service, บัญชี, secret หรือค่าใช้จ่ายรายเดือนอะไรที่ถูกเพิ่มเข้ามาหรือไม่

### Ask for proof, not confidence

AI เขียน code และช่วย test ได้ครับ แต่คำตอบว่า **น่าจะใช้ได้** ยังไม่พอ หลักฐานที่ตรวจได้คือ test result, record ใน database, screenshot หรือขั้นตอนที่คุณลองด้วยตัวเอง

ลองใช้ prompt เหล่านี้ครับ:

- run test แล้วแสดงกรณีที่ไม่ผ่านให้ผมดู อย่าบอกว่าสำเร็จถ้ายังไม่ได้ run
- ทดสอบใน role พนักงาน, manager และ user ที่ไม่มี permission
- ลองใส่ค่าผิด, ค่าว่าง, ค่าซ้ำ และค่าที่ใหญ่มาก
- Review เฉพาะส่วนที่เปลี่ยน มองหา data loss, permission gap, secret ที่หลุด และ error handling ที่ตกหล่น
- บอก assumption ที่เสี่ยงที่สุดซึ่งยังเหลืออยู่

ให้ AI ช่วยลงมือ แต่คุณยังต้องตรวจผลลัพธ์ครับ

prompt ที่ดีช่วยลดการเดาได้มากครับ แต่ยังตัดสิน business rule แทนคุณไม่ได้ ส่วนถัดไปคือพื้นฐานที่ช่วยให้คุณ review งานของ AI ได้ดีขึ้น

## The Computer Science Fundamentals You Actually Touch

คุณไม่จำเป็นต้องลงเรียนคอร์ส Data Structures & Algorithms แบบเต็มรูปแบบก่อนจะสร้าง tool เล็ก ๆ ชิ้นแรกครับ สิ่งที่คุณต้องรู้คือแนวคิดพื้นฐานแบบไหนที่เหมาะกับปัญหาแต่ละประเภท

หลายคำใน Computer Science คือชื่อของปัญหาที่คนทำ software เจอซ้ำ ๆ ครับ คุณไม่ต้องท่องจำตำรา แค่รู้ว่าคำไหนใช้เรียกปัญหาแบบใดและแนวทางแก้มีอะไรบ้าง

### Data has a type, a meaning, and a lifetime

สำหรับ computer ค่า `number: 100`, `text: 100` และ `text: THB 100` ต่างกันครับ ตัวแรกคือตัวเลข ส่วนอีก 2 ตัวคือข้อความ แต่สำหรับธุรกิจของคุณ `100` ก็ยังไม่ชัดเจนอยู่ดีว่าเป็น บาท, ดอลลาร์, จำนวนชิ้น หรือรหัสอะไรบางอย่าง

สำหรับทุกช่องข้อมูล หรือ **field** ที่สำคัญ ให้กำหนดเรื่องต่อไปนี้ครับ:

- **Type:** ข้อความ, ตัวเลข, วันที่และเวลา, ใช่/ไม่ใช่ หรือเลือก 1 ตัวเลือกจากรายการที่กำหนดไว้
- **Meaning:** `approved_at` คือเวลาที่มีการอนุมัติ เกิดขึ้นตอนไหน ไม่ใช่ตอนที่สร้างคำขอ
- **Unit:** THB, กิโลกรัม, นาที หรือเปอร์เซ็นต์
- **Must have or can be empty:** ถ้าค่าว่าง หมายความว่าอย่างไร
- **Valid range:** จำนวนเงินเป็น 0 ได้ไหม วันที่ส่งของเป็นวันที่ในอดีตได้หรือเปล่า
- **How long to keep it:** เก็บไว้ตลอดกาล, 7 ปี หรือ 30 วัน

กฎเล็ก ๆ น้อย ๆ เหล่านี้ช่วยลดข้อผิดพลาดได้มากครับ:

- **Money:** เก็บ currency ไว้ด้วย และใช้ decimal ที่เหมาะกับงานเงิน หรือเก็บเป็นหน่วยเล็กสุด เช่น สตางค์ เพราะ floating-point number อาจมีปัญหาเรื่องการปัดเศษจนยอดรวมคลาดเคลื่อนได้
- **Dates and time:** บันทึกเวลาพร้อมกับ timezone และแสดงผลตาม timezone ของ user การระบุแค่ July 17 ไม่เพียงพอสำหรับการทำงานระหว่างกรุงเทพฯ และลอนดอนครับ
- **ID:** ให้ ID ที่ตายตัวกับทุกแถว อย่าใช้ชื่อหรือเลขลำดับแถว เพราะชื่อเปลี่ยนได้ และแถวสามารถเลื่อนตำแหน่งได้
- **Empty values:** ค่าที่หายไป, 0, false และข้อความว่าง มีความหมายต่างกัน ให้ตัดสินใจว่าในทางธุรกิจตั้งใจให้เป็นแบบไหน

**ถาม AI แบบนี้:** ทำรายการ field ทั้งหมดแบบง่าย ๆ บอก type, ความหมาย, หน่วย, ปล่อยว่างได้หรือไม่, กฎ, ตัวอย่าง และระยะเวลาที่ต้องเก็บ ถ้า field ใดยังไม่ชัด ให้ถามผมก่อน

### Data modeling — turn nouns and relationships into records

**Data modeling** คือการเลือกว่าจะเก็บ record อะไร และ record เหล่านั้นเชื่อมกันแบบไหนครับ ลองกลับมาที่ตัวอย่างอนุมัติค่าใช้จ่ายช่วงต้นบทความ คำนามหลักของงานนี้คือ Employee, Expense, Receipt, Approval และ Payment ซึ่งอาจกลายเป็น table จากนั้นเขียน relationship กับ business rule ออกมาครับ:

- พนักงาน 1 คนสามารถส่งเบิกค่าใช้จ่ายได้หลายรายการ
- ค่าใช้จ่าย 1 รายการต้องเป็นของพนักงาน 1 คน
- Expense 1 รายการมีประวัติการอนุมัติได้หลายครั้ง
- receipt file 1 file ผูกกับ Expense 1 รายการ
- ชุดการจ่ายเงิน 1 ชุดประกอบด้วยค่าใช้จ่ายที่อนุมัติแล้วหลายรายการ

**Constraint** คือกฎที่ database ช่วยบังคับใช้ครับ เช่น email ห้ามซ้ำ, Expense ต้องมี employee และ amount ห้ามว่าง หน้า form ควรแจ้ง error ให้ user เข้าใจ ส่วน database constraint เป็นด่านสุดท้ายที่กันข้อมูลผิดไม่ให้ถูกบันทึก

### Algorithms — a recipe, plus the cost of repeating it

**Algorithm** คือลำดับขั้นตอนที่เปลี่ยน input ให้เป็น output กฎอนุมัติของคุณก็คือ algorithm 1 แบบครับ:

```text
ถ้าจำนวนเงินต่ำกว่า 5,000 บาท:
  ส่งให้ผู้จัดการของพนักงาน
มิฉะนั้น:
  ส่งให้ผู้จัดการ จากนั้นส่งให้ฝ่ายการเงิน
ถ้าผู้ส่งเป็นคนอนุมัติคำขอของตัวเอง:
  ส่งให้ผู้อนุมัติคนถัดไปที่มีสิทธิ์
```

คราวนี้ลองคิดถึงสิ่งที่เกิดขึ้นเมื่อข้อมูลมีจำนวนมากขึ้น เช่น คุณต้องการค้นหาลูกค้า 1 ราย:

- การอ่านทุกแถวตั้งแต่บนลงล่างอาจจะโอเคสำหรับ 100 แถวครับ
- แต่ถ้าทำแบบเดียวกันกับ 10,000,000 rows ทุกครั้งที่เปิด page ระบบจะใช้ resource มากเกินไป
- **Index** ของ database ทำงานเหมือนดัชนีท้ายเล่มหนังสือ ช่วยให้ database กระโดดไปยังแถวที่ถูกต้องได้ทันที

ทีม tech อาจเรียก cost ของ algorithm ว่า **Big O** เช่น `O(n)` คุณไม่ต้องจำสูตรครับ ลองถาม 3 ข้อนี้แทน:

- data เพิ่ม 2 เท่า งานที่ระบบต้องทำเพิ่มแค่ไหน
- code ต้องอ่านทุก record หรือมี index ช่วยหา
- มีจุดไหนเอาทุก record มาเทียบกันเองจนจำนวนงานโตเร็วมากหรือไม่

คุณไม่จำเป็นต้องทำให้ทุกขั้นตอนเร็วที่สุด report ที่ช้าแต่ใช้วันละครั้งอาจจะไม่มีปัญหา แต่การค้นหาที่ช้าซึ่งทุกคนต้องใช้บ่อย ๆ คือปัญหาใหญ่ครับ

เครื่องมือ 3 อย่างที่ช่วยแก้ปัญหาการเติบโตของข้อมูล:

- **Indexing:** ค้นหา record ด้วย field ที่ค้นหาบ่อย เช่น email, สถานะ หรือวันที่สร้าง
- **Pagination:** โหลดทีละ 50 แถว แทนที่จะโหลดทั้งหมด 50,000 แถว โดยให้ user เปลี่ยนหน้าหรือเลื่อนลงเพื่อดูเพิ่ม
- **Precompute:** คำนวณยอดรวมรายวันไว้ก่อนแล้วนำมาใช้ซ้ำ แทนการอ่านข้อมูลเก่าทั้งหมดทุกครั้งที่เปิด Dashboard

**ถาม AI แบบนี้:** สำหรับ list, search, report และงานที่ทำซ้ำแต่ละจุด บอกผมว่าจะเกิดอะไรเมื่อ data โตจาก 100 เป็น 10,000 rows หาจุดที่อ่านทุก row หรือไม่มี limit แล้วแก้เฉพาะส่วนที่สำคัญกับขนาดที่เราคาดไว้

### State — what is true now, and which changes are legal

**State** คือค่าที่บอกว่าตอนนี้ record อยู่ช่วงไหนครับ ในตัวอย่างอนุมัติเดิม Expense อาจเป็น `draft`, `submitted`, `approved`, `rejected` หรือ `paid` อย่าปล่อยให้ user พิมพ์ status เอง ควรกำหนดค่าที่ใช้ได้และ transition ที่อนุญาตไว้

**State machine** คือแผนผังการเปลี่ยนสถานะที่อนุญาต:

ถ้า record มี `status` ลองขอให้ AI วาด State machine แบบนี้ครับ ใช้ได้กับ purchase order, ticket, task, delivery และ invoice

### Concurrency — two valid actions can collide

**Concurrency** คือการที่เหตุการณ์ 2 อย่างเกิดขึ้นในเวลาเกือบจะพร้อมกัน เช่น คน 2 คนเห็นตั๋วใบสุดท้ายใบเดียวกัน และคลิกซื้อพร้อมกัน app อาจจะขายตั๋วใบเดียวให้คน 2 คนถ้าขั้นตอนการตรวจสอบและบันทึกแยกจากกัน

**Transaction** รวมการเปลี่ยนแปลงที่เกี่ยวข้องกันให้ผ่านทั้งหมดหรือล้มทั้งหมดครับ เวลาโอนเงิน การหัก account ต้นทางกับเพิ่มยอด account ปลายทางต้องสำเร็จด้วยกัน ส่วนตั๋วใบสุดท้าย การเช็กว่าตั๋วยังว่างกับการจองต้องอยู่ใน transaction เดียวกัน

อีกปัญหาคือ user double-click เพราะ request แรกยังไม่ตอบกลับ app จึงได้รับ request เดิม 2 ครั้ง

**Idempotency** หมายถึงต่อให้ request เดิมเข้ามา 2 ครั้ง ผลลัพธ์ทางธุรกิจควรเกิดครั้งเดียวครับ วิธีหนึ่งคือใส่ request ID ถ้า ID นี้เคยถูกประมวลผลแล้ว ระบบก็ไม่จ่ายเงิน จอง หรือสร้าง record ซ้ำ

**ถาม AI แบบนี้:** Action ใดเกิดซ้ำหรือเกิดพร้อมกันได้บ้าง ป้องกันรายการซ้ำที่สร้างปัญหาด้วย database transaction, unique rule หรือ request ID แล้วทดสอบด้วยการส่ง request เดิม 2 ครั้ง

### Synchronous, asynchronous, and events

แบบ **Synchronous** คือ user ต้องรอ response เช่น กดบันทึก form แล้วรอจนหน้าจอบอกว่าสำเร็จ

แบบ **Asynchronous** คือรับงานไว้ก่อน แล้วทำต่อข้างหลังครับ เช่น import file 20,000 rows หรือ process video ที่ใช้เวลาหลายนาที ให้ส่ง job เข้า Queue แล้วให้ Worker จัดการ ส่วนหน้าจอแสดง status เป็น `waiting`, `working`, `done` หรือ `failed`

คุณอาจเคยเห็นสิ่งนี้หลังจากอัปโหลด file ขนาดใหญ่หรือขอให้เว็บไซต์สร้าง report หน้าจอจะบอกว่า **Waiting** จากนั้นเป็น **Working** และจบที่ **Done** หน้าจอนี้คือส่วนที่คุณเห็น ส่วน Queue และ Worker คือส่วนที่ทำงานอยู่เบื้องหลังครับ

**Event** คือ message ที่บอกว่ามีบางอย่างเกิดขึ้นแล้ว เช่น `ExpenseSubmitted`, `PaymentReceived` หรือ `CustomerCreated` งานอื่นใช้ Event นี้เป็น trigger ต่อได้ เช่น หลังส่ง Expense ระบบอาจแจ้ง manager และบันทึก audit history แยกกัน

Background job อาจ fail, run ช้า, run ซ้ำ หรือจบไม่ตามลำดับครับ จึงต้องมี status, retry rule, วิธีป้องกันข้อมูลซ้ำ และหน้าสำหรับดู failed job แต่ถ้างานจบได้เร็วพอที่ user จะรอ ก็เริ่มด้วย Synchronous แบบง่ายกว่าก่อนได้

### Input, output, and errors are part of the design

app ของคุณรับข้อมูลจาก form, file, sheet, API, webhook และ AI ครับ คุณควรตรวจสอบข้อมูลทั้งหมดก่อนนำไปใช้งาน

**Validation** เช็กว่า input มีรูปแบบและความหมายที่ใช้ได้ ส่วน **Error handling** กำหนดว่าระบบจะตอบอย่างไรเมื่อเจอข้อมูลผิดหรือ service พัง คุณควรเตรียมข้อมูลให้คน 3 กลุ่มครับ:

- **user:** เกิดอะไรขึ้น, ข้อมูลอะไรที่ถูกบันทึกไว้ และต้องทำอย่างไรต่อ
- **เจ้าของ app:** คำขอไหนที่ล้มเหลว และต้องไปตรวจสอบที่จุดใด
- **ระบบ:** ควรลองใหม่อีกครั้ง, ปฏิเสธคำขอ หรือแยกงานนี้ออกไปเพื่อรอการตรวจสอบ

ข้อความแบบ Something went wrong ไม่ได้บอกขั้นตอนถัดไป ส่วน error จาก database แบบดิบ ๆ อาจแสดงข้อมูลส่วนตัวออกมาครับ คุณควรให้ข้อความที่ปลอดภัยและมีประโยชน์แก่ user แล้วเก็บรายละเอียดทางเทคนิคไว้ใน log ส่วนตัวครับ

### Where your code lives — Git and GitHub

**GitHub เป็นที่เก็บ code กลางของ project และเก็บประวัติการเปลี่ยนแปลงครับ** ให้จำ 2 แนวคิดนี้ไว้:

- **Version history.** ทุกการเปลี่ยนแปลงที่สำคัญใน project จะถูกบันทึกเป็น snapshot ที่เรียกว่า commit ถ้าการเปลี่ยนแปลงของวันนี้ทำให้ทุกอย่างพัง คุณสามารถย้อนกลับไปใช้ snapshot ของเมื่อวานได้ครับ ถ้าคุณเคยสร้าง file ชื่อ `report_final_v2_REAL.xlsx` version control คือแนวคิดเดียวกัน แต่ให้ระบบจัดการแทนการตั้งชื่อ file ครับ
- **One main copy.** อย่าเก็บ code ไว้แค่ใน laptop เพราะเครื่องอาจพังหรือหายได้ GitHub ช่วยให้ทีมใช้ source ชุดเดียวกัน และ hosting service ดึง code ชุดนั้นไป deploy ได้ครับ

AI ของคุณตั้งค่าเรื่องนี้ได้ครับ **ถาม AI แบบนี้:** เชื่อม project นี้กับ private GitHub project หลังการแก้ไขแต่ละรอบที่ใช้งานได้ ให้บันทึก snapshot และเขียน note สั้น ๆ ว่าเปลี่ยนอะไรไปบ้าง

### Deployment — putting it online

ตอนพัฒนา app อาจ run อยู่แค่ใน laptop ของคุณครับ **Deployment** คือการนำ app ไป run บน hosting service เพื่อให้ user เปิดผ่าน URL ได้โดยไม่ต้องเปิดเครื่องของคุณทิ้งไว้

ปกติ hosting service หรือ cloud provider จะดึง code จาก GitHub แล้ว build และ run ให้ตาม flow นี้ครับ:

ไม่มี provider ตัวเดียวที่เหมาะกับทุก app ครับ ให้เลือกระหว่างความง่ายในการเริ่มต้นกับระดับการควบคุมที่คุณต้องการ

#### Path 1: a simple hosting service

[Vercel](https://vercel.com/), [Netlify](https://www.netlify.com/), [Railway](https://railway.com/) และ [Render](https://render.com/) ลดงานตั้งค่า cloud ให้คุณครับ เชื่อม GitHub, ใส่ config ที่จำเป็น แล้วรับ URL ไปใช้ได้เลย วิธีนี้มักพอกับเว็บไซต์ขนาดเล็ก, product รุ่นแรก หรือ internal tool

การเริ่มต้นที่ง่ายนี้ยังมีข้อจำกัดครับ ให้เช็คว่า database และ file ถูกเก็บไว้ที่ไหน เช็คราคาต่อเดือน, กฎการ login, การสำรองข้อมูล และสิ่งที่จะเกิดขึ้นเมื่อมีคนใช้งาน app มากขึ้นครับ

#### Path 2: a large cloud provider

[Amazon Web Services](https://aws.amazon.com/), [Microsoft Azure](https://azure.microsoft.com/) และ [Google Cloud](https://cloud.google.com/) มี service ตั้งแต่ compute, database, storage, private network, identity, logs ไปจนถึง backup ครับ

หลายบริษัทเลือกใช้บริการเหล่านี้เพราะมีบัญชีและทีมเทคนิคอยู่แล้ว หรือเพราะต้องการระบบ login ของบริษัท, network ส่วนตัว, การเลือกภูมิภาคที่เก็บข้อมูล หรือบันทึกการเข้าถึงที่ละเอียดครับ

คุณจะควบคุมได้มากขึ้น แต่ก็มีตัวเลือกให้ตัดสินใจมากขึ้นด้วย ผู้ให้บริการแต่ละรายมี service จำนวนมากและหลายชื่อคล้ายกัน App อาจเปิดใช้งานได้ทั้งที่ตั้งค่าไม่ปลอดภัย หรืออาจมี bill ที่สูงกว่าที่คิดครับ ควรให้ software team หรือ cloud team ตรวจแผนก่อนนำข้อมูลธุรกิจที่สำคัญขึ้น production

#### Path 3: let an AI tool handle part of the cloud work

AI tools บางตัวสามารถสร้างและ deploy ได้ในเวลาเดียวกันครับ

[Google AI Studio](https://aistudio.google.com/) ช่วยสร้าง AI app แบบ full-stack และ deploy ไป [Cloud Run](https://cloud.google.com/run) บน Google Cloud ได้ครับ AI Studio ช่วยฝั่งการสร้าง ส่วน Cloud Run รับหน้าที่ run app ใน production

[Microsoft Foundry](https://ai.azure.com/) เน้นงาน AI model และ agent มากกว่าครับ **prompt agent** ใช้ code น้อย ส่วน **hosted agent** run agent code ของคุณเองได้ Foundry ช่วยดูแล endpoint, identity, scaling และ logs แต่ page, business database และ user login อาจยังต้องใช้ Azure service ตัวอื่น

สรุปสั้น ๆ คือ AI Studio ช่วยจากฝั่งการสร้าง app ส่วน Foundry เน้นการ run และจัดการ agent ครับ ก่อน publish ยังต้อง review ว่าใครเข้าได้, data อะไรถูกส่งเข้า model, data อยู่ region ไหน, ค่าใช้จ่ายมาจากอะไร และย้ายออกได้อย่างไร

จำกฎ 2 ข้อนี้ไว้ครับ:

- **แยก production ออกจากงานทดสอบ** ให้ทดสอบบน computer ของคุณหรือ preview link แล้วนำการเปลี่ยนแปลงขึ้น production เมื่อตรวจแล้วว่าใช้งานได้ครับ
- **Online ไม่ได้แปลว่า public** เครื่องมือภายในที่ deploy แล้วควรมี login ป้องกันไว้ ถ้าลิงก์หลุดออกไป คนที่ไม่มีสิทธิ์ก็ยังเปิดข้อมูลไม่ได้ครับ หัวข้อถัดไปจะอธิบายเรื่องนี้ต่อ

**ถาม AI แบบนี้:** เทียบ simple host, AWS, Azure และ Google Cloud สำหรับ app นี้ ตรวจด้วยว่า Google AI Studio หรือ Microsoft Foundry เหมาะกับส่วน AI หรือไม่ อธิบายว่าแต่ละทาง run อะไร ผมยังต้องตั้งค่าอะไร ค่าใช้จ่ายน่าจะมาจากจุดใด และใครควร review ก่อนเปิดใช้ ยังไม่ต้อง deploy

### The database — where your records live

ถ้าจะจำแนวคิดทางเทคนิคจากโพสต์นี้เพียง 1 อย่าง ขอให้จำเรื่อง database ครับ เพราะข้อผิดพลาดของข้อมูลแก้ไขได้ยากในภายหลัง

**Database ใช้เก็บ record ของ app** เช่น การจอง, คำขอ และข้อมูลลูกค้าครับ หน้าตาคล้าย sheet เพราะมี table, column กับ row แต่ Database ถูกออกแบบมาให้ app ใช้งานหลายแบบพร้อมกัน:

- **ข้อมูลยังอยู่หลัง restart** ถ้าเก็บข้อมูลไว้แค่ใน memory ของ app พอ restart แล้วข้อมูลจะหายครับ
- **รองรับ user หลายคนพร้อมกัน** 2 คนสามารถส่ง form ได้ในเวลาเดียวกันโดยไม่ทำให้ file ที่ใช้ร่วมกันพังครับ
- **กันข้อมูลผิดก่อนถูกบันทึก** เช่น ไม่รับการจองที่ไม่มีวันที่ หรือไม่ยอมให้ 2 account ใช้ email เดียวกันครับ

ชื่อที่คุณอาจเห็น: [**PostgreSQL**](https://www.postgresql.org/) หรือ Postgres เป็น database ที่นิยมมาก [**SQLite**](https://sqlite.org/) เก็บ database ขนาดเล็กไว้ใน file เดียว [**Supabase**](https://supabase.com/), [**Neon**](https://neon.com/), และ [**Firebase**](https://firebase.google.com/) ให้บริการจัดการข้อมูลให้คุณครับ AI รู้จักเครื่องมือเหล่านี้ แต่ AI ไม่รู้กฎธุรกิจของคุณ เช่น ลูกค้า 1 คนสามารถมีการจองได้หลายครั้ง คุณต้องเป็นคนอธิบายส่วนนี้ครับ

**Query** คือคำถามหรือคำสั่งสำหรับ database เช่น ค้นหาค่าใช้จ่ายที่ยังไม่เสร็จ, นับจำนวนการจอง หรือทำเครื่องหมายว่าใบแจ้งหนี้ชำระเงินแล้ว SQL เป็นภาษาที่ใช้เขียนคำสั่งเหล่านี้ครับ code อาจเขียนถูกตามกฎของภาษา แต่ยังตอบคำถามธุรกิจผิดได้ ให้ขอ AI อธิบาย query สำคัญ ๆ ด้วยภาษาที่ทีมของคุณเข้าใจครับ

จดไว้ด้วยว่าใครเป็นเจ้าของ database account และ export ทุก row อย่างไรครับ ต่อให้เลิกใช้ AI builder ตัวเดิม ธุรกิจก็ยังต้องเข้าถึง data ของตัวเองได้

**ถาม AI แบบนี้:** App นี้ต้องเก็บ [record ของคุณ] ระยะยาวใน database ไม่ใช่ใน file หรือ memory ช่วยตั้ง hosted database แล้วอธิบายด้วยคำง่าย ๆ ว่ามี table อะไร และแต่ละ table เก็บอะไร

### Object storage — where files live

สำหรับรูปภาพ, PDF และ file ที่ upload คุณจะได้ยินคำว่า **object storage** ครับ ลองนึกว่าเป็นตู้เก็บเอกสาร online โดย database เก็บแถวข้อมูล ส่วน object storage เก็บตัว file ครับ

ที่เก็บ file จะให้ลิงก์แก่ app ของคุณ และ database จะบันทึกลิงก์นั้นไว้ในแถวที่ถูกต้องครับ ใบเสร็จจะอยู่ในตู้เก็บ file ส่วนแถวค่าใช้จ่ายจะบอกว่าสามารถหาใบเสร็จนั้นได้ที่ไหน

ถ้าคุณเคยแนบใบเสร็จใน form ค่าใช้จ่าย คุณได้ใช้แนวคิดนี้แล้วครับ ช่องใน form จะกลายเป็นแถวใน database ส่วน PDF หรือรูปภาพจะไปอยู่ใน object storage แล้วใช้ลิงก์ที่บันทึกไว้เชื่อมทั้ง 2 อย่างเข้าด้วยกันครับ

file ส่วนตัวต้องมีการป้องกัน **signed URL** คือลิงก์ที่ใช้ได้ในช่วงเวลาสั้น ๆ และเปิดได้เฉพาะ user ที่ได้รับอนุญาต ควรกำหนดประเภทและขนาดสูงสุดของ file ไว้ด้วยครับ

**ถาม AI แบบนี้:** User จะ upload [photo / PDF] ให้เก็บ file ใน object storage ไม่เก็บใน database หรือ disk ของ app และบันทึก link ของแต่ละ file ไว้กับ record ที่เกี่ยวข้อง

### Login — who is this, and what may they do?

tool ที่ไม่ได้เปิด public ต้องตอบ 2 คำถามนี้ให้ได้ครับ และ 2 คำนี้มักถูกสลับกัน:

- **Authentication — คุณคือใคร?** คือขั้นตอน login และพิสูจน์ว่าคนนี้เป็นเจ้าของ account
- **Authorization — คุณได้รับอนุญาตให้ทำอะไรได้บ้าง?** คือกฎหลัง login เช่น พนักงานบัญชีเห็นข้อมูลเงินเดือน แต่เด็กฝึกงานไม่เห็นครับ

คุณเคยใช้ทั้ง 2 ส่วนนี้อยู่แล้วครับ หน้า login เช็กว่าคุณคือใคร ส่วน menu, button และ data ที่เปิดได้หลัง login มาจากกฎ Authorization

**อย่าสร้างระบบรหัสผ่านของคุณเองครับ** ให้ใช้บริการ login ที่มีอยู่แล้ว สำหรับ internal tool ใช้ Sign in with Google หรือ Microsoft อาจเหมาะที่สุด เพราะพนักงานใช้ account ของบริษัทที่มีอยู่แล้วได้ครับ

ในรูปแบบ B2B คุณอาจจะได้ยินคำว่า **SSO (single sign-on)** ซึ่งช่วยให้พนักงานใช้การ login ของบริษัทเพียงครั้งเดียวกับหลายเครื่องมือได้ [Clerk](https://clerk.com/), [Auth0](https://auth0.com/), และ Supabase Auth เป็นบริการที่ช่วยเรื่องการ login ครับ

**ถาม AI แบบนี้:** เพิ่ม login with Google โดยใช้ auth service ที่มีอยู่ ไม่ต้องสร้าง password system เอง อนุญาตให้เฉพาะคนจาก domain ของบริษัท login ได้ หลัง login ให้ใช้กฎนี้: [authorization rule ของคุณ — ใครเห็นอะไรได้บ้าง]

### Secrets — the keys that must never leak

**API key** คล้ายกุญแจเข้าระบบครับ เป็น text ลับที่ service ใช้ยืนยันว่าใครมีสิทธิ์เรียกงาน คนที่ได้ database key อาจอ่านหรือลบ data ได้ ส่วน AI key ที่หลุดอาจถูกนำไปใช้จนเกิดค่าใช้จ่ายครับ

การรั่วไหลที่พบบ่อยคือการที่บางคนใส่กุญแจไว้ใน code แล้วส่ง code นั้นไปยัง GitHub อย่าทำแบบนี้เด็ดขาดครับ

เก็บ secret ไว้นอก code ครับ ตอนพัฒนาบน computer อาจใช้ file `.env` ที่ไม่ถูก commit ส่วน production ให้ตั้งค่าใน settings ของ host อย่าวาง key ลงใน public chat หรือ screenshot

**ถาม AI แบบนี้:** ตรวจ project นี้ว่ามี secret key เขียนอยู่ใน code หรือไม่ ย้ายทั้งหมดไป environment variable ป้องกันไม่ให้ upload ขึ้น GitHub และบอกผมว่าต้องตั้งค่าไว้จุดใดใน hosting service

### A name and a padlock — domains and HTTPS

2 เรื่องเล็ก ๆ นี้ช่วยให้ link ดูน่าเชื่อถือและปลอดภัยขึ้นครับ

- **Domain** คือชื่อที่คุณเช่ามา เช่น `tools.yourcompany.com` เครื่องมือภายในบริษัทอาจใช้ลิงก์ปกติของ host ได้ แต่ app สำหรับลูกค้าควรมีชื่อที่ชัดเจนครับ
- **HTTPS** คือสิ่งที่ทำให้เกิดรูปกุญแจบน browser ซึ่งจะช่วยซ่อนข้อมูลขณะที่เคลื่อนย้ายระหว่าง user กับ app host สมัยใหม่มักจะตั้งค่านี้ให้คุณอยู่แล้ว ถ้าหน้า login ไม่มีรูปกุญแจ ให้หยุดใช้งานทันทีครับ

### Backups — the part you only regret skipping

การคลิกผิดเพียงครั้งเดียวอาจทำให้ข้อมูลหายไปหลายแถว การเปลี่ยน table ข้อมูลอาจล้มเหลว หรือมีคนแฮ็กบัญชีได้ ให้ถามคำถามเดียวเลยครับว่า **เรามีสำเนาสำรองไว้ไหม?**

**Backup** คือการคัดลอก database แยกเก็บไว้จากตัวหลัก ให้ตรวจสอบ 2 เรื่องนี้ครับ:

- บริการที่คุณใช้มีการสำรองข้อมูลให้ใน package ที่จ่ายเงินอยู่หรือไม่?
- คุณสามารถ **restore** ได้หรือไม่? การ restore คือการนำสำเนาข้อมูลกลับมาใช้ ให้ทดสอบเรื่องนี้กับข้อมูลชุดทดลองก่อนที่จะจำเป็นต้องใช้ครับ

**ถาม AI แบบนี้:** ตั้ง automatic daily backup ให้ database แล้วพาผมลอง restore 1 ชุดลงใน test copy เพื่อยืนยันว่า backup ใช้งานได้

### Knowing when it's broken — logs and alerts

ถ้าเครื่องมือของคุณเสียตอน 9 โมงเช้า คุณจะรู้ได้อย่างไร? อย่ารอให้ user ที่กำลังโกรธมาบอกคุณครับ

2 แนวคิดนี้จะช่วยปิดช่องว่างนั้น:

- **Logs** คือบันทึกการทำงานของ app ซึ่งจะแสดงว่า app ทำอะไรไปบ้างและล้มเหลวจุดใด คุณสามารถนำส่วนที่ปลอดภัยของ log ให้ AI ช่วยวิเคราะห์ได้ แต่ต้องลบข้อมูลส่วนตัวและ key ต่าง ๆ ออกก่อนครับ
- **Alerts** จะแจ้งเตือนเมื่อ app มีข้อผิดพลาดหรือหยุดทำงาน ตัวอย่างที่พบบ่อยคือ [Sentry](https://sentry.io/welcome/) และ [UptimeRobot](https://uptimerobot.com/)

คุณอาจเคยเห็น alert ทาง email, ข้อความแชท หรือการแจ้งเตือนในมือถือ ซึ่งจะบอกสั้น ๆ เช่น **App down** ส่วน log จะเก็บรายละเอียดภายในเพื่อช่วยหาเหตุที่ระบบพังครับ

## Testing — Define What Must Stay True

**Testing คือการตรวจสอบว่ากฎสำคัญ ๆ ยังคงทำงานได้ถูกต้องหลังจากมีการเปลี่ยนแปลงทุกครั้งครับ**

การกด app ด้วยตัวเองช่วยเช็กข้อความและ workflow ได้ครับ แต่พอ app โตขึ้น เราไล่กดทุก case หลังแก้ code แต่ละรอบไม่ไหว automated test จึงช่วย run case สำคัญซ้ำให้ได้

### Start with risk, not test quantity

ลองกลับมาที่ตัวอย่างอนุมัติค่าใช้จ่ายช่วงต้นบทความครับ ปัญหาแต่ละข้อต่อไปนี้มีความรุนแรงไม่เท่ากัน:

- ไอคอนบริเวณส่วนหัวเคลื่อนไปเล็กน้อย
- พนักงานสามารถแอบดูใบเสร็จของพนักงานคนอื่นได้
- การ double-click ทำให้จ่ายเงินซ้ำ 2 ครั้ง
- ตัวเลขบน dashboard ไม่ update มา 1 ชั่วโมงแล้ว

ให้ทดสอบจุดที่มีความเสี่ยงสูงสุดก่อนครับ:

1. การกำหนดสิทธิ์และข้อมูลที่ละเอียดอ่อน
2. เรื่องเงิน, action ที่ย้อนกลับไม่ได้ และข้อมูลซ้ำ
3. กฎทางธุรกิจหลักและการเปลี่ยนสถานะต่าง ๆ
4. การบันทึก, การส่งออก และการกู้คืนข้อมูล
5. เส้นทางการใช้งานหลักของ user
6. รายละเอียดการแสดงผล

### Four useful kinds of test

ทีม tech แบ่ง test ตามขนาดและสิ่งที่กำลังเช็กครับ แต่ละทีมอาจใช้ชื่อไม่เหมือนกันทั้งหมด แต่คำเหล่านี้เจอบ่อย:

| Tech term | ความหมายแบบง่าย | ตัวอย่างจาก tool อนุมัติ |
|---|---|---|
| **Unit test** | ตรวจ function หรือ business rule เล็ก ๆ แยกจากส่วนอื่น | ยอด 4,999 บาท ต้องมีผู้อนุมัติ 1 คน, ส่วน 5,000 บาท ต้องผ่านฝ่ายการเงินด้วย |
| **Integration test** | ตรวจสอบว่า 2 ส่วนหรือมากกว่านั้นทำงานร่วมกันได้ | ระบบหลังบ้านบันทึกค่าใช้จ่ายลง database และส่ง email แจ้งเตือนได้ถูกต้อง |
| **End-to-end test (E2E)** | ใช้งาน app ทั้งหมดเหมือนเป็น user ตั้งแต่หน้าแรกจนถึงผลลัพธ์สุดท้าย | ลงชื่อเข้าใช้, ส่งรายการเบิก, อนุมัติ และดูสถานะใหม่ |
| **User acceptance testing (UAT)** | user ทางธุรกิจตรวจสอบว่าเครื่องมือตอบโจทย์การทำงานของทีม | ฝ่ายการเงินลองทำรายการปิดงบเดือนและยืนยันว่าพร้อมใช้งาน |

Unit test มักจะเล็กและเร็ว Integration test จะใช้ส่วนที่ใกล้กับระบบ production มากขึ้น E2E test จะช้ากว่าแต่ช่วยหาจุดเชื่อมที่ขาดระหว่างหน้าจอกับ service ได้ ส่วน UAT ต้องใช้คนที่เข้าใจธุรกิจ Software team พิสูจน์ได้ว่า code ทำตามกฎที่เขียนไว้ แต่ทีมของคุณต้องยืนยันว่ากฎนั้นสอดคล้องกับวิธีทำงานครับ

คุณอาจได้ยินชื่อที่อธิบายว่าทดสอบ **เมื่อไหร่หรือทำไม** ถึงต้องทดสอบ:

- **Smoke test:** เช็กสั้น ๆ หลัง deploy ว่าส่วนหลักยังเปิดได้ user login ได้ไหม เปิดหน้าแรกได้ไหม และบันทึก test data 1 รายการได้หรือเปล่า
- **Regression test:** run test เดิมซ้ำหลังเปลี่ยน code เพื่อหาว่าของที่เคยใช้ได้พังตามไปด้วยหรือไม่

Smoke test กับ Regression test อาจใช้ unit, integration หรือ E2E test ข้างในก็ได้ครับ กลุ่มแรกบอกขอบเขตที่ test ส่วน smoke กับ regression บอกเหตุผลที่ run

คุณไม่จำเป็นต้องทดสอบทุกบรรทัด แต่ให้ทดสอบมากขึ้นในจุดที่ถ้าเกิดข้อผิดพลาดแล้วจะสร้างความเสียหายรุนแรงครับ

### Test when things go wrong

ตอน demo เรามักใช้ข้อมูลที่เตรียมไว้ Internet ไม่หลุด และทุกคนกดตามขั้นตอนครับ แต่พอเปิดใช้จะมีเหตุการณ์นอกเส้นทางปกติ หรือ **happy path** โผล่มา ลอง test เรื่องเหล่านี้ด้วยครับ:

- ใส่ข้อมูลที่ว่างเปล่า, ผิดพลาด, ยาวเกินไป หรือข้อมูลแปลก ๆ
- ใช้ user ที่มีสิทธิ์การเข้าถึงผิดประเภท
- ข้อมูลที่ถูกแก้ไขในแท็บ browser อื่น
- การ double-click หรือการส่งข้อมูลซ้ำ
- บริการภายนอกที่ตอบสนองช้าจนหมดเวลา
- file ที่ใหญ่เกินไปหรือผิดประเภท
- กระบวนการที่ล้มเหลวกลางคัน
- การรีเฟรชหน้าจอ, การกดย้อนกลับ และการกลับมาทำต่อในภายหลัง

มี security test ง่าย ๆ ที่ควรลองกับ app จาก AI ครับ เปลี่ยน ID ใน URL ดู สมมติว่า `/records/123` เป็นข้อมูลของพนักงานอีกคน ถ้าเปลี่ยน URL แล้วเปิดได้ แปลว่า Backend เช็ก authorization ไม่ครบ ถึง Frontend จะซ่อน link ไว้ก็ยังไม่ปลอดภัย

### Keep development, preview, and production separate

**Development** คือที่สำหรับแก้ไขงาน **Preview** คือสำเนา online ที่ปลอดภัยสำหรับการทดสอบ และ **Production** คือ app ที่ user กำลังใช้งานครับ

ควรใช้ test data แยกจาก production data ให้ทดสอบการลบด้วยข้อมูลสำเนา อย่าทดสอบกับข้อมูลเงินเดือนของพนักงานครับ

**Migration** คือขั้นตอนที่บันทึกไว้เพื่อเปลี่ยน database เช่น เพิ่ม column ใหม่ ให้ backup ก่อน ทดสอบกับข้อมูลสำเนา แล้วจึงเปลี่ยน production database ครับ

**ถาม AI แบบนี้:** สร้าง risk-based test plan ให้ core workflow ระบุแต่ละ check เป็น unit, integration, E2E, UAT, smoke หรือ regression เพิ่ม automated test สำหรับ permission, การส่งซ้ำ, state transition และ data validation เขียนขั้นตอน UAT ให้ทีมธุรกิจ ใช้ test data แยก run test แล้ว report ว่าส่วนใดยังไม่ได้ทดสอบ

## Security and Privacy — Control Consequence, Not Fear

Security เริ่มจาก rule ที่ตอบได้ว่าใครทำอะไร เห็น data ชุดไหน เชื่อ input จากที่ใด และจะจำกัดผลกระทบเมื่อมีบางอย่างพังอย่างไรครับ การใส่ prompt ว่า **ทำให้ปลอดภัย** อย่างเดียวยังไม่พอ

คุณไม่จำเป็นต้องเป็นผู้เชี่ยวชาญด้านความปลอดภัย แค่สร้างนิสัยที่ดีไม่กี่อย่างก็พอครับ

### Minimize what you collect

ข้อมูลส่วนตัวที่ปลอดภัยที่สุดคือข้อมูลที่คุณไม่เคยบันทึกไว้เลย ก่อนจะขอเลขบัตรประชาชน ที่อยู่บ้าน วันเกิด หรือ file ต่าง ๆ ให้ถามว่า:

- ขั้นตอนการทำงานไหนที่จำเป็นต้องใช้ข้อมูลช่องนี้?
- ใช้ข้อมูลที่เปิดเผยได้มากกว่านี้แทนได้ไหม?
- ใครบ้างที่ต้องเข้าถึงข้อมูลนี้?
- ต้องเก็บไว้นานแค่ไหน?
- user จะขอส่งออก แก้ไข หรือลบข้อมูลได้อย่างไร?

ความเป็นส่วนตัวเริ่มต้นจากการเลือกสิ่งที่จะบันทึกครับ กฎหมายอย่าง PDPA, GDPR และระเบียบการทำงานอาจแตกต่างกันตามสถานที่และประเภทข้อมูล AI สามารถช่วยทำ checklist เบื้องต้นได้ แต่ควรให้ผู้เชี่ยวชาญด้านกฎหมายและความปลอดภัยตรวจสอบข้อมูลที่มีความเสี่ยงสูง เพราะธุรกิจของคุณยังคงต้องรับผิดชอบทางกฎหมายครับ

### Least privilege

**Least privilege** หมายถึงการให้สิทธิ์การเข้าถึงแก่ user และบริการเท่าที่จำเป็นต่อการทำงานเท่านั้น เช่น พนักงานเห็นเฉพาะคำขอของตนเอง ผู้จัดการเห็นคำขอของทีมตนเอง report สามารถอ่านข้อมูลได้แต่แก้ไขไม่ได้ครับ

อย่าใช้บัญชีเจ้าของร่วมกัน ให้เปิด login แบบ 2 ขั้นตอน (two-step login) สำหรับบริการทางธุรกิจที่สำคัญ ควรมีเจ้าของที่ไว้วางใจได้ 2 คนที่จะสามารถกู้คืนบัญชีได้ และให้ลบสิทธิ์การเข้าถึงทันทีเมื่อพนักงานลาออกครับ

### Validate on the trusted side

ตรวจสอบข้อมูลจาก browser, file, webhook, sheet หรือ AI ก่อนนำไปใช้งาน เพื่อป้องกันข้อผิดพลาดและการโจมตี ดังนี้ครับ:

- **Injection:** ข้อความจาก user ถูก run เป็นคำสั่ง database หรือคำสั่งระบบ
- **XSS:** ข้อความที่ถูกบันทึกไว้ถูก run เป็น code ใน browser ของ user อีกคน
- **Bad file:** file ที่อัปโหลดมีชื่อ ประเภท หรือขนาดที่เสี่ยงอันตราย
- **Prompt injection:** ข้อความใน file สั่งให้ AI ละเมิดกฎของคุณหรือแสดงข้อมูลส่วนตัว

ใช้ library มาตรฐานและ safe default ครับ อย่าต่อข้อความดิบจาก user เข้า SQL ให้ AI agent ใช้เฉพาะ tool ที่จำเป็น และเพิ่ม human confirmation ก่อนลบ, จ่ายเงิน, ส่งข้อมูล หรือ post แบบ public

### Authorization must be server-side and deny by default

สำหรับการกระทำที่สำคัญในระบบหลังบ้าน ให้ถามว่า: คนนี้คือใคร? ต้องการข้อมูลแถวไหน? กฎข้อไหนอนุญาตให้เขาทำได้? ถ้าไม่มีกฎข้อไหนบอกว่าได้ ให้ block การเข้าถึงทันที การซ่อน button หรือใช้ URL ที่เดายากนั้นไม่เพียงพอครับ

ใน Supabase, **Row Level Security (RLS)** จะช่วยเพิ่มกฎการเข้าถึงในระดับแถวของ database เช่น user อาจอ่านได้เฉพาะแถวที่มี `employee_id` ของตนเอง ให้ทดสอบด้วย user 2 คนและลองเปิดข้อมูลของกันและกันดูครับ

### Rate limits and abuse controls

บอทสามารถส่ง form สาธารณะได้มากกว่า 1,000 ครั้ง หรือ feature AI สาธารณะอาจทำให้งบประมาณคุณหมดลง **Rate limiting** คือการกำหนดว่า user หรือ IP แต่ละราย สามารถทำรายการได้กี่ครั้งในระยะเวลาที่กำหนดครับ

ใช้การจำกัดจำนวนร่วมกับการควบคุมค่าใช้จ่ายที่เหมาะสม:

- ตั้งการแจ้งเตือนงบประมาณรายเดือนจากผู้ให้บริการ
- จำกัดจำนวนครั้งต่อ user สำหรับการทำงานที่สิ้นเปลืองทรัพยากร
- กำหนดขนาด file สูงสุดและประเภท file ที่ยอมรับ
- ให้ยืนยันก่อนส่งข้อมูลจำนวนมาก, ลบข้อมูล หรือซื้อของ
- มีสวิตช์ตัวเดียวที่สามารถปิด feature ที่ราคาแพงได้อย่างรวดเร็ว

### Dependencies and updates

app ที่สร้างโดย AI มักดึง code package จากคนอื่นมาใช้ เราเรียกของเหล่านี้ว่า **dependencies** ครับ version เก่าอาจมีช่องโหว่ที่มีรายงานออกมาแล้ว ให้ lock version, เปิด security alert, update ทีละกลุ่มเล็ก ๆ และ run test ทุกครั้ง

AI ช่วย update library เหล่านี้ได้ครับ ให้ update ทีละกลุ่มเล็ก ๆ บันทึก checkpoint, run test, อ่านการเปลี่ยนแปลง แล้วค่อยนำขึ้น production

### Audit trail

Action สำคัญควรบันทึกว่าใครทำอะไร เมื่อไหร่ และทำกับ record ไหนครับ สิ่งนี้เรียกว่า **audit log** ซึ่งช่วยตอบได้ว่าใครอนุมัติคำขอนี้ ใครเปลี่ยนบัญชีธนาคาร หรือใคร export user data

อย่าอนุญาตให้ user ทั่วไปแก้ไข log นี้ และอย่าใส่ secret keys ลงไป สำหรับการเปลี่ยนแปลงที่สำคัญ ให้บันทึกทั้งค่าเก่าและค่าใหม่ครับ

### A focused security review prompt

**ถาม AI แบบนี้:** Review app นี้เรื่อง login, access rule, input, secret key, private data, file, API call และ action ที่ย้อนกลับไม่ได้ ตรวจว่า user A เปิด data ของ user B ไม่ได้ ตรวจ bad input, rate limit, library เก่า, audit log, backup และ recovery เรียง risk จากสูงไปต่ำ และแสดง plan ก่อนแก้ code

นี่คือการตรวจสอบเบื้องต้น ยังแทนการทดสอบความปลอดภัยเต็มรูปแบบไม่ได้ ถ้าเป็นระบบสาธารณะ, ระบบเงิน, ข้อมูลส่วนตัว หรือระบบธุรกิจที่สำคัญ ควรจ้างผู้เชี่ยวชาญมาตรวจครับ

## Tools You Can Use — Start with the Smallest Fit

เครื่องมือเปลี่ยนไปเร็วครับ แต่ประเภทของงานเปลี่ยนช้ากว่า ให้คุณระบุตัวงานก่อน แล้วค่อยเลือกเครื่องมือครับ

### 1. You need to connect existing tools, not build a new app

ถ้างานคือรับ form แล้วเพิ่ม row ใน CRM, แจ้งเตือนผ่าน [Slack](https://slack.com/) และส่ง email ให้เริ่มจาก workflow automation ครับ [**Zapier**](https://zapier.com/), [**Make**](https://www.make.com/en) และ [**n8n**](https://docs.n8n.io/) ใช้ trigger เริ่มงานแล้วต่อ action เป็นลำดับ รองรับเงื่อนไขและ loop ได้ ส่วน n8n เลือก run บน cloud หรือ server ของคุณเองได้

วิธีนี้ได้ผลดีเมื่อ user ใช้งาน [Google Forms](https://workspace.google.com/products/forms/), [Typeform](https://www.typeform.com/), Slack, email หรือเครื่องมือธุรกิจอยู่แล้ว แต่จะเริ่มยากเมื่อคุณต้องการหน้าจอที่ปรับเองจำนวนมาก หรือมีเงื่อนไขแตกแขนงเกิน 100 ทางครับ

**Good first project:** รับ file CSV รายเดือน, ตรวจสอบแต่ละแถว, บันทึกแถวที่ถูกต้อง, ส่งแถวที่มีปัญหาให้เจ้าของงาน และทำ report สรุปสั้น ๆ ครับ

### 2. Your data already looks like a spreadsheet and people need a better interface

[**Glide**](https://www.glideapps.com/docs) เชื่อมกับ sheets และ databases ได้ คุณเพิ่ม form, access rule และ workflow แล้วแชร์ app ได้เลย คู่มือเริ่มจาก data, UI component, action, user และการ publish ครับ

เครื่องมือนี้เหมาะกับ inventory list, งานภาคสนาม, checklist, CRM แบบง่าย และ approval tool ครับ ก่อนเลือกให้เช็ก row limit, access rule, data sync, export และราคาด้วย

### 3. Your team needs screens for company data

[**Retool**](https://retool.com/), [**Microsoft Power Apps**](https://www.microsoft.com/en-us/power-platform/products/power-apps), และ Glide สามารถสร้าง table, form, แผนภูมิ และกฎการเข้าถึงข้อมูลของบริษัทได้ เหมาะกับงานภายในเมื่อคุณไม่ต้องการ design พิเศษสำหรับบุคคลทั่วไปครับ

เครื่องมือกลุ่มนี้เริ่มได้เร็วเพราะมี UI component เตรียมไว้แล้ว แต่คุณจะผูกกับ platform มากขึ้น และราคาอาจเพิ่มตามจำนวน user ก่อนเลือกให้ถามว่า data อยู่ที่ไหน, ย้อน app ไป version เก่าได้ไหม และรองรับ company login หรือเปล่าครับ

### 4. You want to describe and publish a custom web application

คุณอธิบาย web app ให้ **Lovable**, [**Replit Agent**](https://replit.com/ai) หรือ [**v0**](https://v0.app/) ช่วยสร้างได้ครับ แต่ละตัวดูแลขอบเขตไม่เท่ากัน:

- **Lovable** ดูแลเรื่องหน้าจอ, backend, database, login, บริการอื่น ๆ, online hosting และการ sync code กับ GitHub ([Lovable documentation](https://docs.lovable.dev/introduction/welcome)) ถ้าใช้ร่วมกับ Supabase จะช่วยเรื่อง table, การเปลี่ยนโครงสร้าง table, login, file และการเช็คความปลอดภัยด้วยครับ ([Lovable–Supabase documentation](https://docs.lovable.dev/integrations/supabase))
- **Replit Agent** รวม AI builder, online code editor, database, login, secrets และการ publish ไว้ด้วยกัน นอกจากนี้ยังมี private deployment สำหรับ internal tool และงานทดสอบครับ ([Replit private deployments](https://docs.replit.com/references/publishing/private-deployments))
- **v0** สร้าง UI และ web app จากคำอธิบาย แล้วใช้ Vercel สำหรับ deployment, environment settings และ integration ครับ ([v0 documentation](https://vercel.com/docs/v0))

AI builder ช่วยจัดการหลายขั้นตอนให้ครับ แต่คุณยังต้องรู้ว่า code กับ data อยู่ที่ไหน export ได้ไหม ใครเป็นเจ้าของ production account เครื่องมือรับผิดชอบ security ส่วนไหน และถ้าย้ายออกต้องนำอะไรไปรันต่อเอง

### 5. You want to own the code and choose every part

**Claude Code** และ [**Cursor**](https://cursor.com/) ทำงานกับ file code ทันที สามารถอ่านและแก้ไข file, run tests และทำงานกับ Git ได้ Claude Code ครอบคลุมการอ่าน, แก้ไข, ทดสอบ, แก้ error และงาน Git ([Claude Code documentation](https://docs.anthropic.com/en/docs/claude-code/overview)) ส่วน Cursor Agent สามารถค้นหา code, แก้ไข, run คำสั่ง และแสดงการเปลี่ยนแปลงแต่ละจุดให้คุณ review ได้ครับ ([Cursor Agent documentation](https://docs.cursor.com/chat/overview))

คุณจะควบคุมและย้าย code ได้ง่ายขึ้น แต่ต้องเลือก host, database, login, file storage, deployment และ alert เอง ถ้ายังไม่คุ้นกับส่วนเหล่านี้ ลองให้เพื่อนสาย tech ช่วย review ตัวเลือกครับ

### 6. You need ready backend parts

**Supabase** และ **Firebase** มี backend component พื้นฐาน เช่น database, login, file storage และ server function ส่วน **Neon** เน้น Postgres, **Clerk** กับ **Auth0** เน้น login ขณะที่ **Vercel**, **Netlify**, **Render** และ **Railway** ช่วย deploy กับ run app ครับ

service กลุ่มนี้เป็น backend component ไม่ได้สร้าง app ให้ครบทั้งระบบครับ คุณใช้ AI coding tool เชื่อมต่อเข้าด้วยกันได้ แต่อย่าลืมจด account owner, recovery method, cost alert, data location, backup rule และวิธี export

### 7. Use a known service for high-risk work

ใช้บริการที่น่าเชื่อถือสำหรับงานที่มีความเสี่ยงสูงครับ:

- การชำระเงินและระบบสมาชิก: Stripe, [Paddle](https://www.paddle.com/) หรือผู้ให้บริการที่เหมาะสมกับประเทศและธุรกิจของคุณครับ
- email ของ app: Postmark, Resend, [SendGrid](https://www.twilio.com/en-us/sendgrid) หรือเครื่องมือที่คล้ายกันครับ
- login และยืนยันตัวตน: Google หรือ Microsoft sign-in ผ่าน managed auth service ครับ
- การเก็บ file: S3 หรือ managed file store อื่น ๆ ครับ
- การติดตาม error: Sentry หรือบริการที่คล้ายกันครับ
- การเช็คสถานะออนไลน์: UptimeRobot, [Better Stack](https://betterstack.com/) หรือเครื่องมือที่คล้ายกันครับ

service เหล่านี้ช่วยทำงานพื้นฐานที่สร้างเองแล้วเสี่ยงและต้องดูแลต่อครับ แต่ business rule อย่างจำนวนเงิน, ผู้รับ, permission, retention และวิธีรับมือ error ยังเป็นสิ่งที่คุณต้องกำหนด

### Questions to ask before you choose a tool

อย่าเริ่มด้วยคำถามว่า AI builder ตัวไหนดีที่สุดครับ แต่ให้ถามว่า project นี้ต้องการอะไร:

| Question | Why it matters |
|---|---|
| สร้าง workflow และ access rule ของผมได้ไหม? | UI สวยก็ช่วยไม่ได้ถ้า business rule ผิดครับ |
| code, data, file และ user account อยู่ที่ไหน? | แต่ละอย่างอาจอยู่คนละ service และมี owner คนละ account ครับ |
| ผมสามารถ export แต่ละอย่างได้ไหม? | ถ้ามี code แต่เอาข้อมูลธุรกิจออกมาไม่ได้ คุณก็ยังย้ายระบบไม่ได้ครับ |
| ใครเป็นเจ้าของบัญชี vendor, GitHub, domain และ cloud? | ธุรกิจไม่ควรขึ้นอยู่กับ login ส่วนตัวของคนเพียงคนเดียวครับ |
| มีการเข้าถึงแบบส่วนตัว, บทบาท user และการเช็คที่ backend ไหม? | app ภายในและ B2B ต้องการมากกว่าแค่ลิงก์ลับครับ |
| การแยกสภาพแวดล้อม preview และ production ทำอย่างไร? | คุณต้องการที่ปลอดภัยสำหรับทดสอบครับ |
| มีระบบ backup, logs, การแจ้งเตือน และประวัติการทำงานไหม? | คุณต้องดูแล app หลังเปิดใช้ใน production ครับ |
| ค่าใช้จ่ายเพิ่มขึ้นตามจำนวน user, จำนวนครั้งที่ run, จำนวนแถว, file หรือการใช้ AI? | ช่วงทดสอบที่ราคาถูกอาจมีค่าใช้จ่ายสูงขึ้นหลังจากเติบโตครับ |
| Engineer รับช่วงต่อภายหลังได้ไหม? | code ที่อ่านง่าย, data model, documentation และเครื่องมือมาตรฐานช่วยลดเวลารับช่วงครับ |
| จะเกิดอะไรขึ้นถ้าผู้ให้บริการปิดบัญชีของผม? | คุณควรมีวิธี export และ recovery เตรียมไว้ก่อนเกิดปัญหาครับ |

ราคาและ limit เปลี่ยนบ่อยครับ เช็ก official documentation ก่อนตัดสินใจ แล้วทำ prototype เล็ก ๆ: login 1 ครั้ง, สร้าง private record, upload file, run background job, export data และ restore backup การลองครบ flow นี้บอกได้มากกว่าอ่าน feature list ยาว ๆ

### Three simple ways to start

นี่คือตัวอย่าง ไม่ใช่คำตอบตายตัวครับ:

**A five-person internal approval tool:**

```text
Glide or an internal-tool builder
→ company Google/Microsoft login
→ built-in table or an existing data source
→ built-in workflow automation
```

**A public prototype with custom screens:**

```text
Lovable, Replit Agent, or v0
→ managed Postgres/backend platform
→ managed authentication
→ GitHub export/sync
→ managed deployment + error tracking
```

**A product that an engineer may help with later:**

```text
Coding agent working with normal code files
→ GitHub as source of truth
→ managed Postgres + object storage + auth
→ preview and production deployments
→ automated tests, backups, logs, and alerts
```

คำตอบที่เหมาะอาจเป็นการไม่สร้างเลยก็ได้ครับ ถ้ามี SaaS สำเร็จรูปที่ตอบโจทย์ได้ 90% การตั้งค่าระบบนั้นอาจมีต้นทุนน้อยกว่าการดูแล software ที่เขียนเองหลายปี ให้สร้างเมื่อกฎพิเศษหรือการเชื่อมต่อของคุณสร้างคุณค่าได้มากพอครับ

## When to Bring In a Professional

คุณสร้างเองได้หลายส่วนครับ แต่กรณีด้านล่างควรดึงคนที่มีประสบการณ์มาช่วย review

ปรึกษาวิศวกร, เพื่อนสายเทคนิค, ผู้เชี่ยวชาญด้านความปลอดภัย หรือที่ปรึกษา เมื่อ:

- **ระบบรับ จ่าย หรือโอนเงิน** Stripe รับ payment ได้ แต่ถ้า code ต้องคำนวณ balance, ถือเงิน หรือโอนเงินระหว่างคน ให้ขอ system review ก่อนครับ
- **ระบบเก็บหรือใช้ข้อมูลส่วนตัว** เช่น ข้อมูลสุขภาพ, เลขบัตรประชาชน และข้อมูลการเงิน PDPA อาจมีผลแม้ทีมยังเล็กครับ
- **เปลี่ยนจาก internal เป็น public** บน Internet มีทั้ง user, input แปลก ๆ และการโจมตีมากกว่าเดิม ให้ review security ก่อนเปลี่ยนครับ
- **ทีมทำงานไม่ได้ถ้าระบบนี้หยุด** ให้มืออาชีพตรวจสอบเรื่อง backup, การเข้าถึง, การแจ้งเตือน และการกู้คืนระบบครับ

ถ้า tool โตมาถึงจุดนี้ แปลว่ามีคนเริ่มพึ่งระบบแล้วครับ ช่วงนี้ควรตั้ง budget สำหรับ review และการดูแลระยะยาว

ตอนคุยกับ Engineer คุณบอกได้เลยว่า app run บน Railway, ใช้ Postgres บน Supabase, login ด้วย Google ผ่าน Clerk, backup รายวัน และเคยลอง restore แล้ว 1 ครั้ง ข้อมูลชุดนี้ช่วยให้ Engineer เข้าใจระบบและรับช่วงต่อได้ไวขึ้นครับ

## If You Want to Go Deeper

คุณอาจต้องการแค่ tool ที่ช่วยทีมทำงานได้ดีขึ้น แค่นั้นก็เป็นเป้าหมายที่ดีแล้วครับ คุณไม่ต้องเป็น software engineer ถึงจะใช้แนวคิดในคู่มือนี้ได้

### Learn one missing part at a time

บทเรียนสั้น ๆ 2 บทนี้จะช่วยให้คุณทำงานกับ AI tool และทีม software ได้ง่ายขึ้นครับ:

- **Git and GitHub:** [Introduction to GitHub](https://github.com/skills/introduction-to-github) เป็นคอร์สปฏิบัติจาก GitHub Skills ซึ่งจะสอนเรื่อง repository, branch, commit, pull request และ merge ในเวลาไม่ถึง 1 ชั่วโมงครับ
- **APIs:** [Postman quick start](https://learning.postman.com/docs/getting-started/quick-start) ช่วยให้คุณลองส่ง API request 1 ครั้งและดูคำตอบ สิ่งนี้จะทำให้เข้าใจเรื่อง request, response, endpoint และ status code ได้ง่ายขึ้นครับ

คุณไม่จำเป็นต้องเรียนทุกอย่างพร้อมกันครับ ลองสร้างสิ่งเล็ก ๆ 1 อย่าง เรียนรู้ส่วนที่ขาดในขั้นตอนถัดไป ขอความช่วยเหลือ แล้วทำซ้ำครับ

บางทีเครื่องมือเล็ก ๆ นั้นอาจทำให้คุณอยากเข้าใจ code, ทำงานใกล้ชิดกับทีม software มากขึ้น หรือแม้แต่เปลี่ยนสายงานมาทางเทคนิค ถ้าเป็นแบบนั้นก็ยินดีต้อนรับครับ ความรู้ทางธุรกิจและความใส่ใจในตัว user ของคุณมีประโยชน์มากในโลกนี้

แต่ก็เตรียมเจอ bug, error แปลก ๆ, build พัง และ comma หาย 1 ตัวที่หาเป็นชั่วโมงไว้หน่อยนะครับ ยินดีต้อนรับสู่โลกแห่งความหงุดหงิดครับ หยอก ๆ แต่ก็มีบ้าง 😅
