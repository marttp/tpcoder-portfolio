## Introduction

โพสต์นี้เป็นตัวเปิดของซีรีส์ backend system design วิธีเล่าของผมง่าย ๆ เลย คือหยิบระบบนึงขึ้นมา เริ่มจากของที่ทื่อที่สุดเท่าที่มันจะ work ได้ แล้วค่อย ๆ ใส่ข้อจำกัดจากหน้างานเข้าไปทีละอย่าง จนมันค่อย ๆ โตไปเป็น multi-service architecture แบบที่เรารันกันใน production

ที่เปิดด้วย push notification ก็เพราะตอนอยู่ญี่ปุ่นผมหมดเวลาไปกับเรื่องนี้แหละเยอะที่สุด — push pipeline ตัวนึงที่สุดท้ายมันยิงไปแตะระดับ **120M+ notification ต่อวัน** เพราะงั้นอยากให้มองโพสต์นี้เป็นบันทึกว่าผมเดินมายังไงกว่าจะถึงจุดนั้น แล้วทำไมแต่ละ step ถึงโดนบังคับให้ต้องเกิด มากกว่าจะเป็น textbook ที่กางคำตอบสำเร็จรูปมาให้

ของที่ผมว่าสำคัญที่สุดในแต่ละเวอร์ชันข้างล่างคือ **ปัญหาต้องมาก่อน** คือพวก design ใหญ่ ๆ เยอะ ๆ ที่เขาทำกันมาเนี่ย มันมักจะมีที่มาเสมอแหละ อาจจะติดปัญหาอย่างนึง แก้อย่างนึง แล้วก็ไปเจออีกอย่างนึง พอทำไปเรื่อย ๆ มันก็เลยค่อย ๆ ใหญ่ขึ้น ไม่มีใครหรอกที่อยู่ ๆ มาใส่ microservice กันตั้งแต่เริ่ม เปลืองเงินเปล่า ๆ ถ้าวันนึงเราต้องมานั่ง build ของแบบนี้เอง เราก็คงเดินเส้นทางคล้าย ๆ กันนี่แหละ เพราะงั้นเดี๋ยวผมจะชี้ให้ดูก่อนเลยว่าความเจ็บมันมาจากไหน ก่อนจะค่อยโชว์วิธีแก้

เส้นทางมันยาวอยู่นะ เลยขอกาง map ไว้ให้ก่อน แต่ละ step ถูกบังคับด้วย step ก่อนหน้ามันเองทั้งนั้น:

1. ยิงเข้า provider แบบ **direct call**
2. ทำ **provider abstraction** ให้รองรับ iOS กับ web (ใช้ FCM เป็น gateway เดียว แล้ว config แยกราย platform)
3. ย้ายงาน **ออกจาก request ไปไว้บน queue**
4. **Batching**
5. **pipeline สองสเตจที่ partition แล้ว**
6. **Lazy fan-out** สำหรับ campaign ก้อนใหญ่ ๆ
7. **Dedupe**
8. **Rate limiting, delivery tracking แล้วก็ bounded retry**
9. **Token cleanup**

แล้วก็ปิดท้ายด้วย product layer ที่เป็นตัวตัดสินว่าที่อุตส่าห์ส่งไปเนี่ย มันคุ้มจะส่งมั้ย: **templating, personalization แล้วก็การแยกประเภท notification ตาม type**

## The naive version — call FCM directly

ก่อนจะส่งอะไรได้ ฝั่ง backend ต้อง **มี token อยู่ในมือ** ก่อน แล้ว token พวกนี้ก็ไม่ได้เกิดจาก backend นะ — push token มันถูก generate ขึ้นมา **บนเครื่อง** โดย SDK ของ platform (FCM) เพราะงั้น setup ที่ง่ายที่สุดคือ แอปอ่าน token ของตัวเองจาก SDK แล้วพอ user login เสร็จ ฝั่ง client ก็เอา token ไป register — ยิง `POST /devices` ธรรมดา ๆ พก token, platform แล้วก็ app version ไปด้วย — จากนั้น backend ก็ **upsert** มันเข้ากับ `(user, device)` (user คนนึงมีได้หลายเครื่อง แล้วก็ไม่ต้องกลัวว่าจะซ้ำ)

<MermaidDiagram chart={`
flowchart LR
    SDK["FCM SDK\n(on device)"] -->|"generate token"| APP["App"]
    APP -->|"after login: POST /devices\n(token, platform, app version)"| BE["Backend"]
    BE -->|"upsert by (user, device)"| STORE[("Token store")]
`} />

token มันหมุนเวียนแล้วก็หมดอายุได้เหมือนกัน แต่อันนั้นเดี๋ยวค่อยว่ากันทีหลัง — ตอนนี้แค่ register ได้ ก็พอมีของให้ส่งแล้ว

พอมี token อยู่ในมือแล้ว จุดเริ่มแบบซื่อ ๆ ของฝั่ง **send** ก็คือ พอมี event อะไรเกิดขึ้น ก็ไป lookup device token ของ user คนนั้น แล้วเรียก push provider เข้าไปใน request นั้นเลย

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller\n(API / internal event)"] --> SVC["Notification Service"]
    SVC -->|"for each device token"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

เขียนเป็นโค้ดมันก็คือ loop ง่าย ๆ นี่แหละ:

```text
on event:
  tokens = lookup_device_tokens(user_id)
  for token in tokens:
      fcm.send(token, payload)   # synchronous HTTP call
  return 200
```

แล้วก็บอกเลยนะว่า ถ้าเป็น side project หรือ internal tool แบบนี้ก็พอแล้ว มัน work ไฟล์เดียวจบ เขียนบ่ายเดียวก็ขึ้นได้ ยังไม่ต้องไป over-engineer ก่อนถึงเวลา ซอฟต์แวร์หลาย ๆ ตัวบนโลกนี้ก็อยู่แค่กล่องนี้ไปตลอดเลยด้วยซ้ำ — ถ้า volume ไม่ได้เยอะ แล้วการ block ไปสักไม่กี่ร้อย ms ไม่ได้ทำให้ใครเดือดร้อน จุดนี้ก็เป็นที่หยุดที่โอเคมาก ๆ

แต่แบบนี้มันแอบ assume ว่า user ทุกคนใช้ Android ซึ่งมันไม่ใช่ — แล้วก็ยังมีอีกจุดนึงที่คาดได้เลยว่าจะเป็นปัญหาในอนาคต: **request ของคนเรียกมันถูก block ค้างอยู่กับ push provider** เดี๋ยวเรื่อง block ค่อยแก้ทีหลัง ขอเก็บจุดที่พื้นฐานกว่าก่อน

## Not every device is Android — multi-platform from the start

**ปัญหา:** ตอนนี้เราเรียก `fcm.send()` ด้วย payload ที่ hardcode ไว้ก้อนเดียว เล็งไปที่ platform เดียว พอมี user แบบเต็มสนามเข้ามา เราก็มีทั้ง iOS แล้วก็ web ด้วย — แล้วถึง FCM มันจะยิงได้ครบทั้งสาม platform แต่การ `send หาทุกคน` มันไม่ได้เกิดเองอัตโนมัตินะ ฝั่ง iOS จะส่งได้ต่อเมื่อเรา upload **APNs auth key** ของ Apple เข้า Firebase ก่อน, ฝั่ง web ก็ต้องมี **VAPID** key, แล้วแต่ละ platform ก็อยากได้ payload คนละทรง (iOS มี sound กับ badge, Android มี notification กับ data, web มี actions) ถ้า hardcode ทางเดียวไว้ วันที่ platform ที่สองมาถึงเราก็ได้นั่งเขียนใหม่ แล้วเรื่องนี้ไม่ใช่ของ advanced อะไรเลยนะ — แอปส่วนใหญ่หน้างานต้องการทั้ง iOS แล้วก็ Android ตั้งแต่วันแรกอยู่แล้ว คำถามมีแค่ว่าเราจะ build รับมันมาแบบสะอาด ๆ รึเปล่าเท่านั้นเอง

**วิธีแก้:** ใช้ **FCM เป็น gateway เดียว** ไปเลย — มันยิงเข้า Android ได้ทันที, ยิงเข้า iOS ด้วยการ relay ต่อไปที่ APNs (ด้วย key ที่เรา upload ไว้), แล้วก็ยิง web ผ่าน Web Push — จากนั้นวาง **Push Handler** บาง ๆ ไว้ข้างหน้า หน้าที่เดียวของมันคือประกอบ config ของแต่ละ platform ให้ถูกจาก notification model ภายในของเรา เราแค่ส่ง notification ที่เป็นกลาง ๆ ไม่ผูก platform ให้มัน แล้วมันก็ไปเติม block `android` / `apns` / `webpush` ของ FCM ให้เอง แล้วค่อยยิง มันคือ Strategy pattern แหละ แต่ strategy ที่ว่าคือการ **ขึ้นรูป payload ตาม platform** ไม่ใช่การเปลี่ยน provider ตาม platform

<MermaidDiagram chart={`
flowchart LR
    SVC["Notification Service"] --> PH["Push Handler\n(build per-platform config)"]
    PH --> FCM["FCM"]
    FCM -->|"Android (direct)"| AND["Android"]
    FCM -->|"iOS (relayed to APNs)"| IOS["iOS"]
    FCM -->|"Web (Web Push)"| WEB["Web"]
`} />

ขอยกตัวอย่างให้เห็นภาพ — FCM message ก้อนนึงมันพก `notification` กลาง ๆ มาก้อนนึง แล้วก็มี block override ของแต่ละ platform ต่อท้าย เราเติมเฉพาะที่แต่ละ platform ต้องการก็พอ:

```json
{
  "message": {
    "token": "<device-token>",
    "notification": { "title": "Payment complete", "body": "245 THB at Store XXXXX" },
    "android": {
      "priority": "high",
      "notification": { "channel_id": "transactions" }
    },
    "apns": {
      "headers": { "apns-priority": "10", "apns-collapse-id": "txn-9813789127398173921" },
      "payload": { "aps": { "sound": "default", "badge": 1 } }
    },
    "webpush": {
      "headers": { "Urgency": "high" }
    }
  }
}
```

สิ่งที่เราได้จากจุดนี้มีสองอย่าง:

- **integrate ที่เดียว ไม่ใช่สามที่** เรา config แต่ละ platform ครั้งเดียว (APNs key, VAPID) ไม่ต้องไปสร้างแล้วดูแล client ของ APNs กับ Web Push แยกกันเอง
- **ความจุกจิกของแต่ละ platform อยู่ที่เดียว** Push Handler เป็นเจ้าของ logic แบบ `iOS ต้องการอันนี้ web ต้องการอันนั้น` แล้วก็แปลง error code ของ provider — invalid token, rate limited, retryable 5xx — ให้เป็น code ภายในที่ฝั่ง sender ของเราเข้าใจ ซึ่งเรื่องนี้สำคัญมากตอนที่เราเริ่มใส่ retry

มันมีอีกทางนึงนะ: แยก worker (หรือ client แบบ direct) ต่อ provider ไปเลย — คุยกับ APNs กับ Web Push เองโดยไม่ผ่าน FCM เราจะหยิบทางนี้มาใช้ตอนที่อยากเป็นอิสระจาก Google หรืออยากได้ control ที่ละเอียดกว่าแล้วก็ latency ที่ต่ำกว่าตอน volume สูงมาก ๆ แต่มันก็ของที่ต้อง build แล้วก็ดูแลเพิ่ม เพราะงั้น default ของเราคือใช้ FCM ทำทุกอย่างไปก่อน จนกว่าจะมีอะไรเป็นรูปธรรมมาบังคับให้ต้องแยก

ไม่ว่าทางไหน เราก็ต้องรู้ platform กับ token ปัจจุบันของแต่ละเครื่องอยู่ดี เพราะงั้นจุดนี้แหละที่ **device information** (token, platform, app version, last-seen) กลายเป็น lookup ตัวหลักที่ขาดไม่ได้

ทีนี้เราก็ยิงครบทุก platform แล้ว — แต่เรื่อง block ที่เกริ่นไว้ตั้งแต่แรกเรายังไม่ได้แตะเลยนะ: ทุกการส่งมันยังเกิดแบบ **inline อยู่ใน request ของคนเรียก**

## Stop blocking the caller — async worker + queue

**ปัญหา:** FCM มันคือ network call ไปหาระบบของคนอื่น พอมันช้า API ของ**เรา**ก็ช้าตาม พอมัน timeout คนที่เรียก**เรา**ก็กิน timeout นั้นไปเต็ม ๆ แล้วพอมี notification spike (เช่น event ใหญ่ ๆ ที่ fan out ไปหา user เยอะ ๆ) มันก็กลายเป็นกอง synchronous HTTP call ที่จับ request thread ของเราไว้เป็นตัวประกัน แถมเวลา send fail เราก็ retry ไม่ได้เลยถ้าไม่ทำให้คนเรียกต้องรอ retry ไปด้วย

**วิธีแก้:** ทำให้ API แทบไม่ทำอะไรเลย — validate request, โยนลง queue, แล้ว return `202 Accepted` กลับไปทันที จากนั้นมี background worker แยกอีกตัวคอยดึงงานจาก queue มารัน Push Handler

<MermaidDiagram chart={`
flowchart LR
    CALLER["Caller"] --> SVC["Notification Service"]
    SVC -->|"enqueue, return 202"| Q["Message Broker (Queue)"]
    Q --> WK["Delivery Worker"]
    WK --> PH["Push Handler"]
    PH -->|"send"| FCM["FCM"]
    FCM --> DEV["Device"]
`} />

ของที่เราได้มาแบบแทบฟรีมีสองอย่าง:

- **คนเรียกหลุดออกจาก provider แล้ว** API ตอบกลับในระดับ millisecond ส่วน latency ของ provider กลายเป็นปัญหาของ worker ไม่ใช่ของคนเรียกอีกต่อไป
- **worker scale แยกของมันเองได้** พอมี spike มันไปกอง queue แทนที่จะทำ API ละลาย — แล้วเราก็แค่เพิ่ม worker มาช่วยระบาย

อันนี้เป็นการเปลี่ยนที่ leverage สูงที่สุดในโพสต์นี้เลย เกือบทุกอย่างหลังจากนี้คือการขัดเกลาไอเดีย `ทำงานแบบ async โดยอิงกับ durable queue` ให้เนียนขึ้นเฉย ๆ

แล้วสำหรับแอปส่วนใหญ่ จุดนี้คือเส้นชัยแล้วนะ ของ naive บวก config แบบ multi-platform บวก queue กับ worker ก็ครอบคลุม product ส่วนใหญ่บนโลกนี้ได้แล้ว — ที่เหลือข้างล่างคือกลุ่มเคสที่เล็กกว่า ที่ volume มันมหาศาลแบบสุด ๆ หรือต้องการ reliability เข้มข้นจนบังคับให้ต้องมี structure เพิ่ม ถ้านั่นไม่ใช่เรา ก็หยุดจุดนี้ได้เลยแบบสบายใจ

แต่การส่งทีละ token — ต่อให้ย้ายมาไว้บน worker แล้วก็เถอะ — มันพังทันทีที่ notification ก้อนเดียวต้องไปให้ถึงเครื่องเป็นจำนวนเยอะ ๆ พร้อมกัน

## One HTTP call per token doesn't scale — batching

**ปัญหา:** พอมีงานแบบ `feature ใหม่ออกแล้ว บอกทุกคนเลย` มันก็ fan out ไปเป็น token หลักล้าน ถ้ายิงทีละ HTTP request ต่อ token เราก็จ่ายค่า round-trip overhead เป็นล้านรอบ, เปลือง connection ของ provider, แล้วก็ชน rate limit เร็วมาก ตัวการ send เองนี่แหละกลายเป็น bottleneck

**วิธีแก้:** เลิกส่งทีละอันแบบทื่อ ๆ ซะ — แต่ต้องเข้าใจก่อนว่า `batching` มันช่วยอะไรเรากันแน่ ตัว batch helper ของ FCM (`sendEachForMulticast`, สูงสุด 500 token) มันก็สะดวกอยู่ แต่ลึก ๆ ข้างใต้มันไม่ได้ยุบ 500 token ให้เหลือ network call เดียวนะ — มันยังยิงทีละ token เป็น request ของใครของมันอยู่ดี แค่ทำแบบ concurrent บน HTTP client ตัวเดียวกันเฉย ๆ เพราะงั้นมันก็มีดีไซน์อยู่สองแบบที่ค่าเท่ากัน:

- **ใช้ batch helper ของ SDK** — มันจัดการ fan-out ทีละ token กับเรื่อง concurrency ให้เราเอง
- **วน loop ส่งเอง** — ผลลัพธ์เหมือนกัน แต่เราคุม control ได้มากกว่าว่าจะ group token ยังไง แล้วจะยิงทีละกี่ตัว

ใช้ได้ทั้งคู่แหละ แต่ตัวที่ตัดสินว่ามันจะ **stable** รึเปล่าคือ **connection pool ของ HTTP client** ต่างหาก: reuse connection (keep-alive), จำกัด max connection, ตั้ง timeout ให้สมเหตุสมผล ไม่ว่าจะเลือกทางไหน pool ที่ tune มาดีคือสิ่งที่กันเราไม่ให้ไปนั่ง thrash TCP/TLS handshake แล้วก็ socket หมดตอนโหลดหนัก ๆ — ถ้าพลาดจุดนี้ไป ไม่ว่าจะ SDK หรือ loop ที่เขียนเองก็ช่วยไม่ได้ (ถ้าไปทาง direct-to-APNs หลักการเดียวกันนี้จะออกมาในรูปของ HTTP/2 connection ตัวเดียวที่อยู่ยาว ๆ แล้วยิงหลาย stream แบบ multiplex บนนั้น)

เพราะงั้นเราเลยใส่ **Notification Batcher** เข้ามา: มันจัดกลุ่ม token — ตาม platform แล้วก็ตามอะไรก็ตามที่ควรจับกลุ่ม — แล้วส่ง batch ไปให้ Push Handler ซึ่งจะยิงออกไปแบบ concurrent บน client ที่ pool ไว้

    NB -->|"batch of ≤500 tokens"| PH["Push Handler"]
    PH -->|"per-token sends,\npooled HTTP client"| FCM["FCM"]

ส่วนที่ไม่ค่อยมีใครนึกถึงคือ **partial failure** การ multicast 500 ตัวมันไม่ได้สำเร็จหรือ fail เป็นก้อนเดียว — มันตอบกลับมาเป็น list ผลลัพธ์รายตัว: บางตัว delivered, บางตัว failed, บางตัวเป็น invalid token ที่เราควรลบทิ้ง เพราะงั้น batcher จะมาเช็คแค่ status 200 เฉย ๆ ไม่ได้ มันต้องกาง response ออกมาแล้วตัดสินว่าตัวไหน retry, ตัวไหนทิ้ง, ตัวไหน prune ออก จำจุดนี้ไว้ดี ๆ — เพราะมันคือเหตุผลว่าทำไม delivery tracking ทีหลังถึงไม่ใช่ของ optional

มีกลไกสองอย่างที่ทำให้ batching มันอยู่ในร่องในรอย:

- **ซอยเป็น chunk แล้วคุม concurrency** ซอย token list เป็น chunk ละ 500 ตัวสำหรับ FCM (หรือ N stream แบบ concurrent ถ้าไปทาง direct-to-APNs) — แต่อย่ายิงทุก chunk พร้อมกันทีเดียว ไม่งั้นเราก็แค่ย้ายปัญหา rate-limit จาก `ราย token` ไปเป็น `ราย chunk` เฉย ๆ ใช้ worker pool ที่มีลิมิตคอยระบาย chunk ในจังหวะที่ provider รับไหว
- **map ผลลัพธ์รายตัวให้เป็น action ทุกตัว** `Unavailable` / 5xx → retry; `TooManyRequests` → ถอย back off ทั้ง chunk; `NotRegistered` / `InvalidRegistration` → prune token ตัวนั้นทิ้ง การ map รายตัวแบบนี้แหละคือ signal ที่ไปป้อน token cleanup ทีหลัง — response ของ batch คือที่แรกที่ token ตาย ๆ จะโผล่ตัวออกมาให้เห็น

Batching ทำให้การ send ก้อนใหญ่ก้อนเดียวมันมีประสิทธิภาพ กำแพงถัดไปคือตัว pipeline เองแล้ว: worker ตัวเดียวที่ทำทั้ง lookup, เช็ค preference, ทำ template แล้วก็ส่ง มัน scale แต่ละสเตจแยกกันไม่ได้ — แถมเราไม่มีทั้ง ordering แล้วก็ replay

## Split the work — a two-stage Kafka pipeline

**ปัญหา:** ตอนนี้ worker ตัวเดียวของเราทำงานเยอะมาก — โหลด user info, โหลด device info, validate, เช็ค preference, render template, batch, แล้วก็ส่ง งานพวกนี้ต้นทุนต่างกันลิบเลย DB lookup ที่ช้า ๆ ตัวเดียวก็ block worker ที่ควรจะได้ส่งงานอยู่ เราแยก scale `enrichment` ออกจาก `delivery` ไม่ได้ แล้ว queue ธรรมดาก็ไม่ให้ทั้ง ordering แล้วก็ replay — ถ้า worker crash กลาง batch เราก็ได้แต่เดาว่าเกิดอะไรขึ้น

**วิธีแก้:** ผ่ามันเป็น **สองสเตจ** โดยมี durable log (Kafka) มาคั่นระหว่างสองสเตจ แล้วก็ **partition ด้วย user ID**

- **Pre-Delivery Worker** (สเตจ enrichment): user information, **device/token enrichment** (แปลง user ให้กลายเป็น device token + platform ของเขา), validator, notification batcher, user-preference checker, แล้วก็ template parser มันเปลี่ยน event ดิบ ๆ แบบ `notify user X` ให้กลายเป็น payload ที่ resolve ครบพร้อมส่ง แล้วค่อย publish ไป topic ถัดไป
- **Delivery Worker** (สเตจ send): consume payload ที่พร้อมแล้ว แล้วไม่ทำอะไรเลยนอกจากส่งต่อให้ FCM ผ่าน Push Handler

    SVC --> Q1["Kafka topic 1\n(partition by user_id)"]
    subgraph PRE["Pre-Delivery Worker"]
        ENR["user info\ndevice/token enrichment\n(user → device tokens)\nvalidator\npreference checker\ntemplate parser\nbatcher"]
    end
    Q1 --> PRE

ทำไมต้อง Kafka แล้วทำไมต้อง partition ด้วย `user_id`:

- **scale แยกกันได้** enrichment ติดที่ DB ส่วน delivery ติดที่ network — ทีนี้แต่ละฝั่ง scale บน consumer group ของตัวเองได้เลย
- **ordering ในจุดที่มันสำคัญ** event ทั้งหมดของ user คนนึงจะตกลงไปที่ partition เดียวกัน เลยถูกประมวลผลตามลำดับ (เราคงไม่อยากให้ `order ของคุณจัดส่งแล้ว` มันแซง `order ของคุณกำลังแพ็คอยู่` หรอก)
- **replay แล้วก็ durability** consumer ที่ crash ไปก็กลับมาทำต่อจาก offset เดิมได้ ถ้ามี bug ปลายน้ำทำข้อมูลพัง เราก็ reset offset แล้ว reprocess ใหม่ได้ — queue ธรรมดาทำแบบนี้ไม่ได้

มีชิ้นนึงในสเตจ enrichment ที่ควรดึงออกมาพูดหน่อย: ตัว **device/token enricher** คนเรียกบอกว่า `notify user X` — แต่เราไม่เคยส่ง push หา **user** เราส่งหา **device** ต่างหาก step นี้แหละคือตัวที่แปลง `user_id` ให้กลายเป็นเป้าหมายที่ยิงได้: ทุก `(device_token, platform, locale)` ที่ user คนนั้นยัง active อยู่ มันแอบทำงานอยู่สามอย่างเงียบ ๆ:

![fan-out สองแบบ: campaign กระจายไปหา user หลายคน ส่วน user คนนึงกระจายไปหา device ของตัวเอง](/assets/images/blogs/design-push-notification-system/two-fan-outs.png)

- **มันคือ fan-out อันที่สอง** campaign จะกระจาย message นึงออกไปหา **user** เป็นล้าน ส่วนอันนี้กระจาย user คนนึงออกไปหา **N device** ของเขา `notification หนึ่งอัน` กลายเป็น `การส่ง N ครั้งไปที่ provider` ก็จุดนี้แหละ — เป็น fan-out อีกชั้นที่ซ้อนทับอยู่บน fan-out ของ campaign อีกที
- **มันแปะ platform ให้** ซึ่งเป็นสิ่งเดียวที่ Push Handler ปลายน้ำเอาไปใช้ตัดสินใจได้ — ไม่มี platform tag ก็ประกอบ payload ของ iOS / Android / web ให้ถูกไม่ได้
- **มันคัด device ที่ตายแล้วทิ้งตั้งแต่ตอน read** ข้าม token ที่ `last_seen` เก่าเก็บไปก่อนที่จะเปลือง send ฟรี ๆ อันนี้เป็นคู่หูฝั่ง read ของ token cleanup ที่อยู่ถัดลงไปข้างล่าง

เพราะงั้นกล่อง `device info` ที่ดูเรียบ ๆ นั่น แท้ ๆ แล้วมันคือ token enricher: เส้นแบ่งที่ event ซึ่งมองเป็น user เปลี่ยนร่างเป็นการส่งที่มองเป็น device

นี่แหละคือรูปร่างของระบบตัวเต็ม — แล้วมันก็เอาอยู่ ไปจนถึงวันที่ campaign ก้อนเดียวพยายามยิงให้ถึง user เกือบทั้ง base ในช็อตเดียว

## One campaign, 60M recipients — fan-out without flooding the queue

**ปัญหา:** ฝั่ง marketing อยากยิง campaign ไปหา active base ทั้งหมด — ตอน peak คือ **60–70M คนในช็อตเดียว** อันนี้คือ fan-out ระดับ **campaign** — message เดียวไปหา user เป็นล้าน เป็นพี่ใหญ่ของ device fan-out รายคนจาก pipeline เมื่อกี้ ท่าแบบ naive คือกาง 60–70M นั่นออกเป็น notification event รายตัว แล้วเทลง broker ทีเดียวหมด ซึ่งมันทำให้ queue ท่วม, อย่างอื่นอดตายหมด (MFA code ของเราตอนนี้ติดอยู่หลัง promo 60M ตัว) แล้วถ้ามีใคร cancel campaign กลางคัน เราก็มี message ค้างใน flight อยู่หลายสิบล้านที่เรียกคืนกลับมาง่าย ๆ ไม่ได้

**วิธีแก้:** อย่าเพิ่งกาง audience ทั้งก้อนออกมาตั้งแต่แรก เก็บเป็น **campaign definition + audience query** ไว้ แล้วให้ pipeline **ค่อย ๆ กางมันแบบ lazy** — ไล่ page ผู้รับทีละ chunk: enqueue chunk นึง, ปล่อยให้มันระบายจนหมด, แล้วค่อยดึง chunk ถัดไป มี fan-out step เฉพาะของมัน (producer ที่นั่งอยู่หน้าสเตจ pre-delivery) คอยเดิน audience ทีละ page แทนที่จะระเบิดมันลง broker ในช็อตเดียว

![lazy audience fan-out: กาง campaign เป็น chunk ทีละก้อนแทนที่จะเทหมดทีเดียว](/assets/images/blogs/design-push-notification-system/fan-out.png)

สิ่งที่เราได้:

- **queue depth มีขอบเขต** broker ถือแค่ working set — ไม่กี่ chunk — ไม่ใช่ทั้ง campaign traffic อื่น ๆ ก็ไหลต่อได้เรื่อย ๆ
- **ได้ backpressure มาฟรี ๆ** consumer ช้าก็แค่แปลว่าเรา page ช้าลง ตัว cursor ของ audience รอแทนที่จะปล่อยให้ broker ล้น
- **ปุ่ม pause/cancel ที่กดแล้วเอาอยู่** สมมติมีคนเจอ typo ใน copy หรือ marketing ตัดสินใจกลางทางว่าจะดึง campaign กลับ — แล้วเพราะ audience ส่วนใหญ่ยังไม่ได้ถูก enqueue การหยุดมันก็แค่ `หยุดเลื่อน cursor` ไม่ใช่ `ไล่ตามล่า message 60M ตัวที่อยู่บน broker ไปแล้ว` อันไหนที่ยิงออกไปแล้วก็เลยตามเลย แต่เราจำกัดรัศมีความเสียหายได้ แทนที่จะปล่อยให้ send ที่รู้อยู่ว่าผิดมันวิ่งจนจบ

แล้วจุดนี้แหละที่เรื่อง **priority** เริ่มสำคัญ — marketing blast 60M ตัว กับ MFA code อันเดียว ไม่ควรถูกต่อคิวแบบเท่ากัน — แต่เดี๋ยวเราจะวกกลับมาเรื่องนี้ทีหลัง ก่อนอื่น: ที่ volume ระดับนี้ ทุก hop มันเป็น at-least-once หมด แล้ว push ตัวเดียวกันก็ยิงซ้ำได้สองรอบ ตอน volume น้อย ๆ เราก็แค่ยักไหล่ แต่จุดนี้ user สังเกตเห็นแน่นอน

## Stop the double-buzz — a dedupe processor

**ปัญหา:** ทุก hop ที่เชื่อถือได้ใน pipeline เส้นนี้มันเป็น at-least-once กันหมดเลยนะ Kafka เองพอ consumer restart ทีมันก็ redeliver ซ้ำ ส่วน delivery worker ที่ดันไป crash เอา**ตอนที่**เรียก provider ไปแล้ว แต่ยัง**ไม่ทัน** commit offset เนี่ย พอมันฟื้นขึ้นมาก็ reprocess แล้วยิงซ้ำอีกดอก ยังไม่นับ retry (ที่จะมากันเวอร์ชันหน้า) ที่ยิ่งเพิ่มโอกาสซ้ำเข้าไปอีก สรุปคือเราได้สิ่งที่ user เกลียดที่สุดของ push มาเต็ม ๆ — notification อันเดิม เด้งซ้ำสองที

![ปัญหา double-buzz: push notification ที่เด้งซ้ำ](/assets/images/blogs/design-push-notification-system/double-buzz.png)

จะเอา exactly-once แบบเป๊ะ ๆ ข้ามเน็ตเวิร์กเนี่ย บอกเลยว่าฝันไปเถอะ (มันคือ Two Generals Problem — ตัวเดียวกับที่เขียนไว้ใน[โพสต์ payment backend](/blog/payment-backend-stripe-integration-th/)) ทางออกที่เวิร์คในทางปฏิบัติก็เลยเหมือนเดิม: **at-least-once ตอนส่ง + idempotent ตอนรับ**

**วิธีแก้:** ก็จับทุก notification ไปผูกกับ **idempotency key** นิ่ง ๆ ตัวนึง (เช่น `notificationId:deviceToken`) แล้วก่อนจะยิงเข้า provider ก็แวะเช็คกับ store เร็ว ๆ อย่าง Redis ก่อนสักนิด รอบแรกที่วิ่งผ่านมา ก็ claim key นั้นไว้พร้อมตั้ง TTL แล้วก็ยิงได้เลย แต่ถ้า key มันมีอยู่แล้ว — อ้าว ซ้ำนี่หว่า — ก็ข้ามมันไป

แล้วตำแหน่งที่เราวางตัวเช็คเนี่ยสำคัญนะ: มันต้องอยู่ **ที่ Push Handler** ไม่ใช่ที่ Delivery Worker เพราะ key มันเป็นราย device token แล้วฝั่ง batching ก็เกิดไปก่อนหน้านี้แล้วที่ต้นน้ำ — เพราะงั้นกว่า payload จะไหลมาถึงสเตจ delivery มันอาจจะมาเป็น batch ของหลาย token เราเลยอยากไป claim ราย token ตอนที่ batch ถูกแกะออกมาแล้ว ที่ประตูด่านสุดท้ายก่อนยิงออกนอกบ้านพอดี ซึ่งเป็นจุดที่ดักของซ้ำได้จากทุกทางเลย (ทั้ง Kafka redelivery, retry loop, แล้วก็ตอนกาง batch) ในระดับความละเอียดที่กำลังพอดี

<MermaidDiagram chart={`
flowchart LR
    Q2["Kafka topic 2"] --> DEL["Delivery Worker"]
    DEL --> PH["Push Handler"]
    PH <-->|"SETNX notif:device (TTL)"| REDIS["Redis (dedupe)"]
    PH -->|"first send only"| FCM["FCM"]
    FCM --> DEV["Devices\n(Android / iOS / Web)"]
`} />

เขียนเป็นโค้ดแล้ว การ claim มันก็แค่ `SET` แบบมีเงื่อนไขเองนี่แหละ:

```text
# key มาจาก identity ของ event ไม่ใช่ text ของ message
key = "notif:" + sha256(userId + ":" + type + ":" + eventId + ":" + deviceToken)

# claim ก่อนส่ง: SET เฉพาะตอนที่ยังไม่มี key พร้อมตั้ง TTL
if redis.set(key, "1", nx=True, ex=DEDUPE_TTL):   # true แค่รอบแรกรอบเดียว
    push_handler.send(deviceToken, payload)
else:
    skip()                                          # มีคน claim ไปแล้ว -> ของซ้ำ
```

ของที่สำคัญตอนเอาไปใช้งานก็มีอยู่ไม่กี่อย่าง:

- **claim ก่อนส่ง อย่าไป claim หลังส่ง** ใช้การ claim แบบ atomic สไตล์ `SETNX` ไว้ เพื่อให้ worker สองตัวที่บังเอิญแย่ง key เดียวกันอยู่ ไม่มีทางชนะพร้อมกันได้ทั้งคู่
- **ตั้ง TTL ให้พอดีกับ retry window** key มันแค่ต้องอยู่ให้นานกว่าช่วงที่ของซ้ำมีโอกาสโผล่มา (หลักชั่วโมงก็พอ ไม่ต้องอยู่ยันชาติหน้า) ไม่งั้น Redis จะกลายเป็นตารางที่ใหญ่ที่สุดแล้วก็แพงที่สุดของเราซะเอง
- **มันเป็นแค่ guard แบบ best-effort ไม่ใช่ ledger** dedupe มีหน้าที่ปัดของซ้ำที่หลุดมาจาก retry กับ redelivery ออกไปเฉย ๆ มันไม่ใช่ระบบบัญชีนะ (ถ้าเป็นเรื่องเงิน เราต้องไป reconcile กันอีกที — แต่กับ push ที่บังเอิญซ้ำมาทีนึง มันไม่คุ้มหรอกที่จะไปลงทุนทำ distributed transaction)

แล้วก็มีอีกสองเรื่องที่ถ้าข้ามไปละก็ ได้โดนกัดทีหลังแน่ ๆ:

- **key คือ identity ของ event ไม่ใช่ตัว content** ปั้น key ขึ้นมาจากสิ่งที่ทำให้ notification นั้น **unique ในฐานะ event** สิ — ประมาณ `hash(userId + type + eventId + deviceToken)` (ก้อนนี้แหละที่กลายเป็น `notificationId` ของเรา) — อย่าไปปั้นจาก text ของ message เด็ดขาด เรื่องนี้กัดเจ็บตอนใช้งานเลยนะ: user คนนึงจ่ายเงินสอง payment ที่ **ยอดเท่ากันเป๊ะ** ได้สบาย ๆ แล้วสองอันนั้นมันคนละ event กัน ควรได้ push ทั้งคู่ ถ้า key เราดันไปอิงกับ content อันที่สองมันจะหน้าตาเหมือนกันเป๊ะ แล้วโดนกลืนหายไปในฐานะของซ้ำปลอม ๆ — เพราะงั้น key ต้องมี id ของ event ตัวเองอยู่ด้วย (`eventId` / paymentId) แล้วก็ทำให้มัน deterministic ด้วยนะ: คำนวณตั้งแต่ตอน ingest, พกติดตัวไปทุก retry, ห้าม regen ใหม่ราย attempt เด็ดขาด แล้วก็ scope ให้เป็นราย device ด้วย จะได้ไม่เผลอยุบ notification อันเดียวกันที่ส่งไปมือถือ **กับ** แท็บเล็ต เหลือเด้งเดียว
- **claim-ก่อน-ส่ง มันมีกับดักซ่อนอยู่** ลองคิดดูดี ๆ ถ้าเรา claim key ไปแล้วดันมา send fail ทีหลัง — หรือ worker ตายซะดื้อ ๆ หลัง claim ไปแล้ว — key ตัวนั้นมันจะไป block retry ที่ควรจะได้ส่งซะงั้น สรุปคือของซ้ำที่เรากันไว้ได้ กลายเป็น notification ที่เรา **ทำหาย** ไปแทน วิธีแก้คือ ให้ key มันชนะถาวรเฉพาะตอนที่ยืนยันแล้วว่า success เท่านั้น ส่วนเคสที่รู้ผลแล้วว่า fail ก็ปล่อย key คืนไป ให้เส้น bounded-retry มันมา re-claim ใหม่ได้ ส่วน TTL ก็เป็นตัวกันท้ายไว้ — ตั้งให้นานพอที่ redelivery มาช้า ๆ จะ re-send ของที่สำเร็จไปแล้วไม่ได้ แต่ก็สั้นพอที่ Redis จะไม่กลายเป็นป่าช้า

อีกทริคนึงที่ใช้ได้ดี: ถ้าเป็น notification ประเภทที่ข้อมูลข้างในมันเปลี่ยนบ่อย ๆ อยู่แล้ว (พวก transaction เนี่ยตัวดีเลย) เราแอบยัด `transactionId` ลงไปใน content ได้เลย พอ content มันมี id ที่ unique ติดไปด้วย โอกาสที่สอง notification จะออกมาหน้าตาชนกันก็แทบเหลือศูนย์ — กลายเป็นว่าตัว content เองก็ช่วยการันตี uniqueness ให้เราอีกแรงนึง

ทีนี้การ send ของเราก็ idempotent แล้วล่ะ คำถามต่อไปเป็นคำถามที่ระบบระดับ 120M ต่อวันมันบังคับให้เราต้องตอบ: ทำยังไงถึงจะไม่โดน throttle จนน่วม แล้วก็เราจะไปรู้ได้ยังไงว่าตกลงอะไรส่งถึงไปแล้วบ้าง?

## Surviving 120M+ a day — rate limits, tracking, and bounded retry

**ปัญหา:** พอมาถึง volume ระดับนี้ มีอยู่สามอย่างที่เลิกเป็นของ optional ได้เลย

1. **provider มันจะ throttle เรา** FCM เองก็มี send limit ของมัน (แล้ว capacity ของ APNs ที่มัน relay ผ่านก็ไม่ได้มีไม่จำกัดนะ) ยิงทะลุเพดานเมื่อไหร่ก็เจอ 429 แล้วก็โดน drop message ทิ้ง — เพราะงั้นเราต้องรู้จักคุมจังหวะตัวเอง ไม่ใช่สาด fan out ให้เร็วที่สุดเท่าที่ worker จะไหว
2. **เรามองไม่เห็นว่าเกิดอะไรขึ้นเลย** พอส่งกันทีหลักล้าน คำถามว่า `มันส่งสำเร็จมั้ย` เนี่ย มันเป็น log line ที่นั่ง grep เอาไม่ไหวแล้ว เราอยากได้ delivery result ออกมาเป็น data — delivered / failed / invalid-token — ไว้ debug, ไว้ทำ report, แล้วก็ไว้ prune token ที่ตายไปแล้ว
3. **retry ได้ แต่ต้องจำกัดจำนวนครั้ง** เจอ 5xx แว้บ ๆ หรือ network สะดุดแป๊บนึง ก็ควร retry แหละ แต่ retry ต้องมีเพดานจำนวนครั้งกำกับไว้ด้วย ไม่งั้นถ้ามันวน retry ไปเรื่อย ๆ ไม่มีจุดจบ นั่นแหละคือสูตรสร้าง infinite loop แบบไม่ได้ตั้งใจ ที่เอาแต่กระหน่ำ provider แล้วก็ไม่มีวันระบายหมดสักที

**วิธีแก้** — ก็คือครึ่งขวาของ architecture ตัวเต็มนั่นแหละ:

- **rate limiting + sharding** ให้ worker เคารพ rate budget ราย provider (token bucket) แล้วก็ shard pipeline ตาม partition เราจะได้ scale แนวนอนไปได้โดยไม่มี consumer ตัวไหนกลายเป็น hot spot
- **delivery tracking** ทุกการส่งจะโยน result (success/fail + เหตุผล) ลง result queue ตัว **Delivery Track Aggregator** ก็มา consume แล้วเขียนลง **columnar DB** ไว้ทำ analytics/report (แล้วก็ update state ลง operational DB) แล้วจุดนี้แหละที่ result แบบ invalid-token จะกลายร่างไปเป็น token cleanup ต่อ
- **Retry Delivery Worker** payload ที่ fail ก็ไหลไปเข้าเส้น retry ที่ re-queue มันด้วย **exponential backoff บวกตัวนับ max-attempt** พอตัวไหนข้าม threshold จำนวนครั้งไปเมื่อไหร่ มันก็หยุด (แล้วไปตกที่ dead-letter store ไว้ให้เรามาส่องทีหลัง) แทนที่จะวน loop ไปไม่จบไม่สิ้น

    PH --> DEV["Devices"]
    DEL -->|"result (success / fail)"| RQ["Result Queue"]

    AGG --> COL[("Columnar DB\n(analytics)")]

ส่วนที่ยากกว่าที่คิดคือเรื่อง rate limiting เพราะมันเป็นเรื่องของ **budget ก้อนเดียวที่ต้องแชร์กันทั้ง fleet** คือ quota ของ provider มันมีก้อนเดียว — ของ FCM ก็ราย project (หรือ limit ราย connection/stream ของ APNs ถ้าไปทาง direct) — แต่ต้องเอามาแบ่งให้ delivery worker **ทุกตัว** ใช้ร่วมกัน ทีนี้ถ้าให้ worker แต่ละตัวถือ token bucket ของตัวเองแบบ local เราจะติดหล่มทันที: หาร quota ด้วยจำนวน worker ก็ใช้ได้น้อยกว่าที่ควรแบบน่าเสียดาย แต่ถ้าให้แต่ละตัวคิดว่าตัวเองมี budget เต็มก้อน พอรวมกันมันก็ทะลุเพดานแล้วโดน 429 อยู่ดี เพราะงั้น bucket มันเลยอยากเป็นแบบ **แชร์กัน** — เอา token bucket กลางไปวางไว้ใน Redis ให้ worker มาดึงไปใช้ — หรือมี coordinator คอยแบ่ง slice ให้แต่ละ worker อย่างยุติธรรมแล้วก็ rebalance ได้ แล้วก็อย่าลืมทำให้มัน **adaptive** ด้วยนะ: พอ provider เริ่มตอบ 429 พร้อม `Retry-After` กลับมา ท่าที่ถูกคือ **หรี่ refill rate ลง** ไม่ใช่ดันทุรังกระหน่ำต่อที่เพดานที่เราเดาเอาเอง

รายละเอียดที่ผมอยากขีดเส้นใต้จากที่เจอมาเอง: **ตัวนับ max-attempt คือวาล์วนิรภัย** ถ้าไม่มีมัน payload ตัวเดียวที่ fail ดื้อ ๆ (token เสีย, template พัง, provider ล่ม) ก็จะกลายเป็น worker ที่ retry ไม่จบ เผา budget ทิ้ง แล้วก็อุดตัน partition ที่ต่อแถวอยู่ข้างหลัง หลักการ `retry ได้ แต่ต้องมีเพดาน` คือหัวใจทั้งหมดเลย

ขอแปะ caveat แบบเปิดอกอันนึงเรื่องคำว่า **delivered** หน่อย result ที่ provider ส่งกลับมาเนี่ย มันแปลว่า **provider รับ push ไปแล้ว** เฉย ๆ — ไม่ได้แปลว่ามันไปถึงเครื่อง แล้วก็ยิ่งไม่ได้แปลว่ามีใครเห็นมันเลยนะ ทั้ง APNs แล้วก็ FCM ไม่ได้การันตี last mile ให้เราหรอก เพราะงั้นสิ่งที่เราวัดได้เองมันเลยเป็นแค่ funnel ที่แต่ละชั้นมันมีรั่ว:

![funnel ของ delivery: accepted กับ delivered กับ opened ที่รั่วในแต่ละชั้น](/assets/images/blogs/design-push-notification-system/delivery-funnel.png)

- **Accepted** — provider รับไป เราวัดอันนี้ได้เลยจาก send result
- **Delivered** — มันถึงเครื่อง เราจะรู้อันนี้ก็ต่อเมื่อ device/SDK รายงานกลับมาเท่านั้น ส่วน network ที่ drop เงียบ ๆ, แอปที่โดน force-quit, แล้วก็ notification ที่ถูกปิด หายวับไปจุดนี้หมด
- **Opened** — user กดมัน อันนี้ต้องมี callback ฝั่ง client ยิงกลับเข้า tracking ของเรา

นัยเชิง design อันนี้มันฟังดูไม่ค่อยสบายใจเท่าไหร่ แต่ก็ควรพูดออกมาดัง ๆ: ถ้าเราไม่ได้ใส่ channel สำหรับ client ack เข้าไป ไอ้ที่เราเรียกกันว่า `delivery tracking` เนี่ย แท้ ๆ แล้วมันคือ **acceptance tracking** ต่างหาก dashboard หลาย ๆ ตัวก็รายงาน acceptance rate เงียบ ๆ แล้วก็ติดป้ายให้มันว่าเป็น delivery rate ถ้าอยากได้ engagement funnel ตัวเต็มแบบครบ ๆ เราต้องไป instrument ฝั่ง client เอง แล้วป้อน event พวกนั้นกลับเข้า aggregator ตัวเดิม

## Tokens rot — device registry and cleanup

**ปัญหา:** เรื่องของ device token คือมันเปลี่ยนได้ตลอดเวลา — user reinstall หรือ uninstall แอปทีนึง token ก็เปลี่ยนแล้ว หรือบางทีมันก็เปลี่ยนเงียบ ๆ อยู่เบื้องหลังเองโดยที่ user ไม่รู้ตัวด้วยซ้ำ ผลคือ user คนนั้นหลุดจากการรับ notification ไปเลยทั้ง ๆ ที่ยังใช้แอปอยู่ดี ๆ ส่วนฝั่งเรา token store ที่เอาแต่ **เพิ่ม** อย่างเดียวก็จะค่อย ๆ เต็มไปด้วย token ที่ตายไปแล้ว ทุก campaign เลยเปลือง send ส่วนนึงไปกับ token ที่ยิงยังไงก็ไม่มีวันถึง แล้วที่แย่กว่านั้น พวก token ตาย ๆ เนี่ยมันทำให้ตัวเลข `sent` ของเราพองโตผิดส่วน แล้วก็ค่อย ๆ วางยา delivery metric ทุกตัวที่เรา report

**วิธีแก้:** มอง token store เป็น **registry ที่มีชีวิต** ที่มี feedback loop ไม่ใช่ list ที่เอาแต่ append อย่างเดียว

![lifecycle ของ device token: register, ใช้งาน, แล้วก็ loop cleanup ที่รักษาตัวเองจาก delivery failure](/assets/images/blogs/design-push-notification-system/token-lifecycle.png)

- **Registration / refresh** client SDK ส่ง token ให้เราตอน install แล้วก็ส่งตัวใหม่ทุกครั้งที่มันหมุน เราก็ upsert มันเข้ากับ user + device แล้วเก็บ `last_seen` / app version ไว้ เพื่อให้ entry เก่า ๆ ถูกปลดตามอายุได้
- **Cleanup จาก delivery result** อันนี้คือผลตอบแทนของการทำ delivery tracking: พอ FCM บอกว่า token ตายแล้ว (`UNREGISTERED` / `registration-token-not-registered`) result นั้นก็ลอยไปบน result queue เดียวกันเข้าสู่ aggregator ซึ่งจะ **prune token ตายตัวนั้น** ออกจาก registry ระบบมันค่อย ๆ รักษาตัวเองทีละนิดทุกครั้งที่ส่ง
- **key ด้วย (user, device) ไม่ใช่ raw token** user คนนึงมีหลายเครื่อง แล้วเครื่องนึงก็ re-register ด้วย token ใหม่ได้ก่อนตัวเก่าจะหมดอายุ การ key ด้วย device กันเราไม่ให้เด้งหาเครื่องเดียวกันซ้ำสองทีจาก row เก่าสองแถว

สรุปคือต้องปรับมุมคิดใหม่: token ไม่ใช่ค่าที่เก็บไว้ครั้งเดียวแล้วจบ มันคือ **lease ที่ต้องคอยต่ออายุ** ต่างหาก — แล้ว delivery failure ของเราเองนั่นแหละคือ signal ที่คอยบอกว่า lease ตัวไหนหมดอายุไปแล้ว

พอ token ตายมัน prune ตัวเองได้แล้ว ฝั่ง infrastructure ก็ถือว่าเสร็จสมบูรณ์ ที่เหลือไม่ใช่เรื่อง infra แล้ว — มันคือการทำให้ notification มันคุ้มค่าพอที่จะส่งออกไปต่างหาก

## Make it worth sending — templating + personalization

**ปัญหา:** pipeline ที่ส่ง text เดียวกันเป๊ะหาทุกคนได้แบบไร้ที่ติ มันก็ยังเป็น product ที่กลาง ๆ อยู่ดี ยิง push ที่เหมือนกันหมดแล้วไม่เกี่ยวกับคนรับ คนก็ปิดเสียง mute แอปหรือ uninstall ทิ้ง ส่งตอนตีสามตามเวลาท้องถิ่นก็ยิ่งเสียเขาไว เด้งหา user คนเดิมห้าครั้งในชั่วโมงเดียวก็เท่ากับฝึกให้เขาปัดเราทิ้ง เรื่อง reliability ทำให้ message **ออกไป**ได้ ส่วน relevance เป็นตัวตัดสินว่ามันควรออกไปตั้งแต่แรกมั้ย

**วิธีแก้:** เจ้า **template parser** กับ **user-preference checker** ใน Pre-Delivery Worker เนี่ยแหละที่มีไว้เพื่อเรื่องนี้เลย ก่อนที่ notification จะถูกต่อคิวไปส่ง มันต้องวิ่งผ่านด่าน content-กับ-consent ก่อน:

- **Preference check** — user คนนี้ opt-in หมวดนี้ไว้มั้ย? ตอนนี้อยู่ในช่วง quiet hours ของเขา (ตาม timezone ของเขา) รึเปล่า? ถ้าอันใดอันนึงไม่ผ่าน ก็ drop หรือ defer notification ทิ้งที่จุดนี้เลย ก่อนที่มันจะเปลือง send
- **Template render** — resolve template ราย user: locale/ภาษา, แล้วก็ merge field อย่างชื่อ, ยอดเงิน, หรือ status ของ order
- **Personalization** — เลือกว่า user คนนี้ควรได้ content **อันไหน** ซึ่งเป็นจุดที่ pipeline นี้ไปต่อกับ audience/segmentation service — เรื่องนี้ใหญ่พอที่จะแยกไปเป็นหัวข้อของมันเองได้เลย

<MermaidDiagram chart={`
flowchart TD
    EV["Notify user X"] --> PREF{"Opted in?\nWithin quiet hours?"}
    PREF -->|"no"| DROP["Drop / defer"]
    PREF -->|"yes"| TMPL["Render template\n(locale + merge fields)"]
    TMPL --> READY["Ready-to-send payload"]
`} />

ที่ทำเรื่องนี้ในสเตจ **pre-delivery** เป็นความตั้งใจ: notification ที่ไม่ควรถูกส่ง ควรตายก่อนที่มันจะไปแตะ provider, rate budget, หรือ dedupe key เรื่อง consent กับ relevance มันบังคับใช้ได้ถูกที่สุดตั้งแต่หน้าด่าน

**อีกทางเลือกนึงในการออกแบบ — ตกลงใครเป็นคนปั้น payload?** ทั้งหมดข้างบนมันตั้งอยู่บนสมมติฐานว่า Pre-Delivery Worker เป็นคน enrich เอง คือไป lookup ข้อมูลแล้วก็ render template ให้ ของที่วิ่งอยู่บนสายเลยมีขนาดเล็ก ซึ่งเป็น default ที่ดีนะ แต่มันไม่ใช่ทางเดียว แล้วจะเลือกทางไหนมันก็ขึ้นกับว่า ใครเป็นเจ้าของข้อมูล:

- **ปล่อยให้ upstream ส่ง payload มาทั้งก้อนเลย** ในเคสที่ service **อื่น** เป็นเจ้าของข้อมูลที่เปลี่ยนไปทุก event — อย่างพวก transaction activity ที่เนื้อหามันเป็น dynamic — บางทีปล่อยให้ service นั้นปั้น notification มาทั้งก้อนแบบพร้อมส่งเลยมันง่ายกว่า เราแลกมากับ payload ที่ใหญ่ขึ้นบนสาย แต่ก็คุ้ม เพราะถ้าจะมานั่ง enrich เองที่ส่วนกลาง สุดท้ายมันก็ต้องวิ่งกลับไปถาม service นั้นอยู่ดี
- **ใช้ template id แล้วให้ enrich เป็น optional** ส่วนพวก content ที่ซ้ำ ๆ — campaign, เตือน abandoned cart — อย่าไปขน body เดิม ๆ ส่งซ้ำไปซ้ำมา ลงทะเบียน `templateId` ไว้ตัวนึง แล้วให้ upstream ส่งมาแค่ไม่กี่ field ที่จำเป็นต้องเอาไปเติมในช่องว่างก็พอ การ enrich ก็จะกลายเป็นแค่ option แล้วก็ทำน้อย ๆ: payload เล็ก แล้ว upstream ก็เป็นเจ้าของทั้ง template แล้วก็ข้อมูลที่เอามา merge

ขอยกตัวอย่างให้เห็นภาพความต่างของสองแบบนี้:

```json
// transaction activity — เนื้อหา dynamic, upstream ปั้น body มาทั้งก้อน
{
  "userId": "u_123",
  "type": "payment_completed",
  "eventId": "9813789127398173921",
  "notification": {
    "title": "Payment complete",
    "body": "245 THB at Store XXXXX (txn 9813789127398173921)"
  }
}
```

```json
// campaign / abandoned cart — ลงทะเบียน templateId ไว้ครั้งเดียว ส่งมาแค่ field ที่ต้องเติม
{
  "userId": "u_123",
  "type": "abandoned_cart",
  "templateId": "cart_reminder_v3",
  "data": { "item": "Sony WH-1000XM5", "price": "13,990 THB" }
}
```

## Not all notifications are equal — type, priority, and frequency caps

**ปัญหา:** ปฏิบัติกับ MFA code, `payment received`, แล้วก็ `ลด 50% สุดสัปดาห์นี้` แบบเหมือนกันหมด ถ้าทุก marketing campaign ออกไปด้วย priority เต็มแบบไม่มีเพดาน มันจะมีของพังสองอย่าง อย่างแรกคือ user ปิดเสียงแอปทิ้ง — แล้วพอเขาปิดปุ๊บ transactional notification ที่เขาต้องการมันก็ตายไปพร้อม promo ด้วย อย่างที่สองคือ campaign ที่ไม่ได้ด่วนอะไรก็ไปแย่ง delivery budget ก้อนเดียวกันกับ alert ที่ด่วนแบบเป็นเรื่องเป็นราว

**วิธีแก้:** จัดประเภท notification ทุกตัวตาม **type** แล้วให้ type นี่แหละเป็นตัวกำหนด priority กับ limit

- **Transactional / activity** (MFA, ยืนยัน payment, มีคนทักมา) — priority สูง ส่งทันที ใน FCM เราเซ็ตอันนี้ผ่าน block ราย platform — `apns-priority: 10` ใน header ของ `apns`, แล้วก็ตั้ง priority สูงใน block `android` พวกนี้ได้ประโยชน์จากการ **collapse** ด้วย: `balance updated` อันเก่าที่ค้างอยู่ ควรถูก **แทนที่** ด้วยอันใหม่ ไม่ใช่กองทับ ๆ กัน — ใช้ `apns-collapse-id` ฝั่ง iOS, `collapse_key` ฝั่ง Android — user จะได้เห็น state ล่าสุด ไม่ใช่กองห้าอันซ้อนกัน
- **Marketing / campaign** — priority ต่ำลงมา (`apns-priority: 5` ใน block `apns` — ถนอมแบต แล้วก็เปิดให้ OS จับ batch ได้) แล้วก็โดน **frequency cap** คุมไว้ เกิน cap เมื่อไหร่ → drop หรือ defer

เรื่อง cap เนี่ย ตอนนี้เก็บเป็น **global policy ไปก่อนเถอะ** — เช่น อย่างมาก N marketing push ต่อ user ต่อวัน ใช้ N เดียวกันหมดทุกคน มันคือ counter ถูก ๆ ตัวนึง: key ราย user ใน Redis ที่มี TTL รายวัน, `INCR` แล้วก็เช็คในสเตจ pre-delivery จะทำ cap เป็นราย user หรือราย segment ทีหลังก็ได้ แต่นั่นมันคือการ tune ที่ยังไม่ต้องมีตั้งแต่วันแรกหรอก แค่ global cap ตัวเดียวก็ตัดพฤติกรรมที่แย่ที่สุด (spam รัว ๆ ไม่หยุด) ออกไปได้แล้ว โดยแทบไม่เพิ่มความซับซ้อนเลย

สังเกตว่าอันนี้มันคนละแกนกับหัวข้อก่อนหน้านะ consent ราย user (opt-in, quiet hours) คือ personalization — มันว่าด้วยความต้องการของ user คนนี้ ส่วน priority ตาม type กับ global cap คือ platform policy — มันว่าด้วยว่าระบบจะปฏิบัติกับ notification ทั้ง **class** ยังไง โดยไม่สนว่าใครเป็นคนรับ ทั้งคู่อยู่ในสเตจ pre-delivery เหมือนกัน แต่มันตอบคนละคำถามกัน

**ก้าวถัดไปของเรื่องนี้ — แยก lane** ตอนแรกแค่ติด **type tag** บน pipeline ที่ใช้ร่วมกันก็พอแล้ว แต่พอ volume ของ marketing เริ่มใหญ่ — campaign 60–70M พวกนั้น — แค่ tag มันไม่พอแล้ว เพราะ campaign กับ MFA code ยังใช้ topic แล้วก็ worker ร่วมกันอยู่ blast เลยไป head-of-line-block ตัว code ได้ วิธีแก้คือแยก lane ออกจากกันทางกายภาพไปเลย: **transactional lane** ที่มี topic กับ delivery worker ของตัวเอง จัดทรัพยากรให้แบบเหลือเฟือแล้วก็ไม่โดน throttle, กับ **bulk/marketing lane** ที่มี pool ของตัวเอง โดน rate-limit แล้วก็ frequency-cap ตอนรันที่ scale ระดับนั้น อันนี้คือสิ่งที่เราทำกันมาเป๊ะ ๆ เลย — notification ฝั่ง payment กับ transactional เดินคนละเส้นกับ campaign blast เพราะงั้นต่อให้ marketing พุ่งขึ้นมา มันก็ไปกอง queue ของตัวเองได้ โดยไม่หน่วง push ที่ user กำลังนั่งรออยู่เลยสักนิด

## Wrapping up

เราเริ่มจาก loop ที่เรียก FCM อยู่ใน request แล้วมาจบที่ pipeline สองสเตจที่ partition แล้ว รับงานระดับ 120M+ ต่อวัน เส้นทางทั้งหมด เรียงตามลำดับ:

1. **ยิงให้ถึงทุก platform ผ่าน FCM** — gateway เดียวบวก config ราย platform ไม่ใช่ client แยกราย provider เป็นของพื้นฐานที่ต้องมี ไม่ใช่ step ของการ scale
2. **ทำให้เป็น async โดยอิง durable log** — การเปลี่ยนอันเดียวที่ทุกอย่างหลังจากนี้สร้างต่อยอดอยู่บนมัน
3. **Batch แล้วก็ pool connection** เพื่อให้การ send ก้อนใหญ่วิ่งเป็น request แบบ concurrent บน HTTP client ที่ reuse ไม่ใช่ handshake ใหม่ทุก token
4. **ผ่าเป็นสเตจแล้วก็ partition** เพื่อให้ enrichment กับ delivery scale แยกกันได้ พร้อมทั้ง ordering แล้วก็ replay
5. **Fan out แบบ lazy** เพื่อให้ campaign 60M คน กางออกเป็น chunk แทนที่จะท่วม queue
6. **Dedupe** เพื่อไม่ให้ transport แบบ at-least-once กลายเป็น double-buzz
7. **Rate-limit, track, แล้วก็ retry แบบมีเพดาน** — แล้วก็ยอมรับกันแบบเปิดอกว่าถ้าไม่มี client ack เราวัดได้แค่ acceptance ไม่ใช่ delivery
8. **ทำให้ token registry รักษาตัวเองได้** ด้วยการให้ delivery failure คอย prune token ที่ตายแล้วออก
9. **มีด่าน consent กับ relevance** เพื่อให้ของที่เราส่งถึงอย่างมั่นใจ มันคุ้มค่าพอที่จะส่ง
10. **ปฏิบัติกับ notification ตาม type** — priority กับ collapse สำหรับ transactional, frequency cap สำหรับ marketing, แล้วก็แยก lane พอ volume เรียกร้อง เพื่อไม่ให้ promo ไปกลบ alert ที่สำคัญ

ประเด็นใหญ่ที่อยากฝากไว้: **ไม่มีใครควร build เวอร์ชันสุดท้ายตั้งแต่วันแรกหรอก** ทุกกล่องมันได้ที่ทางของมันมาจากการแก้ปัญหาที่เวอร์ชันก่อนหน้าสร้างขึ้น ถ้าเราเริ่มจากง่าย ๆ แล้วปล่อยให้ข้อจำกัดหน้างานมันคอยผลักเราไปทีละสเตป สุดท้ายเราก็มักจะมาโผล่ที่ใกล้ ๆ จุดนี้แหละ — แล้วเราจะเข้าใจทุกชิ้นส่วน เพราะเรารู้สึกถึงความเจ็บที่ทำให้มันถูกเพิ่มเข้ามาด้วยตัวเอง

## Fun fact: my favorite feedback loop

ขอเล่าเรื่อง love–hate ของการทำงานกับ push หน่อย: มันเป็นหนึ่งในไม่กี่ระบบที่ให้ feedback loop กับผมแบบเกือบ ๆ **ทันที** หลัง deploy ผม deploy ของลงไป แล้วก็เดินไปเป็น user ซะเอง — สั่งของออนไลน์สักอย่าง — แล้วก็นั่งรอเสียงเด้ง notification เด้งมามั้ย? เยี่ยมเลย เงียบกริบเหรอ? จบเห่ แล้วก็รู้ทันทีเลยว่าวันที่เหลือของผมจะเป็นยังไงต่อ

ตอนที่ย้าย FCM integration ไปใช้ SDK ตัวใหม่ ผมเทสต์ทุกเส้นทางเท่าที่เทสต์ได้ในโค้ดเลยนะ… แล้วสุดท้ายก็ยังเดินไป 7-11 หัวมุมถนน กดซื้อของขึ้นมาซะงั้น เพื่อจะดูว่า transactional push มันเด้งเข้ามือถือตัวเองมั้ย ก่อนจะกล้าเชื่อใจมัน automated test มันก็ดีงามแหละ — แต่ไม่มีอะไรทำให้ใจผมนิ่งลงได้เท่ากับใบเสร็จตัวเป็น ๆ ที่เด้งอยู่ในกระเป๋าหรอก 😅
