## Vibe Coding - Zero to One

Session แรกเปิดด้วยหัวข้อ Vibe Coding ซึ่งพูดถึงแนวคิดการสร้าง application ด้วย Natural Language เป็นหลัก แทนที่จะเขียน code ทีละบรรทัด

แนวคิดหลักคือ ตอนนี้เป็นช่วงเวลาที่ดีที่สุดสำหรับการเริ่มสร้าง app ด้วย AI เพราะ tool ต่าง ๆ พร้อมมากขึ้น คนจำนวนมากกำลังใช้ vibe coding เพื่อ showcase ไอเดียและ creativity ของตัวเอง โดยเริ่มจากง่ายไปยาก

โดย Google ได้วาง flow สำหรับ vibe coding ไว้เป็น 3 ขั้นตอนหลัก

![Think → Build → Publish](/assets/images/blogs/build-with-ai-bkk-2026/session1-vibe-coding/CleanShot_2026-03-22_at_13.44.592x.jpg)

1. **Think (The Intent)** — define ปัญหาหลักที่อยากแก้ให้ชัดก่อน จะเป็น productivity app หรือ retro game ก็ได้ แต่ต้องโฟกัสที่ intent ที่จะอธิบายให้ AI ฟัง
2. **Build (The Vibe)** — ใช้ AI Studio สำหรับ instant prototype หรือ Antigravity สำหรับ agentic coding แค่ describe ไอเดีย แล้วให้ agent สร้าง logic และ UI ให้แบบ real-time
3. **Publish (The Launch)** — deploy app ขึ้น Google Cloud ได้ทันที จาก local preview ไปเป็น public URL ที่แชร์ให้ทีมหรือใครก็ได้

### เครื่องมือจาก Google ที่น่าสนใจ

Session นี้แนะนำ 3 เครื่องมือหลักจาก Google สำหรับ vibe coding

**AI Studio** — เป็น rapid prototyping tool ที่ทำงานคล้าย Lovable โดยมี live preview ในตัว สามารถสร้าง application ได้จาก single prompt และ preview ผลลัพธ์ได้เลยในbrowser รวมถึงดู responsive บนมือถือได้ด้วย

![Google AI Studio - Build your ideas with Gemini](/assets/images/blogs/build-with-ai-bkk-2026/session1-vibe-coding/CleanShot_2026-03-22_at_13.53.362x.jpg)

**Antigravity** — เป็น agentic IDE สำหรับ developer ที่ต้องการ control มากขึ้น สามารถ spawn multi-agent และใช้ tools ต่าง ๆ ได้ มี agentic browser ที่มี built-in Chrome ให้ agent โต้ตอบกับ component ภายใน app ได้โดยตรง และมี Files & Artifacts panel ที่แสดง file ทั้งหมดที่ agent สร้างขึ้นมา

![Antigravity](/assets/images/blogs/build-with-ai-bkk-2026/session1-vibe-coding/antigravity.png)

**Gemini CLI** — สำหรับคนที่ถนัดทำงานผ่าน terminal เป็น command-line interface ที่ใช้ Gemini model

![Gemini CLI](/assets/images/blogs/build-with-ai-bkk-2026/session1-vibe-coding/gemini-cli.png)

นอกจากนี้ยังมีการพูดถึง [Stitch](https://blog.google/innovation-and-ai/models-and-research/google-labs/stitch-ai-ui-design/) ซึ่งเป็น AI-native platform จาก Google Labs ที่ช่วยให้ทุกคนสร้าง high-fidelity UI ได้ เรียกว่าเป็นแนวคิด "vibe design" เลยก็ว่าได้

### Anatomy of a Powerful Prompt

![The anatomy of a Powerful Prompt](/assets/images/blogs/build-with-ai-bkk-2026/session1-vibe-coding/photo_2026-03-22%2016.03.15.jpeg)

ส่วนที่ผมคิดว่าสำคัญที่สุดใน session นี้คือเรื่อง prompt engineering สำหรับ vibe coding โดย speaker ได้แบ่ง prompt ออกเป็นหลายระดับ

- **Lazy prompt** — ไม่มี context ไม่มี definition ของ feature ผลลัพธ์ก็ออกมาไม่ค่อยตรง แถม token ก็ถูกใช้ไปเยอะมากโดยไม่จำเป็น
- **Good prompt** — มี context พอสมควร เช่น "Help me build an app to track my food and water intake"
- **Powerful prompt** — expand ออกเป็นหลาย layer ได้แก่ การกำหนด role, target audience, context, definition ของ feature ที่ต้องการ และ optional อย่าง visual reference

ประเด็นที่ชัดเจนคือ AI ทำงานไม่ดีกับ lazy prompt และเมื่อเรามี context ที่ดี iteration จะเร็วขึ้นมาก ในขณะที่ prompt ที่ไม่มี context จะ consume token มากกว่าจำเป็น

ผมคิดว่าเรื่องนี้สอดคล้องกับหลักการเขียน requirement spec หรือ technical design doc อยู่แล้ว ยิ่งเรา define scope ชัดเจนเท่าไหร่ ผลลัพธ์ก็ยิ่งตรงเป้ามากขึ้นเท่านั้น ไม่ว่าจะเป็นการสื่อสารกับคนหรือกับ AI

## Software Delivery in the AI Age (Datadog Session)

Session นี้จาก Datadog เริ่มด้วยคำถามที่ฟังดูง่ายแต่คิดแล้วน่าสนใจ

> ถ้าคุณเขียน code ได้เร็วขึ้น 10 เท่า แปลว่าคุณจะ deliver ได้เร็วขึ้น 10 เท่าไหม?

คำตอบคือ ไม่จำเป็น เพราะ bottleneck ของ software delivery ไม่ได้อยู่แค่ที่การเขียน code แต่อยู่ที่ทั้ง pipeline ตั้งแต่ testing, deployment, monitoring ไปจนถึง incident response

### AI เป็น Amplifier ไม่ใช่ Magic

ประเด็นหลักที่ speaker เน้นคือ Agentic AI ทำหน้าที่เป็น amplifier

- **องค์กรที่ทำได้ดีอยู่แล้ว** → AI จะช่วยให้ดียิ่งขึ้น
- **องค์กรที่ process ยุ่งเหยิง** → AI จะ amplify ความยุ่งเหยิงนั้นให้หนักขึ้นไปอีก

ผมว่าตรงนี้ตรงประเด็นมาก เพราะเห็นหลายทีมที่ adopt AI tooling แล้วคาดหวังว่าทุกอย่างจะดีขึ้นอัตโนมัติ แต่จริง ๆ แล้วถ้า foundation ไม่ดี ยิ่งเร็วก็ยิ่งพัง

### ผลกระทบต่อ SDLC

การ develop ที่เร็วขึ้นด้วย AI ส่งผลหลายอย่าง

- Code changes เข้า production เร็วขึ้นและถี่ขึ้น
- การหาต้นตอของปัญหายากขึ้น เพราะ code ที่ AI สร้างมาอาจไม่มีใครเข้าใจ context ทั้งหมด
- Expectation จากองค์กรสูงขึ้น ถ้าใช้ AI แล้ว ทุกอย่างก็ต้องดีขึ้นสิ

สิ่งที่ Datadog เสนอคือ ในยุค AI, **data คือ differentiated advantage** ที่แท้จริง และ observability คือสิ่งที่ทำให้เรามองเห็นได้ว่าเกิดอะไรขึ้นจริง ๆ

## Lab: Getting Started with MCP, ADK and A2A

Session นี้เป็น hands-on lab ที่ให้ผู้เข้าร่วมลองทำ [codelab](https://codelabs.developers.google.com/codelabs/currency-agent?hl=en#0) สร้าง currency conversion agent โดยใช้ 3 protocol/framework หลักของ Google

![Codelab Architecture - MCP, ADK and A2A](/assets/images/blogs/build-with-ai-bkk-2026/session3-lab-mcp-adk-a2a/codelab.png)

### ทำความเข้าใจ 3 ตัวย่อ

**MCP (Model Context Protocol)** — เป็น open protocol ที่กำหนดมาตรฐานการเชื่อมต่อระหว่าง AI model กับ tools ภายนอก คิดง่าย ๆ ว่าเป็นเส้นทาง Agent → Tools ที่ standardized และเป็น agent-agnostic หมายความว่าไม่ว่าจะใช้ agent framework ไหนก็เรียกใช้ MCP tools เดียวกันได้ ใน lab นี้สร้างด้วย FastMCP

**ADK (Agent Development Kit)** — เป็น framework สำหรับ develop AI agent ที่ orchestrate การใช้ tools ต่าง ๆ ผ่าน LLM (ในที่นี้คือ Gemini) เป็นตัวกลางที่เอา MCP tools มาประกอบเข้าด้วยกันเป็น agent ที่ทำงานได้

**A2A (Agent-to-Agent Protocol)** — เป็น open standard สำหรับการสื่อสารระหว่าง agent กับ agent คิดง่าย ๆ ว่า

- MCP คือ Agent → Tools
- A2A คือ Agent ↔ Agent

### Agent Card

**Agent Card** ซึ่งเปรียบได้กับนามบัตรของ agent หรือจะมองเป็น resume สำหรับสมัครงานก็ได้ มันบอกว่า agent ตัวนี้ทำอะไรได้บ้าง มี skill อะไร มี capability อะไร

ใน [A2A Python SDK](https://github.com/google-a2a/a2a-python) มี Pydantic models สำหรับ [AgentSkill](https://github.com/google-a2a/a2a-python/blob/a1894aecb27486baa2d19ec965dbfcb9be6b215b/src/a2a/types.py#L84), [AgentCapabilities](https://github.com/google-a2a/a2a-python/blob/a1894aecb27486baa2d19ec965dbfcb9be6b215b/src/a2a/types.py#L827) และ [AgentCard](https://github.com/google-a2a/a2a-python/blob/a1894aecb27486baa2d19ec965dbfcb9be6b215b/src/a2a/types.py#L1394) ให้ใช้งานได้เลย

### สิ่งที่ได้ลองทำใน Lab

Lab นี้ให้สร้าง currency conversion agent ตั้งแต่ต้นจนจบ

1. สร้าง MCP server ด้วย FastMCP ที่มี tool `get_exchange_rate` สำหรับดึงอัตราแลกเปลี่ยนจาก Frankfurter API
2. Deploy MCP server ขึ้น Cloud Run
3. สร้าง ADK agent ที่เชื่อมต่อกับ remote MCP server ผ่าน HTTP
4. Expose agent เป็น A2A server ด้วย function `to_a2a()`
5. ทดสอบด้วย A2A Client

สิ่งที่น่าสนใจคือ flow ทั้งหมดแสดงให้เห็นว่า 3 protocol นี้ทำงานร่วมกันอย่างไร

- MCP server ห่อหุ้ม tool capabilities เอาไว้
- ADK orchestrate การเรียกใช้ tools ผ่าน LLM
- A2A ทำให้ agent ตัวอื่นค้นพบและสื่อสารกับ agent ตัวนี้ได้แบบ standardized

ผมมองว่า architecture แบบนี้คล้ายกับ microservices pattern มาก MCP server เหมือน individual service ที่ expose API, ADK เหมือน orchestration layer และ A2A เหมือน service discovery + inter-service communication protocol

## Best Practice to Design Agentic AI System in Google ADK

Session นี้เข้มข้นมาก พูดถึงหลักการออกแบบ agentic AI system ด้วย Google ADK ตั้งแต่ concept พื้นฐานไปจนถึง design patterns ระดับ production

### Assistants vs. Agents

![Agents vs. Assistants](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.14.jpeg)

จุดเริ่มต้นที่ดีคือการแยกแยะระหว่าง Assistants กับ Agents ให้ชัด

- **Assistants** → Task-oriented คือเราถาม มันตอบ มันช่วยเราทำงาน เช่น "Draft an email for me"
- **Agents** → Goal-oriented คือเราให้เป้าหมาย มันคิดเองว่าจะทำยังไง เช่น "Book meetings with all leads who emailed me today"

พูดง่าย ๆ คือ Assistants เป็นแบบ Chatting ส่วน Agents เป็นแบบ Doing

### Agentic Design Patterns

Speaker ได้อ้างอิง [Google Cloud Architecture Center - Agent Design Patterns](https://docs.cloud.google.com/architecture/choose-design-pattern-agentic-ai-system) ซึ่งเป็น proven architectural blueprint สำหรับออกแบบ agentic system

หลักการสำคัญที่สุดสำหรับ agent ตัวเดียวคือ **Single Responsibility** อย่าสร้าง "God Agent" ที่ทำทุกอย่าง เพราะจะเจอปัญหา context overflow และ AI confusion ให้ decompose task ออกเป็น agent เล็ก ๆ ที่ specialized แทน

![Core Design Principle: Single Responsibility](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.17.jpeg)

### Single-Agent vs. Multi-Agent

**Single-Agent System** คือ AI model ตัวเดียวที่มี tools ให้ใช้ pattern ที่นิยมคือ ReAct (Reason + Act) ที่ agent จะ "คิดออกเสียง" ก่อนลงมือทำ เหมาะกับปัญหาที่ซับซ้อนและต้องการ continuous planning

![Single Agent](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-single-agent.svg)

![ReAct Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-react.svg)

![Multi-Agent Systems](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.18.jpeg)

**Multi-Agent System** คือหลาย agent ทำงานร่วมกัน แต่ละตัวมีความเชี่ยวชาญเฉพาะด้าน โดยมี key requirements สองข้อ

- Context Engineering: แต่ละ agent ต้องมี specific information (history, documents, rules) และต้อง manage data flow ระหว่าง agent ให้ดี
- Orchestration: ต้องมีระบบที่ทำให้ agent สื่อสารกันได้อย่างถูกต้อง

patterns ที่น่าสนใจสำหรับ multi-agent

- **Sequential** — agent ทำงานตาม pipeline ที่กำหนดไว้ เหมาะกับ structured processes

![Sequential Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-sequential.svg)

- **Parallel** — หลาย agent ทำงานพร้อมกัน ลด wait time แต่ cost สูงขึ้น

![Parallel Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-parallel.svg)

- **Loop** — ทำซ้ำจนกว่าจะได้ผลลัพธ์ที่ดี ใช้เยอะใน review and critique pattern

![Loop Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-loop.svg)

![Review & Critique Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-review-critique.svg)

- **Coordinator** — มี main agent คอย orchestrate sub-agents

![Coordinator Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-coordinator.svg)

- **Human-in-the-Loop (HITL)** — agent หยุดรอให้คนยืนยันก่อน เหมาะกับงานที่มี risk สูง เช่น financial/legal (approve เงิน) หรือ safety/privacy (handling sensitive data) ซึ่งพอ user approve แล้ว responsibility ก็ตกเป็นของ human ทั้งหมด

![Human-in-the-Loop Pattern](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/choose-design-pattern-agentic-ai-system-human-in-the-loop.svg)

![Taxonomy of Agent Patterns](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.19.jpeg)

สรุป taxonomy ได้ 3 กลุ่ม

- **Iterative** → ReAct สำหรับปัญหาที่ ambiguous
- **Procedural** → Sequential, Loop สำหรับงานที่ต้องการ reliability
- **High Stakes** → HITL สำหรับงานที่ต้องมีคน approve

### ADK Agent Architecture

![ADK Agent - Base](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.16.jpeg)

ADK Agent ประกอบด้วย Base Agent ที่มี Name, Model, Instruction และ Tools โดยสามารถมี Sub Agents ได้ รองรับ input หลากหลาย (Video, Database, Files, Audio, Cache, Vector) และ output ได้หลายแบบ (API, HTTP, MQTT, gRPC) นอกจากนี้ยังมี out-of-box tools อย่าง Google Search และ Vertex AI Search ให้ใช้ได้เลย

### Managing State & Memory

Agents ต้อง remember past interactions และ keep track specific data ซึ่ง ADK มี built-in Session object ที่ persist across turns ให้อยู่แล้ว

### Adding Safety — Callbacks & Guardrails

![Callbacks](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.20.jpeg)

ส่วนที่สำคัญมากคือเรื่อง safety ซึ่ง ADK มี callback system ให้แทรก logic ก่อนและหลังแต่ละ step

- **before_model_callback** — Input Guardrail ทำงานก่อนส่ง request ไป LLM ใช้สำหรับ input validation, filtering PII, block harmful content หรือแม้แต่ dynamic prompt modification
- **before_tool_callback** — Tool Argument Guardrail ทำงานก่อนเรียก tool ใช้ validate tool arguments
- **Model Armor** — architecture ระดับ application ที่ inspect ทั้ง prompt และ response เพื่อ sanitize ก่อนส่งถึง user

![Model Armor Architecture](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.21%20(1).jpeg)

### Summary & Key Takeaways

![Summary](/assets/images/blogs/build-with-ai-bkk-2026/session4-adk-best-practice/photo_2026-03-22%2017.20.22.jpeg)

1. **Start Simple** — เริ่มจาก ReAct pattern ก่อน ค่อย evolve เป็น Multi-Agent เมื่อจำเป็น
2. **Specialization** — ออกแบบ agent ให้ทำหน้าที่เดียว หลีกเลี่ยง "God Agent"
3. **Tooling** — ใช้ robust tools และเขียน docstrings ให้ชัด เพราะ model ใช้ docstrings ในการ reasoning
4. **Control** — ใช้ SequentialAgent สำหรับงานที่ต้องการ predictable workflow และ LlmAgent สำหรับ dynamic reasoning
5. **Safety First** — implement HITL pattern สำหรับ high-stakes actions เสมอ

จากมุม backend engineer session นี้ให้ mental model ที่ดีมากสำหรับการออกแบบ agent system เพราะ patterns เหล่านี้คล้ายกับ design patterns ใน distributed systems ที่เราคุ้นเคยอยู่แล้ว Sequential เหมือน pipeline, Coordinator เหมือน orchestrator, HITL เหมือน approval workflow

![Gemini AI - Your Cute Multi-Tool](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.24.jpeg)

### The Chat Wall Problem

![The Chat Wall Problem](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.25.jpeg)

ปัญหาที่ speaker หยิบมาเปิด session คือสิ่งที่เรียกว่า "Chat Wall Problem" ซึ่งหลายคนน่าจะเคยเจอ

- **Cognitive Overload** — ถามไปนิดเดียว อยากรู้แค่นิดเดียว แต่ AI ตอบมาเป็นหน้า ๆ
- **Low Information Density** — text ไม่ได้ตอบโจทย์เสมอไป บางทีแค่กราฟหรือรูปมาก็เข้าใจเลย
- **Poor Scannability** — ต้องไล่อ่านหา สิ่งที่ต้องการจาก wall of text
- **Contextual Blindness** — AI ไม่รู้ว่า user อยากเห็นข้อมูลในรูปแบบไหน
- **Passive vs. Active UX** — text เป็น passive ที่ user อ่านอย่างเดียว ไม่ได้ interact
- **Text Fatigue** — อ่านมากไปก็เหนื่อย

![The Vision: From Wall of Text to Dynamic UI](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.25%20(1).jpeg)

### GenUI คืออะไร และทำงานอย่างไร

![What is GenUI and How it works](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.26.jpeg)

GenUI เป็น open-source protocol สำหรับ "Speaking UI" โดยมีหลักการสำคัญคือ

- เป็น data ไม่ใช่ executable code
- ไม่ได้ draw pixel แต่ส่ง JSON blueprint กลับมา
- ใช้ JSON map กับ predefined UI Catalog ที่ฝั่ง client
- ใช้ Tool-Calling เพื่อสร้าง UI
- Client re-render UI จาก JSON ที่ AI เลือกมาให้

พูดง่าย ๆ คือแทนที่ AI จะตอบเป็น text ยาว ๆ AI จะเลือก UI component ที่เหมาะกับข้อมูลนั้นแล้วส่ง JSON กลับมาให้ frontend render

### Safety & Boundaries

![Safety & Boundaries](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.27.jpeg)

เรื่อง safety เป็นจุดที่ speaker เน้นมาก

- Declarative data, not code ทำให้ไม่มี code execution risk
- ไม่ใช้ iframes — client render ด้วย UI framework ของตัวเอง
- JSON เดียวกัน render ได้บนทุก platform: web (Lit/Angular/React), mobile (Flutter/SwiftUI/Jetpack Compose), desktop

### A2UI — Agent-to-User Interface

![A2UI: Cross-Platform Power](/assets/images/blogs/build-with-ai-bkk-2026/session5-genui-flutter/photo_2026-03-22%2017.20.28.jpeg)

concept ที่น่าสนใจคือ A2UI (Agent-to-User Interface) ที่เปลี่ยนจาก chat wall เป็น dynamic workflow ทำให้ user สามารถ interact กับข้อมูลผ่าน UI component ที่เหมาะสมได้โดยตรง

แต่ก็ยังมีข้อจำกัด — A2UI ยังต้องการ user consent และฝั่ง frontend ต้องมี predefined component มาในระดับหนึ่ง A2UI เป็นแค่สิ่งที่ return ให้ frontend แต่ frontend ต้องรู้ว่าจะ render อะไรได้บ้าง

ถึงแม้จะมี limitation แต่การเปลี่ยนจาก wall of text เป็น dynamic workflow ก็ทำให้ UX ดีขึ้นมาก user interact กับข้อมูลได้จริง ๆ ไม่ใช่แค่อ่าน
