## Introduction

โพสต์นี้เป็นตัวที่ห้าของซีรีส์ backend system design แล้วนะครับ สูตรเดิมเหมือนเคย: เริ่มจากของที่ทื่อที่สุดเท่าที่มันจะทำงานได้ แล้วก็ปล่อยให้ข้อจำกัดจากหน้างานค่อย ๆ ดันมันให้เข้ารูปเข้ารอยไปเอง

รอบนี้ทิศทางของระบบจะต่างจากโพสต์ก่อน ๆ อยู่บ้างครับ สุดท้ายมันจะแตกออกเป็น **สองระบบในตัวเดียว** โดยมีเป้าหมายตรงข้ามกันอย่างสิ้นเชิง ก่อนจะไปดูการพัฒนาไปทีละสเต็ป ผมขอปูภาพจำลองในหัวกันก่อนสักหน่อย

ลองนึกภาพหน้าแรกของ super-app: มีตารางไอคอน, แถบเมนูด้านล่าง, แบนเนอร์โปรโมชัน ส่วนประกอบที่เราต้องโฟกัสคือ **slot** ครับ หน้าเว็บหรือหน้าแอปจะประกอบด้วย **block** หลายชิ้น ซึ่งแต่ละ block จะมี **slot** อยู่ข้างใน และ slot ก็คือช่องสำหรับแสดงไอคอน, รูปภาพ หรือแบนเนอร์ตัวนึง ตัวโครงร่าง (layout) ว่าจะมี block ไหน อยู่ตรงไหนบ้าง เป็นโครงสร้างที่เราออกแบบไว้ล่วงหน้าแล้ว แต่ตัวที่จะมาอยู่ในแต่ละ slot นี่แหละคือจุดที่น่าสนใจ: เพราะมันต้องหาคำตอบ **แยกรายคน (per user)** ผู้ใช้ A อาจเห็นแบนเนอร์ A ในช่องนี้ ส่วนผู้ใช้ B จะเห็นแบนเนอร์ B ในช่องเดียวกันนี้เอง และหากใครไม่ตรงตามเกณฑ์เลย ระบบก็จะแสดงผลด้วย **default** ของ slot นั้นแทน

งานทั้งหมดสรุปสั้น ๆ ในบรรทัดเดียวคือ: เมื่อส่งข้อมูล `(user, page)` มา ระบบต้องหาข้อสรุปให้ได้ว่าแต่ละ slot บนหน้านั้นจะแสดง asset ตัวไหน หรือต้องใช้ค่า default ฟังดูไม่กี่คำ แต่มันจะซับซ้อนทันทีเมื่อเราเริ่มใส่เรื่องสิทธิ์เข้าถึง (permissions), การแบ่งกลุ่มผู้ใช้ (segmentation), กฎความเหมาะสมในการแสดงผล (eligibility rules) และต้องรับโหลดหลักพัน request ต่อวินาที โดยห้ามปล่อยให้มีหน้าจอว่างเปล่าโผล่มาให้ผู้ใช้เห็นเป็นอันขาด

เส้นทางที่จะเดินต่อจากนี้: ครึ่งแรกจะว่าด้วยการพัฒนาแบบค่อยเป็นค่อยไปที่จะบังคับให้ระบบต้องกลายเป็นแบบเจาะกลุ่มผู้ใช้ ส่วนครึ่งหลังจะพาไปดูระบบที่แยกย่อยออกเป็นสองฝั่ง

การพัฒนาระบบ (Evolution):

1. **Static icons** ฝังไปกับตัว client เลยตั้งแต่แรก
2. เริ่มมีเรื่องสิทธิ์ใช้งานเข้ามา → ต้อง **ซ่อน/แสดงผลแยกตามคน**
3. จัดกลุ่มการแสดงผลแยกตามคนบนแอปไม่ไหว → **ดึงข้อมูลผ่าน API แทน**
4. แสดงผลของไม่เหมือนกันตามประเภทผู้ใช้ → **ทำระบบแบ่งกลุ่ม (segments) และระดับสมาชิก (tiers)**
5. หน้าจอห้ามโล่งเด็ดขาด → **ตรวจสอบสิทธิ์การแสดงผล (eligibility) ร่วมกับ fallback**

จากนั้นแยกออกเป็นสองส่วนที่มี SLA ต่างกันโดยสิ้นเชิง:

- **Front-facing serving** (ระบบฝั่งหน้าร้าน) — รับ RPS สูง, เน้นตอบสนองไว, ห้ามหน้าจอว่างเปล่าเด็ดขาด **เน้น Availability เป็นหลัก**
- **Backoffice** (ระบบหลังบ้าน) — ใช้เฉพาะภายใน, ทราฟฟิกน้อย, เน้นความถูกต้องเป็นหลัก ข้อมูลดีเลย์ไปสักสองสามนาทีก็ไม่มีปัญหา

จุดสำคัญของโพสต์นี้คือความแตกต่างตรงนี้แหละครับ: งานในขอบเขตเดิม แต่ถูกดึงแยกออกเป็นสองบริการที่เราต้องจูนคนละแนวทางเลย เรามาเริ่มจากแบบง่าย ๆ กันก่อน

## เวอร์ชันแรกสุดแบบง่าย ๆ — ฝังไอคอนไว้ในตัวแอปเลย

คนเราต้องเริ่มจากที่ไหนสักแห่งครับ และจุดเริ่มต้นที่ตรงไปตรงมาที่สุดคือ: hardcode ตารางไอคอนและเมนูนำทางลงไปใน client แต่ละตัวตรง ๆ เลย แอปบนมือถือก็ใส่รายชื่อไอคอนแบบนึง หน้าเว็บก็ใส่อีกแบบนึง ไม่ต้องมีระบบหลังบ้านมาเกี่ยว — โครงร่างการแสดงผลถูกมัดรวมไปกับขั้นตอนการสร้างแอปนั่นแหละ

```
// mobile build — hardcoded
val homeIcons = listOf("wallet", "scan", "transfer", "rewards", "topup")
```

สำหรับแอปขนาดเล็ก วิธีนี้ได้ผลดีเลยครับ — ปล่อยขึ้นใช้งานได้เลย ทุกคนเห็นตารางแบบเดียวกัน โหลดแสดงผลทันที และไม่มีอะไรต้องดูแลหลังบ้านเลย

แต่ไม่นานรอยร้าวสองจุดก็เริ่มโผล่มา:

- **ทุกการแก้ไขคือการปล่อยแอปเวอร์ชันใหม่** อยากเปลี่ยนไอคอน สลับลำดับตาราง หรือจะขึ้นแบนเนอร์สัปดาห์หน้า? บนมือถือคุณต้องส่งแอปขึ้น app store (และต้องนั่งรอรีวิว) แถมฝั่งเว็บก็ต้องทำ deploy ใหม่ ทีมการตลาดขยับตัวไวกว่ารอบการปล่อยแอปของคุณเยอะครับ
- **หน้าจอแต่ละ platform เริ่มไม่ตรงกัน (Platform drift)** ฝั่งเว็บกับฝั่งแอปมือถือต่างกุมความถูกต้องของตัวเองไว้คนละชุด สุดท้ายก็หลุดจากกัน — แบนเนอร์โผล่บนเว็บแต่หายไปจากแอปมือถือ หรือไอคอนที่มือถือเอาออกไปแล้วแต่หน้าเว็บยังมีอยู่

## เมื่อระบบสิทธิ์ (Permissions) เข้ามาเกี่ยว — ต้องซ่อนหรือแสดงผลแยกรายคน

**ปัญหา:** สมมุติว่าฟีเจอร์ต่าง ๆ เริ่มมีข้อกำหนดเรื่องสิทธิ์เข้าใช้งาน เช่น เครื่องมือสำหรับร้านค้าต้องเห็นเฉพาะบัญชีร้านค้าเท่านั้น, เมนูขอสินเชื่อแสดงเฉพาะจุดที่เปิดให้กู้ได้ หรือปุ่มลัดของแอดมินที่จำกัดไว้ให้เฉพาะทีมงาน จากตารางไอคอนแบบเดียวใช้ร่วมกันทั้งหมด จู่ ๆ ก็กลายเป็นว่าสิ่งที่ควรขึ้นบนหน้าจอต้องขึ้นอยู่กับว่า *ใครกำลังเปิดดู*

เรา *อาจจะ* ค้างของพวกนี้ไว้ฝั่ง client ก็ได้ ด้วยการส่งไอคอนทุกตัวไป แล้วค่อยไปกดซ่อนอันที่ผู้ใช้ไม่มีสิทธิ์ แต่แบบนั้นจะพังได้สองทางครับ ทางแรกคือคุณกำลังส่งข้อมูลการมีอยู่ของฟีเจอร์ต่าง ๆ ไปให้คนที่ไม่ควรรับรู้ (เปิดแกะโค้ดใน bundle ออกมาก็ส่องเจอหมด) และทางที่สองคือฝั่ง client ต้องมารับรู้เงื่อนไขสิทธิ์ทั้งหมดเพื่อตัดสินใจ ซึ่งแปลว่าแอปบนมือถือหรือเว็บกำลังทำหน้าที่ตัดสินใจเรื่องความปลอดภัย (authorization) ทั้งที่ไม่ใช่งานของตัวเองเลย

**วิธีแก้:** หาข้อสรุปเงื่อนไขการแสดงผลจากฝั่ง server แยกตามคนไปเลย เมื่อผู้ใช้เปิดเข้ามา ให้ไล่เช็ครายการ slot ทั้งหมดใน catalog แล้วเก็บไว้เฉพาะส่วนที่ระดับบัญชี หรือบทบาทหน้าที่ของผู้ใช้รายนั้นได้รับอนุญาต ฝั่ง client ไม่ต้องคิดอะไรต่อครับ มีหน้าที่แค่วาดของตามที่ได้รับมาก็พอ

## เมื่อโครงร่างหน้าจอแปรผันตามผู้ใช้ — จึงต้องดึงข้อมูลผ่าน API

**ปัญหา:** ถึงโครงร่างหน้าจอจะแปรผันตามผู้ใช้แล้ว แต่มันก็ยังถูกประกอบขึ้นบนเครื่องฝั่งผู้ใช้เองอยู่ดี การจะทำแบบนั้นได้แปลว่า client ต้องดาวน์โหลดรายการ catalog ทั้งชุด พร้อมเงื่อนไขกฎเกณฑ์ในการคัดกรองทั้งหมดลงมาที่เครื่อง ซึ่งทั้งช้า เสี่ยงทำให้ข้อมูลที่ควรเป็นความลับรั่วไหล แถมยังไม่สามารถแก้ไขเงื่อนไขใด ๆ ได้เลยหากไม่ยอมปล่อยแอปเวอร์ชันใหม่ ปัญหาเรื่องข้อจำกัดในการอัปเดตของเวอร์ชันแรก กับเรื่องการจัดหน้ารายคนของเวอร์ชันถัดมาเลยเข้าชนกันตรงนี้พอดีครับ

**วิธีแก้:** ย้ายขั้นตอนการหาคำตอบไปไว้หลัง API ครับ ตัว client มีหน้าที่แค่เรียก endpoint ตัวเดียวพร้อมแนบการยืนยันตัวตน (auth) จากนั้นหลังบ้านจะคำนวณสรุปช่องแสดงผล (slot) สำหรับผู้ใช้รายนั้น ๆ ออกมา และส่งกลับมาเป็นหน้าตาสำหรับนำไปวาดบนหน้าจอได้ทันที — ว่าต้องแสดงผลอะไร เรียงลำดับอย่างไร โดยไม่ต้องประมวลผลเพิ่มเลย

```
GET /pages/home          (auth: this user)
->
{
  "blocks": [
    { "id": "shortcuts", "slots": [
      { "id": "s1", "asset": { "icon": "wallet",   "deeplink": "app://wallet" } },
      { "id": "s2", "asset": { "icon": "scan",     "deeplink": "app://scan"   } }
    ]}
  ]
}
```

client จะถูกลดหน้าที่ลงให้ไม่ซับซ้อนโดยตั้งใจ: มีหน้าที่แค่วาด block และใส่ asset ลงใน slot ตามข้อมูลที่ได้รับมา สิ่งที่เราจะได้กลับมาทันทีโดยไม่ต้องลงแรงเพิ่มคือ:

- **แก้ไขข้อมูลโดยไม่ต้องปล่อยแอปใหม่** จะเปลี่ยนรูปไอคอนหรือแบนเนอร์ ก็ทำผ่านการแก้ไขชุดข้อมูลฝั่ง backend ได้เลย ไม่ต้องคอยกดส่งแอปขึ้น store
- **เว็บกับแอปบนมือถือไม่หลุดจากกันอีกต่อไป** เพราะมีจุดหาคำตอบ (resolver) เดียวกัน และใช้ฐานข้อมูลร่วมกัน ตัว API จะส่งคำตอบที่จัดรูปมาให้เหมาะกับแต่ละแพลตฟอร์มเอง (แอปมือถือได้แบบตาราง หน้าเว็บได้แบบเมนูตามหน้างาน)
- **กฎเกณฑ์และรายการ catalog ทั้งหมดจะอยู่บน server** ตัว client จะไม่มีโอกาสส่องเห็นข้อมูลหรือเงื่อนไขของส่วนที่ไม่มีสิทธิ์เข้าใช้งาน

นี่คือจุดเริ่มต้นของ API จัดส่งข้อมูลครับ และเนื้อหาที่เหลือทั้งหมดในโพสต์นี้จะต่อยอดมาจากจุดนี้: ทุกสเต็ปถัดจากนี้จะเข้ามาช่วยทำให้การเรียก `(user, page) → layout` ทำงานได้ฉลาดขึ้น เร็วขึ้น หรือล่มได้ยากขึ้น

## แสดงของต่างกันตามผู้ใช้ — ระบบแบ่งกลุ่ม (Segments) และระดับสมาชิก (Tiers)

**ปัญหา:** สิทธิ์เข้าใช้งานตอบได้เพียงแค่ ใช่/ไม่ใช่ — ผู้ใช้คนนี้มีสิทธิ์มองเห็น slot นี้ได้รึเปล่า? แต่ทางทีมธุรกิจต้องการมากกว่านั้น ช่องเดียวกัน ตำแหน่งเดียวกันบนหน้าจอ ควรแสดงผล **asset ต่างตัวกัน** ตามเงื่อนไขของแต่ละคน สมมุติว่าเป็นช่องแบนเนอร์ช่องเดิม — ผู้ใช้ระดับ Gold ได้รับแบนเนอร์ของรางวัลสุดพิเศษ, ผู้ใช้ใหม่เอี่ยมเจอข้อแนะนำตอนเริ่มต้นใช้งาน หรือผู้ใช้เก่าที่เงียบหายไปเห็นโปรโมชันดึงความสนใจกลับมา ช่องเดิมเป๊ะ แต่เนื้อหาไม่เหมือนกันตามกลุ่มคนดู

**วิธีแก้:** เปลี่ยนระบบประมวลผลให้เป็นการค้นหาจับคู่ (lookup) แทนที่จะเป็นแค่การกรองขอบเขต (filter) โดยแต่ละ slot จะมีรายการ asset หลายตัวที่เป็นตัวเลือก และตัวเลือกเหล่านั้นจะถูกติดป้ายกำกับไว้ว่าทำมาเพื่อเป้าหมายกลุ่มไหน — เช่น ระดับสมาชิก, ค่าคุณสมบัติของผู้ใช้ หรือแคมเปญที่กำลังทำอยู่ เมื่อมี request เข้ามา ระบบจะเลือกจับคู่ asset ที่มีเงื่อนไขตรงกับผู้ใช้คนนั้นขึ้นมาแสดงผล

ประเด็นที่ควรออกแบบให้รัดกุม:

- **เงื่อนไขกลุ่มเป้าหมาย (Targeting) ต้องอยู่ในรูปข้อมูล ไม่ใช่โค้ด** เช่น เงื่อนไข 'เป็นสมาชิกระดับ Gold, อยู่ในไทย, เพิ่งสมัครไม่เกิน 30 วัน' จะแนบอยู่กับตัว asset และนำมาคำนวณเทียบกับตัวแปรของผู้ใช้ตอนที่เรียกหา ทำให้ทีมธุรกิจสามารถสร้างกลุ่มเป้าหมายใหม่ได้เรื่อย ๆ โดยฝั่งผู้พัฒนาไม่ต้องทำ deploy ใหม่
- **ลำดับความสำคัญ (Priority) สำคัญมากเมื่อตรวจเจอมากกว่าหนึ่งเงื่อนไข** ผู้ใช้คนหนึ่งอาจตรงกับเงื่อนไขหลายกลุ่มในเวลาเดียวกัน คุณเลยต้องมีข้อกำหนดการเรียงลำดับผลลัพธ์ที่แน่นอน (เช่น เงื่อนไขที่เฉพาะเจาะจงที่สุดชนะ หรือระบุเลขอันดับลงไปตรง ๆ) มิฉะนั้น ตัว slot อาจดึงข้อมูลแถวที่ระบบฐานข้อมูลสุ่มคืนค่ามาตัวแรกสุด ซึ่งหน้าจอก็จะเปลี่ยนไปเรื่อย ๆ ทุกครั้งที่กดโหลด
- **แคมเปญก็คือ asset ที่มีเวลากำหนดปิดเปิด** มันคือเงื่อนไขกลุ่มเป้าหมายบวกกับกำหนดเวลาเริ่มต้นและสิ้นสุดของกิจกรรม การหาแบนเนอร์ที่เหมาะสมที่สุดให้ผู้ใช้แต่ละคนจึงเปลี่ยนเป็น: การสืบค้นหา asset ลำดับความสำคัญสูงสุด ที่ตรงกับเงื่อนไขของผู้ใช้คนนั้น และมีเงื่อนไขวันเวลาครอบคลุมอยู่ ณ วินาทีนั้นพอดี

## หน้าจอห้ามแหว่งเป็นรู — ตรวจสอบเกณฑ์แสดงผลและแผนสำรอง (Fallback)

**ปัญหา:** เมื่อทุก slot ต้องมาคอยหาคำสั่งจับคู่แบบนี้ ย่อมมีผู้ใช้จำนวนมากที่ไม่ตรงเงื่อนไขตัวเลือกใดเลย — และมีโอกาสสูงที่บาง request จะทำงานไม่สำเร็จระหว่างประมวลผล เช่น ตัวเช็คเกณฑ์แสดงผลใช้เวลาเกินเวลาที่กำหนด (timeout), ระบบเช็คกลุ่มเป้าหมายสะดุด หรือแคมเปญหมดเวลาไปก่อนหน้านั้นเพียงเสี้ยววินาที เมื่อหน้าร้านต้องรับโหลดพันครั้งต่อวินาที ความผิดพลาดพวกนี้จะกลายเป็นรอยโหว่บนตารางเมนู — เป็น slot ที่โล่งไปเฉย ๆ หรือร้ายแรงที่สุดคือหน้าจอไม่ขึ้นอะไรเลย ซึ่งนั่นคือสิ่งที่คุณห้ามปล่อยให้เกิดขึ้นเป็นอันขาด

**วิธีแก้:** ทุก slot ต้องผูก asset แบบ **default** ไว้ตัวหนึ่งเสมอ และหุ้มกระบวนการประมวลผลเอาไว้ให้มันต้องส่งของบางอย่างกลับมาเสมอ แยกเป็นสองกรณี แต่ใช้คำตอบเดียวกัน:

- **ไม่ตรงเกณฑ์ (Ineligible)** — ผู้ใช้ไม่ตรงกับตัวเลือกกลุ่มเป้าหมายใดเลย → ให้แสดงผลด้วย asset ตัว default ของ slot นั้น
- **ระบบทำงานผิดพลาด (Timeout / ระบบอื่นล่ม)** — หาคำตอบไม่ทันตามเวลาที่จำกัดไว้ → ให้แสดงผลด้วย asset ตัว default แทนทันที

ข้อแม้เล็ก ๆ คือ ค่า default เองก็ตั้งให้เป็น **ค่าว่าง (blank)** ได้เหมือนกันครับ — บาง slot ถ้า default คือว่างเปล่า ช่องนั้นก็จะถูกปล่อยให้ว่าง (ยุบช่องทิ้งไป) อย่างตั้งใจ จุดนี้ต่างจากรูโหว่ที่เกิดจากความผิดพลาดนะครับ เพราะมันคือช่องว่างที่เราเลือกออกแบบเอง ไม่ใช่ของหล่นหายระหว่างทาง

## แยกส่วนออกแบบ — สองระบบที่มีเป้าหมายต่างกันคนละขั้ว

สิ่งที่เล่ามาทั้งหมด ดูเหมือนเกิดขึ้นในการเรียก `(user, page) → layout` เพียงชั้นเดียว แต่งานที่ซ่อนอยู่ข้างในกลับแบ่งออกเป็นสองฝั่งที่ต่างกันอย่างสิ้นเชิง: **ฝั่งจัดส่งแสดงผล** (สืบค้นบ่อย, ถี่มาก, เกิดขึ้นในทุก request) กับ **ฝั่งจัดการ** asset และเงื่อนไขต่าง ๆ (เขียนนาน ๆ ครั้ง, งานหลังบ้าน) สองเรื่องนี้มีความต้องการที่วิ่งสวนทางกัน เราจึงควรสร้างแบบแยกเป็นสองระบบ แล้วเชื่อมต่อกันด้วยจุดเชื่อมต่อที่บางที่สุดเท่าที่จะทำได้

## ระบบ A — Front-facing serving (ฝั่งประมวลผลข้อมูลขาอ่าน)

นี่คือฝั่งที่ต้องคอยแบกรับทราฟฟิกหลักพัน RPS ดังนั้น การออกแบบทั้งหมดจึงเน้นไปที่กฎข้อเดียวคือ: **ดึงคำตอบจาก Memory เท่านั้น และต้องไม่มีวันโดนบล็อกด้วยกระบวนการทำงานที่ช้า**

**อ่านคำตอบจากแคช แทนที่จะมานั่งคำนวณสด** การหาข้อสรุปโครงร่างหน้าจอให้ผู้ใช้แต่ละรายต้องดึงข้อมูลทั้งจากคลังเงื่อนไขกฎเกณฑ์, ข้อมูลวันเวลาแคมเปญ และรายการ asset ทั้งหมด ซึ่งมีขั้นตอนเยอะเกินกว่าจะทำในแต่ละ request ด้วยโหลดระดับนี้ งานหนัก ๆ อย่างการประเมินกฎเกณฑ์และการจัดเรียงลำดับตัวเลือกจึงถูกทำ **ล่วงหน้าก่อนจะมี request เข้ามา** แล้วเก็บผลไว้ในแคช ส่วนตอนรับ request เข้ามาก็แค่หยิบคำตอบสุดท้ายจากของที่เตรียมไว้แบบเบา ๆ

**การหาคำตอบของ slot คือ เจอตัวแรกที่เข้าเกณฑ์ก็จบ** ใน slot นึงไม่ได้เก็บ asset ไว้แค่ตัวเดียวครับ แต่เก็บเป็น *list ของตัวเลือก (candidates)* แล้วการหาคำตอบก็คือการไล่ลงมาตาม list สั้น ๆ แค่นั้นเอง:

- ตัวเลือกถูกเรียงตามลำดับความสำคัญ (priority) ไว้ล่วงหน้า แล้วแคชไว้ในลำดับนั้นเลย
- พอมี request เข้ามา serving API จะไล่ list จากบนลงล่าง แล้วหยุดทันทีที่เจอตัวเลือกตัวแรกที่ผู้ใช้คนนั้นเข้าเกณฑ์ — ไม่ต้องไล่ให้ครบทั้ง list ไม่ต้องมานั่งให้คะแนนหาตัวที่ดีที่สุด
- ตัวอย่าง: ผู้ใช้คนนึงเข้าเกณฑ์ campaign 1, 2 และ 5 ส่วน slot นั้นมีตัวเลือกเรียงไว้เป็น `[banner ของ campaign 2, banner ของ campaign 5, default]` → loop ก็จะหยิบ banner ของ campaign 2 ตั้งแต่ตัวแรกที่เข้าเกณฑ์ แล้วเลิกหาต่อทันที

**แล้วที่ว่า 'ผู้ใช้คนนี้เข้าเกณฑ์มั้ย' รู้ได้จากไหน** ตัว serving cache เก็บไว้แค่ตัวเลือก ไม่ได้เก็บคำตัดสินว่าใครเข้าเกณฑ์ครับ:

- แคชเก็บ asset ที่เรียงลำดับไว้ของแต่ละ slot พร้อมข้อมูลว่าแต่ละตัวผูกกับ campaign ไหน — แต่มันไม่ได้เป็นคนตัดสินเรื่องสิทธิ์การเข้าเกณฑ์
- ส่วนนั้นมาจาก Audience domain (ขอติดไว้เป็นโพสต์ของตัวเองในซีรีส์นี้นะครับ): มี rule engine คอยประเมินเงื่อนไขกลุ่มเป้าหมาย แล้วเขียนผลลง membership store ที่ใช้ userId เป็น key — จากที่ผมเคยทำมาก็คือ Redis ที่เก็บเป็น set ของ campaign ที่ผู้ใช้คนนั้นเข้าเกณฑ์อยู่ ณ ตอนนั้น เช่น `10001 -> {1, 2, 3, 4, 5}`
- เพราะงั้นการหาคำตอบของ slot จึงเหลือแค่อ่าน memory **สองที่**: ดึง list ตัวเลือกจาก serving cache กับดึง set ของ campaign ที่ผู้ใช้เข้าเกณฑ์จาก membership store สมมุติ banner A ผูกกับ campaign 1 — มันก็เข้าเกณฑ์ของผู้ใช้ 10001 เพราะเลข 1 อยู่ใน set ของเขา

**ใช้ Worker คอยอุ่นแคช (Cache Warm) แบบ Asynchronous** เราจะใช้ worker ตัวอื่นแยกต่างหาก — อาจสร้างเป็นหน่วยประมวลผลเดียว หรือแยกกันทำตามกลุ่มหน้าเพจ/กลุ่มผู้ใช้ — คอยไปอ่านแหล่งข้อมูลหลัก (source of truth) แล้วนำมาประกอบเป็นชุดคำตอบเก็บไว้บนแคช: ประมวลผลแยกตามกลุ่ม (segment) เท่าที่ทำได้ หรือเจาะรายคนกรณีที่เลี่ยงไม่ได้ ฝั่งจัดส่งหน้าร้านและฝั่งคอยอุ่นข้อมูลจะไม่มีวันมาแชร์ทราฟฟิก request เดียวกัน ฝั่งอุ่นข้อมูลทำงานอยู่หลังบ้าน ส่วนหน้าร้านแค่อ่านค่าจาก memory เท่านั้น

**เขียนทับแคชเสมอ ห้ามลบทิ้ง (Overwrite, never delete)** ตอนที่ตัวอุ่นแคชสร้างชุดคำตอบขึ้นมาใหม่ ขอให้มีกติกาเดียวคือ เขียนทับได้ก็ต่อเมื่อสร้างสำเร็จเท่านั้นครับ ถ้ารอบนั้นประมวลผลพลาด ตรวจสอบไม่ผ่าน หรือโหลดข้อมูลตั้งต้นไม่ได้ ให้ทิ้งค่าเดิมที่อยู่ในแคชไว้แบบนั้นเลย ข้อมูลเก่าที่ยังใช้งานได้ก็ยังเสิร์ฟให้ผู้ใช้ต่อไปได้อยู่ แต่ถ้าเผลอลบคีย์นั้นทิ้ง หน้าจอจะกลายเป็นช่องว่างทันที (หรือดันทราฟฟิกทั้งก้อนไปถล่มเส้นทางประมวลผลที่ช้าพร้อมกัน) แคชฝั่งให้บริการจะขยับไปข้างหน้าได้ด้วยการเขียนทับเท่านั้น — ไม่มีคำว่าลบทิ้งอยู่ในตัวเลือกเลยครับ

**ล็อกจังหวะ deploy ให้รอจนแคชอุ่นเสร็จ (Gate the deploy)** นี่คือรายละเอียดจุดเล็ก ๆ ที่ผมเห็นหลายคนมักข้ามไป: เมื่อเราสั่งเปิดเครื่องเซิร์ฟเวอร์ตัวใหม่เพื่อเตรียมสลับมาใช้งาน มันจะต้องโหลดข้อมูลใส่แคชของตัวเอง **ให้เสร็จสิ้นเสียก่อน** จึงจะแจ้งระบบว่าพร้อมรับทราฟฟิก หากเครื่องที่ยังเตรียมแคชไม่เสร็จเริ่มรับทราฟฟิก มันจะปล่อยให้ผู้ใช้เห็นหน้าจอว่างเปล่า (หรือหลุดไปเรียกเส้นทางประมวลผลที่ช้า) ในช่วงนาทีแรก ๆ ซึ่งมักจะเกิดขึ้นจังหวะที่กำลังปล่อยแอป (rollout) พอดี ซึ่งนั่นคือจังหวะที่แย่ที่สุด ดังนั้น ความพร้อมของระบบ (readiness) จึงนิยามด้วยการ 'โหลดแคชเสร็จสมบูรณ์' นี่คือการแลกเปลี่ยนตรงไปตรงมา: จังหวะทำ deploy จะช้าลง และถ้าข้อมูลเยอะมาก การเปิดเครื่องรอบแรก (cold start) จะใช้เวลานานขึ้น — แก้ไขด้วยการดึงข้อมูลจาก **snapshot** ที่บันทึกไว้ล่าสุด แทนที่จะประมวลผลใหม่ทั้งหมดจากศูนย์ทุกครั้งที่เปิดเครื่อง

**ถอยกลับมาใช้ Default เสมอเมื่อมีข้อสงสัย** ต่อให้อ่านค่าจากแคช ก็ยังมีโอกาสที่ข้อมูลจะไม่ตรงกับคีย์ที่มี — เช่น ผู้ใช้รายใหม่เอี่ยมที่ประมวลผลไม่ทัน หรือช่องแสดงผลที่เพิ่งถูกเพิ่มเข้ามาใหม่ เส้นทางขาอ่านจึงต้องจบลงด้วยกฎความปลอดภัยเดิม: หากหาไม่เจอหรือเกิดความผิดพลาดใด ๆ → ให้ดึงข้อมูล default ของ slot นั้นมาแสดงผล ทันที ตัวแคชช่วยเพิ่มความเร็วในการตอบสนอง ส่วนค่า default จะช่วยรับประกันความปลอดภัยให้หน้าจอไม่มีวันพัง

**การเลือกจุดเก็บแคชเป็นประเด็นที่ต้องตัดสินใจให้ดี:**

- **เก็บไว้บนหน่วยความจำของแต่ละเครื่อง (In-memory, per instance)** — อ่านข้อมูลได้รวดเร็วที่สุดเพราะไม่ต้องกระโดดข้ามเครือข่าย แต่แลกกับการที่เซิร์ฟเวอร์ทุกเครื่องต้องกุมข้อมูลไว้ทั้งเซ็ต และการทำ deploy ทุกครั้งต้องเริ่มอุ่นของกันใหม่หมด ก่อนจะเลือกทางนี้ ลองกะคร่าว ๆ แบบ back-of-the-envelope ดูก่อนครับ: สิ่งที่เก็บคือ JSON ที่ประมวลผลแล้ว — ลิงก์ CDN กับ metadata ของ slot ไม่ใช่ตัวไฟล์รูป — เพราะงั้นคิดเป็น (จำนวนชุดคำตอบที่แคชไว้) × (ขนาด JSON ต่อหนึ่ง slot) หนึ่ง slot อาจหนักไม่กี่ร้อยไบต์ ทั้งหน้าที่มีสัก 12 slot ก็แค่ไม่กี่ KB ถ้าแคชแบบรายกลุ่ม (segment) ตัวเลขจะเล็กนิดเดียว แต่พอแคชแบบรายคน มันจะกลายเป็น users × pages ซึ่งจุดนั้นแหละที่เริ่มยัดลงแรมไม่ไหว วิธีนี้จะเหมาะเมื่อข้อมูลคำตอบไม่ใหญ่เกินกว่าจะยัดลงในแรม และคุณสามารถจัดระบบดึงค่าจาก snapshot เพื่อช่วยให้เครื่องเปิดใช้งานได้เร็วขึ้น
- **ใช้ระบบแคชส่วนกลางร่วมกัน (Shared cache เช่น Redis / Memcached)** — ข้อมูลอุ่นเสร็จจะอยู่ตรงกลางตัวเดียวให้เซิร์ฟเวอร์หลายเครื่องมาเปิดอ่านร่วมกัน ทำ deploy ใหม่ไม่ต้องนั่งอุ่นข้อมูลซ้ำ แต่แลกกับการมี overhead ในการยิงข้ามระบบเครือข่ายในทุกครั้งที่อ่าน และระบบแคชตัวนี้จะกลายเป็นจุดเชื่อมต่อสำคัญที่ต้องดูแลอย่าให้ล่มเด็ดขาด

## ระบบ B — Backoffice (ฝั่งจัดการบันทึกข้อมูลและตั้งค่า)

นี่คือจุดที่ทีมงานใช้ในการทำหน้าที่ควบคุมระบบ: สร้าง asset, กำหนดกลุ่มเป้าหมาย, จัดแคมเปญลงแต่ละ slot และกำหนดช่วงเวลาแสดงผล ระบบนี้ใช้กันเฉพาะภายใน, ทราฟฟิกน้อย และ — ข้อดีอันสำคัญคือ — **อนุญาตให้ข้อมูลทยอยอัปเดตตามรอบได้ (eventually consistent)** การแก้ไขตัวแปรต่าง ๆ ไม่จำเป็นต้องแสดงผลบนหน้าจอผู้ใช้ในเสี้ยววินาทีนั้นทันที รอดีเลย์ไปสักไม่กี่นาทีก็ได้ ความยืดหยุ่นจุดนี้แหละที่ช่วยรักษาโครงสร้างฝั่งหลังบ้านให้เรียบง่ายและไม่ซับซ้อนเกินจำเป็น

**หน้าจอจัดการภายในที่คุยกับฐานข้อมูลหลักโดยตรง** ทีมงานจะบริหารจัดการระบบไอคอน / แบนเนอร์ / แคมเปญต่าง ๆ แล้วนำมาจับคู่กับกลุ่มผู้ใช้, ตำแหน่งจัดวาง (ว่าจะให้อยู่ slot ไหน) รวมถึงตารางเวลา มันคือฟังก์ชันจัดการข้อมูลพื้นฐาน (CRUD) ทั่วไปที่คุยตรงกับ rule + asset store — ไม่จำเป็นต้องใช้โครงสร้างระบบพื้นฐานที่ซับซ้อนหรือล้ำยุค ความถูกต้องและการตรวจสอบค่าที่บันทึก (validation) ให้รัดกุมคือสิ่งที่ต้องใส่ใจมากกว่าเรื่องระยะเวลาตอบสนอง (latency) หรือการทำงานโดยไม่มีล่ม (uptime)

**รูปแบบการนำ asset เข้าระบบ — สามแนวทาง** ตัว asset คือข้อมูลภาพบวกกับคำอธิบายประกอบ (metadata) ซึ่งสามารถนำเข้าระบบได้หลายแนวทางดังนี้:

- **จัดการผ่านระบบอัปโหลด (Managed upload)** — ทีมงานอัปโหลดไฟล์เข้าระบบ → ระบบตรวจสอบไฟล์ (validate ทั้งนามสกุลไฟล์, ขนาดไฟล์ และสัดส่วน) → นำไฟล์ไปเก็บใน object storage → แสดงผลผ่าน CDN โดยตัว URL ของ CDN จะสืบค้นขึ้นมาในแบบ **asynchronous** (หน้าจอจัดการจึงอาจขึ้นว่ากำลังประมวลผลสักพัก แทนที่จะปล่อยให้แสดงลิงก์เสีย)
- **นำลิงก์ที่มีอยู่แล้วมาวางตรง ๆ (Bring-your-own URL, direct)** — ทีมงานระบุลิงก์จากแหล่งเก็บของตัวเอง แล้วระบบจะบันทึกตามนั้นทันที วิธีนี้รวดเร็วและประหยัดแรงพัฒนา แต่คุณจะแบกรับปัญหาของแหล่งอื่นมาด้วย: ลิงก์เสีย, โฮสต์ปลายทางทำงานช้า หรือภาพแอบหายไปดื้อ ๆ ซึ่งสุดท้ายแล้วรอยแหว่งบนหน้าจอนั้นจะกลายเป็นปัญหาที่เกิดบนระบบของคุณเอง
- **วางลิงก์ส่วนตัวแต่ให้ระบบคัดลอกไฟล์เข้ามาเก็บ (Bring-your-own URL, ingested)** — ทีมงานระบุลิงก์ภาพเข้ามา แต่ระบบหลังบ้านจะช่วยดาวน์โหลดไฟล์ภาพนั้น นำมาอัปโหลดขึ้น object storage ของเราเอง และให้บริการจัดส่งผ่าน CDN ของเราเองแทน จากนั้นระบบเราจะเป็นผู้ดูแลความพร้อมใช้งานและแคชของไฟล์นั้นอย่างสมบูรณ์ โฮสต์ภายนอกจะมีบทบาทแค่รอบเดียวตอนนำเข้าเท่านั้น

ทางเลือกตรงกลางคือกับดักแสนหวานครับ — ทำเสร็จส่งมอบงานง่ายที่สุด แต่ก็จะเป็นจุดแรกสุดที่ย้อนกลับมาปลุกให้คุณต้องตื่นมาแก้ปัญหาระบบล่ม การเลือกว่าจะใช้แนวทางไหนคือการตัดสินใจที่ต้องไตร่ตรองให้ถี่ถ้วน

**การกดปล่อยข้อมูล (Publishing) = การล้างค่าหรืออัปเดตแคช** เมื่อทีมงานกดบันทึก ข้อมูลใหม่จะถูกบันทึกลงฐานข้อมูลหลัก — และนั่นคืองานทั้งหมดของฝั่งหลังบ้าน จากนั้นระบบจะส่งสัญญาณไปหาฝั่งประมวลผลเพื่อบริการหน้าร้าน (ให้ล้างคีย์แคชที่มีผลกระทบ หรือกระตุ้นให้หน่วยอุ่นแคชทำคำนวณใหม่) ข้อมูลใหม่ก็จะไปแสดงผลบนหน้าจอผู้ใช้ในเวลาต่อมา คำว่า 'คอยอีกสักพัก' คือหัวใจของ SLA ระดับ 10 นาทีที่ไม่ต้องรีบร้อน ฝั่งหลังบ้านไม่จำเป็นต้องทนทานสูงมากหรือตอบสนองไวสูงสุด เพราะมันไม่ได้อยู่ในเส้นทางการประมวลผลหลักของหน้าร้านนั่นเอง

แถมอีกนิดแบบสนุก ๆ ครับ: บางคนอาจอยากให้ระบบนี้ consistency สูง ๆ ไปเลย — พอ operator กดเซฟปุ๊บ แคชต้องอัปเดตตามทันทีแบบการันตีว่าสดใหม่ ผมว่าทางนี้ไม่ค่อยเข้าท่าเท่าไหร่ครับ เพราะมันลากเอาฝั่งหลังบ้านที่สบาย ๆ กลับเข้าไปอยู่ใต้ SLA อันโหดของฝั่งหน้าร้าน แล้วก็มัดสองระบบที่เราอุตส่าห์แยกกันมาทั้งโพสต์ให้กลับมาผูกกันใหม่ ลำพังตัวแคชเองก็เหนื่อยพอแล้วครับ ไม่ต้องไปจับมันมัดรวมกับ distributed transaction เพิ่มอีก ปล่อยให้การรีเฟรชแบบ async กับค่า default ทำงานไป — แค่ eventually consistent ก็เหลือเฟือสำหรับงานนี้แล้ว 😅

ดังนั้น ระบบหลังบ้านจึงถูกออกแบบมาให้เป็นส่วนที่จืดชืดธรรมดาอย่างตั้งใจ: มีแค่หน้าจอสำหรับป้อนค่า, การตรวจสอบความถูกต้องเบื้องต้น, บันทึกลงฐานข้อมูล และส่งสัญญาณไปล้างแคช ความจืดชืดเรียบง่ายนี้แหละคือปริมาณงานวิศวกรรมที่เหมาะสมที่สุดสำหรับมันแล้ว

## จุดเชื่อมต่อ (The connection point) — พื้นที่ที่สองฝั่งมาประจบกัน

สองระบบที่ทำงานแยกกัน แต่ไม่ได้ตัดขาดโดยสิ้นเชิง — มีจุดเชื่อมต่ออยู่เพียงเส้นเดียว และการคุมให้มีเพียงเส้นเดียวนี้เองคือจุดที่ทำให้ระบบแยกส่วนกันได้อย่างสะอาดตา

จุดเชื่อมต่อที่ว่าคือ: **หลังบ้านบันทึกฐานข้อมูลหลัก → ส่งสัญญาณเตือน (ให้ล้างคีย์หรือให้เริ่มคำนวณใหม่) → หน่วยอุ่นข้อมูลเริ่มรีเฟรชแคชแสดงผล** นี่คือการคุยกันทั้งหมดที่มี ฝั่งหลังบ้านจะไม่มีวันยิงเรียกใช้ API หน้าร้าน และฝั่งหน้าร้านก็จะไม่มีวันมาอ่านข้อมูลจากฐานข้อมูลของหลังบ้านตรง ๆ สองฝั่งนี้จะประจบกันผ่านฐานข้อมูลหลักและตัวแคชที่วางคั่นอยู่ตรงกลางเท่านั้น

ประเด็นสองข้อที่ช่วยให้การบริหารจัดการระบบแนวทางนี้ง่ายขึ้น:

- **ทำงานแบบ asynchronous และทนทานต่อข้อมูลตกหล่น** สัญญาณเตือนสามารถส่งผ่านระบบคิว, การดึง CDC stream หรือปล่อยให้ตัวอุ่นข้อมูลมาดึงตามรอบเวลาก็ได้ ต่อให้สัญญาณเตือนหลุดหายไปหนึ่งรอบ รอบทำงานถัดไปก็จะสแกนมาพบการอัปเดตในที่สุด — เพราะฝั่งหน้าร้านมีค่า default ที่เตรียมไว้คอยรองรับได้อยู่แล้ว SLA ระดับ 10 นาทีที่เปิดขอบเขตไว้กว้างจึงเป็นจุดช่วยให้ระบบคงความเรียบง่ายเอาไว้ได้
- **ฐานข้อมูลหลักทำหน้าที่เป็นข้อตกลงร่วมกัน (Contract)** ตราบใดที่ทั้งสองฝั่งตกลงเรื่องรูปแบบโครงร่างข้อมูลที่ประมวลผลแล้วตรงกัน คุณจะปรับปรุงระบบใหม่, ทำ re-shard หรือทำ redeploy ฝั่งใดฝั่งหนึ่งได้อิสระโดยไม่กระทบต่ออีกฝั่งเลย

เป็นบทเรียนเดียวกับเรื่อง 'ห้ามทำข้อมูลหล่นหาย' ในโพสต์ webhook ครับ — คือจำกัดการผูกกันให้เหลือเพียงจุดเชื่อมต่อเดียวที่เข้าใจตรงกันอย่างรัดกุม แล้วแต่ละฝั่งจะได้รับความอิสระในการปรับแต่ง จูนประสิทธิภาพ หรือแม้แต่การเกิดปัญหาล่มและการขยายสเกลโดยไม่ไปส่งผลกระทบต่อกัน

## ส่งท้าย

เราเริ่มต้นจากตารางไอคอนแบบ hardcode ลงบน client และเดินทางมาถึงระบบสองฝั่งที่แชร์ข้อมูลผ่านแคชร่วมกัน สรุปเส้นทางการพัฒนาตามลำดับ:

1. **ฝังไอคอนตรง ๆ บน Client (Static icons)** — เวิร์กดีจนกระทั่งทุกแก้ไขกลายเป็นเรื่องต้องปล่อยแอปเวอร์ชันใหม่
2. **ระบบสิทธิ์ (Permissions)** บังคับให้โครงสร้างการแสดงผลแปรผันตามตัวผู้ใช้ — เป็นแรงขับสำคัญที่ทำให้เกิดการเปลี่ยนแปลง
3. **โครงสร้างแบบเฉพาะตัวอยู่บน client ไม่ไหว** → ย้ายมาดึงข้อมูลผ่าน API
4. **ระบบแบ่งกลุ่มและระดับสมาชิก (Segments + tiers)** ทำให้ slot เปลี่ยนสถานะเป็นการค้นหาจับคู่กลุ่มเป้าหมาย
5. **ตรวจสอบสิทธิ์และแผนสำรอง (Eligibility + fallback)** — ทุก slot มีค่า default เสมอ ทำให้หน้าจอไม่มีวันโล่งหรือแหว่ง
6. **แยกเป็นสองระบบการทำงาน** — ฝั่งให้บริการแบบ cache-first (อุ่นแคชเสร็จพร้อมสรรพก่อนส่งทราฟฟิกเข้า) และระบบจัดการหลังบ้านที่มีระดับ SLA ไม่ตึงเครียด เชื่อมกันด้วยจุดประสานงานแบบ async เส้นทางเดียว

ประเด็นภาพกว้างซึ่งเป็นหัวใจสำคัญของซีรีส์นี้: **ไม่มีใครออกแบบเริ่มต้นที่ระบบแบบนี้ครับ** ถ้าแอปคุณยังมีอันเดียว ตารางแสดงผลไม่มีความซับซ้อน? hardcode ไปเลยแล้วเอาเวลาไปทำอย่างอื่นดีกว่า แต่ละขั้นตอนที่เพิ่มเติมขึ้นมาจะทำงานเพื่อแก้ไขปัญหาที่เกิดขึ้นในขั้นตอนก่อนหน้า — เรื่องสิทธิ์ใช้งานบีบให้สร้าง API, ขนาดผู้ใช้ที่โตขึ้นบีบให้สร้างแคช และเงื่อนไข SLA ที่ตรงข้ามกันก็บีบให้ต้องแยกสองฝั่งออกจากกัน

หากมีประเด็นสำคัญข้อเดียวที่อยากฝากไว้ในโพสต์นี้คือ: ฝั่งบริการหน้าร้านกับระบบหลังบ้าน *ดูเหมือน* เป็นเรื่องเดียวกันในมุมของธุรกิจ แต่มันเป็นปัญหาทางวิศวกรรมคนละหัวข้อกันอย่างสิ้นเชิง — และชัยชนะที่ยิ่งใหญ่คือการยินยอมให้ข้อกำหนดการออกแบบของสองฝั่งนี้แยกจากกันได้ ฝั่งขาอ่านต้องมุ่งเน้นเรื่อง latency ต่ำและต้องห้ามมีหน้าจอแหว่งเด็ดขาด ส่วนฝั่งหลังบ้านสามารถทำงานได้แบบไม่เร่งรีบ เน้นข้อมูลถูกต้อง และอนุญาตให้อัปเดตล่าช้ากว่าได้ไม่กี่นาที การไปวางมาตรฐานความเข้มงวดของ SLA ไว้ในระดับเดียวกันทั้งสองฝั่ง จะจบลงด้วยการสร้างระบบฝั่งหนึ่งที่ซับซ้อนเกินควร ในขณะที่อีกฝั่งกลับได้รับบริการที่แย่กว่าที่ควรจะเป็น
