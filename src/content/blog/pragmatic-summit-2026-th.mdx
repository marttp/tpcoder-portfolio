## Chip Huyen: Building when it feels like there's nothing left to build

![Building when it feels like there's nothing left to build](/assets/images/blogs/the-pragmatic-summit-2026/Building%20when%20it%20feels%20like%20there's%20nothing%20left%20to%20build.jpg)

Chip Huyen พูดถึงความรู้สึกที่ engineer หลายคนเจอในยุค AI คือ AI ทำให้สร้างอะไรก็ได้ง่ายขึ้น แต่คนอื่นก็ clone สิ่งที่เราสร้างได้ง่ายพอกัน เธอเล่าว่าเคยปล่อย side project ได้ 300,000 views ใน 1 สัปดาห์ แล้ววันถัดมามีคนส่ง email มาบอกว่า clone ได้หมดแล้วด้วย Claude

คำถามคือ ถ้า software อะไรก็ตามที่ describe ได้ AI ก็ generate ได้ แล้วจะสร้างอะไรดี?

สิ่งที่ Chip แนะนำคือมองไปที่ long-tail ของปัญหา foundation model อย่าง OpenAI โฟกัสที่ปัญหา generic ของคนส่วนใหญ่ โอกาสของ engineer อยู่ที่ปัญหาเฉพาะทางที่ big tech ไม่สนใจ เหมือนกับว่าอย่าไปแข่งกับ hedge fund ในตลาดใหญ่ แต่หา sweet spot ที่ใหญ่พอจะทำกำไรแต่เล็กพอที่ big tech จะไม่มายุ่ง

ตัวอย่างที่น่าสนใจคือเรื่อง cultural nuance เช่น ในเวียดนามคนขับมอเตอร์ไซค์เยอะจึงต้องใช้ voice bot แทน text หรือเรื่อง awkward silence ที่คน US คาดหวัง response ใน 80ms แต่ฝั่ง Asia รอได้ 200-300ms ถ้า engineer ไม่เข้าใจ nuance พวกนี้ UX ก็พัง

อีกเรื่องที่น่าคิดคือ เราเอา AI มายัดใส่ workflow เดิมแทนที่จะออกแบบใหม่ ทำไมยังแยก terminal กับ IDE ในเมื่อทั้งคู่รับ instruction เหมือนกัน? หรือเรื่อง code review ที่ senior ยังนั่ง review AI-generated code ทีละบรรทัด ทั้งที่ควร mentor junior ให้เขียน prompt ที่ดีกว่า และเรื่องการสร้าง environment ที่พร้อมสำหรับ agent เพราะ website กับ API ปัจจุบันออกแบบมาสำหรับคน ไม่ใช่สำหรับ agent-to-agent

เรื่อง cultural nuance ผมเห็นด้วยมาก ลองนึกถึงอาหารตามสั่งในไทย ไม่มีทาง generic agent ตัวไหนจะเข้าใจรสชาติของแต่ละคนได้ บาง culture อาจคล้ายญี่ปุ่นที่เน้นความสุภาพ แต่คนไทย freestyle มาก nuance แบบนี้ AI จับไม่ได้ง่าย ๆ

ส่วนเรื่อง "nothing left to build" ผมรู้สึกตรงกับ Chip เลย เหตุผลหลักที่ยังไม่เปิดบริษัทเองเพราะสิ่งที่อยากสร้าง คนอื่นสร้างไปแล้ว big tech ก็ครอบคลุมหมด แต่พอคิดถึงธุรกิจของพ่อที่เป็นช่างแอร์ ไม่มีทาง big tech จะกระโดดมาทำตรงนี้ แม้ในไทยจะมี platform จ้างงานช่างอยู่แล้ว แต่ก็ไม่ได้เฉพาะเจาะจงสำหรับช่างแอร์โดยตรง

## Nicole Forsgren: Leading high-performing engineering teams in the age of AI

![Leading high-performing engineering teams in the age of AI](/assets/images/blogs/the-pragmatic-summit-2026/Leading%20high-performing%20engineering%20teams%20in%20the%20age%20of%20AI.jpg)

Nicole Forsgren (ผู้เขียน Frictionless และเป็นคนเบื้องหลัง DORA, SPACE, DevEx metrics) พูดถึง impact ของ AI ต่อ engineering velocity

ประเด็นหลักคือ AI ช่วยเขียน code ได้เร็วขึ้นมาก (inner loop) แต่กลับทำให้ bottleneck ใน outer loop เช่น code review, security check, release management ที่ยังพึ่งคน New joiner ใช้ AI commit code ได้ตั้งแต่วันแรก แต่ถ้ารอ database access หรือ human review อีก 2 สัปดาห์ ความเร็วที่ได้จาก AI ก็ถูก friction กลืนหมด

เรื่อง DevEx ก็น่าสนใจ Forsgren แบ่งเป็น 3 อัน คือ Flow State, Cognitive Load, Feedback Loops ปัญหาคือ AI agent ให้ feedback เร็วจนคน rebuild mental model หลายสิบรอบต่อชั่วโมง ซึ่งกลับเพิ่ม cognitive load แทน คนเรา focus แบบ deep work ได้แค่ 3-4 ชั่วโมงต่อวัน ถ้า AI interrupt ตลอด flow ก็พัง

สำหรับเรื่อง productivity ถ้าถามว่า เราทำงานได้ productive ขึ้นไหม? ต้องถามกลับว่า productive ในมิติไหน จำนวน PR หรือ lines of code ไม่มีความหมายถ้าไม่ได้ ship feature ที่ถูกต้อง

SPACE framework (Satisfaction, Performance, Activity, Communication, Efficiency) ช่วยให้มองรอบด้านขึ้น และถ้า developer adopt AI tool ด้วยตัวเอง นั่นเป็น signal ที่ดีว่า tool มันสามารถใช้ได้

สุดท้าย engineering leader ต้องให้ permission ชัดเจนว่าทีมทดลอง AI ได้โดยไม่ต้องกลัว และถ้าจะให้ agent ทำงาน outer loop ได้ ระบบต้อง observable ก่อน ไม่มี telemetry ที่ดี agent ก็ self-correct ไม่ได้

## Leading Product Engineering Teams in an AI-native World

![Product-minded engineers in an AI-native world](/assets/images/blogs/the-pragmatic-summit-2026/Product-minded%20engineers%20in%20an%20AI-native%20world.jpg)

Panel session พูดถึง product-minded engineer ในยุคที่ AI เขียน code ได้ ทักษะที่มีค่าที่สุดไม่ใช่ how to build แต่คือ what to build และ why

Product-minded engineer คือคนที่สนใจ user impact มากกว่า technical elegance ใน startup คนแบบนี้เป็น force multiplier เพราะทำได้ตั้งแต่ define problem ไปจนถึง deploy และ test กับ user แม้แต่ infra engineer ก็เป็น product-minded ได้ เพราะ API หรือ module ก็คือ product ที่มี user เป็น developer คนอื่น

เรื่อง taste เป็นทักษะที่ฝึกได้ผ่านการ expose ตัวเองกับ product คุณภาพดีในหลาย domain - ตอนสัมภาษณ์ คนที่เป็น product engineer จะถามเรื่อง user motivation ก่อน ไม่ใช่กระโดดไป database schema ทันที

สิ่งที่น่าสนใจคือ ritual ของ Linear อย่าง Quality Wednesdays ที่ engineer ต้องหา defect เล็ก ๆ มาแก้ทุกสัปดาห์ ไม่ใช่ bug ใหญ่ แต่เป็นเรื่อง polish เช่น hover state ที่หาย หรือ animation ที่ไม่ smooth ส่วน Stripe ใช้วิธีให้ engineer เขียน developer flow เพื่อบังคับให้ลองใช้ API ของตัวเองจากมุม external developer

ในยุค AI workflow เปลี่ยนไปมาก engineer ปล่อย 3-4 AI agent แก้ minor bug ไปพร้อมกับทำ feature หลัก Designer และ PM เริ่มเขียน PR แล้ว ship UI tweak ไป preview environment ได้เลย เส้นแบ่งระหว่าง role เริ่มค่อย ๆ หายไป

สุดท้าย สำหรับ IC แล้วไม่มีอะไรทดแทนการคุยกับ customer ได้

AI สรุป call ให้ได้ แต่ empathy ต้องสร้างจากการเจอด้วยตัวเอง

สำหรับ leader ต้องให้เวลาทีม polish งาน quality วัดด้วย A/B test ระยะสั้นไม่ได้ แต่ถ้าไม่ให้ priority ระยะยาว product จะเสื่อมแน่

## Simon Willison: Engineering practices that make coding agents work

![Engineering practices that make coding agents work](/assets/images/blogs/the-pragmatic-summit-2026/Engineering%20practices%20that%20make%20coding%20agents%20work.jpg)

Simon Willison (co-creator of Django) แชร์วิธีใช้ AI coding agent แบบ practical ตอนนี้เขาเขียน code บนมือถือมากกว่า laptop แล้ว โดยใช้ agent ship feature ได้ในไม่กี่นาที

เรื่อง trust เปลี่ยนไปมาก จากให้ AI แนะนำ กลายเป็น AI เขียน code เกือบทั้งหมด จนถึงจุดที่ไม่มีคนเขียนและไม่มีคนอ่าน code วิธีคิดคือให้มอง AI agent เหมือนทีม engineer อีกทีมที่เรา verify output ด้วย test ไม่ใช่ด้วยการอ่านทีละบรรทัด

วิธีที่ Simon ใช้ verify คือ Red/Green TDD บังคับให้ agent เขียน failing test ก่อน แล้วสร้าง code ที่ทำให้ test ผ่าน ป้องกันไม่ให้ AI สร้าง feature ที่ไม่จำเป็น นอกจากนี้ยังให้ agent boot server แล้วยิง curl เพื่อพิสูจน์ว่า API ทำงานได้ ไม่ใช่แค่ test ผ่าน

เรื่อง code quality ถ้า agent ให้ spaghetti code 2,000 บรรทัดแล้วเรารับ นั่นคือ choice ของเรา สามารถ prompt ให้ refactor ได้ และถ้าตั้ง repo template ที่มี CI/CD กับ test คุณภาพดีไว้ agent จะเลียนแบบคุณภาพนั้น

เรื่อง security Simon มี framework ชื่อ [Lethal Trifecta](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/) สำหรับ prompt injection ต้องมี 3 ขาครบถึงจะอันตราย คือ access private data, เจอ malicious instruction จากภายนอก และมีช่องทาง exfiltrate data ออก ทางแก้คือ sandboxing ให้ agent ทำงานใน isolated container เพื่อจำกัด blast radius

สุดท้าย barrier ในการเรียนภาษาใหม่หายไปแล้ว Simon ship 3 project ด้วย Go โดยไม่เคย fluent แต่ปัญหาใหม่คือ open source ถูก AI คุกคาม ทั้ง library ที่ถูกแทนด้วย AI-generated component และ maintainer ที่โดน low-quality AI PR ท่วม

Key takeaways:

- TDD
- Code quality เป็น choice ของเรา สั่ง refactor ได้เสมอ
- ถ้า repo มีคุณภาพดี AI จะให้ output คุณภาพดีตาม แม้จะเป็น model ธรรมดา

## Uber: Leading engineering through an agentic shift

![Leading engineering through an agentic shift](/assets/images/blogs/the-pragmatic-summit-2026/Leading%20engineering%20through%20an%20agentic%20shift.jpg)

เป้าหมายของ Uber ไม่ใช่ให้ AI มาแทน engineer แต่ให้ engineer ทำงานได้เหมือน "superhuman" โดยโยนงาน toil (dead code cleanup, migration, library upgrade) ให้ AI แล้วคนโฟกัสกับงานที่สร้าง business value

การเปลี่ยนแปลงเกิดเป็น 2 phase คือ pair programming (2022-2023) ที่ใช้ GitHub Copilot แบบ sync auto-complete ให้ velocity เพิ่ม 10-15% กับ peer programming (ปัจจุบัน) ที่ developer ทำตัวเป็น Tech Lead ส่งงานให้ AI agent แบบ async แล้วกลับมา review ทีหลัง พอเปิดให้ใช้ 70% ของงานที่ส่งมาคือ toil เพราะงานพวกนี้มี start/end state ชัดเจน AI ทำได้แม่นยำ สร้าง trust loop ที่ดี

![uber-additional-img-1](/assets/images/blogs/the-pragmatic-summit-2026/uber-additional-img-1.jpg)

### Infrastructure ที่ Uber สร้างเอง

Uber ไม่ได้ใช้ tool สำเร็จรูป แต่สร้าง infrastructure ทั้ง stack เอง เริ่มจาก Model Gateway ผ่าน [Michelangelo](https://www.uber.com/gb/en/blog/from-predictive-to-generative-ai/) (ML platform เดิมของ Uber) สำหรับ proxy ไปยัง external model อย่าง OpenAI, Anthropic พร้อมรักษาความปลอดภัยของ internal infrastructure

จากนั้นสร้าง MCP gateway และ registry สำหรับเปิด internal tools (Jira, Slack, source code) ให้ agent เข้าถึงได้อย่างปลอดภัย มี AIFX เป็น CLI กลางสำหรับ provision และจัดการ agent clients ทั่วทั้งองค์กร

ที่สำคัญที่สุดคือ Minions ซึ่งเป็น background agent platform ที่รันใน CI pipeline ของ Uber เอง ไม่ใช่บน infrastructure ของ vendor ภายนอก developer แค่วาง error log ผ่าน web UI หรือ Slack แล้ว Minions จะ checkout monorepo, รัน Claude, แก้ไข แล้วส่ง co-authored PR กลับมาภายในไม่กี่นาที

![uber-additional-img-2](/assets/images/blogs/the-pragmatic-summit-2026/uber-additional-img-2.jpg)

### จัดการ Code Sprawl

![uber-additional-img-3](/assets/images/blogs/the-pragmatic-summit-2026/uber-additional-img-3.jpg)

Agent สร้าง code เร็วกว่าที่คนจะ review ทัน Uber แก้ด้วยหลาย tool คือ Code Inbox ที่ route PR ตาม ownership, time zone และ calendar พร้อมคำนวณ risk profile ของแต่ละ change, U-Review เป็น AI code review ที่กรอง nit comment ออกเหลือแค่ feedback ที่มีค่าจริง ๆ, Auto-Cover ที่ generate unit test ได้ ~5,000 merged tests ต่อเดือนด้วยคุณภาพ 3 เท่าของ generic agent มี Critic Engine คอยตรวจว่า test ไม่ใช่แค่ change detector และ Shepherd สำหรับจัดการ migration ขนาดใหญ่ เช่น upgrade Java version โดยแตก PR เป็นร้อย ๆ ตัวแล้วให้ agent จัดการ lifecycle ทั้งหมด

![uber-additional-img-4](/assets/images/blogs/the-pragmatic-summit-2026/uber-additional-img-4.jpg)

### ความท้าทายที่ไม่ใช่เรื่อง tech

ฝั่ง culture การให้คนในองค์กรเปลี่ยนมาใช้ AI ช้ากว่าที่คิด สั่งจากข้างบนไม่ work สิ่งที่ได้ผลคือให้ engineer ที่ใช้แล้วมาโชว์ให้เพื่อนร่วมทีมเห็นเองว่า ship งานได้เร็วขึ้นแค่ไหน

เรื่อง ROI developer satisfaction และ code output สูงขึ้นมาก แต่ CFO ต้องการเห็น revenue impact ไม่ใช่แค่ activity metrics Uber กำลังวัด "Time to Launch" สำหรับ product experiment ใหม่เพื่อพิสูจน์ financial ROI

ส่วน cost AI API ขึ้น 6x ตั้งแต่ 2024 ต้องขอ CFO approve แล้ว Uber แก้โดยใช้ model แพงสำหรับ plan และ model ถูกสำหรับ execute

สิ่งที่น่าสนใจอีกอย่างคือ Loyalty Dilemma ตลาด AI เปลี่ยนทุกสัปดาห์ Uber จึงสร้าง architecture แบบ abstract อย่าง Minions เพื่อ swap model หรือแทน internal tool ด้วย SaaS ที่ดีกว่าได้ทันทีโดยไม่ผูกติด vendor

## Ramp: Lessons from Building a New AI Product

![Lessons from Building a New AI Product](/assets/images/blogs/the-pragmatic-summit-2026/Lessons%20from%20Building%20a%20New%20AI%20Product.jpg)

Ramp เล่าประสบการณ์สร้าง AI agent สำหรับ automate งาน corporate finance ตอนแรก Ramp สร้าง micro-agent แยกกันเป็นพัน ๆ ตัว แต่ละทีมสร้างเอง สุดท้ายก็ pivot มาเป็น agent ตัวเดียว (Omnihat) ที่มี 1,000 skills แทน แนวคิดเปลี่ยนจาก software ที่บอกว่าต้องทำอะไร เป็นระบบที่ react, reason และ act เองโดยไม่ต้องรอคน

![ramp-1](/assets/images/blogs/the-pragmatic-summit-2026/ramp-1.jpg)

ตัวอย่างที่ลงลึกคือ Policy Agent ที่วิเคราะห์รูป receipt แล้ว cross-reference กับ expense policy ของบริษัทเพื่อ approve/reject อัตโนมัติ สิ่งที่เรียนรู้คือ model ไม่ค่อยพลาดเรื่อง reasoning แต่พลาดเพราะขาด context เช่น ไม่รู้ว่าพนักงานคนนี้ระดับไหน limit เท่าไหร่ และเรื่อง ground truth ก็สำคัญ ตอนแรกใช้ user behavior เป็น ground truth แต่พบว่า user เองก็ approve ผิดบ่อย สุดท้ายต้องตั้ง cross-functional labeling session ขึ้นมาสร้าง dataset เอง

![ramp-2](/assets/images/blogs/the-pragmatic-summit-2026/ramp-2.jpg)

ฝั่ง infrastructure Ramp สร้าง Applied AI Service เป็น proxy layer สำหรับ standardize API call และ swap model ได้ด้วย config บรรทัดเดียว มี Tool Catalog กลางหลายร้อย tool และสร้าง Ramp Inspect ซึ่งเป็น internal coding agent (open-sourced เป็น Open Inspect) ที่รันใน secure sandbox มี access ถึง test suite 150k+ ตัว และ integrate กับ DataDog, Slack, Notion ตอนนี้ Ramp Inspect รับผิดชอบ PR ที่ merge เข้า production มากกว่า 50% และไม่ใช่แค่ engineer ใช้ แต่ทีม product, design, marketing ก็ใช้แก้ copy และ logic ง่าย ๆ ได้

![ramp-3](/assets/images/blogs/the-pragmatic-summit-2026/ramp-3.jpg)

เรื่อง culture Ramp แบ่ง engineer เป็น 2 แบบ Team A คือคนที่กล้าเผชิญปัญหาที่ยังไม่ชัดเจน เข้าใจธุรกิจ เปิดรับ tool ใหม่ และใส่ใจประสบการณ์ผู้ใช้ ส่วน Team B คือคนที่ชอบเถียงเรื่อง library จุกจิกรายละเอียดเล็ก ๆ แล้วลงมือสร้างก่อนเข้าใจปัญหา AI พิสูจน์แล้วว่าการเขียน code ไม่เคยเป็นส่วนยากที่สุด สิ่งที่มีค่าจริง ๆ คือวิจารณญาณ ความเข้าใจบริบท และการพา Project ฝ่าช่วงที่วุ่นวายไปให้ได้

## Frameworks for reinventing software, again and again

![Frameworks for reinventing software, again and again](/assets/images/blogs/the-pragmatic-summit-2026/Frameworks%20for%20reinventing%20software,%20again%20and%20again.jpg)

Session นี้เป็นบทสนทนาระหว่างสองตำนานของวงการ Software Engineering ที่ย้อนมองประสบการณ์ 25 ปีหลัง Agile Manifesto แล้วมองไปข้างหน้าว่า AI กำลังเปลี่ยนอะไรบ้าง

**TDD ถูกพิสูจน์แล้วในยุค AI** — สิ่งที่เคยเป็นเรื่องถกเถียงกันมา 20 ปี กลายเป็นทักษะที่จำเป็นมาก เพราะเมื่อเรามี powerful genie เขียนโค้ดให้ สิ่งที่ขาดไม่ได้คือความสามารถในการ verify ว่ามันทำถูก ซึ่งเป็นสิ่งที่คนทำ TDD ฝึกมาตลอด

**AI ไม่เหมือนการเปลี่ยนแปลงใด ๆ ในอดีต** — ไม่ว่าจะ OOP หรือ Internet ไม่มีอะไรมาด้วย magnitude และความเร็วแบบนี้ แต่สิ่งที่ใกล้เคียงที่สุดคือการเกิดของ microprocessor (Intel 4004) ที่เปลี่ยนคอมพิวเตอร์จากอันใหญ่ราคาแพง ให้กลายเป็นสิ่งที่อยู่ได้ทุกบ้าน และในปัจจุบันคือทุกที่ — AI ก็กำลังสร้าง **expansion of imagination** แบบเดียวกัน

**Skepticism ต้องมาคู่กับ Curiosity** — ถ้าลอง AI แล้วได้ผลแย่แล้วตัดสินทันที อาจพลาดสิ่งสำคัญ ต้อง skeptical about your own skepticism ด้วย คำถามที่สำคัญคือ _"What's the smallest experiment I can run to verify this?"_

**Agile Industrial Complex กำลังเกิดซ้ำ** — สมัยก่อน Agile เริ่มต้นจากแนวคิดดี ๆ แต่สุดท้ายกลายเป็นธุรกิจขาย certification และ consulting ที่ไม่ได้ช่วยอะไรจริง ตอนนี้ AI ก็กำลังเข้าสู่วงจรเดียวกัน มีคนขาย tool และ solution ที่โฆษณาเกินจริงเต็มไปหมด ต้องรู้จักแยกแยะว่าอะไรคือของจริงที่ใช้งานได้ อะไรคือแค่ hype

**Junior จะเติบโตเร็วขึ้น แต่ mid-level อาจถูกบีบ** — AI เป็น amplifier ที่ทำให้คนเรียนรู้เร็วยิ่งเร็วขึ้น และคนเก่งยิ่งทำงานได้มากขึ้น แต่กลุ่มกลางที่เข้ามาเพราะรายได้ดีอาจเจอแรงกดดันคล้าย dot-com crash

**อย่ากลับไปทำงานคนเดียว (re-siloing)** — จุดแข็งของ XP คือการทำให้ engineer พูดคุยกันเยอะขึ้น ทำงานเป็นทีมจริง ๆ ผ่านการ pair programming แต่ตอนนี้มีแนวคิดว่า developer คนเดียวสั่ง AI agents 10 ตัวก็เท่ากับมีทีมแล้ว ซึ่งจริง ๆ แล้วมันไม่เหมือนกัน เพราะการคุยกับคนจริงที่มีมุมมองต่างกันยังสำคัญอยู่ — รูปแบบที่ได้ผลดีกว่าคือ คนสองคน pair กัน แล้วใช้ AI agent เป็นตัวช่วย

**Craft ไม่ได้ตาย แค่ขยับขึ้นไปอีกชั้น** — ความสุขจากการ perfect function เดียวอาจหมดไป แต่ความสุขใหม่คือการเข้าใจ domain ทั้งระบบ และ Venn diagram ระหว่าง Developer Experience กับ Agent Experience คือ **วงกลมเดียวกัน** — โค้ดที่ดีสำหรับคนก็ดีสำหรับ AI เช่นกัน

## Lessons from building Vercel v0 and the d0 agent

![Lessons from building Vercel v0 and the d0 agent](/assets/images/blogs/the-pragmatic-summit-2026/Lessons%20from%20building%20Vercel%20v0%20and%20the%20d0%20agent.jpg)

Malte Ubl (CTO ของ Vercel) เล่าเรื่อง AI tooling ทั้ง internal และ external ของ Vercel

เรื่องแรกคือ D0 ซึ่งเป็น internal data agent สำหรับ Text-to-SQL ที่ query Snowflake ตอนแรกสร้างแบบ tools-in-a-loop ซับซ้อนมาก ใช้ได้แต่มันก็พังง่ายด้วย สุดท้ายตัดสินใจลบทิ้งหมดแล้วทำใหม่ โดยแค่ extract business semantics ของทุก column ใน Snowflake เป็น YAML ภาษาอังกฤษธรรมดา ตอนนี้ D0 เหลือแค่ไม่กี่บรรทัด กับ tool แค่ 2 ตัว (Bash กับ SQL execute) แล้วให้ LLM reasoning จาก YAML context เอง

เรื่อง v0 ตั้งใจทำให้ frontend developer แต่กลับได้ traction จาก backend engineer ก่อน เพราะคนกลุ่มนี้ต้องการ UI แต่มี technical depth พอจะแก้ข้อผิดพลาดของ model ได้ breakthrough สำคัญคือการสั่งให้ model ใช้ Tailwind CSS แทนการ generate HTML กับ CSS แยกกัน พอ styling อยู่ inline เป็น class model ก็ reasoning ได้ดีขึ้นทันที พอ model อย่าง Claude 3.5 Sonnet มา v0 ก็ขยับจาก UI prototyping เป็น full-stack app generation

เรื่อง deployment Vercel ใช้แนวคิด Optimistic Locking คือ engineer คนไหนก็ ship ได้โดยไม่ต้องรอ approval แค่ broadcast intent ให้ทีม Legal, Security รู้ ทีมพวกนี้มี power ถ้ามีปัญหา แทนที่จะเป็น gatekeeper ที่ต้อง approve ทุกอย่าง เพื่อรองรับ velocity นี้ Vercel แยก state เป็น 20 region อิสระ deploy เป็น wave ถ้าพังก็พังแค่ region เดียว rollback ได้เร็ว

สุดท้ายเรื่องที่น่าคิดคือ code ที่ AI สร้างมานั้น คือได้มาฟรีแต่ยังต้อง maintain อยู่ดี เรากำลังเข้าสู่ยุคที่ software จะเกิดขึ้นมหาศาล แต่ภาระ maintenance ก็จะตามมา engineering leader ต้องเตรียมใช้ agent สำหรับ software lifecycle management ไม่ใช่แค่สร้างมันขึ้นมาเฉย ๆ

## Building world-class engineering teams in the age of AI

![Building world-class engineering teams in the age of AI](/assets/images/blogs/the-pragmatic-summit-2026/Building%20world-class%20engineering%20teams%20in%20the%20age%20of%20AI.jpg)

Panel ระหว่าง Rajeev (CTO, Atlassian) กับ Thomas Dohmke (CEO, GitHub) พูดถึงการสร้างทีม AI-native

Rajeev บอกว่า Atlassian มีทีมที่ไม่เขียน code เองเลยสำหรับ greenfield project แต่ Thomas ก็ชี้ว่าสำหรับ process อย่าง HR หรือ month-end close ยังไม่สามารถ fully adopt ได้ สิ่งที่ต้องฝึกคือบังคับตัวเองไม่อ่าน code ทีละบรรทัด แต่ express intent แล้วให้ agent จัดการ เรื่องที่ยังไม่มีคนพูดถึงมากคือ language barrier พอเปลี่ยนมาใช้ English prompt ทีมที่ไม่ใช่ native English speaker จะเจอ friction เพิ่ม

เพราะ implementation เริ่มเข้าใกล้ zero cost bottleneck จึง shift left ไปที่ planning, spec และ shift right ไปที่ CI/CD, deployment, incident Atlassian สร้าง agent ชื่อ [Rovo Dev](https://www.atlassian.com/software/rovo-dev) (ใช้ Anthropic model) ที่ได้เปรียบเพราะมี Teamwork Graph รู้ว่าใครทำงานกับใคร Jira ticket ล่าสุดคืออะไร ส่วน role ก็เริ่มรวมกัน PM กลายเป็น product engineer ที่สร้าง prototype เองใน Replit designer กลายเป็น design engineer

สำหรับ engineering leader CTO กลับมาเขียน code ได้อีกครั้งเพราะ agent ลด friction ของการ setup environment สัดส่วน manager ต่อ engineer จะเปลี่ยนจาก 1:5-8 เป็น 1:30-50 เพราะ AI จัดการงาน admin ได้ เส้นทาง Junior → Senior → Staff → Manager แบบเดิมกำลังหายไป คนที่ยังลงมือทำและ master agent orchestration จะได้เปรียบ

ฝั่งตัวเลข Atlassian เห็น PR ต่อ engineer เพิ่ม 89% cycle time ลด 42% และ agent จับ security vulnerability ได้ 51% แต่ปัญหาใหม่คือ token cost ที่เป็น variable OPEX engineer ที่ productive มากอาจเผา token เป็นหลาย ๆ ดอลลาร์ leader ต้องหาวิธี fund โดยไม่ไป cap velocity

## OpenAI: How AI is reshaping the craft of building software

![How AI is reshaping the craft of building software](/assets/images/blogs/the-pragmatic-summit-2026/How%20AI%20is%20reshaping%20the%20craft%20of%20building%20software.jpg)

Vijaye กับ Tibo จาก OpenAI เล่าว่า workflow ภายในเปลี่ยนไปมากแค่ไหน ภายใน 6 เดือน AI tooling วิวัฒนาการจาก tool → IDE extension → agent → teammate ถึงขั้นที่ engineer ตั้งชื่อให้ agent ของตัวเอง

OpenAI deploy Codex Box ให้ engineer จอง server instance ส่ง complex prompt และปิด laptop ไปประชุม แล้วกลับมาเจอ feature เสร็จ - Top engineer เผา token หลายแสนล้านต่อสัปดาห์ มีคนที่มาสมัครงานถามว่า ผมจะได้ compute เท่าไหร่สำหรับสร้าง product? ซึ่งเมื่อก่อนมีแต่ researcher ที่ถามแบบนี้

แทนที่จะเขียน design doc เถียงกันเป็นวัน ตอนนี้สั่ง agent สร้าง implementation หลายแบบพร้อมกัน test ทุกตัว แล้วเลือกตัวที่ดีที่สุด bottleneck ขยับจากการเขียน code ไปที่ code review แล้วจะขยับต่อไปที่ CI/CD ทีม Codex ยังปล่อย agent รัน QA loop ข้ามคืนหา regression โดยไม่ต้องมีคนสั่ง และในประชุมถ้ามีคำถามที่ไม่มีใน dashboard ก็ยิง Codex thread ไป query data แล้วได้คำตอบก่อนประชุมจบ

Tibo ดูแล 33 direct reports แบบ flat org คน onboard ใหม่ใช้ Codex เป็นตัวอธิบาย codebase ให้เลย OpenAI จ้าง intern กับ new grad เยอะเพราะคนรุ่นนี้ไม่มีนิสัยเก่าที่ต้องทิ้ง adopt agentic workflow ได้ทันที มี new grad คนหนึ่งกลายเป็นคนที่ productive ที่สุดในทีมภายใน 6 เดือน แต่ foundation อย่าง architecture กับ product intuition ยังเป็นสิ่งที่ AI แทนไม่ได้

เรื่อง cost Vijaye แนะนำว่าอย่าคิดเป็น API cost แต่ให้คิดเป็นเงินเดือนของ teammate ที่ทำงาน 24/7 ซึ่งถูกมากเทียบกับคน ส่วน prediction ใน 6-24 เดือน จะเห็น multi-agent network ทำงานร่วมกัน codebase จะใหญ่จนคนหยุดอ่าน code แล้วทำงานผ่าน constraint, input/output กับ guardrail แทน และจะมี "Chief of Staff" agent ตัวเดียวที่คุม sub-agent ทั้งหมดแทนการจัดการ agent ร้อยตัว

## Data vs Hype: How Orgs Actually Win with AI

![Data vs Hype: How Orgs Actually Win with AI](/assets/images/blogs/the-pragmatic-summit-2026/Data%20vs%20Hype:%20How%20Orgs%20Actually%20Win%20with%20AI.jpg)

ใช้ Space Race เป็น analogy การไปดวงจันทร์ไม่ใช่เพื่อไปอยู่บนนั้น แต่เพื่อเอา technology อย่าง barcode, space blanket กลับมาใช้บนโลก เหมือนกับการทดลอง AI สุดขั้วอย่าง autonomous multi-agent swarm ที่จะให้ practical tool กลับมาใช้ในองค์กร

### ตัวเลข Q1 2026

![data-vs-hype-1](/assets/images/blogs/the-pragmatic-summit-2026/data-vs-hype-1.jpg)

- 92.6% ของ developer ใช้ AI coding assistant อย่างน้อยเดือนละครั้ง, 75% ใช้ทุกสัปดาห์
- Time savings ค้างอยู่ที่ ~4 ชั่วโมงต่อสัปดาห์ (productivity เพิ่ม ~10%) ไม่ขยับมา 3 quarter แล้ว
- Code ที่ AI เขียนแล้ว merge เข้า production โดยคนแทบไม่แก้ ขึ้นมาอยู่ที่ 26.9% คนที่ใช้ทุกวันเกิน 30%
- AI ลด onboarding time (วัดจาก Time to 10th PR) ลงครึ่งหนึ่ง และคนที่ onboard เร็วยัง maintain productivity สูงกว่าตลอด 2 ปีแรก

![data-vs-hype-2](/assets/images/blogs/the-pragmatic-summit-2026/data-vs-hype-2.jpg)

### AI amplify สิ่งที่มีอยู่

ประเด็นสำคัญคือ AI ไม่ได้แก้ระบบที่พัง แต่ amplify สิ่งที่มีอยู่ data แสดงให้เห็นว่าบาง company incident ลด 50% แต่บางที่ incident เพิ่ม 2 เท่า เพราะ AI ช่วย ship code ที่มีปัญหาได้เร็วขึ้น ถ้าองค์กรพัง AI ก็ช่วยพังเร็วขึ้น

MIT study ชี้ว่าใช้ AI แค่ระดับ individual coding ได้ productivity ceiling ต่ำมาก ต้อง apply กับ organizational workflow ถึงจะ transform ได้ ในกลุ่มบริษัทที่ advanced แล้ว 80% ของ developer ใช้ agentic workflow ทุกสัปดาห์ และ 50% ใช้ทุกวัน

![data-vs-hype-3](/assets/images/blogs/the-pragmatic-summit-2026/data-vs-hype-3.jpg)

### Case Study

- **Cisco** มี 18,000 engineer ใช้ Codex ทุกวัน ลด code review time 50%
- **JP Morgan Chase** สร้าง multi-agent framework (MAFA) ที่ agent หลายตัว annotate, rank และ validate data มี consensus algorithm สำหรับแก้ conflict ระหว่าง agent
- **Haven Headache & Migraine Center** (healthcare startup) ใช้ agentic workflow สร้าง patient portal train HIPAA-compliant model บน symptom logs ได้ customer satisfaction เพิ่ม 3x พร้อม measurable clinical outcomes

![data-vs-hype-4](/assets/images/blogs/the-pragmatic-summit-2026/data-vs-hype-4.jpg)

### สิ่งที่ต้องทำ for the win

- **Set goals and measure your progress** หยุด spray and pray ไม่ใช่แค่แจก license แล้วหวัง ROI ต้องชี้ AI ไปที่ปัญหาเฉพาะแล้ววัดผลด้วย adoption, impact และ cost
- **DevEx matters more than ever** DevEx เป็น prerequisite ของ AI ถ้าอยากได้ budget แก้ tech debt ให้เปลี่ยนชื่อเป็น "Agent Experience" เพราะ agent ต้องการสิ่งเดียวกับคน: CI/CD ที่เร็ว, documentation ที่ดี, architecture ที่สะอาด AI ที่เร็วแค่ไหนก็ข้ามไม่ได้ถ้า meeting culture แย่หรือ deployment pipeline พังง่าย หรือ flaky

![data-vs-hype-5](/assets/images/blogs/the-pragmatic-summit-2026/data-vs-hype-5.jpg)

- **Treat AI as an organizational problem**
- **Experiment by solving real customer problems** โฟกัสที่ปัญหาของลูกค้า ไม่ใช่ทดลอง AI ไปเรื่อย ๆ

Framework ที่แนะนำ: DORA AI Capabilities Model และ Thoughtworks FOREST Framework สำหรับ audit ความพร้อมขององค์กร

## สิ่งที่ผมได้จากการดู

- สิ่งที่ช้าตอนนี้คือ review, testing, deployment และ organizational friction เกือบทุก session พูดตรงกันว่าการเขียน code ไม่ใช่คอขวดอีกต่อไป

- AI amplify ไม่ได้ fix ถ้าองค์กรดีอยู่แล้ว AI จะช่วยทำให้ดีขึ้น ถ้าองค์กรพัง AI ช่วยพังเร็วขึ้น DevEx ที่ดีเป็น prerequisite ไม่ใช่ nice-to-have สำหรับยุคที่ใช้กันแต่ AI

- TDD เป็น safety net สำหรับ verify output ของ AI

- ทุก session ชี้ไปทางเดียวกันว่า judgment, context และ product sense สำคัญกว่าความสามารถในการเขียน code

- ระวังเรื่อง Cost ให้มาก ๆ - AI API cost ที่พุ่งขึ้น token budget ที่กลายเป็น variable OPEX นี่ไม่ใช่แค่เรื่อง tech แต่เป็นเรื่องเงิน ๆ ทอง ๆ ที่ถ้าไม่มีเงินจ่ายก็จบ 55555
