## Introduction

โพสต์นี้เป็นตัวที่สี่ของซีรีส์ backend system design — สูตรเดิมเหมือนเคย: เริ่มจากของที่ทื่อที่สุดเท่าที่มัน work ได้ แล้วก็ปล่อยให้ข้อจำกัดหน้างานค่อย ๆ ดันมันให้เข้ารูปเข้ารอย

คราวก่อนใน[โพสต์ payment](/blog/payment-backend-stripe-integration-th/) ผมอยู่ฝั่ง **คนรับ** webhook — Stripe ยิง event มา *หาเรา* แล้วเราก็สร้าง receiver โง่ ๆ ที่เอา event ไปพักไว้บน queue ก่อนแล้วค่อยประมวลผลแบบปลอดภัย โพสต์นี้ขอสลับฝั่งกันบ้าง: คราวนี้ **เราเป็นคนส่ง** มีอะไรเกิดขึ้นในระบบเรา แล้วเราต้องส่ง callback ไปหา **endpoint ของคนอื่น — ตัวที่เราคุมไม่ได้ มองไม่เห็น แล้วก็เชื่อใจอะไรไม่ได้เลยตั้งแต่แรก** คราวที่แล้วเรารับ คราวนี้เราส่ง

ขอปูพื้นเร็ว ๆ เพราะมันคือเหตุผลทั้งหมดที่ webhook มีอยู่: ก่อนจะมีมัน พาร์ตเนอร์ที่อยากรู้ว่า `order นี้จ่ายเงินรึยัง` ต้องมานั่ง **poll** เรา — ยิง `GET` มาทุก ๆ ไม่กี่นาทีแล้วก็มา diff ผลดูเอง เปลือง, หน่วง, แล้วก็ scale ไม่ไหวเอาซะเลย ส่วน webhook มันพลิกเป็น **push** — วินาทีที่ `order.paid` เกิดขึ้น *เรา* นี่แหละโทรหา *เขา* ถูกกว่าแล้วก็ real-time ด้วย แต่ทีนี้ปัญหายาก ๆ มันก็มาตกอยู่ที่เรา: endpoint ของเขาเดี๋ยวก็ล่ม เดี๋ยวก็ช้า เดี๋ยวก็ตอบมาแบบเชื่อไม่ได้ แล้วเราก็ยังต้องส่งให้ถึงอยู่ดี

**แล้วก็มีจุดพลิกของปี 2026 ที่จะวิ่งอยู่ตลอดทั้งโพสต์นี้** ของที่มารับ webhook ของเรา เมื่อก่อนมันคือ backend ของ merchant ที่สุดท้ายก็มีคนมานั่งดู dashboard อีกที แต่เดี๋ยวนี้มันเริ่มกลายเป็น **AI agent ที่ทำงานเองแล้วลงมือกับ event นั้นทันที** — สั่ง order, ขยับเงิน, เริ่ม fulfillment ให้เลย เพราะงั้น webhook ที่ซ้ำ, มาผิดลำดับ, หรือโดนปลอม มันเลิกเป็นแค่ความผิดเล็ก ๆ น่ารำคาญแล้ว — มันกลายเป็น agent ที่ลงมือทำสิ่งที่ผิดพลาดออกไปในโลกภายนอกโดยไม่มีมนุษย์คอยกั้นสักคน จุดนี้แหละที่ดันมาตรฐานของทุกชิ้นส่วนเรื่องความน่าเชื่อถือข้างล่างให้สูงขึ้น

เส้นทางทั้งหมด แต่ละ step ถูกบังคับด้วย step ก่อนหน้ามันเอง:

1. **POST แบบ synchronous** ทำ inline
2. มันพังบ้างเป็นบางครั้ง → ใส่ **sync retry** (ยังทำ inline อยู่)
3. sync retry มัน **บล็อก thread เรา** → ย้ายไป **async** (queue + worker)
4. **Backoff + max attempts** จะได้ไม่กระหน่ำหรือวนไม่จบ
5. **เซ็นมันซะ (HMAC)** เพื่อให้ consumer เชื่อใจได้ว่าเป็นเราแน่ ๆ
6. **Idempotency / event id** จะได้ไม่ให้ retry ของเราทำให้เขาลงมือซ้ำ
7. **DLQ + redrive** สำหรับ delivery ที่ส่งยังไงก็ไม่สำเร็จ
8. **Ordering** — แล้วก็ dependency **DAG** ที่ฝั่งคนรับต้องเคารพ
9. ปลายทาง: แยก delivery ออกมาอยู่ tier ของมันเอง (**acceptor → queue → consumer**, บวก dispatcher ตอนที่ routing เริ่ม dynamic)

## The naive version — just POST it

ต้องเริ่มจากที่ไหนสักที่ แล้วจุดเริ่มแบบซื่อ ๆ ก็คือ: พอมีอะไรเกิดขึ้น ก็ `POST` ไปที่ URL ของพาร์ตเนอร์ในตัว request handler นั้นเลย

```
on order_paid(order):
  resp = http.post(partner_url, body=event(order))   # synchronous
  return 200
```

สำหรับพาร์ตเนอร์รายเดียว volume น้อย ๆ แบบนี้มัน work ได้โอเคเลย — ปล่อยขึ้นได้ แต่ลองสังเกตสิ่งที่เราเพิ่งมัดติดกันไว้: **ตอนนี้ endpoint ของพาร์ตเนอร์มันมานั่งอยู่บนเส้นทางวิกฤตของ request เราเองแล้ว** วันที่ server เขามีปัญหา ก็เลยกลายเป็นวันที่เรามีปัญหาไปด้วย

## It failed — add a retry

**ปัญหา:** endpoint ของคนอื่นมัน *จะ* ตอบ 500 หรือ timeout กลับมา — ไม่ใช่เพราะเราทำอะไรผิดนะ แค่เพราะมันคือ server ตัวนึงบนอินเทอร์เน็ตเฉย ๆ การ POST แบบยิงทีเดียวจบ มันทำ event หล่นหายตั้งแต่ครั้งแรกที่เครื่องเขาสะดุด

**วิธีแก้:** อันที่นึกออกง่ายสุด — retry สักสองสามครั้งก่อนจะยอมแพ้

```
on order_paid(order):
  for attempt in 1..3:
      resp = http.post(partner_url, body=event(order))
      if resp.ok: return 200
      sleep(1s)
  return 200   # gave up
```

แบบนี้ดีขึ้นนะ แล้วสำหรับ integration เล็ก ๆ มันอาจจะพอแค่นี้เลยก็ได้ แต่มีอย่างนึง — เวลาเขียนโค้ดเอาไปใช้งานเอง เราจะไม่มานั่งเขียน loop นี้เองหรอก ทุก ecosystem มี retry library ที่ผ่านสนามมาแล้วคอยจัดการเรื่องนับจำนวนครั้ง, จัดการ delay, แล้วก็ (ที่สำคัญ) ดูว่า error แบบไหนที่ retry แล้วถึงจะคุ้ม:

- **Java** — resilience4j (`Retry`), Spring Retry (`@Retryable`), หรือ Failsafe
- **Go** — `avast/retry-go` หรือ `cenkalti/backoff`
- **Python** — `tenacity` (หรือ `Retry` adapter ที่มีมาให้ใน `requests` / `urllib3`)

คว้าตัวพวกนี้มาใช้ดีกว่ามานั่ง reinvent เอง — มันจัดการรายละเอียดจุกจิกได้ดีกว่า เดี๋ยวเรื่อง tune policy เอง (backoff, jitter, เพดาน) ค่อยว่ากันอีกไม่กี่ step ข้างล่าง แต่ไม่ว่าจะใช้ library ตัวไหน ลองอ่านโค้ดดี ๆ: **retry พวกนี้มันก็ยังเกิดแบบ inline อยู่ข้างใน request ของคนเรียกอยู่ดี** นั่นแหละคือจุดที่ step ถัดไปจะเข้ามาขยายต่อ

## Sync retries block you — go async

**ปัญหา:** sync retry มันทำให้การมัดติดกัน *แย่ลง* ไม่ใช่ดีขึ้น พอพาร์ตเนอร์ช้า เราไม่ได้ fail เร็ว ๆ — เรานั่งถือ request นั้นไว้ (พร้อมกับ worker thread ตัวนึง) เป็นเวลา `timeout × จำนวนครั้ง` แถมหลับรอระหว่างแต่ละครั้งอีก แล้วส่วนที่โหดร้ายคือ: เรา retry *หนักที่สุด* ก็ตอนที่พาร์ตเนอร์ *ช้าที่สุด* พอดี ซึ่งเป็นจังหวะที่เราเสีย thread ไปแบบไม่คุ้มที่สุด พอเจอ load หนัก ๆ ของหน้างาน พาร์ตเนอร์ที่กำลังแย่อยู่รายเดียวก็ดูด thread pool เราจนหมด แล้วก็ลากเอา request อื่นที่ไม่เกี่ยวกับ webhook เลยล้มตามไปด้วย

**วิธีแก้:** เอางาน delivery ออกไปจาก thread ของคนเรียกให้หมดเลย ตัว request handler ทำให้น้อยที่สุด — แค่บันทึกว่า `event นี้ต้องถูกส่งออกไปนะ` ลง durable queue แล้วก็ return ส่วน **delivery worker** แยกอีกตัวค่อยดึงจาก queue มาทำ POST (แล้วก็ retry)

```
on order_paid(order):
  queue.put(event(order))   # durable, returns in ms
  return 202

# elsewhere, a worker:
loop:
  e = queue.take()
  deliver(e)                # POST + retries live here now
```

อันนี้คือการเปลี่ยนที่ leverage สูงที่สุดในโพสต์เลย latency ของพาร์ตเนอร์กลายเป็นปัญหาของ *worker* ไม่ใช่ของ API เราแล้ว — แถมงานมันก็ durable ด้วย crash ไปก็ไม่ทำ event หาย เกือบทุกอย่างข้างล่างนี้คือการขัดเกลาไอเดีย `ส่งแบบ async โดยอิงกับ durable queue` ให้เนียนขึ้นเฉย ๆ

แล้วก็พูดกันแบบเปิดอกนะ ถ้าลองนั่งคิดดูดี ๆ async ในเคสนี้มันไม่ได้ดีแค่เรื่องความไวอย่างเดียว — แต่มันเป็นโมเดลที่เข้ากับเนื้องานมากกว่าด้วย: ตัวที่ **ทำให้เกิด** event กับตัวที่ **ควรได้รับ** callback มันคนละฝ่ายกันเลย ลูกค้าที่จ่ายเงินจนเกิด `order.paid` ไม่ได้แคร์หรอกว่า endpoint ของพาร์ตเนอร์จะ ack อะไรกลับมามั้ย แล้วก็ไม่ควรต้องมานั่งรอมันด้วย ต้นทางของ request กับปลายทางของการ notify มันคนละที่กัน — เพราะงั้นฝั่งคนเรียกมันไม่เคยมีอะไรที่ต้องใช้ผลการส่งแบบ inline เลยตั้งแต่แรก จะไปบล็อกมันไว้ก็ไม่เคยมีความหมายมาตั้งแต่ต้น งาน notify แบบนี้มันคือ fire-and-forget โดยธรรมชาติของมันอยู่แล้ว

## Don't hammer, don't loop forever — backoff + max attempts

**ปัญหา:** retry ของเราเมื่อกี้มันยังกิ๊กก๊อก — delay คงที่ แล้วก็ (พูดกันตามเนื้อผ้า) ไม่มีอะไรห้ามไม่ให้มันวนไปเรื่อย ๆ ไม่จบ มันมีโหมดพังซ่อนอยู่สองอย่าง อย่างแรก การ retry กระหน่ำ endpoint ที่กำลังแย่อยู่ซ้ำ ๆ แบบทันที มันก็เหมือนไปกระทืบซ้ำตอนที่เขากำลังล้มอยู่ อย่างที่สอง ถ้า endpoint มันตายสนิทไปแล้ว retry แบบไม่มีเพดานก็คือ worker ที่หมุนอยู่กับ event ตายตัวเดียวไปตลอด ไม่ยอมระบาย queue ที่ต่อแถวอยู่ข้างหลังเลย (แล้วขอตอบคำถามที่หลายคนน่าจะสงสัย — *ทำไม backoff ถึงเพิ่งมาโผล่จุดนี้ ไม่ไปอยู่ตั้งแต่ step retry*: การรอ 1s, 2s, 4s, 8s ระหว่างแต่ละครั้ง ถ้ามันรันแบบ inline ก็เท่ากับพิน request thread ไว้ 15 วินาทีกว่า ๆ เพราะงั้น backoff กับเพดานจำนวนครั้งข้างล่าง ถึงเพิ่งปลอดภัยพอจะเอามาใช้ได้ ก็ตอนที่เราย้าย retry ไปอยู่บน background worker แล้ว)

**วิธีแก้:** **exponential backoff พร้อม jitter แล้วก็เพดาน max-attempt แบบตายตัว**

- **Exponential backoff** — รอนานขึ้นเรื่อย ๆ ในแต่ละครั้ง (1s, 2s, 4s, 8s, …) เพื่อให้ endpoint ที่กำลังฟื้นมีที่หายใจ
- **Jitter** — ใส่ความสุ่มเข้าไปใน delay ถ้าไม่มีมัน ทุก event ที่กองไว้ตอน outage ก็จะ retry พร้อมกันเป๊ะ ๆ ในวินาทีที่ endpoint กลับมา — กลายเป็น thundering herd ที่เราสร้างขึ้นเองแล้วก็ทุบเขาล้มซ้ำอีกรอบ
- **Max attempts** — หลังจาก N ครั้ง (หรือหมด time budget) ก็หยุด event ไม่ได้หายนะ — มันไปอยู่ที่ที่เราจะจัดการทีหลัง (ตัว DLQ อีกไม่กี่ step ข้างล่าง) หลักการ `retry ได้ แต่ต้องมีเพดาน` คือหัวใจทั้งหมด

## Anyone can POST to them — sign it (HMAC)

**ปัญหา:** endpoint ที่รับ webhook ของพาร์ตเนอร์มันคือ **URL สาธารณะบนอินเทอร์เน็ต** ใครก็ตามที่รู้ URL นี้ก็ POST เข้าไปได้ แล้วก็แกล้งทำเป็นเราได้ด้วย — `order.paid` ปลอม ๆ, `refund.completed` ปลอม ๆ ฝั่ง consumer ไม่มีทางแยกออกเลยว่าอันไหน event ของเราอันไหนของปลอม แล้วถ้าเราเผลอเอารายละเอียดไปยัดไว้ใน **URL หรือ query string** (`?amount=5000&user=123`) ก็เท่ากับทำมันรั่วไปแล้ว — query string มันไปโผล่ใน access log, proxy, แล้วก็ประวัติ browser

**วิธีแก้:** พิสูจน์ว่า message มาจากเรา แล้วก็เก็บข้อมูลออกไปจาก URL

- **เซ็น payload** คำนวณ HMAC (เช่น HMAC-SHA256) ของ raw request body ด้วย **shared secret** ที่มีแค่เรากับ consumer รู้ แล้วก็ส่งไปใน header (`X-Signature`) ฝั่ง consumer ก็คำนวณ HMAC จาก body ที่เขาได้รับมาใหม่แล้วเทียบ — ถ้าไม่ match ก็แปลว่าโดนปลอมหรือโดนแก้ไขระหว่างทาง เขาก็ทิ้งไป
- **เซ็น timestamp ไปด้วย** แล้วก็รวมมันเข้าไปในสิ่งที่เซ็น ฝั่ง consumer จะ reject อะไรก็ตามที่เก่าเกินไม่กี่นาที ซึ่งกัน **replay attack** ได้ (พวกที่ดักจับ call ที่ valid แล้วเอามายิงซ้ำ)
- **ทุกอย่างที่มีความหมายอยู่ใน body** ไม่ใช่ URL ตัว URL เป็นแค่ที่อยู่ ส่วน body ที่เซ็นแล้วคือตัว message

```
sig = hmac_sha256(secret, timestamp + "." + raw_body)
POST /webhooks/orders
  X-Timestamp: 1718000000
  X-Signature: sha256=...
  { "id": "...", "type": "order.paid", ... }   # data lives here
```

อันนี้คือด่านแรกสุดของการประมวลผลฝั่ง consumer เลย: ก่อนจะไปดูเนื้อหาด้วยซ้ำ เขาถามก่อนว่า `อันนี้มาจากเขาคนนั้นแน่นะ?` ยืนยันตัวตนก่อน แล้วค่อยว่ากันเรื่องอื่น

**ขอถอยมามองภาพรวมหน่อย — สุดท้ายแล้วสิ่งที่เราต้องการคือ trust แล้ว HMAC ก็เป็นแค่ทางนึงที่ทำให้ได้ trust นั้นมา ไม่ใช่ทางเดียว** ในคำว่า `เชื่อ webhook อันนี้ได้มั้ย` มันมีคำถามซ่อนอยู่สองข้อที่ต่างกัน แล้วก็ตอบด้วยเครื่องมือคนละตัว:

- **การเซ็น (HMAC)** ตอบคำถามว่า *payload ก้อนนี้ของแท้แล้วก็ไม่โดนแก้รึเปล่า?* — integrity + origin ของ message ทำแบบ stateless ไม่ต้องมี token infra เลย เป็น default ของโลก webhook (Stripe, GitHub)
- **Auth (OAuth2 client-credentials Bearer token หรือ mTLS)** ตอบคำถามว่า *คนที่เรียกมาเป็น sender ที่ได้รับอนุญาตมั้ย?* — เป็นเรื่อง identity ระดับ connection มันต่อกับ IdP ที่ผู้รับมีอยู่แล้ว, token ก็อายุสั้นแล้วก็ revoke ได้ — แต่แลกมากับที่ฝั่งคนส่งต้องไปขอแล้วก็ cache token ไว้ก่อนยิงทุกครั้ง

ขอแปะ caveat อันนึงถ้าจะไปทาง OAuth2: flow ที่เข้ากับงานนี้คือ **client credentials** — machine-to-machine ไม่มีคนมาเกี่ยวกลางทาง ส่วน *authorization code* เอาไว้ทำ user-delegated consent (คนกดอนุญาตให้แอปเข้าถึงข้อมูล) ซึ่งมันไม่เข้ากับ callback แบบ server-to-server เพราะงั้นไม่ใช่เครื่องมือสำหรับงานนี้

สองอย่างนี้ไม่ได้แข่งกัน — มันตอบคนละคำถาม แล้วหลายระบบก็ใช้ทั้งคู่เลย: token ไว้พิสูจน์ว่า *ใครเรียกมา* ส่วนลายเซ็นไว้พิสูจน์ว่า *message ไม่โดนแตะ* เลือกตาม threat model ของเรา สุดท้ายแล้วทุกอย่างมันก็รับใช้เป้าหมายเดียวกัน — ทำให้ผู้รับเชื่อ event มากพอที่จะลงมือกับมัน

## Your retries cause doubles — idempotency

**ปัญหา:** ทุกอย่างที่เราสร้างมามันเป็น **at-least-once** หมด worker ก็ retry; queue ก็ redeliver ได้หลัง crash; ส่วน delivery ที่ส่งสำเร็จไปแล้วแต่ ack ดันหายระหว่างทาง ก็โดนส่ง *ซ้ำ* อีกที เพราะงั้น consumer มัน **จะ** ได้รับ event เดิมซ้ำมากกว่าหนึ่งครั้งแน่ ๆ เราเจอกำแพงนี้มาแล้วเป๊ะ ๆ ทั้งในโพสต์ payment แล้วก็ push — exactly-once แบบเป๊ะข้ามเครือข่ายมันเป็นเรื่องเพ้อฝัน (Two Generals Problem) สำหรับ dashboard ของซ้ำมันก็แค่เรื่องผิวเผิน แต่สำหรับ **AI agent** `order.paid` ที่ซ้ำ คือมีการสั่ง order รอบสอง คือเงินขยับสองเด้ง

**วิธีแก้:** ให้ทุก event พก **`event_id` ที่นิ่งแล้วก็ไม่ซ้ำใคร** (กำหนดครั้งเดียวตั้งแต่ต้นทาง — แล้วพกติดตัวไปไม่เปลี่ยนทุก retry) แล้วก็ทำสัญญาเรื่องการส่งให้เคลียร์ ๆ ไปเลยว่า `เราส่งแบบ at least once นะ เพราะงั้นช่วยไป dedupe ด้วย event_id เอาเองนะ` ฝั่ง consumer ก็จดว่า id ไหนประมวลผลไปแล้วบ้าง แล้วก็ข้ามตัวที่ซ้ำ เรารับประกันการส่งถึง เขารับประกันการประมวลผลแบบ idempotent มันคือสัญญาร่วมกัน แล้วก็ควรเขียนไว้ดัง ๆ ใน webhook docs ของเราด้วย

```
{
  "id": "evt_7f3c8a",
  "type": "order.paid",
  "data": { "order_id": "ord_123", "amount": "245 THB" }
}
```

สังเกตลำดับกับ step ก่อนหน้า: ฝั่ง consumer **ยืนยันตัวตน (HMAC) ก่อน แล้วค่อย dedupe (`event_id`) แล้วค่อยประมวลผล** เชื่อใจมันก่อน แล้วเช็คว่าเคยเห็นรึยัง แล้วค่อยลงมือ

## Some never succeed — DLQ + redrive

**ปัญหา:** backoff กับเพดาน max-attempt หยุดการวนไม่จบได้ก็เถอะ แต่มันสร้างคำถามใหม่ขึ้นมา: *แล้ว event ที่ retry จนหมดโควต้าไปแล้วล่ะ ไปไหนต่อ?* พาร์ตเนอร์ล่มเป็นชั่วโมง ๆ ได้ เรา retry ตลอดกาลไม่ได้ (ตกลงกันไปแล้ว) แต่การปล่อย event หล่นหายเงียบ ๆ ก็แปลว่า notification ของ order หายไปดื้อ ๆ — แล้วสำหรับ consumer ที่ขับเคลื่อนด้วย agent มันคือ state ที่หายไปแล้วกู้กลับมาไม่ได้เลย

**วิธีแก้:** **dead-letter queue (DLQ)** พอ event ข้ามเพดาน max-attempt ก็ย้ายมันไปเก็บที่ store แยกแทนที่จะทิ้ง ทีนี้:

- **operator มองเห็นแล้วก็ลงมือได้** — ส่องดูว่าทำไมมันถึง fail แล้วก็ **redrive** (เอากลับเข้า queue ใหม่) ทั้งก้อนตอนที่พาร์ตเนอร์กลับมาปกติ
- **ให้ทางกู้คืนกับ consumer ด้วย** — API แบบ `list/replay event ตั้งแต่จุด X` เพื่อให้พาร์ตเนอร์ที่เคยล่มดึงของที่พลาดไปกลับมาได้ (อันนี้คือด้านกลับของเหรียญเดียวกับโมเดล polling เก่า — เป็น channel ไว้ตามเก็บโดยเฉพาะ ไม่ใช่ทางหลัก)

รายละเอียดเชิงปฏิบัติจากระบบหน้างาน: ผูกการย้ายเข้า DLQ ไว้กับ **receive-count** (message นี้ถูกดึงออกมาแล้ว fail ไปกี่ครั้งแล้ว) แล้วก็กำหนด **redrive policy** ไว้ล่วงหน้าเลย — ว่า message ที่หมดโควต้าจะไปไหนแล้วก็จะกลับมายังไง อย่าเพิ่งไปนั่งคิดสูตรมันตอนเกิด incident

**การรัน DLQ บนหน้างานมันมีอะไรมากกว่าแค่ `queue ตัวที่สอง` เยอะ** มีเรื่องที่กัดเราตอนมันขึ้นใช้งานอยู่หลายอย่าง:

- **queue ไม่ใช่ที่เก็บของระยะยาว** broker ส่วนใหญ่มันลบ message ทิ้งตามอายุ (SQS เก็บได้สูงสุด 14 วัน) — เพราะงั้น DLQ ที่เป็น *แค่* queue เฉย ๆ มันจะค่อย ๆ ทำ failed event หล่นหายเงียบ ๆ พอเลยกำหนด ซึ่งมันก็เสียจุดประสงค์ไปเลย ของอะไรที่หายไม่ได้ ให้ **persist event ที่ตกลง DLQ ลง store ที่ทนทาน** — ตาราง `failed_deliveries` (หรือ object storage) สไตล์ Outbox — ที่เรา query, audit, แล้วก็ redrive ได้ทุกเมื่อ ไม่ใช่ queue ที่กำลังแข่งกับนาฬิกานับอายุอยู่
- **ต้องมีคนคอยจับตาดูมัน** operator ส่อง DLQ ด้วยการ **poll ดูเองแบบ manual** (จะเป็น admin view เล็ก ๆ หรือ cloud console ก็ได้) — list, filter ตาม endpoint หรือ error, แล้วก็เปิดอ่าน payload ที่ fail ทีละอัน DLQ ที่ไม่มีใครเปิดดูเลย ก็เป็นแค่วิธีทำข้อมูลหายแบบช้าลงเฉย ๆ
- **redrive แบบเจาะจง อย่า redrive มั่ว ๆ** ตอนเกิด incident เราแทบไม่อยากกด `replay ทั้งหมด` หรอก — batch เสีย ๆ มันก็แค่ fail ซ้ำหรือยิงซ้ำ ให้ **consume / เลือกแบบ manual** เฉพาะ message ที่เกี่ยวข้องเท่านั้น (endpoint นี้, ช่วงเวลานี้) แล้วก็ redrive เฉพาะพวกนั้น
- **redrive ด้วยความระวังเท่ากับตอนส่งครั้งแรก** เอากลับเข้า queue โดยพก **idempotency key เดิม** ไปด้วย (consumer จะได้ยัง dedupe ได้อยู่), **throttle** การ redrive ด้วย (endpoint ที่เพิ่งฟื้นมาไม่อยากได้ backlog ทั้งกองทีเดียว — บทเรียน thundering herd อันเดิม) แล้วก็คงเพดานจำนวนครั้งไว้
- **ตั้ง alert ที่ความลึกของ DLQ** DLQ ที่โตขึ้นเรื่อย ๆ คือ signal แรกสุดที่บอกว่าพาร์ตเนอร์ — หรือชั้น delivery ของเราเอง — กำลังพัง เปิด page แจ้งเตือนไว้ อย่าไปรู้เอาตอนพาร์ตเนอร์ส่งเมลมาด่า
- **ให้ store ที่ persist ไว้มี lifecycle ด้วย** สำเนา audit ของ failed event ต้องมี TTL / นโยบาย archive ไม่งั้นตารางนั้นมันโตไม่หยุด

## They arrive out of order — sequencing, and the DAG

**ปัญหา:** การส่งแบบ async + retry + worker หลายตัวขนานกัน แปลว่า event ของ resource เดียวกันมา **ผิดลำดับ** ได้ `order.paid` แล้วตามด้วย `order.refunded` แต่ consumer ดันเห็น `refunded` มาก่อน หรือ event เก่าที่โดน retry ดันมาถึงทีหลังตัวที่ใหม่กว่า คนเรายังพอทนได้ แต่ agent ที่ลงมือตามลำดับที่ได้รับ มันทำผิดเลย

**วิธีแก้ (เคสเชิงเส้น):** แปะ **sequence number หรือ version** ที่เพิ่มขึ้นเรื่อย ๆ ต่อ resource ไว้ในทุก event ฝั่ง consumer ก็เก็บ version ล่าสุดที่เห็น แล้วใช้ **last-write-wins** — อะไรที่เก่ากว่าก็ข้ามไป ถ้าอยากได้ลำดับการประมวลผล ไม่ใช่เอาแค่ state ล่าสุด ก็ **partition ตาม resource id** เพื่อให้ทุก event ของ order เดียวไหลผ่าน lane เดียวที่เรียงลำดับแล้ว ขอแปะหมายเหตุเรื่อง scope ไว้: เราแทบไม่เคยต้องการ *global ordering* เลย — ต้องการแค่ลำดับ **ต่อ resource** เท่านั้น อย่าไปสร้างระบบ total-order ที่ไม่ได้ใช้

**แต่ consumer หน้างานมันไม่ได้เป็นแค่เส้นเดียว — มันมีกราฟของ dependency** ส่วนนี้แหละที่ควรค่อย ๆ ดู ลองมอง event เป็น **DAG**: แต่ละ event คือ node, เส้น directed ชี้จากตัวที่เป็น prerequisite ไปยังตัวที่ต้องพึ่งมัน แล้วก็เป็น acyclic (ไม่มีวนกลับ) ส่วนใหญ่หน้าตามันเป็นเชิงเส้น — จนกระทั่งไปเจอ **fan-in (จุดรวม)**: event ที่ต้องรอ prerequisite *ตั้งแต่สองตัวขึ้นไป*

`shipment.created` คือจุดรวม — มันต้องรอ **ทั้ง** `order.paid` *และ* `inventory.reserved` ระบบเราส่งสองตัวนี้แยกกันแล้วก็มาผิดลำดับได้ เพราะงั้น consumer อาจจะได้ `shipment.created` มาก่อนที่ `inventory.reserved` จะมาถึงด้วยซ้ำ นี่คือเส้นแบ่งที่ทำให้เรื่องนี้ยังอยู่ในร่องในรอย:

- **งานของฝั่งคนส่ง** (อยู่ใน scope ทำเลย): ลำดับต่อ resource บวกกับ **causation metadata** ในทุก payload — `event_id`, `causation_ids` (id ของ event แม่), แล้วก็ `correlation_id` ของทั้ง flow (เช่น `order_id`) นั่นคือส่วนร่วมทั้งหมดที่เรามีต่อกราฟนี้
- **งานของฝั่งคนรับ** (อยู่นอก scope ของเรา แต่ขอพูดไว้): เอา link พวกนั้นมาประกอบ DAG ขึ้นใหม่แล้วประมวลผลตาม **topological order** — พัก node ไว้ก่อนจนกว่า parent ทุกตัวจะมาถึง/ประมวลผลเสร็จ พร้อม timeout ที่จะ trigger ให้ไปดึงเองหรือ alert ถ้า parent ไม่มาสักที **ระบบ webhook ของเราไม่ใช่ workflow engine — อย่าไปทำให้มันกลายเป็นแบบนั้น**

แล้วนี่แหละคือจุดที่มุม agent มันพุ่งถึงขีดสุด: agent ที่ทำงานเองแล้วไล่ DAG นี้เพื่อจะ *ลงมือ* มันต้องเคารพ topological order ไม่งั้นมันจะไปสั่ง logic เรื่อง shipment กับ order ที่ไม่เคยถูกจ่ายเงินเลยด้วยซ้ำ ตัว DAG นี่แหละที่กั้นไม่ให้ agent ไปลงมือกับ state ที่ยังประกอบไม่เสร็จ ถ้าเป็นคนเหลือบมอง dashboard แป๊บเดียวก็คง... เอ๊ะ ทันแล้ว ส่วน agent มันไม่เอ๊ะหรอก

## Pull it apart — acceptor / dispatcher / consumer

**ปัญหา:** มาถึงตอนนี้ service ตัวเดียวทำงานเยอะมาก — รับ event ภายใน, หาว่า *ใคร* subscribe ไว้บ้าง, แล้วก็ทำ delivery ออกนอกที่ช้า ๆ พร้อม retry ทั้งหมด งานสามอย่างนี้รูปร่างคนละแบบกันเลย พอเอามาแปะรวมกัน ตัวที่ช้าแล้วก็พังบ่อยที่สุดมันก็ลากตัวอื่นจมตามไปด้วย พาร์ตเนอร์ endpoint ช้า ๆ ที่ถาโถมเข้ามา ก็ไปทำให้งาน delivery ตันแล้วก็ไปอุดการรับ event ใหม่ที่ต่อแถวอยู่ข้างหลังด้วย

**แต่ขอพูดให้เคลียร์ ๆ หน่อย — service ตัวเดียวที่ทำครบทั้งสามอย่างเนี่ย มันคือจุดที่แอปบนหน้างานส่วนใหญ่อยู่กัน แล้วก็โอเคแบบนั้นด้วยนะ** ไม่ว่าจะ package เป็น deployable ตัวเดียว หรือเป็นสแตก **serverless** (event → SQS / EventBridge → Lambda ที่เซ็นแล้วก็ส่ง → DLQ) เครื่องจักร queue-worker-retry-DLQ จากไม่กี่ step ที่ผ่านมาก็แบก traffic หน้างานได้เยอะแล้ว ไม่ต้องไปแยกอะไรเลย โดยเฉพาะ serverless เนี่ยมันแถม retry, การ scale, แล้วก็ DLQ มาให้แบบสำเร็จรูปด้วย **จะงัดการแยกข้างล่างมาใช้ ก็ต่อเมื่อโหลดหรือรูปแบบการพังของ tier นึงมันเริ่มลากตัวอื่นให้จมตามเท่านั้น** — ก่อนหน้านั้นยังไม่ต้อง

**วิธีแก้ — แล้วการแยกที่สำคัญแท้ ๆ มันมีแค่สอง tier:** **acceptor → queue → consumer**

- **Acceptor** (ขาเข้า) — รับ event แบบ `มีอะไรเกิดขึ้น` มา **เก็บให้ไว durable แล้วก็ return** เน้นงาน write เป็นหลัก ห้ามทำ event หายเด็ดขาด ตั้งใจทำให้มันผอม ๆ ไว้ เพื่อให้การรับเข้าไม่ติดขัดเพราะอะไรที่อยู่ปลายน้ำ
- **Consumer** (delivery) — ดึงจาก queue แล้วทำ POST ที่เซ็นแล้วช้า ๆ นั่นแหละ: HMAC, backoff, idempotency id, track ผล, DLQ เป็นงานที่ติด I/O โดน endpoint ข้างนอกที่ช้าบล็อกไว้ แล้วก็ fail บ่อย อยากได้ตัวนี้เยอะ ๆ

การแยกสอง tier นี้แหละคือหัวใจ: แยกฝั่งรับเข้าที่ durable-แล้ว-ไว ออกจากฝั่ง delivery ที่ช้า-เพราะของข้างนอก เพื่อให้ตัวนึงลากอีกตัวจมไม่ได้

**แล้ว dispatcher อยู่ไหนล่ะ?** สังเกตว่าผมไม่ได้ใส่มันเข้าไป งานที่ dispatcher ทำ — *พา event ที่ถูกไปหา endpoint ที่ถูก* — มันก็ยังต้องเกิดขึ้นอยู่ดี แต่มันไม่จำเป็นต้องเป็น component แยกของตัวเอง การ route มันมีอยู่สองทาง แล้วการเลือกระหว่างสองทางนี้แหละคือจุดตัดสินใจที่แท้:

- **route ผ่าน broker (ไม่ต้องมี dispatcher)** partition queue/stream ตาม route หรือชนิดของ event แล้วก็ให้แต่ละ **consumer subscribe เฉพาะ route ที่ตัวเองดูแล** — การ route ไปอยู่ในโครงสร้างของ topic ไม่ใช่ใน service deploy ได้สองแบบ:
  - **service ตัวเดียว รัน N consumer** — consumer ละ route อยู่ใน deployable เดียวกัน หรือ
  - **N service แยกกัน อันละ route** — อิสระ deploy แล้วก็ scale ของใครของมัน

  อันนี้ง่ายสุดตอนที่ routing มัน **fix แล้วก็รู้แน่ชัด** — event ไม่กี่ชนิดวิ่งไปที่ที่รู้กันอยู่แล้ว
- **dispatcher (tier ที่สาม)** component กลางที่ไปหาว่า *ใคร subscribe ไว้* แล้ว apply subscription config แล้วก็กาง event ออกเป็น task ราย endpoint มันคุ้มก็ตอนที่ routing มัน **dynamic** — ลูกค้าลงทะเบียน endpoint กับ event filter เองตอน runtime หรือเราต้องทำ transform / rate cap ราย subscription นั่นแหละคือ logic การ route ที่มีน้ำหนักพอจะอยากมีบ้านหลังเดียว บังคับใช้ที่เดียว แทนที่จะกระจายไปทั่วทุก consumer

การแลกแบบเปิดอก: consumer ที่ route ผ่าน broker มันง่ายแล้วก็ decouple สุด ๆ — เพิ่ม consumer ได้โดยไม่ต้องแตะ upstream, ได้ replay จาก offset มาฟรี ๆ — แต่ไม่มีตัวกรองกลาง เพราะงั้นแต่ละ consumer ก็อ่านของมากกว่าที่ตัวเองต้องใช้ แล้วก็ต้องพึ่ง event schema ที่นิ่ง ๆ เพราะทุกตัวตีความ raw stream เอง (ถ้า contract ผิดเมื่อไหร่ มันพังพร้อมกันหมด) ส่วน dispatcher ก็รวมศูนย์การ route กับ filtering ไว้ แลกกับ component ที่เราต้อง build แล้วก็ scale **routing ที่ลูกค้า config เองแบบเยอะ ๆ → dispatcher; routing ที่เราคุมเองแบบ fix → consumer ที่ route ผ่าน broker**

ไม่ว่าจะ route แบบไหน ผลตอบแทนของการดึง delivery ออกมาอยู่ tier ของมันเองก็เหมือนกัน:

- **scale แยกกันได้** — delivery พุ่งหรือเจอพาร์ตเนอร์ช้า ๆ มาเป็นพรวด ก็ scale แค่ *consumer*
- **แยก failure ออกจากกัน** — พาร์ตเนอร์ที่งอแงทำให้ตันแค่ฝั่ง consumer ส่วนการรับเข้าก็ยังรับต่อไปได้เรื่อย ๆ
- **deploy แยกกันได้** — เปลี่ยน routing หรือตัว consumer โดยไม่ต้องไปแตะการรับเข้า

มันคือเส้นเรื่องเดียวกับโพสต์ก่อน ๆ: แยกส่วนที่ durable-แล้ว-ไว ออกจากส่วนที่ช้า-เพราะของข้างนอก เพื่อให้ตัวนึงลากอีกตัวจมไม่ได้

นี่คือรูปร่างคร่าว ๆ ที่ผมเคย build มากับมือ — ย้ายการต่อกับพาร์ตเนอร์ออกจากการ poll แบบ batch `GET` เก่า ๆ มาเป็น pipeline แบบ push real-time พร้อม retry ที่ขับด้วย queue (receive-count + redrive) ในเคสของผม routing มันซับซ้อนพอที่ผมเลยรัน **acceptor / dispatcher / consumer** เต็มรูปแบบ โดยแยกชั้น delivery ออกมาเพื่อให้ scale ของมันเองได้ — แต่ถ้า routing ของใครมันง่ายกว่านั้น dispatcher นี่แหละคือชิ้นที่ควรข้ามไป

## Wrapping up

เราเริ่มจาก `POST` ตัวเดียวแบบ inline แล้วมาจบที่ระบบ delivery ที่แยกชั้นแล้วก็ทนทาน เส้นทางทั้งหมด เรียงตามลำดับ:

1. **เอา delivery ออกจาก thread ของคนเรียก** ไปไว้บน durable queue — การเปลี่ยนอันเดียวที่ทุกอย่างต่อยอดจากมัน
2. **Back off ด้วย jitter แล้วก็มีเพดาน max-attempt** จะได้ไม่กระหน่ำหรือวนไม่จบ
3. **เซ็นด้วย HMAC** (แล้วก็เก็บข้อมูลออกจาก URL) เพื่อให้ consumer เชื่อใจได้ว่าเป็นเรา
4. **`event_id` ที่นิ่ง** จะได้ไม่ให้ at-least-once กลายเป็นการลงมือซ้ำสองเด้ง
5. **DLQ + redrive** เพื่อให้ outage ยาว ๆ ไม่ทำ event หาย
6. **ลำดับต่อ resource** แล้วก็ส่งมอบ **causation metadata** ให้ฝั่งคนรับเอาไปประกอบ dependency DAG ขึ้นใหม่ — โดยที่เราไม่ต้องกลายเป็น workflow engine
7. **แยก delivery ออกมาอยู่ tier ของมันเอง** (acceptor → queue → consumer; เพิ่ม dispatcher ตอนที่ routing เริ่ม dynamic) เพื่อให้ส่วนข้างนอกที่ช้าลากการรับเข้าจมตามไม่ได้

ประเด็นใหญ่ก็ยังเหมือนตลอดทั้งซีรีส์: **ไม่มีใครควร build เวอร์ชันที่แยกชั้นออกมาตั้งแต่วันแรกหรอก** พาร์ตเนอร์รายเดียว volume น้อย ๆ เหรอ? `POST` แบบ sync พร้อม retry สักสองครั้งก็โอเคสุด ๆ แล้ว ทุกกล่องข้างบนได้ที่ทางของมันมาจากการแก้ปัญหาที่เวอร์ชันก่อนหน้าสร้างขึ้น

สิ่งเดียวที่ **เพิ่ง** กลายเป็นเรื่องของปี 2026 คือ: consumer ปลายทางมีโอกาสมากขึ้นเรื่อย ๆ ที่จะเป็น **agent ที่ทำงานเองแล้วลงมือทันทีที่ webhook เราไปถึง** — เพราะงั้นของซ้ำที่เราไม่ได้ dedupe, event ที่เราส่งผิดลำดับ, call ปลอมที่เราไม่ได้กันด้วยลายเซ็น ทั้งหมดนี้กลายเป็นการลงมือทำผิดพลาดออกไปในโลกภายนอก แทนที่จะเป็นแค่ความผิดเล็ก ๆ บน dashboard งานเรื่องความน่าเชื่อถือมันเป็น engineering ที่ดีมาตลอดอยู่แล้ว ตอนนี้มันแค่กลายเป็นเส้นแบ่งระหว่าง agent ที่เชื่อใจ event ของเราได้ กับตัวที่เชื่อไม่ได้
