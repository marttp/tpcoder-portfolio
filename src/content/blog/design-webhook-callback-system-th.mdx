## Introduction

โพสต์นี้เป็นตัวที่สี่ของซีรีส์ backend system design แล้วนะครับ สูตรเดิมเหมือนเคย: เริ่มจากของที่ทื่อที่สุดเท่าที่มัน work ได้ แล้วก็ปล่อยให้ข้อจำกัดหน้างานค่อย ๆ ดันมันให้เข้ารูปเข้ารอยไปเอง

คราวก่อนใน[โพสต์ payment](/blog/payment-backend-stripe-integration-th/) ผมอยู่ฝั่ง **คนรับ** webhook ครับ — Stripe ยิง event มา *หาเรา* แล้วเราก็สร้าง receiver ที่เอา event ไปพักไว้บน queue ก่อน แล้วค่อยประมวลผลแบบปลอดภัย คราวนี้ผมขอสลับฝั่งกันบ้าง: **คราวนี้เราเป็นคนส่ง** มีอะไรเกิดขึ้นในระบบเรา แล้วเราต้องส่ง callback ไปหา **endpoint ของคนอื่น — ตัวที่เราคุมไม่ได้ มองไม่เห็น แล้วก็เชื่อใจอะไรไม่ได้เลยตั้งแต่แรก** คราวที่แล้วเรารับ คราวนี้เราส่งครับ

ขอปูพื้นเร็ว ๆ นะครับ สมมุติว่า partner เขาอยากรู้ว่า `order นี้จ่ายเงินรึยัง` ถ้าไม่มี webhook เขาก็ต้องมานั่ง **poll** เราเอง — ยิง `GET` เข้ามาเรื่อย ๆ ทุก ๆ ไม่กี่นาที ขอข้อมูลทีละอัน หรือขอมาเป็น list ทั้งก้อน แล้วก็มา diff ดูเองว่ามีอะไรเปลี่ยนไปบ้าง เปลือง หน่วง แล้วก็ scale ไม่ไหวเอาซะเลย ส่วน webhook พลิกเป็น **push** — วินาทีที่ `order.paid` เกิดขึ้น *เรา* นี่แหละเป็นฝ่ายยิงไปหา *เขา* เอง ถูกกว่าแล้วก็ real-time ด้วย แต่ทีนี้ปัญหายาก ๆ ก็มาตกอยู่ที่เราครับ: endpoint ของเขาเดี๋ยวก็ล่ม เดี๋ยวก็ช้า เดี๋ยวก็ตอบมาแบบเชื่อไม่ได้ แล้วเราก็ยังต้องส่งให้ถึงอยู่ดี

**ตอนนี้ในปี 2026** ตัวที่มารับ webhook ของเรา เมื่อก่อนคือ backend ของ partner ที่สุดท้ายก็มีคนมานั่งดู dashboard อีกที แต่จุดที่แตกต่างคือ ตัวที่มารับ webhook อาจจะไม่ใช่ backend อย่างเดียวแล้วนะครับ — **AI agent ก็เป็นตัวรับ webhook ได้เหมือนกัน** แล้วพอคนรับเป็น agent มันก็ลงมือกับ event ที่ได้รับทันที — สั่ง order, โอนเงิน, เริ่ม fulfillment ด้วยตัวเอง โดยไม่ได้รอให้คนมากดยืนยันก่อน

นี่แหละครับคือจุดที่ทุกอย่างเปลี่ยนไป webhook ที่ซ้ำ มาผิดลำดับ หรือโดนปลอม เมื่อก่อนก็เป็นแค่ข้อผิดพลาดเล็ก ๆ ที่เดี๋ยวคนก็มาเห็นแล้วตามแก้ทัน แต่พอปลายทางเป็น agent มันแปลว่า agent จะลงมือทำสิ่งที่ผิดออกไปเลย โดยไม่มีใครมาเบรกไว้ทัน เพราะงั้นความน่าเชื่อถือของทุก step ที่เหลือในโพสต์นี้ เลยยิ่งสำคัญกว่าเดิมมากครับ

เส้นทางทั้งหมด แต่ละ step ถูกบังคับด้วย step ก่อนหน้าเองครับ:

1. **POST แบบ synchronous** ทำ inline
2. พังบ้างเป็นบางครั้ง → ใส่ **sync retry** (ยังทำ inline อยู่)
3. sync retry **บล็อก thread เรา** → ย้ายไป **async** (queue + worker)
4. **Backoff + max attempts** จะได้ไม่กระหน่ำหรือวนไม่จบ
5. **Signed payload (HMAC)** เพื่อให้ consumer เชื่อใจได้ว่าเป็นเราแน่ ๆ
6. **Idempotency / event id** retry มาได้ แต่ไม่ใช่ปล่อยให้ผลลัพธ์เกิดซ้ำ
7. **DLQ + redrive** สำหรับ delivery ที่ส่งยังไงก็ไม่สำเร็จ
8. **Ordering** — maintain state เพื่อให้การทำงานถูกต้องตามลำดับที่ถูกต้อง
9. ปลายทาง: แยก delivery ออกมาอยู่ tier ของมันเอง (**acceptor → queue → consumer**, บวก dispatcher ตอนที่ routing เริ่ม dynamic)

## The naive version — just POST it

ต้องเริ่มจากที่ไหนสักที่ก่อนเนอะครับ จุดเริ่มแบบซื่อ ๆ ที่สุดเลยก็คือ: พอมีอะไรเกิดขึ้น ก็ `POST` ไปที่ URL ของ partner ในตัว request handler นั้นเลย

```
on order_paid(order):
  resp = http.post(partner_url, body=event(order))   # synchronous
  return 200
```

สำหรับ partner รายเดียว volume น้อย ๆ วิธีนี้ก็ work ได้สบาย ๆ ครับ ปล่อยขึ้น prod ได้เลย แต่ลองสังเกต coupling ที่เราเพิ่งสร้างขึ้นมาให้ดีนะครับ: **ตอนนี้ endpoint ของ partner เข้ามาอยู่ในเส้น request ของเราเรียบร้อยแล้ว** วันที่ server ของเขามีปัญหา ก็กลายเป็นวันที่ระบบเรามีปัญหาตามไปด้วยทันที

## It failed — add a retry

**ปัญหา:** endpoint ของคนอื่น ยังไงมันก็มีโอกาสตอบ 500 หรือ timeout กลับมาครับ ไม่ใช่เพราะเราทำอะไรผิด แต่เพราะมันก็เป็น server ตัวนึงบนอินเทอร์เน็ต ที่มีวันแย่ ๆ ของมันได้ การ POST แบบยิงทีเดียวจบ พอเครื่องเขาสะดุดขึ้นมาครั้งแรก event ก็หล่นหายไปเลย

**วิธีแก้:** อันที่นึกออกง่ายสุด — retry สักสองสามครั้งก่อนจะยอมแพ้

```
on order_paid(order):
  for attempt in 1..3:
      resp = http.post(partner_url, body=event(order))
      if resp.ok: return 200
      sleep(1s)
  return 200   # gave up
```

แบบนี้ดีขึ้นนะครับ แล้วสำหรับ integration เล็ก ๆ มันอาจจะพอแค่นี้เลยก็ได้ แต่มีอย่างนึง — เวลาเขียนโค้ดเอาไปใช้งานเอง คุณจะไม่มานั่งเขียน loop นี้เองหรอกครับ ทุก ecosystem มี retry library ที่ผ่านสนามมาแล้วคอยจัดการเรื่องนับจำนวนครั้ง จัดการ delay แล้วก็ (ที่สำคัญ) ดูว่า error แบบไหนที่ retry แล้วถึงจะคุ้ม:

- **Java** — resilience4j (`Retry`), Spring Retry (`@Retryable`), หรือ Failsafe
- **Go** — `avast/retry-go` หรือ `cenkalti/backoff`
- **Python** — `tenacity` (หรือ `Retry` adapter ที่มีมาให้ใน `requests` / `urllib3`)

คว้าตัวพวกนี้มาใช้ดีกว่ามานั่ง reinvent เองครับ — มันจัดการรายละเอียดจุกจิกได้ดีกว่าเยอะ เดี๋ยวเรื่อง tune policy เอง (backoff, jitter, เพดาน) ค่อยว่ากันอีกไม่กี่ step ข้างล่าง แต่ไม่ว่าคุณจะใช้ library ตัวไหน ลองอ่านโค้ดดี ๆ นะครับ: **retry พวกนี้มันก็ยังเกิดแบบ inline อยู่ข้างใน request ของคนเรียกอยู่ดี** นั่นแหละครับคือจุดที่ step ถัดไปจะเข้ามาขยายต่อ

## Sync retries block you — go async

**ปัญหา:** sync retry มันยิ่งทำให้ coupling *แย่ลง* ไม่ใช่ดีขึ้นครับ พอ partner ช้า เราไม่ได้ fail เร็ว ๆ — เรานั่งถือ request นั้นไว้ (พร้อมกับ worker thread ตัวนึง) เป็นเวลา `timeout × จำนวนครั้ง` แถมหลับรอระหว่างแต่ละครั้งอีก แล้วจุดที่แย่ที่สุดคือ เรา retry *หนักที่สุด* ก็ตอนที่ partner *ช้าที่สุด* พอดี ซึ่งเป็นจังหวะที่เราเสีย thread ไปแบบไม่คุ้มที่สุด พอโดน load หนัก ๆ เข้า partner ที่กำลังแย่อยู่รายเดียวก็ดูด thread pool เราจนหมด แล้วก็ลากเอา request อื่นที่ไม่เกี่ยวกับ webhook เลยล้มตามไปด้วย

**วิธีแก้:** เอางาน delivery ออกไปจาก thread ของคนเรียกให้หมดเลยครับ ตัว request handler ทำให้น้อยที่สุด — แค่บันทึกว่า `event นี้ต้องถูกส่งออกไปนะ` ลง durable queue (SQS, RabbitMQ, Kafka topic, แล้วแต่จะเลือก) แล้วก็ return จากนั้นมี **delivery worker** แยกอีกตัวค่อยดึงจาก queue มาทำ POST (แล้วก็ retry)

```
on order_paid(order):
  queue.put(event(order))   # durable, returns in ms
  return 202

# elsewhere, a worker:
loop:
  e = queue.take()
  deliver(e)                # POST + retries live here now
```

อันนี้คือการเปลี่ยนที่ leverage สูงที่สุดในโพสต์เลยครับ latency ของ partner กลายเป็นปัญหาของ *worker* ไม่ใช่ของ API เราแล้ว — แถมงานมันก็ durable ด้วย crash ไปก็ไม่ทำ event หาย เกือบทุกอย่างข้างล่างนี้คือการขัดเกลาไอเดีย `ส่งแบบ async โดยอิงกับ durable queue` ให้เนียนขึ้นเฉย ๆ

แล้วก็พูดกันแบบเปิดอกนะครับ ถ้าลองนั่งคิดดูดี ๆ async ในเคสนี้มันไม่ได้ดีแค่เรื่องความไวอย่างเดียว — แต่มันเป็นโมเดลที่เข้ากับเนื้องานมากกว่าด้วย: ตัวที่ **ทำให้เกิด** event กับตัวที่ **ควรได้รับ** callback มันคนละฝ่ายกันเลยครับ ลูกค้าที่จ่ายเงินจนเกิด `order.paid` ไม่ได้แคร์หรอกว่า endpoint ของ partner จะ ack อะไรกลับมามั้ย แล้วก็ไม่ควรต้องมานั่งรอมันด้วย ต้นทางของ request กับปลายทางของการ notify มันคนละที่กัน — เพราะงั้นฝั่งคนเรียกมันไม่เคยมีอะไรที่ต้องใช้ผลการส่งแบบ inline เลยตั้งแต่แรก จะไปบล็อกมันไว้ก็ไม่เคยมีความหมายมาตั้งแต่ต้น งาน notify แบบนี้มันคือ fire-and-forget โดยธรรมชาติของมันอยู่แล้วครับ

## Don't hammer, don't loop forever — backoff + max attempts

**ปัญหา:** retry แบบเมื่อกี้ของเรามันยังหยาบอยู่ครับ — delay คงที่ แล้วก็ (พูดกันตามเนื้อผ้า) ไม่มีอะไรห้ามไม่ให้มันวนไปเรื่อย ๆ ไม่จบ ซึ่งมันมีโหมดพังซ่อนอยู่สองอย่าง อย่างแรก การ retry ซ้ำ ๆ ใส่ endpoint ที่กำลังแย่อยู่แบบรัว ๆ ทันที ก็เหมือนไปซ้ำเติมตอนที่เขากำลังจะล้มอยู่แล้ว อย่างที่สอง ถ้า endpoint มันตายสนิทไปแล้ว retry แบบไม่มีเพดานก็คือ worker ที่วนอยู่กับ event ตายตัวเดียวไปตลอด ไม่ยอมระบาย queue ที่ต่อแถวอยู่ข้างหลังเลย (แล้วขอตอบคำถามที่หลายคนน่าจะสงสัย — *ทำไม backoff ถึงเพิ่งมาโผล่จุดนี้ ไม่ไปอยู่ตั้งแต่ step retry*: การรอ 1s, 2s, 4s, 8s ระหว่างแต่ละครั้ง ถ้ามันรันแบบ inline ก็เท่ากับพิน request thread ไว้ 15 วินาทีกว่า ๆ เพราะงั้น backoff กับเพดานจำนวนครั้งข้างล่าง ถึงเพิ่งปลอดภัยพอจะเอามาใช้ได้ ก็ตอนที่เราย้าย retry ไปอยู่บน background worker แล้วครับ)

**วิธีแก้:** **[exponential backoff พร้อม jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/) แล้วก็เพดาน max-attempt แบบตายตัว**

- **Exponential backoff** — รอนานขึ้นเรื่อย ๆ ในแต่ละครั้ง (1s, 2s, 4s, 8s, …) เพื่อให้ endpoint ที่กำลังฟื้นมีที่หายใจ
- **Jitter** — ใส่ความสุ่มเข้าไปใน delay ถ้าไม่มีมัน ทุก event ที่กองไว้ตอน outage ก็จะ retry พร้อมกันเป๊ะ ๆ ในวินาทีที่ endpoint กลับมา — กลายเป็น thundering herd ที่เราสร้างขึ้นเองแล้วก็ทุบเขาล้มซ้ำอีกรอบ
- **Max attempts** — หลังจาก N ครั้ง (หรือหมด time budget) ก็หยุด event ไม่ได้หายนะครับ — มันไปอยู่ที่ที่เราจะจัดการทีหลัง (ตัว DLQ อีกไม่กี่ step ข้างล่าง) หลักการ `retry ได้ แต่ต้องมีเพดาน` คือหัวใจทั้งหมด

### Circuit breaker — stop dialing a dead endpoint

backoff กับเพดาน max-attempt มันคุมการ retry ของ *event ตัวเดียว* ได้ครับ แต่ลองนึกภาพ partner ที่ล่มสนิทมาเป็นชั่วโมงดูสิครับ: **ทุก event ใหม่** ก็ยังเปิด connection รอจน timeout แล้วก็ไล่ backoff ครบทั้งชุดกว่าจะยอมแพ้ — เท่ากับเราเผาเวลา worker เผา connection แล้วก็ถ่วง latency ของทั้ง queue ไปกับการกระแทก endpoint ที่เรารู้อยู่แล้วว่ามันตายไปแล้ว แถมยังไปทำให้ partner ที่ *ยังดี ๆ* ที่ต่อแถวรออยู่ข้างหลังอดส่งไปด้วย

**circuit breaker** แก้จุดนี้ด้วยการคอยจับชีพจรของแต่ละ endpoint ครับ:

- **Closed** (ปกติ) — delivery ไหลได้ตามปกติ เราก็แค่นับ failure เก็บไว้
- **Open** — พอ failure rate เกิน threshold ปุ๊บ เราก็ *หยุดเรียก endpoint นั้นไปเลย* ชั่วคราว event ใหม่ ๆ fail fast (เด้งไป retry-later / DLQ เลย) แทนที่จะเสีย timeout เต็ม ๆ ทีละตัว
- **Half-open** — พอครบ cooldown ก็ปล่อย delivery ทดลองผ่านไปสักสองสามตัว ผ่าน → กลับไป close แล้วเดินต่อ; ยังไม่ผ่าน → เปิดวงจรค้างไว้แล้วรอใหม่

แบ่งหน้าที่กันแบบนี้ครับ: **backoff คอยถ่างจังหวะ retry ของ delivery ตัวเดียว ส่วน breaker คอยตัดทั้งสายของ delivery ที่จะวิ่งไปหา host ที่ล่มชัด ๆ** สองตัวนี้ทำงานเสริมกัน — library ชุดเดิมจาก step ที่แล้วมีมาให้ทั้งคู่เลย (resilience4j มีทั้ง `Retry` แล้วก็ `CircuitBreaker`) แล้ว breaker มันก็แยกตาม endpoint ด้วย partner ที่ตายไปรายนึงก็ตัดวงจรของตัวเองไป โดยไม่ไปกระทบ delivery ของรายอื่นเลยครับ

## Anyone can POST to them — sign it (HMAC)

**ปัญหา:** endpoint ที่รับ webhook ของ partner มันคือ **URL สาธารณะบนอินเทอร์เน็ต** ครับ ลองนึกภาพว่ามีใครสักคนไปรู้ URL นี้เข้า เขาก็ POST เข้าไปได้เลย แล้วก็แกล้งทำเป็นเราได้ด้วย — ยิง `order.paid` ปลอม ๆ, `refund.completed` ปลอม ๆ เข้าไป ฝั่ง consumer ไม่มีทางแยกออกเลยว่าอันไหน event ของเราอันไหนของปลอม แล้วถ้าคุณเผลอเอารายละเอียดไปยัดไว้ใน **URL หรือ query string** (`?amount=5000&user=123`) ก็เท่ากับทำมันรั่วไปแล้ว — query string มันไปโผล่ใน access log, proxy, แล้วก็ประวัติ browser

**วิธีแก้:** พิสูจน์ว่า message มาจากเรา แล้วก็เก็บข้อมูลออกไปจาก URL ซะ

- **เซ็น payload** คำนวณ [HMAC](https://en.wikipedia.org/wiki/HMAC) (เช่น HMAC-SHA256) ของ raw request body ด้วย **shared secret** ที่มีแค่เรากับ consumer รู้ แล้วก็ส่งไปใน header (`X-Signature`) ฝั่ง consumer ก็คำนวณ HMAC จาก body ที่เขาได้รับมาใหม่แล้วเทียบ — ถ้าไม่ match ก็แปลว่าโดนปลอมหรือโดนแก้ไขระหว่างทาง เขาก็ทิ้งไป
- **เซ็น timestamp ไปด้วย** แล้วก็รวมมันเข้าไปในสิ่งที่เซ็น ฝั่ง consumer จะ reject อะไรก็ตามที่เก่าเกินไม่กี่นาที ซึ่งกัน **replay attack** ได้ (พวกที่ดักจับ call ที่ valid แล้วเอามายิงซ้ำ)
- **ทุกอย่างที่มีความหมายอยู่ใน body** ไม่ใช่ URL ตัว URL เป็นแค่ที่อยู่ ส่วน body ที่เซ็นแล้วคือตัว message

```
sig = hmac_sha256(secret, timestamp + "." + raw_body)
POST /webhooks/orders
  X-Timestamp: 1718000000
  X-Signature: sha256=...
  { "id": "...", "type": "order.paid", ... }   # data lives here
```

อันนี้คือด่านแรกสุดของการประมวลผลฝั่ง consumer เลยครับ: ก่อนจะไปดูเนื้อหาด้วยซ้ำ เขาถามก่อนว่า `อันนี้มาจากเขาคนนั้นแน่นะ?` ยืนยันตัวตนก่อน แล้วค่อยว่ากันเรื่องอื่น

**ขอถอยมามองภาพรวมหน่อยนะครับ — สุดท้ายแล้วสิ่งที่เราต้องการคือ trust แล้ว HMAC ก็เป็นแค่ทางนึงที่ทำให้ได้ trust นั้นมา ไม่ใช่ทางเดียว** ในคำว่า `เชื่อ webhook อันนี้ได้มั้ย` มันมีคำถามซ่อนอยู่สองข้อที่ต่างกัน แล้วก็ตอบด้วยเครื่องมือคนละตัว:

- **การเซ็น (HMAC)** ตอบคำถามว่า *payload ก้อนนี้ของแท้แล้วก็ไม่โดนแก้รึเปล่า?* — integrity + origin ของ message ทำแบบ stateless ไม่ต้องมี token infra เลย เป็น default ของโลก webhook (Stripe, GitHub)
- **Auth ([OAuth2 client-credentials](https://oauth.net/2/grant-types/client-credentials/) Bearer token หรือ [mTLS](https://en.wikipedia.org/wiki/Mutual_authentication))** ตอบคำถามว่า *คนที่เรียกมาเป็น sender ที่ได้รับอนุญาตมั้ย?* — เป็นเรื่อง identity ระดับ connection มันต่อกับ IdP ที่ผู้รับมีอยู่แล้ว, token ก็อายุสั้นแล้วก็ revoke ได้ — แต่แลกมากับที่ฝั่งคนส่งต้องไปขอแล้วก็ cache token ไว้ก่อนยิงทุกครั้ง

ขอแปะ caveat อันนึงนะครับ ถ้าคุณจะไปทาง OAuth2: flow ที่เข้ากับงานนี้คือ **client credentials** — machine-to-machine ไม่มีคนมาเกี่ยวกลางทาง ส่วน *authorization code* เอาไว้ทำ user-delegated consent (คนกดอนุญาตให้แอปเข้าถึงข้อมูล) ซึ่งมันไม่เข้ากับ callback แบบ server-to-server เพราะงั้นไม่ใช่เครื่องมือสำหรับงานนี้

สองอย่างนี้ไม่ได้แข่งกันนะครับ — มันตอบคนละคำถาม แล้วหลายระบบก็ใช้ทั้งคู่เลย: token ไว้พิสูจน์ว่า *ใครเรียกมา* ส่วนลายเซ็นไว้พิสูจน์ว่า *message ไม่โดนแตะ* เลือกตาม threat model ของคุณเอง สุดท้ายแล้วทุกอย่างมันก็รับใช้เป้าหมายเดียวกัน — ทำให้ผู้รับเชื่อ event มากพอที่จะลงมือกับมัน

## Your retries cause doubles — idempotency

**ปัญหา:** ทุกอย่างที่เราสร้างมามันเป็น **at-least-once** หมดเลยครับ worker ก็ retry; queue ก็ redeliver ได้หลัง crash; ส่วน delivery ที่ส่งสำเร็จไปแล้วแต่ ack ดันหายระหว่างทาง ก็โดนส่ง *ซ้ำ* อีกที เพราะงั้น consumer มัน **จะ** ได้รับ event เดิมซ้ำมากกว่าหนึ่งครั้งแน่ ๆ เราเจอกำแพงนี้มาแล้วเป๊ะ ๆ ทั้งในโพสต์ payment แล้วก็ push — exactly-once แบบเป๊ะข้ามเครือข่ายมันเป็นเรื่องเพ้อฝัน ([Two Generals Problem](https://en.wikipedia.org/wiki/Two_Generals%27_Problem)) สำหรับ dashboard ของซ้ำมันก็แค่เรื่องผิวเผิน แต่สำหรับ **AI agent** `order.paid` ที่ซ้ำ คือมีการสั่ง order ซ้ำอีกรอบ คือเงินถูกตัดสองรอบ

**วิธีแก้:** ให้ทุก event พก **`event_id` ที่นิ่งแล้วก็ไม่ซ้ำใคร** (กำหนดครั้งเดียวตั้งแต่ต้นทาง — แล้วพกติดตัวไปไม่เปลี่ยนทุก retry) แล้วก็ทำสัญญาเรื่องการส่งให้เคลียร์ ๆ ไปเลยว่า `เราส่งแบบ at least once นะ เพราะงั้นช่วยไป dedupe ด้วย event_id เอาเองนะ` ฝั่ง consumer ก็จดว่า id ไหนประมวลผลไปแล้วบ้าง แล้วก็ข้ามตัวที่ซ้ำ เรารับประกันการส่งถึง เขารับประกันการประมวลผลแบบ idempotent มันคือสัญญาร่วมกัน แล้วก็ควรเขียนไว้ดัง ๆ ใน webhook docs ของเราด้วยนะครับ

```
{
  "id": "evt_7f3c8a",
  "type": "order.paid",
  "data": { "order_id": "ord_123", "amount": "245 THB" }
}
```

สังเกตลำดับกับ step ก่อนหน้านะครับ: ฝั่ง consumer **ยืนยันตัวตน (HMAC) ก่อน แล้วค่อย dedupe (`event_id`) แล้วค่อยประมวลผล** เชื่อใจมันก่อน แล้วเช็คว่าเคยเห็นรึยัง แล้วค่อยลงมือ

## Some never succeed — DLQ + redrive

**ปัญหา:** backoff กับเพดาน max-attempt หยุดการวนไม่จบได้ก็เถอะครับ แต่มันสร้างคำถามใหม่ขึ้นมา: *แล้ว event ที่ retry จนหมดโควต้าไปแล้วล่ะ ไปไหนต่อ?* partner ล่มเป็นชั่วโมง ๆ ได้ เรา retry ตลอดกาลไม่ได้ (ตกลงกันไปแล้ว) แต่การปล่อย event หล่นหายเงียบ ๆ ก็แปลว่า notification ของ order หายไปดื้อ ๆ — แล้วสำหรับ consumer ที่ขับเคลื่อนด้วย agent มันคือ state ที่หายไปแล้วกู้กลับมาไม่ได้เลย

**วิธีแก้:** **dead-letter queue (DLQ)** ครับ พอ event ข้ามเพดาน max-attempt ก็ย้ายมันไปเก็บที่ store แยกแทนที่จะทิ้ง ทีนี้:

- **operator มองเห็นแล้วก็ลงมือได้** — ส่องดูว่าทำไมมันถึง fail แล้วก็ **redrive** (เอากลับเข้า queue ใหม่) ทั้งก้อนตอนที่ partner กลับมาปกติ
- **ให้ทางกู้คืนกับ consumer ด้วย** — API แบบ `list/replay event ตั้งแต่จุด X` เพื่อให้ partner ที่เคยล่มดึงของที่พลาดไปกลับมาได้ (อันนี้คือด้านกลับของเหรียญเดียวกับโมเดล polling เก่า — เป็น channel ไว้ตามเก็บโดยเฉพาะ ไม่ใช่ทางหลัก)

รายละเอียดเชิงปฏิบัติจากระบบหน้างานครับ: ผูกการย้ายเข้า DLQ ไว้กับ **receive-count** (message นี้ถูกดึงออกมาแล้ว fail ไปกี่ครั้งแล้ว) แล้วก็กำหนด **redrive policy** ไว้ล่วงหน้าเลย — ว่า message ที่หมดโควต้าจะไปไหนแล้วก็จะกลับมายังไง อย่าเพิ่งไปนั่งคิดสูตรมันตอนเกิด incident นะครับ

**การรัน DLQ บนหน้างานมันมีอะไรมากกว่าแค่ `queue ตัวที่สอง` เยอะครับ** มีเรื่องที่กัดเราตอนมันขึ้นใช้งานอยู่หลายอย่าง:

- **queue ไม่ใช่ที่เก็บของระยะยาว** broker ส่วนใหญ่มันลบ message ทิ้งตามอายุ (SQS เก็บได้สูงสุด 14 วัน) — เพราะงั้น DLQ ที่เป็น *แค่* queue เฉย ๆ มันจะค่อย ๆ ทำ failed event หล่นหายเงียบ ๆ พอเลยกำหนด ซึ่งมันก็เสียจุดประสงค์ไปเลย ของอะไรที่หายไม่ได้ ให้ **persist event ที่ตกลง DLQ ลง store ที่ทนทาน** — ตาราง `failed_deliveries` (หรือ object storage) สไตล์ [Outbox](https://microservices.io/patterns/data/transactional-outbox.html) — ที่เรา query, audit, แล้วก็ redrive ได้ทุกเมื่อ ไม่ใช่ queue ที่กำลังแข่งกับนาฬิกานับอายุอยู่
- **ต้องมีคนคอยจับตาดูมัน** operator ส่อง DLQ ด้วยการ **poll ดูเองแบบ manual** (จะเป็น admin view เล็ก ๆ หรือ cloud console ก็ได้) — list, filter ตาม endpoint หรือ error, แล้วก็เปิดอ่าน payload ที่ fail ทีละอัน DLQ ที่ไม่มีใครเปิดดูเลย ก็เป็นแค่วิธีทำข้อมูลหายแบบช้าลงเฉย ๆ ครับ
- **redrive แบบเจาะจง อย่า redrive มั่ว ๆ** ตอนเกิด incident เราแทบไม่อยากกด `replay ทั้งหมด` หรอกครับ — batch เสีย ๆ มันก็แค่ fail ซ้ำหรือยิงซ้ำ ให้ **consume / เลือกแบบ manual** เฉพาะ message ที่เกี่ยวข้องเท่านั้น (endpoint นี้, ช่วงเวลานี้) แล้วก็ redrive เฉพาะพวกนั้น
- **redrive ด้วยความระวังเท่ากับตอนส่งครั้งแรก** เอากลับเข้า queue โดยพก **idempotency key เดิม** ไปด้วย (consumer จะได้ยัง dedupe ได้อยู่), **throttle** การ redrive ด้วย (endpoint ที่เพิ่งฟื้นมาไม่อยากได้ backlog ทั้งกองทีเดียว — บทเรียน thundering herd อันเดิม) แล้วก็คงเพดานจำนวนครั้งไว้
- **ตั้ง alert ที่ความลึกของ DLQ** DLQ ที่โตขึ้นเรื่อย ๆ คือ signal แรกสุดที่บอกว่า partner — หรือชั้น delivery ของเราเอง — กำลังพัง เปิด page แจ้งเตือนไว้ อย่าให้ต้องไปรู้เอาตอน partner ส่งเมลมาทวงนะครับ
- **ให้ store ที่ persist ไว้มี lifecycle ด้วย** สำเนา audit ของ failed event ต้องมี TTL / นโยบาย archive ไม่งั้นตารางนั้นมันโตไม่หยุด

## They arrive out of order — sequencing and the state machine

**ปัญหา:** การส่งแบบ async + retry + worker หลายตัวขนานกัน แปลว่า event ของ resource เดียวกันมา **ผิดลำดับ** ได้ครับ `order.paid` แล้วตามด้วย `order.refunded` แต่ consumer ดันเห็น `refunded` มาก่อน หรือ event เก่าที่โดน retry ดันมาถึงทีหลังตัวที่ใหม่กว่า คนเรายังพอทนได้ แต่ agent ที่ลงมือตามลำดับที่ได้รับ มันทำผิดเลย

**วิธีแก้:** มีอยู่สองชิ้น แล้วก็ไม่ต้องมีมากกว่านี้ครับ

- **sequence/version ต่อ resource + last-write-wins** — แปะ sequence number หรือ version ที่เพิ่มขึ้นเรื่อย ๆ ต่อ resource ไว้ในทุก event ฝั่ง consumer ก็เก็บ version ล่าสุดที่เห็นไว้ แล้วข้ามอะไรที่เก่ากว่าทิ้งไป ถ้าอยากได้ลำดับการประมวลผล ไม่ใช่เอาแค่ state ล่าสุด ก็ **partition ตาม resource id** ให้ทุก event ของ order เดียวไหลผ่าน lane เดียวที่เรียงแล้ว เราแทบไม่เคยต้องการ *global ordering* เลยนะครับ — ต้องการแค่ลำดับ **ต่อ resource** เท่านั้น อย่าไปสร้างระบบ total-order ที่ไม่ได้ใช้
- **state machine ที่บังคับ transition** — อันนี้แหละคือตัวกันหลักครับ แล้วก็เป็นวิธีเดียวกับที่เราทำฝั่งคนรับในโพสต์ payment เลย ฝั่ง consumer มอง lifecycle ของแต่ละ resource เป็น state ที่มี transition ที่ถูกต้องของมัน แล้วก็ปฏิเสธอันที่ไม่สมเหตุสมผล — `refunded` ที่มาก่อน `paid` มันก็ไม่ใช่ transition ที่ออกจาก state ปัจจุบันได้ ก็เลยโดนข้ามหรือพักไว้ ไม่ใช่เอาไปลงมือ ลำดับการ *มาถึง* เลยไม่สำคัญ สำคัญแค่ลำดับของ *state* เท่านั้น

**พอ flow เริ่มมี dependency ซ้อนกัน** การมองมันเป็น **DAG** ก็ช่วยได้ครับ — ไม่ใช่เพื่อเอาไป *สร้าง* อะไร แต่เป็นวิธีคิดว่าอะไรต้องมาก่อนอะไร ส่วนใหญ่ lifecycle มันก็เป็นเส้นเดียว จนกระทั่งไปเจอ **fan-in (จุดรวม)**: event ที่ต้องรอ prerequisite *ตั้งแต่สองตัวขึ้นไป*

`shipment.created` คือจุดรวม — มันต้องรอ **ทั้ง** `order.paid` *และ* `inventory.reserved` ซึ่งระบบเราส่งสองตัวนี้แยกกัน แล้วก็มาผิดลำดับได้ การมอง event เป็น DAG ก็แค่ช่วยให้เราเห็นว่า transition ไหนบ้างที่ state machine ต้องคอยกันไว้ ส่วนงานของฝั่ง *คนส่ง* ก็ยังเล็กเหมือนเดิมครับ: ลำดับต่อ resource บวกกับ id ในแต่ละ payload ให้พอ — `event_id` แล้วก็ `correlation_id` (แบบ `order_id`) ไว้ผูก flow เดียวกันเข้าด้วยกัน — เพื่อให้ฝั่งคนรับเอาไปโยง event กันได้ ที่เหลือเป็นงานของ state machine ฝั่งเขา **ระบบ webhook ของเราไม่ใช่ workflow engine อย่าไปทำให้มันกลายเป็นแบบนั้นนะครับ**

แล้วนี่แหละคือจุดที่มุม agent สำคัญที่สุด: agent ที่ลงมือกับทุก event ทันทีที่มันมาถึง มันจะไปสั่ง logic เรื่อง shipment ของ order ที่ยังไม่เคยถูกจ่ายเงินเลย — ถ้าไม่มี state machine คอยปฏิเสธ transition นั้นไว้ ตัวกันอันนี้แหละที่ไม่ให้ agent ไปลงมือกับ state ที่ยังประกอบไม่เสร็จ ถ้าเป็นคนนั่งดู dashboard อยู่ เห็นอะไรแปลก ๆ ก็คงทักได้ทัน แต่ agent มันไม่ได้รู้จักเอะใจแบบนั้นครับ

## Pull it apart — acceptor / dispatcher / consumer

**ปัญหา:** มาถึงตอนนี้ service ตัวเดียวทำงานเยอะมากครับ — รับ event ภายใน, หาว่า *ใคร* subscribe ไว้บ้าง, แล้วก็ทำ delivery ออกนอกที่ช้า ๆ พร้อม retry ทั้งหมด งานสามอย่างนี้รูปร่างคนละแบบกันเลย พอเอามาแปะรวมกัน ตัวที่ช้าแล้วก็พังบ่อยที่สุดมันก็ลากตัวอื่นจมตามไปด้วย partner endpoint ช้า ๆ ที่ถาโถมเข้ามา ก็ไปทำให้งาน delivery ตันแล้วก็ไปอุดการรับ event ใหม่ที่ต่อแถวอยู่ข้างหลังด้วย

**แต่ขอพูดให้เคลียร์ ๆ หน่อยนะครับ — service ตัวเดียวที่ทำครบทั้งสามอย่างเนี่ย มันคือจุดที่แอปบนหน้างานส่วนใหญ่อยู่กัน แล้วก็โอเคแบบนั้นด้วยนะ** ไม่ว่าจะ package เป็น deployable ตัวเดียว หรือเป็นสแตก **serverless** (event → SQS / EventBridge → Lambda ที่เซ็นแล้วก็ส่ง → DLQ) เครื่องจักร queue-worker-retry-DLQ จากไม่กี่ step ที่ผ่านมาก็แบก traffic หน้างานได้เยอะแล้ว ไม่ต้องไปแยกอะไรเลย โดยเฉพาะ serverless เนี่ยมันแถม retry, การ scale, แล้วก็ DLQ มาให้แบบสำเร็จรูปด้วย **จะงัดการแยกข้างล่างมาใช้ ก็ต่อเมื่อโหลดหรือรูปแบบการพังของ tier นึงมันเริ่มลากตัวอื่นให้จมตามเท่านั้น** — ก่อนหน้านั้นยังไม่ต้องครับ

### The split that matters: acceptor → queue → consumer

**วิธีแก้: ในงานสามอย่างนั้น มีแค่สองอย่างที่ต้องแยกเป็น tier ของตัวเอง — acceptor → queue → consumer**

- **Acceptor** (ขาเข้า) — รับ event แบบ `มีอะไรเกิดขึ้น` มา **เก็บให้ไว durable แล้วก็ return** เน้นงาน write เป็นหลัก ห้ามทำ event หายเด็ดขาด ตั้งใจทำให้มันผอม ๆ ไว้ เพื่อให้การรับเข้าไม่ติดขัดเพราะอะไรที่อยู่ปลายน้ำ
- **Consumer** (delivery) — ดึงจาก queue แล้วทำ POST ที่เซ็นแล้วช้า ๆ นั่นแหละ: HMAC, backoff, idempotency id, track ผล, DLQ เป็นงานที่ติด I/O โดน endpoint ข้างนอกที่ช้าบล็อกไว้ แล้วก็ fail บ่อย อยากได้ตัวนี้เยอะ ๆ

การแยกสอง tier นี้แหละคือหัวใจครับ: แยกฝั่งรับเข้าที่เก็บของได้ไวแล้วก็ทนทาน ออกจากฝั่ง delivery ที่ช้าเพราะต้องไปง้อของข้างนอก พอแยกกันแล้ว ตัวนึงล่มก็ไม่ลากอีกตัวจมไปด้วย

### Knowing where to send — the metadata store

มีอยู่ชั้นนึงที่เราแอบข้ามมาตลอดเลยครับ: จะส่ง webhook ได้ ตัว consumer มันต้องรู้ก่อนว่า **ส่งไปไหน** แล้วก็ **ส่งยังไง** — endpoint URL, secret ที่ใช้เซ็น, แล้วก็ partner รายนี้ subscribe event type ไหนไว้บ้าง ในเวอร์ชันซื่อ ๆ ตอนแรกมันคือ URL ที่ hardcode ไว้ แต่พอมี partner หลายรายเข้า มันก็กลายเป็น **subscription / partner metadata store** — ตารางนึงที่ฝั่ง delivery ต้องอ่านในทุก event:

- *ส่งไปไหน* — endpoint URL ของแต่ละ partner
- *ส่งยังไง* — secret ของ HMAC (หรือ OAuth client id), API version, custom header
- *ส่งอะไรบ้าง* — เซ็ตของ event type ที่เขา subscribe ไว้
- *ลิมิตเท่าไหร่* — rate cap ราย partner แล้วก็ flag active/paused

**แต่การอ่าน store นี้ในทุก ๆ event มันก็เป็นปัญหาของมันเองครับ** ที่ระดับหลายพัน delivery ต่อวินาที เราก็จะไปกระแทก database ด้วย lookup เดิม ๆ ที่แทบไม่เปลี่ยนเลยซ้ำ ๆ เพราะงั้น **cache มันซะ** — จะเป็น in-process cache ในแต่ละ worker สำหรับ hot path หรือจะใช้ **Redis / Memcached** ที่แชร์กันตอนอยากได้ view เดียวที่เหมือนกันทั้งหลาย worker หลาย instance ก็ได้ ใส่ TTL สั้น ๆ หรือ invalidate ตอนมีการเปลี่ยน เพื่อให้ partner ที่หมุน secret ใหม่หรือกดหยุด delivery มีผลไว ๆ ไม่ใช่รอจนพ้นช่วง cache ค้าง ชั้น metadata นี้แหละที่ทั้งสองแบบของการ route ข้างล่างต้องอ่านจากมัน — ทั้ง consumer ที่ route ผ่าน broker แล้วก็ตัว dispatcher

### So where's the dispatcher?

สังเกตว่าผมไม่ได้ใส่มันเข้าไปนะครับ งานที่ dispatcher ทำ — *พา event ที่ถูกไปหา endpoint ที่ถูก* — มันก็ยังต้องเกิดขึ้นอยู่ดี แต่มันไม่จำเป็นต้องเป็น component แยกของตัวเอง การ route มันมีอยู่สองทาง แล้วการเลือกระหว่างสองทางนี้แหละคือจุดตัดสินใจที่แท้:

- **route ผ่าน broker (ไม่ต้องมี dispatcher)** partition queue/stream ตาม route หรือชนิดของ event แล้วก็ให้แต่ละ **consumer subscribe เฉพาะ route ที่ตัวเองดูแล** — การ route ไปอยู่ในโครงสร้างของ topic ไม่ใช่ใน service deploy ได้สองแบบ:
  - **service ตัวเดียว รัน N consumer** — consumer ละ route อยู่ใน deployable เดียวกัน หรือ
  - **N service แยกกัน อันละ route** — อิสระ deploy แล้วก็ scale ของใครของมัน

  อันนี้ง่ายสุดตอนที่ routing มัน **fix แล้วก็รู้แน่ชัด** — event ไม่กี่ชนิดวิ่งไปที่ที่รู้กันอยู่แล้ว
- **dispatcher (tier ที่สาม)** component กลางที่ไปหาว่า *ใคร subscribe ไว้* แล้ว apply subscription config แล้วก็กาง event ออกเป็น task ราย endpoint มันคุ้มก็ตอนที่ routing มัน **dynamic** — ลูกค้าลงทะเบียน endpoint กับ event filter เองตอน runtime หรือเราต้องทำ transform / rate cap ราย subscription นั่นแหละคือ logic การ route ที่มีน้ำหนักพอจะอยากมีบ้านหลังเดียว บังคับใช้ที่เดียว แทนที่จะกระจายไปทั่วทุก consumer

การแลกแบบเปิดอกครับ: consumer ที่ route ผ่าน broker มันง่ายแล้วก็ decouple สุด ๆ — เพิ่ม consumer ได้โดยไม่ต้องแตะ upstream, ได้ replay จาก offset มาฟรี ๆ — แต่ไม่มีตัวกรองกลาง เพราะงั้นแต่ละ consumer ก็อ่านของมากกว่าที่ตัวเองต้องใช้ แล้วก็ต้องพึ่ง event schema ที่นิ่ง ๆ เพราะทุกตัวตีความ raw stream เอง (ถ้า contract ผิดเมื่อไหร่ มันพังพร้อมกันหมด) ส่วน dispatcher ก็รวมศูนย์การ route กับ filtering ไว้ แลกกับ component ที่เราต้อง build แล้วก็ scale **routing ที่ลูกค้า config เองแบบเยอะ ๆ → dispatcher; routing ที่เราคุมเองแบบ fix → consumer ที่ route ผ่าน broker**

### What the split buys you

ไม่ว่าจะ route แบบไหน ผลตอบแทนก็เหมือนกันครับ — แล้วก็เป็นเส้นเรื่องเดียวกับทั้งซีรีส์ คือแยกฝั่งที่ไวแล้วก็ทนทาน ออกจากฝั่งที่ช้าเพราะต้องไปง้อของข้างนอก จะได้ไม่ลากกันจม:

- **scale แยกกันได้** — delivery พุ่งหรือเจอ partner ช้า ๆ มาเป็นพรวด ก็ scale แค่ *consumer*
- **แยก failure ออกจากกัน** — partner ที่มีปัญหาทำให้ตันแค่ฝั่ง consumer ส่วนการรับเข้าก็ยังรับต่อไปได้เรื่อย ๆ
- **deploy แยกกันได้** — เปลี่ยน routing หรือตัว consumer โดยไม่ต้องไปแตะการรับเข้า

สามข้อนั้นคือข้อดีแบบนามธรรม แต่คุณจะยังไม่ *รู้สึก* ถึงมันเลย จนกว่าจะลองไล่ดูว่าพอแต่ละชิ้นมันล้ม จะเกิดอะไรขึ้น:

- **acceptor (ฝั่งรับเข้า) ล่ม?** ก็ไม่มีของใหม่เข้ามา — ไม่มี event ลง queue เพราะงั้นก็ไม่มีอะไรให้ส่งต่อด้วย ฟังดูแย่ แต่ลองสังเกตสิ่งที่ *ไม่* เกิดขึ้น: ของที่ค้างอยู่ระหว่างทางไม่ได้หายไปไหนเลย เราแค่หยุดรับเข้าเฉย ๆ ส่วนอะไรที่อยู่ใน queue แล้วมันก็ไหลออกต่อไปเรื่อย ๆ พอ acceptor กลับมาปุ๊บก็ทำต่อจากเดิมได้เลย
- **consumer (delivery) ล่ม?** อันนี้แหละที่ queue ได้ทำหน้าที่สมชื่อมัน event มันก็แค่ *นอนรออยู่ใน queue แบบปลอดภัย* จนกว่า consumer จะกลับมาดึงไปส่ง การส่งแค่ดีเลย์ ไม่ได้หาย — ฝั่งผู้รับอาจจะสังเกตเห็นแล้ว ping มาถามเราด้วยซ้ำ (`เอ๊ะ event X หล่นหายไปรึเปล่า?`) แต่ของไม่ได้หายไปไหนเลย queue มันคือ **buffer** ตัวนึง — consumer ล่มเนี่ยแหละคือสถานการณ์ที่ buffer ตัวนั้นมีไว้รับมือพอดี
- **broker เองล่ม?** อันนี้แหละคือส่วนที่เรานิ่งนอนใจไม่ได้ครับ broker มันคือกระดูกสันหลัง — ถ้ามันล่ม acceptor ก็ไม่มีที่ให้วาง message ทั้ง flow ก็หยุดนิ่งสนิท เพราะงั้น broker เลยเป็นชิ้นเดียวที่เราต้องทำให้ HA สูงที่สุดเท่าที่งบจะไหว มันคือสิ่งเดียวที่ถ้าล่มแล้วเรารับไม่ได้เลย

นี่แหละครับหน้าตาแท้ ๆ ของสิ่งที่เราแลกมา: เรารวมชะตากรรมของทั้งระบบไปไว้ที่ component ตัวเดียวที่เข้าใจมันดีแล้วก็ replicate ได้ (ตัว broker) แล้วทำให้ทุกอย่างรอบ ๆ มันทิ้งแล้วเกิดใหม่ได้ acceptor กับ consumer จะ crash, redeploy, scale ขึ้นลงทั้งวันก็ได้ ระบบมันก็แค่รออยู่ใน buffer แล้วค่อยตามเก็บทีหลัง นี่แหละคือสิ่งที่คำว่า `independent` มันซื้อให้เรา

นี่คือรูปร่างคร่าว ๆ ที่ผมเคย build มากับมือครับ — ย้ายการต่อกับ partner ออกจากการ poll แบบ batch `GET` เก่า ๆ มาเป็น pipeline แบบ push real-time พร้อม retry ที่ขับด้วย queue (receive-count + redrive) ในเคสของผม routing มันซับซ้อนพอที่ผมเลยรัน **acceptor / dispatcher / consumer** เต็มรูปแบบ โดยแยกชั้น delivery ออกมาเพื่อให้ scale ของมันเองได้ — แต่ถ้า routing ของคุณมันง่ายกว่านั้น dispatcher นี่แหละคือชิ้นที่ควรข้ามไปครับ

## Wrapping up

เราเริ่มจาก `POST` ตัวเดียวแบบ inline แล้วมาจบที่ระบบ delivery ที่แยกชั้นแล้วก็ทนทานครับ เส้นทางทั้งหมด เรียงตามลำดับ:

1. **เอา delivery ออกจาก thread ของคนเรียก** ไปไว้บน durable queue — การเปลี่ยนอันเดียวที่ทุกอย่างต่อยอดจากมัน
2. **Back off ด้วย jitter แล้วก็มีเพดาน max-attempt** — แล้วก็ทริป **circuit breaker** กับ endpoint ที่ล่มชัด ๆ — จะได้ไม่กระหน่ำหรือวนไม่จบ
3. **เซ็นด้วย HMAC** (แล้วก็เก็บข้อมูลออกจาก URL) เพื่อให้ consumer เชื่อใจได้ว่าเป็นเรา
4. **`event_id` ที่นิ่ง** จะได้ไม่ให้ at-least-once กลายเป็นการลงมือซ้ำสองรอบ
5. **DLQ + redrive** เพื่อให้ outage ยาว ๆ ไม่ทำ event หาย
6. **ลำดับต่อ resource** แล้วก็ให้ฝั่งคนรับมี **state machine** คอยปฏิเสธ event ที่มาผิดลำดับ แทนที่จะลงมือตามนั้น — โดยที่เราไม่ต้องกลายเป็น workflow engine
7. **แยก delivery ออกมาอยู่ tier ของมันเอง** (acceptor → queue → consumer; เพิ่ม dispatcher ตอนที่ routing เริ่ม dynamic) เพื่อให้ส่วนข้างนอกที่ช้าลากการรับเข้าจมตามไม่ได้

ประเด็นใหญ่ก็ยังเหมือนตลอดทั้งซีรีส์ครับ: **ไม่มีใครควร build เวอร์ชันที่แยกชั้นออกมาตั้งแต่วันแรกหรอก** partner รายเดียว volume น้อย ๆ เหรอ? `POST` แบบ sync พร้อม retry สักสองครั้งก็โอเคสุด ๆ แล้ว ทุกกล่องข้างบนได้ที่ทางของมันมาจากการแก้ปัญหาที่เวอร์ชันก่อนหน้าสร้างขึ้น

สิ่งเดียวที่ **เพิ่ง** กลายเป็นเรื่องของปี 2026 คือ: consumer ปลายทางมีโอกาสมากขึ้นเรื่อย ๆ ที่จะเป็น **agent ที่ทำงานเองแล้วลงมือทันทีที่ webhook เราไปถึง** — เพราะงั้นของซ้ำที่เราไม่ได้ dedupe, event ที่เราส่งผิดลำดับ, call ปลอมที่เราไม่ได้กันด้วยลายเซ็น ทั้งหมดนี้กลายเป็นการลงมือทำผิดพลาดออกไปในโลกภายนอก แทนที่จะเป็นแค่ความผิดเล็ก ๆ บน dashboard งานเรื่องความน่าเชื่อถือมันเป็น engineering ที่ดีมาตลอดอยู่แล้ว ตอนนี้มันแค่กลายเป็นเส้นแบ่งระหว่าง agent ที่เชื่อใจ event ของเราได้ กับตัวที่เชื่อไม่ได้ครับ
