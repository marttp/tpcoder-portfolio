## Introduction

This is the fourth post in the backend system design series — same format: start with the most naive thing that works, then let real-world constraints push it into shape.

Back in the [payment post](/blog/payment-backend-stripe-integration-en/) we sat on the **receiving** end of webhooks — Stripe fired events *at us*, and we built a dumb receiver that parked them on a queue and processed them safely. This post flips the wire around: now **we're the sender**. Something happens in our system, and we have to deliver a callback to **someone else's endpoint — one we don't control, can't see, and can't trust by default.** Last time we received; this time we deliver.

Quick grounding, because it's the whole reason webhooks exist: before them, a partner who wanted to know "did this order get paid?" had to **poll** you — hit a `GET` every few minutes and diff the result. Wasteful, laggy, and it scales terribly. A webhook flips it to **push** — the moment `order.paid` happens, *you* call *them*. Cheaper and real-time, but now the hard problems are yours: their endpoint will be down, slow, or lying, and you still have to deliver.

**The 2026 twist that runs through this whole post:** the thing receiving your webhook used to be a partner's backend, with a human eventually reviewing a dashboard. Not anymore — increasingly the receiver is an **autonomous AI agent**, and the instant your webhook lands it *acts*: places an order, moves money, kicks off fulfillment, with no human confirming anything in between.

And that's what changes everything. A duplicate, out-of-order, or forged webhook used to be a small glitch someone would catch on a dashboard and fix. With an agent on the other end, it becomes the agent doing the wrong thing for real — with no one to hit the brakes. So the reliability of every step below matters far more than it used to.

The road, each step forced by the one before it:

1. A **synchronous POST**, inline
2. It fails sometimes → a **sync retry** (still inline)
3. Sync retries **block your thread** → go **async** (queue + worker)
4. **Backoff + max attempts** so you don't hammer or loop forever
5. **Sign it (HMAC)** so the consumer can trust it's really you
6. **Idempotency / event id** so your retries don't make them act twice
7. **DLQ + redrive** for deliveries that never succeed
8. **Ordering** — and a **state machine** so out-of-order events can't trigger the wrong action
9. End-state: split delivery onto its own tier (**acceptor → queue → consumer**, plus a dispatcher when routing turns dynamic)

## The naive version — just POST it

You have to start somewhere, and the honest start is: when something happens, `POST` it to the partner's URL right there in the request handler.

```
on order_paid(order):
  resp = http.post(partner_url, body=event(order))   # synchronous
  return 200
```

<MermaidDiagram chart={`
flowchart LR
    EV["order.paid happens"] --> SVC["Our service"]
    SVC -->|"POST (inline)"| EP["Partner endpoint"]
`} />

For one partner at low volume, this genuinely works — ship it. But notice what you've coupled: **the partner's endpoint is now on the critical path of your own request.** Their server's bad day is now your bad day.

### Where does that URL come from? — the metadata store

One foundational thing before we harden anything, because every step below quietly leans on it: that `partner_url` doesn't appear out of nowhere. The moment you have more than one partner, *where* and *how* to deliver lives in a **subscription / partner metadata store** — a table the delivery path looks up per event:

- *where* — the endpoint URL(s) per partner
- *how* — the signing secret (or OAuth client id), API version, custom headers
- *what* — the event types they subscribed to
- *limits* — per-partner rate caps, and an active/paused flag

Every delivery reads it, so once you're sending at any real volume you'll **cache it** — an in-process cache in each worker, or a shared **Redis / Memcached** for one consistent view across instances — with a short TTL or invalidation on change, so a rotated secret or a paused partner takes effect quickly. Everything from retries to routing reads off this layer.

## It failed — add a retry

**The problem:** someone else's endpoint *will* return a 500 or time out — not because you did anything wrong, just because it's a server on the internet. A one-shot POST drops the event on the floor the first time their box hiccups.

**The fix:** the obvious one — retry a few times before giving up.

```
on order_paid(order):
  for attempt in 1..3:
      resp = http.post(partner_url, body=event(order))
      if resp.ok: return 200
      sleep(1s)
  return 200   # gave up
```

This is better, and for a tiny integration it might be all you need. One thing though — in real code you wouldn't hand-write that loop. Every ecosystem has a battle-tested retry library that handles the attempt counting, the delays, and (importantly) *which* errors are even worth retrying:

- **Java** — resilience4j (`Retry`), Spring Retry (`@Retryable`), or Failsafe
- **Go** — `avast/retry-go` or `cenkalti/backoff`
- **Python** — `tenacity` (or the `Retry` adapter built into `requests` / `urllib3`)

Reach for one of those instead of reinventing it — they get the fiddly parts right, and we'll tune the actual policy (backoff, jitter, a ceiling) a few steps down. But read the code closely either way: **the retries are still happening inline, inside the caller's request.** That's the crack the next step widens.

## Sync retries block you — go async

**The problem:** sync retries make the coupling *worse*, not better. When the partner is slow, you don't fail fast — you sit there holding the request (and a worker thread) for `timeout × attempts`, sleeping between tries. And the cruel part: you retry *most* exactly when the partner is *slowest*, which is exactly when you can least afford to hold threads. Under any real load, a single struggling partner exhausts your thread pool and takes down requests that have nothing to do with webhooks.

**The fix:** get the delivery off the caller's thread entirely. The request handler does the minimum — record "this event needs to go out" onto a durable queue — SQS, RabbitMQ, a Kafka topic, take your pick — and return. A separate **delivery worker** pulls from the queue and does the POST (and the retries).

```
on order_paid(order):
  queue.put(event(order))   # durable, returns in ms
  return 202

# elsewhere, a worker:
loop:
  e = queue.take()
  deliver(e)                # POST + retries live here now
```

<MermaidDiagram chart={`
flowchart LR
    EV["order.paid"] --> SVC["Our service"]
    SVC -->|"enqueue, return 202"| Q["Queue"]
    Q --> WK["Delivery worker"]
    WK -->|"POST"| EP["Partner endpoint"]
`} />

This is the highest-leverage change in the post. The partner's latency is now the *worker's* problem, not your API's — and the work is durable, so a crash doesn't lose the event. Almost everything below is a refinement of "deliver asynchronously, off a durable queue."

And to be frank, if you sit with it a second, async isn't just *faster* here — it's the *honest* model. The thing that **triggered** the event and the thing that should **receive** the callback are different parties: the customer whose payment fired `order.paid` doesn't care whether a partner's endpoint ack'd anything, and shouldn't have to wait on it. The request's origin and the notification's target aren't the same place — so there was never anything on the caller's side that needed the delivery result inline. Blocking on it was meaningless from the start; a notify like this is fire-and-forget by nature.

## Don't hammer, don't loop forever — backoff + max attempts

**The problem:** that retry was naive — fixed delay, and nothing stopping it from looping forever. Two failure modes: retrying a struggling endpoint on repeat just **kicks it while it's down**; and if an endpoint is gone for good, an unbounded retry is a worker spinning on one dead event forever, never draining the queue behind it. (Why backoff lands *here* and not at the retry step: waiting 1s, 2s, 4s, 8s inline would pin a request thread for 15+ seconds — it's only safe once retries moved onto a background worker.)

**The fix:** **[exponential backoff with jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/), and a hard max-attempt ceiling.**

- **Exponential backoff** — wait longer between each try (1s, 2s, 4s, 8s, …) so a recovering endpoint gets room to breathe.
- **Jitter** — add randomness to the delay. Without it, every event queued during an outage retries at the *same* instant the endpoint comes back — a self-inflicted thundering herd that knocks it over again.
- **Max attempts** — after N tries (or a time budget), stop. The event isn't lost; it goes somewhere we deal with later (the DLQ, a few steps down). "Retry, but with a ceiling" is the whole game.

### Circuit breaker — stop dialing a dead endpoint

Backoff and a max-attempt ceiling tame *one event's* retries. But picture a partner who's been hard-down for an hour: **every new event** still opens a connection, waits out the timeout, and grinds through its whole backoff schedule before giving up. You're burning worker time, connections, and queue latency hammering an endpoint you already know is a corpse — and starving every *healthy* partner queued behind it.

A **circuit breaker** fixes that by tracking each endpoint's recent health:

- **Closed** (normal) — deliveries flow; you just count failures.
- **Open** — once the failure rate crosses a threshold, you *stop calling that endpoint entirely* for a cooldown. New events fail fast (straight to retry-later / DLQ) instead of each wasting a full timeout.
- **Half-open** — after the cooldown, let a few trial deliveries through. They succeed → close and resume; they fail → re-open and wait again.

<MermaidDiagram chart={`
stateDiagram-v2
    [*] --> Closed
    Closed --> Open: failure rate > threshold
    Open --> HalfOpen: cooldown elapsed
    HalfOpen --> Closed: trial succeeds
    HalfOpen --> Open: trial fails
`} />

The division of labor: **backoff spaces out the retries of one delivery; the breaker stops the whole stream of deliveries to a host that's clearly down.** They compose — the same libraries from the last step ship both (resilience4j has `Retry` *and* `CircuitBreaker`). And the breaker is per-endpoint, so one dead partner trips only its own circuit and leaves everyone else's deliveries flowing.

## Anyone can POST to them — sign it (HMAC)

**The problem:** the partner's webhook endpoint is a **public URL on the internet**. Anyone who learns it can POST to it, pretending to be you — a fake `order.paid`, a fake `refund.completed`. The consumer has no way to tell your real event from a forged one. And if you ever stuffed details into the **URL or query string** (`?amount=5000&user=123`), you've leaked them — query strings end up in access logs, proxies, and browser history.

**The fix:** prove the message is from you, and keep the data out of the URL.

- **Sign the payload.** Compute an [HMAC](https://en.wikipedia.org/wiki/HMAC) (e.g. HMAC-SHA256) of the raw request body using a **shared secret** only you and the consumer know, and send it in a header (`X-Signature`). The consumer recomputes the HMAC over the body they received and compares — if it doesn't match, the request is forged or tampered, and they drop it.
- **Sign a timestamp too**, and include it in what's signed. The consumer rejects anything older than a few minutes, which kills **replay attacks** (someone capturing a valid call and resending it).
- **Everything meaningful goes in the body**, never the URL. The URL is just the address; the signed body is the message.

```
sig = hmac_sha256(secret, timestamp + "." + raw_body)
POST /webhooks/orders
  X-Timestamp: 1718000000
  X-Signature: sha256=...
  { "id": "...", "type": "order.paid", ... }   # data lives here
```

![HMAC trust: a forged request is rejected, a validly signed one is accepted, using a secret shared between our system and the partner](/assets/images/blogs/design-webhook-callback-system/hmac-trust.png)

This is the consumer's *first* line of processing: before they look at the content at all, they ask "is this actually from them?" Authenticate first, then everything else.

**Step back, though — what you're really buying is *trust*, and HMAC is one way to earn it, not the only one.** There are two questions hiding inside "can I trust this webhook," plus a coarser gate you can put in front of both — a different tool for each:

- **Signing (HMAC)** answers *"is this exact payload genuine and untampered?"* — message integrity + origin. Stateless, no token infrastructure; the webhook-world default (Stripe, GitHub).
- **Auth ([OAuth2 client-credentials](https://oauth.net/2/grant-types/client-credentials/) Bearer token, or [mTLS](https://en.wikipedia.org/wiki/Mutual_authentication))** answers *"is the caller an authorized sender?"* — identity at the connection level. It plugs into the receiver's existing IdP, tokens are short-lived and revocable — but now the sender has to fetch and cache a token before each delivery.
- **Network allowlist (defense in depth)** — a coarser gate you bolt on *in front* of both: restrict which sources can even reach the endpoint. An **IP allowlist** of the sender's published egress IPs (Stripe, GitHub and friends publish theirs), or lock the route behind mTLS / a private link. It won't prove a payload is genuine on its own — IPs get shared, spoofed, and rotated — but it keeps random internet traffic from ever reaching your verification logic.

One caveat if you go OAuth2: **client credentials** is the flow that fits — machine-to-machine, no human in the loop. *Authorization code* is for user-delegated consent (a person granting an app access), which doesn't map onto a server-to-server callback, so it's not the tool here.

These aren't competitors, and it's not all-or-nothing — run none, one, or all three by your threat model: an allowlist so only known sources reach you, a token to prove *who's calling*, a signature to prove *the message wasn't touched*. In the end it all serves the same goal — the receiver trusting the event enough to act on it.

## Your retries cause doubles — idempotency

**The problem:** everything we've built is **at-least-once**. The worker retries; the queue can redeliver after a crash; a delivery that succeeded but whose acknowledgement got lost gets sent *again*. So the consumer **will** see the same event more than once — we hit this same wall in the payment and push posts, and true exactly-once over a network just isn't on the table. For an **AI agent**, a duplicate `order.paid` means a second order placed, maybe money charged twice — if it isn't checking idempotency.

**The fix:** make every event carry a **stable, unique `event_id`** (assigned once, at the source — carried unchanged through every retry), and make the delivery contract explicit: *"we deliver at least once; dedupe on `event_id`."* The consumer records the ids it has processed and skips repeats. You guarantee delivery; they guarantee idempotent handling. It's a shared contract, and you should document it loudly in your webhook docs.

```
{
  "id": "evt_7f3c8a",
  "type": "order.paid",
  "data": { "order_id": "ord_123", "amount": "245 THB" }
}
```

Note the order with the previous step: the consumer **authenticates (HMAC) first, then dedupes (`event_id`), then processes.** Trust it, then check if you've seen it, then act.

## Some never succeed — DLQ + redrive

**The problem:** backoff and a max-attempt ceiling stop the looping, but they create a new question: *what happens to the event that exhausted its retries?* A partner can be down for hours, and silently dropping the event means a lost order notification — for an agent-driven consumer, lost state it never recovers (you could dig through the logs, sure, but you won't refill it completely).

**The fix:** a **dead-letter queue (DLQ)**. When an event crosses the max-attempt threshold, move it to a separate store instead of discarding it. Now:

- **Operators can see and act** — inspect why it failed, and **redrive** (re-enqueue) the batch once the partner is healthy again.
- **Give the consumer a recovery path too** — a "list/replay events since X" API, so a partner who was down can pull what they missed. (This is the push side of the same coin as the old polling model — a deliberate catch-up channel, not the default path.)

<MermaidDiagram chart={`
flowchart LR
    WK["Delivery worker"] -->|"delivered"| OK["done"]
    WK -->|"attempt >= max"| DLQ[("Dead-letter queue")]
    DLQ -->|"operator redrive (partner healthy)"| WK
    DLQ -.->|"or consumer replays"| API["Replay API"]
`} />

A practical detail from real systems: tie the move-to-DLQ to a **receive-count** (how many times this message has been pulled and failed), and define a **redrive policy** up front — where exhausted messages go and how they come back. Don't invent it during the incident.

**Running a DLQ in production is more than "a second queue."** A few realities that bite once it's live:

- **A queue is not long-term storage.** Most brokers expire messages (SQS caps retention at 14 days) — so a DLQ that's *only* a queue silently drops your failed events after a while, which defeats the whole point. For anything you can't lose, **persist DLQ'd events to a durable store** — a `failed_deliveries` table (or object storage), [Outbox](https://microservices.io/patterns/data/transactional-outbox.html)-style — that you can query, audit, and redrive from at any time, not a queue racing a retention clock.
- **You need eyes on it.** Operators inspect the DLQ by **polling it manually** — a small admin view or the cloud console — to list, filter by endpoint or error, and read the individual failed payload. A DLQ nobody looks at is just a slower way to lose data.
- **Redrive surgically, not blindly.** During an incident you rarely want "replay everything" — a bad batch just re-fails or double-fires. **Manually consume / select** the messages that matter (this endpoint, this time window) and redrive only those.
- **Redrive with the same care as the first send.** Re-enqueue carrying the **original idempotency keys** (so the consumer still dedupes), **throttle** it (a just-recovered endpoint doesn't want your whole backlog at once — same thundering-herd lesson), and keep the attempt ceiling.
- **Alert on DLQ depth.** A growing DLQ is your earliest signal that a partner — or your own delivery tier — is broken. Page on it; don't find out from the partner's angry email.
- **Give the persisted store a lifecycle.** Audit copies of failed events need a TTL / archival policy, or that table grows forever.

## They arrive out of order — sequencing and the state machine

**The problem:** async delivery + retries + parallel workers means events for the same resource can land **out of order**. `order.paid` then `order.refunded`, but the consumer sees `refunded` first; or a *retried* old event arrives after a newer one. A human tolerates it. An agent acting in receipt order does the wrong thing.

**The fix:** two pieces, and you don't need more than this.

- **Per-resource sequence/version + last-write-wins.** Stamp a monotonic version per resource on every event; the consumer keeps the last-seen one and ignores anything older. Need processing order too? **Partition by resource id** so one order's events share a single lane. You only ever need *per-resource* ordering, not global — don't build a total-order system you won't use.
- **A state machine with enforced transitions.** This is the real defense — the same move we made on the *receiving* side in the payment post. The consumer models each resource's lifecycle as states with *legal* transitions and rejects the ones that don't make sense: a `refunded` that lands before `paid` simply isn't a valid transition out of the current state, so it's ignored or parked, not applied. Arrival order stops mattering; only state order does.

But that *parked* word hides a real engineering decision: when the state machine can't apply an event yet, where do you hold it? The DLQ is the tempting answer — except a DLQ that's a plain **FIFO queue** doesn't reorder anything. Re-enqueue a too-early event and it just lands at the back; the prerequisite you're waiting on still can't jump the line. Parking out-of-order events only works if that spot is a **persistent, queryable store keyed by resource** — hold the future event there, then apply it once the state catches up. It's the same lesson as the DLQ section: a queue moves things along, a store lets you *hold and reorder* — and which one you reach for is a tradeoff you design on purpose, not something the broker hands you.

**When a flow has real dependencies, it helps to picture it as a DAG** — not as something to *build*, but as a way to reason about what has to come before what. Most lifecycles are a simple line, until a **fan-in (join)**: an event that needs *two or more* prerequisites.

<MermaidDiagram chart={`
flowchart TD
    OC["order.created"] --> PA["payment.authorized"]
    OC --> IR["inventory.reserved"]
    PA --> OP["order.paid"]
    OP --> SC["shipment.created"]
    IR --> SC
    SC --> SD["shipment.delivered"]
`} />

`shipment.created` is the join — it needs **both** `order.paid` *and* `inventory.reserved`, and your system delivers those independently and possibly out of order. Seeing the events as a DAG is simply how you spot which transitions the state machine has to guard. Your job as the *sender* is small, and worth keeping strict: **enrich the event and send it** — `event_id`, per-resource ordering, a `correlation_id` (like `order_id`) to tie a flow together — then stop. State belongs to the *receiver*; you just hand over enough for them to rebuild it. **Your webhook system is not a workflow engine — don't turn it into one.**

And if that growing dependency graph looks familiar, it is: it's the same shape as a **[SAGA](https://microservices.io/patterns/data/saga.html)** in microservices — specifically the **choreography** flavor, where each participant reacts to events and emits its own, with no central brain sequencing them. The other flavor, **orchestration**, *is* a central workflow engine — exactly the thing you (the sender) are staying out of. Keeping delivery dumb is choosing choreography on purpose.

And this is where the agent angle peaks: an agent that acts on each event the instant it lands will fire shipment logic on an order that was never actually paid — unless a state machine refuses that transition. That guard is the only thing standing between the agent and acting on half-built state. A human glancing at a dashboard would have just... noticed. The agent won't.

## Pull it apart — acceptor / dispatcher / consumer

**The problem:** by now one service is doing a lot — accepting the internal event, figuring out *who* subscribed to it, and doing the slow external delivery with all its retries. Those three jobs have completely different shapes, and bolting them together means the slowest, least reliable one drags down the others. A flood of slow partner endpoints backs up the delivery work *and* the ingestion of new events behind it.

**To be clear, though — one service doing all three has been production-grade for ages.** Packaged as a single deployable, or as a **serverless** stack (event → SQS / EventBridge → a Lambda that signs-and-delivers → a DLQ), the queue-worker-retry-DLQ machinery from the last few steps already carries a lot of real traffic — serverless especially hands you retries, scaling, and a DLQ off the shelf. When it needs to grow, you scale it the usual way: **horizontal or vertical**.

**But what if you want to scale the three jobs independently?** *That's* where splitting the tiers starts to earn its keep — scaling the whole box up and down is a different thing from scaling each job on its own. If you don't need that, you don't need the split.

### The split that matters: acceptor → queue → consumer

**The fix: of those three jobs, only two need their own tier — acceptor → queue → consumer.**

- **Acceptor** (ingress) — receives the internal "this happened" event, **durably captures it fast, and returns.** Write-bound; must never drop an event. Kept deliberately lean so ingestion never blocks on anything downstream.
- **Consumer** (delivery) — pulls from the queue and does the slow signed POST: HMAC, backoff, idempotency id, result tracking, DLQ. **I/O-bound, blocked on slow external endpoints, fails a lot.** You want many of these.

That two-tier split *is* the point: the durable-fast ingest is isolated from the slow-external delivery, so one can't drag the other down.

### So where's the dispatcher?

Notice I didn't put one in. The job a dispatcher does — *getting the right event to the right endpoint* — still has to happen, but it doesn't have to be its own component. There are two ways to route, and choosing between them is the real decision:

- **Route through the broker (no dispatcher).** Partition the queue/stream by route or event type, and let each **consumer subscribe to the routes it handles** — routing lives in the topic structure, not a service. Deploy it either way:
  - **one service running N consumers** — one consumer per route, all in the same deployable; or
  - **N services, each owning one route** — independent, deployed and scaled on their own.

  Simplest when routing is **fixed and known** — a handful of event types going to known places.
- **A dispatcher (a third tier).** A central component that looks up *who's subscribed*, applies subscription config, and fans the event out into per-endpoint tasks. It earns its keep when routing is **dynamic** — customers register their own endpoints and event filters at runtime, or you apply per-subscription transforms and rate caps. That's routing logic that wants one home, enforced in one place, rather than smeared across every consumer.

Head to head:

| | **Broker-routed consumer** | **Dispatcher** |
| --- | --- | --- |
| Where routing logic lives | baked into the topic/partition structure | in one central component |
| Adding a route | subscribe a new topic, touch nothing else | add its config in the dispatcher first |
| Filtering | none — each consumer takes everything and filters itself | centralized; only the events you need get sent |
| Replay | pull from offsets, free | build a replay mechanism yourself |
| Coupling to event schema | tight — every consumer reads the raw stream, so one schema change breaks them all | looser — the dispatcher absorbs a schema change in one place |
| Added cost | no extra component | another component to build + scale |
| Best when | routing is fixed and fully yours | routing changes often / customers configure it |

**Rich, customer-configured routing → dispatcher. Fixed routing you control → broker-routed consumers.**

### What the split buys you

- **Independent scaling** — a delivery surge or a wave of slow partners means you scale *consumers* only.
- **Failure isolation** — a flaky partner backs up only the consumer side; ingestion keeps accepting regardless.
- **Independent deploy** — change routing or a consumer without touching ingestion.

Now trace what happens when each piece falls over:

- **Acceptor (ingress) down?** New events can't get in — and yes, that's critical: you're refusing intake at the front door. But the split contains it. You know immediately it's the acceptor (not delivery, not the broker), and the blast radius stops there — nothing already in the queue is lost, the consumer keeps draining it, and intake resumes the moment ingress is back.
- **Consumer (delivery) down?** This is where the queue earns its name. The events just *sit there, safe*, until a consumer comes back to drain them. Delivery is delayed, not lost — the receiver might even notice and ping you ("hey, did event X get dropped?"), but nothing's actually gone. A queue is a **buffer**; a consumer outage is precisely the situation that buffer exists for.
- **Broker itself down?** *This* is the one you can't shrug off. The broker is the spine — if it goes, the acceptor has nowhere to put messages and the whole flow stops cold. Which is exactly why the broker is the one piece you make as highly-available as you can afford: it's the single thing whose outage you genuinely can't absorb.

Roughly the shape I've built myself. 🙇‍♂️

## Wrapping up

Notice it all starts with one question — what's the destination this system delivers to? From there, this Webhook Callback System grew from a single inline `POST` (with that destination pulled from a partner metadata store) into a delivery system decoupled behind a queue/buffer:

1. **Get delivery off the caller's thread** onto a durable queue.
2. **Back off with jitter and a max-attempt ceiling** — and trip a **circuit breaker** when an endpoint is down.
3. **Sign with HMAC**, plus an **allowlist** or **auth**, so the consumer can trust it's you.
4. **A unique `event_id`** so at-least-once doesn't turn into double-work.
5. **DLQ + redrive** so a long outage doesn't lose events.
6. **Per-resource ordering**, plus a **state machine** on the receiver that rejects out-of-order events.
7. **Split delivery onto its own tier** (acceptor → queue → consumer; add a dispatcher when routing turns dynamic).

The meta-point holds from the rest of the series: **nobody should build this on day one.** One partner at low volume? A synchronous POST with a couple of retries is genuinely fine.

The one thing that's *newly* true in 2026: the consumer on the other end is more and more likely to be an **autonomous AI agent that executes the instant your webhook lands** — so a duplicate you didn't dedupe, an out-of-order event, or a call you didn't sign all turn into real risk. This reliability work was always good engineering. The difference now is that we lean on agents to act for us — so we want them acting on our actual intent, and these guarantees are what keep that intact.
