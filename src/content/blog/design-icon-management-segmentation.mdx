## Introduction

This is the fifth post in the backend system design series — same format as always: start with the most naive thing that works, then let real-world constraints push it into shape.

This one's shaped a little differently from the earlier posts, though. It ends up as **two systems in one** — with deliberately opposite priorities — so before the usual step-by-step evolution, let me set the mental model.

Picture the home screen of a super-app: the grid of icons, the nav bar, the promo banners. The unit we actually care about is the **slot**. A page is made of **blocks**, each block holds **slots**, and a slot is where a single icon / asset / banner renders. The *layout* — which blocks, which slots, where they sit — is a design choice you make up front. What *fills* each slot is the interesting part: it's resolved **per user**. User A sees asset A in a slot, user B sees asset B in that same slot, and a user who matches nothing gets the slot's **default**.

So the whole job, in one line: given a `(user, page)`, resolve every slot to an asset — or its default. Sounds small. It stops being small the moment you add permissions, segmentation, eligibility rules, and a few thousand requests a second that can never, ever render a blank screen.

Here's the road. The first half is the evolution that *forces* personalization on you; the second half is the two systems it splits into.

The evolution:

1. **Static icons** baked into each client
2. Permissions arrive → **show/hide per user**
3. A per-user layout can't live in the client → **serve it from an API**
4. Different users, different assets → **segments + tiers**
5. Never render a blank slot → **eligibility + fallback**

Then the split — two systems with opposite SLAs:

- **Front-facing serving** — high-RPS, latency-critical, must-never-blank. **Availability first.**
- **Backoffice** — internal, low-traffic, correctness-first. A few minutes of staleness is fine.

The punchline of the whole post is that contrast: the same domain pulls apart into two services you tune *differently*. Let's start simple.

---

## The naive version — icons baked into each client

You have to start somewhere, and the honest start is: hardcode the icon grid and the nav bar straight into each client. The mobile app ships a list of icons; the web app ships its own. No backend involved — the layout is just part of the build.

```
// mobile build — hardcoded
val homeIcons = listOf("wallet", "scan", "transfer", "rewards", "topup")
```

For a small app this genuinely works — ship it. Everyone sees the same grid, it renders instantly, and there's nothing to operate.

But two cracks show up fast:

- **Every change is a release.** Want to swap an icon, reorder the grid, or run a banner next week? That's an app-store submission (and its review wait) on mobile, plus a deploy on web. Marketing moves faster than your release train.
- **The platforms drift.** Web and mobile each hold their own copy of the truth, so they fall out of sync — a banner that's live on web but missing in the app, an icon mobile dropped but web still shows.

Both cracks point the same way: the layout shouldn't live in the client. But before we move it, something else forces our hand first.

---

## Permissions arrive — show/hide per user

**The problem:** features start carrying permissions. A merchant tool only for merchant accounts, a "loans" entry only where lending is enabled, an admin shortcut only for staff. Suddenly the grid isn't one fixed list — what belongs on the screen depends on *who's looking*.

You *could* keep this in the client: ship every icon, then hide the ones the user can't use. That breaks two ways. You're shipping the existence of entries to people who shouldn't even know they're there (open the bundle and read them off), and the client now needs the full permission picture to make the call — which means it's making an authorization decision it has no business making.

**The fix:** resolve visibility on the server, per user. For a given user, walk the catalog of slots and keep only the ones their level / role / entitlements allow. The client stops deciding; it just renders what it's handed.

The shift that matters here isn't the filtering itself — it's that the answer is now **per-user**. The same page, requested by two different people, comes back as two different layouts. And a per-user answer is exactly the thing you can't bake into a client build.

---

## A per-user layout can't live in the client — serve it from an API

**The problem:** the layout is per-user now, but it's still being assembled on the device. To do that the client has to download the whole catalog *and* the rules to filter it — which is slow, leaks what it shouldn't, and still can't change without a release. The v1 cracks and the v2 per-user need collide right here.

**The fix:** move resolution behind an API. The client calls one endpoint with its auth, the server resolves the slots for that user, and the response *is* the finished layout — what to render, in order, nothing else.

```
GET /pages/home          (auth: this user)
->
{
  "blocks": [
    { "id": "shortcuts", "slots": [
      { "id": "s1", "asset": { "icon": "wallet",   "deeplink": "app://wallet" } },
      { "id": "s2", "asset": { "icon": "scan",     "deeplink": "app://scan"   } }
    ]}
  ]
}
```

The client goes dumb on purpose: it draws blocks and fills slots with whatever came back. A few things fall out of this for free:

- **Change without a release** — swap an icon or a banner by changing data, not by shipping an app.
- **Web and mobile stop drifting** — one resolver, one source of truth; the response just carries a per-platform shape (mobile gets its grid, web gets its layout).
- **Catalog and rules stay server-side** — the client never sees what it isn't allowed to.

This is the serving API in its first form. It's also where the whole rest of the post lives: every step from here makes that one `(user, page) → layout` call either smarter, faster, or harder to knock over.

---

## Different users, different assets — segments and tiers

**The problem:** permissions only answer yes/no — *can* this user see this slot at all? The business wants more. The same slot should show **different assets** to different people: a gold-tier user gets a premium-rewards banner, a brand-new user gets an onboarding nudge, a lapsed user gets a win-back promo. Same slot, same position on screen — different content per audience.

**The fix:** make slot resolution a *lookup*, not just a filter. Each slot has a set of candidate assets, and each candidate is tagged with the audience it's meant for — a tier, some attributes, an active campaign. At request time, for each slot, you pick the asset whose targeting matches this user.

A few things worth being precise about:

- **Targeting is data, not code.** "gold tier, in TH, signed up under 30 days ago" is a rule attached to an asset and evaluated against the user's attributes — so the business can add an audience without a deploy.
- **Priority matters when more than one matches.** A user can fall into several audiences at once, so you need a deterministic order (most-specific wins, or an explicit rank). Otherwise the slot shows whichever row the database happened to return first — different on every request.
- **A campaign is just a time-boxed asset** — targeting plus a start/end window. "The right banner for this user" becomes: the highest-priority asset whose targeting matches *and* whose window is open right now.

So a slot's answer for one user is: of all candidates for that slot, take the matching, eligible, highest-priority one. Which begs the obvious question — what happens when **nothing** matches?

---

## Never render a blank slot — eligibility + fallback

**The problem:** now that each slot is a targeted lookup, plenty of users will match nothing — and plenty of *requests* will fail mid-resolution. The eligibility check times out, the targeting store blips, a campaign ended a second ago. On a home screen at thousands of requests a second, every one of those turns into a hole in the grid — a blank slot, or worse, a blank page. That's the one outcome you can't ship: the app opening to nothing.

**The fix:** every slot carries a **default** asset, and resolution is wrapped so it always returns *something*. Two cases, one rule:

- **No match (ineligible)** — the user fits no targeted candidate → render the slot's default.
- **Failure (timeout / dependency down)** — resolution couldn't finish inside its budget → render the slot's default.

<MermaidDiagram chart={`
flowchart TD
    S["Resolve slot for user"] --> M{"Match + eligible?"}
    M -->|"yes"| A["Render matched asset"]
    M -->|"no"| D["Render slot default"]
    S -.->|"timeout / error"| D
`} />

The mindset shift: the request path's job isn't "compute the perfect asset." It's "return a good-enough asset, always, fast." Personalization is the upside; the default is the floor you never fall through. Put a tight timeout on the fancy path and let it fail *to the default*, never to an error.

It's the same instinct as the webhook post's "never lose an event" — here it's "never render a blank." And it's the hinge that splits this into two systems. The moment *always fast, always something* becomes the contract on the read path, that path wants completely different treatment from the backoffice that *manages* the assets. So let's pull them apart.

---

## Pull it apart — two systems, opposite priorities

Everything so far has been one `(user, page) → layout` call. But two very different jobs hide inside it: **serving** the layout (read, hot, every request) and **managing** the assets and rules behind it (write, rare, internal). They pull in opposite directions, so we build them as two systems with one thin connection.

<MermaidDiagram chart={`

    UI["Operator UI"] --> SOT[("Source of truth\nrules + assets")]
    SOT --> W["Cache warmer /\ninvalidation"]
    W --> C[("Serving cache")]
    API["Serving API"] --> C
    CL["Client"] --> API
`} />

Same domain, mirror-image requirements:

| | **Front-facing serving** | **Backoffice** |
| --- | --- | --- |
| Traffic | thousands of RPS | a handful of operators |
| Priority | availability + latency | correctness + usability |
| Shape | read-heavy | write / config |
| Freshness | instant, every request | minutes of staleness is fine |
| Cost of failure | a blank screen for users | an operator waits and retries |

The rest of the post is designing each to its own contract — and being honest that the backoffice gets to be the *simpler* one, because we let it.

---

## System A — Front-facing serving (the read path)

This is the side that eats the thousands of RPS, so the whole design bends toward one sentence: **answer from memory, and never block on anything slow.**

**Serve from a cache, not a live computation.** Resolving a user's layout touches the rule store, the campaign/eligibility data, and the asset catalog — far too much to do per request at this volume. So the request path doesn't resolve anything live; it reads an already-resolved view out of a cache and returns it. Resolution happens *ahead* of the request, not during it.

**A worker keeps the cache warm, asynchronously.** A separate cache-warming worker — one deployable, or several partitioned by page/segment — reads the source of truth and (re)builds the resolved view: per segment where it can, per user where it must. Serving and warming never share a request. Warming is background work; serving is a memory read.

<MermaidDiagram chart={`
flowchart LR
    SOT[("Source of truth")] --> WK["Cache warmer\n(background)"]
    WK --> C[("Serving cache")]
    CL["Client"] -->|"GET /pages/home"| API["Serving API"]
    API -->|"read"| C
    API -.->|"miss / error"| DEF["Slot default"]
`} />

**Gate the deploy on a warm cache.** Here's the detail people miss: when a new serving instance boots, it has to finish loading its cache **before** it's marked ready and put into rotation. A half-warm instance that starts taking traffic serves blanks (or falls through to the slow path) for its first minutes — exactly during a rollout, the worst moment. So readiness *is* "cache loaded." The tradeoff is honest: deploys get slower, and a big dataset makes cold start long — so warm from a **snapshot** instead of recomputing from scratch on every boot.

**Fall back to the default, always.** Even reading from cache, a key can miss — a brand-new user, a just-added slot. The read path still ends in the v5 rule: miss or doubt → the slot default. The cache makes you fast; the default keeps you safe.

**Where the cache lives is a real decision:**

- **In-memory, per instance** — the fastest possible read (no network hop), but every instance holds the whole set and every deploy re-warms it. Fine when the resolved set fits in memory and you can snapshot it for a quick load.
- **Shared (Redis / Memcached)** — one warm copy that many instances read; deploys don't re-warm, but you pay a network hop per read and the cache is now a dependency to keep alive.

Plenty of setups land on a blend: a shared cache as the warmed source, plus a small in-process layer for the hottest keys. Whichever way, the theme holds — the read path does as little as possible, as fast as possible, and degrades to a default rather than an error.

---

## System B — Backoffice (the write/config path)

This is where operators actually do the work: create assets, define audiences, map a campaign to a slot, set its window. It's internal, low-traffic, and — the key decision — **allowed to be eventually consistent.** A change doesn't have to hit screens this millisecond; a few minutes of propagation is fine. That one permission is what lets the backoffice stay simple.

**An internal UI over the source of truth.** Operators manage icons / banners / campaigns and map them to segments, placements (which slot), and schedules. It's plain CRUD against the rule + asset store — no exotic infrastructure. Correctness and clear validation matter far more here than latency or uptime.

**Getting an asset in — three paths.** An asset is an image plus metadata, and there's more than one way it arrives:

- **Managed upload** — the operator uploads the file → you **validate** it (format, size, dimensions) → store it in object storage → serve it via CDN, with the final CDN URL filling in **asynchronously** (so the UI shows "processing" for a beat, not a broken link).
- **Bring-your-own URL, used directly** — the operator pastes a link they already host, and you store it as-is. Fast and zero-effort, but you inherit their problem: a broken link, a slow host, an asset that quietly disappears — and now it's *your* blank slot.
- **Bring-your-own URL, ingested** — the operator pastes a link, but in the background you download it, re-upload to your own object storage, and serve from your CDN. You own availability and caching from then on; the external host only mattered once.

The middle option is the honest trap — easiest to ship, first to page you. Which paths you allow is a call; just make it a deliberate one.

**Publishing = invalidate / refresh the cache.** When an operator saves, the write lands in the source of truth — and that's where the backoffice's job ends. A signal goes to the serving side (invalidate the affected keys, or nudge the warmer to rebuild) and the new state shows up on screens shortly after. "Shortly" is the whole point: this is the ~10-minutes-is-fine SLA in action. No need to make the backoffice highly available or low-latency; it isn't on anyone's hot path.

So the backoffice is deliberately the boring one: a UI, some validation, writes to a store, and a poke at the cache. Boring is the *correct* amount of engineering for it.
