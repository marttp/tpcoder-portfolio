ในโพสต์ที่แล้วเรื่อง [fundamentals](/blog/fundamentals-how-much-you-need-early-career-2026-th) ผมทิ้งประเด็นหนึ่งไว้ และอยากหยิบกลับมาต่อครับ: floor ของ entry-level ไม่ได้ขยับไปไหนมากนัก แต่ ceiling ขยับขึ้นไปแล้ว รอบนี้เราเริ่มกันช้ากว่านั้นหน่อย คุณมีงานทำแล้ว ship feature ได้ debug ปัญหาส่วนใหญ่ที่หล่นมาบนโต๊ะได้ และผ่าน sprint ปกติไปได้โดยไม่ต้องมีคนบอกทุกขั้น แล้วคำถามใหม่ก็ค่อย ๆ โผล่ขึ้นมาว่า จากจุดนี้ bar ข้างหน้าหน้าตาเป็นยังไง?

ก่อนจะไปต่อ ผมอยากวางความสัมพันธ์ของสองโพสต์นี้ให้เคลียร์ก่อนครับ:

- **การเป็น entry-level ไม่ได้แปลว่าต้องรู้ทุกข้อในโพสต์ fundamentals** โพสต์นั้นเป็นแผนที่ของสิ่งที่ค่อย ๆ สร้างต่อได้ ไม่ใช่ checklist ที่ต้องเก็บครบก่อนสมัครงาน
- **พอถึง mid-level พื้นที่ส่วนใหญ่ในแผนที่นั้นไม่ควรเป็นของแปลกหน้าแล้ว** คุณยังเปิด documentation ขอความช่วยเหลือ หรือกลับไปทบทวนได้ตามปกติ แต่ควรเข้าใจ concept ส่วนใหญ่พอที่จะหยิบมาใช้ในงานประจำวัน
- **สิ่งที่เปลี่ยนในโพสต์นี้คือ mindset ไม่ใช่ syllabus ชุดใหม่** fundamentals ยังอยู่ครบครับ เพียงแต่คราวนี้คุณต้องใช้มันในวันที่มีคำสั่งน้อยลง โจทย์คลุมเครือขึ้น และคุณรับผิดชอบผลลัพธ์มากขึ้น

โพสต์ที่แล้วยังจบ section เรื่อง Reading a Ticket ด้วยประโยคที่แอบเป็น setup ของบทความนี้ทั้งบทความ: ยิ่งคุณโต input ที่ได้รับจะยิ่งมีโครงสร้างน้อยลง และความสามารถในการสร้างโครงสร้างจากความคลุมเครือ คือสิ่งที่แยก engineer ที่ execute ออกจาก engineer ที่ lead — โพสต์นี้คือสิ่งที่เกิดขึ้นเมื่อคุณเอาประโยคนั้นไปทำต่อแบบเอาเรื่องสักสองสามปีครับ

ก่อนเข้าเรื่อง ขอวางโน้ตไว้สองข้อครับ

เรื่องแรก เรื่อง title นี่คุยกันยาวครับ คนที่ถูกประเมินเป็น mid-level ในบริษัทหนึ่ง อาจทำงานในระดับ senior ได้เมื่อย้ายไปอีกบริษัท และคนที่ถือ title senior อยู่ ก็อาจถูกประเมินเป็น mid-level เมื่อย้ายไปบริษัทที่มี scale, system complexity หรือ ownership bar อีกแบบ การขยับทั้งสองทิศไม่ได้บอกอัตโนมัติว่าคนนั้นเก่งขึ้นหรือถอยหลังครับ ระบบที่ใช้วัดเปลี่ยนไปต่างหาก

ทั้ง mid-level และ senior ครอบตลาดเป็นช่วงกว้างมาก งาน mid-level สองที่อาจต่างกันเยอะทั้ง scope, expectation และ compensation ฝั่ง senior ก็เหมือนกัน senior ที่ startup 10 คนกับ senior ที่ธนาคารอาจใช้เวลาทั้งสัปดาห์กับงานคนละแบบ บริษัทใหญ่บางแห่งก็มีพื้นที่ให้ mid-level ที่แข็งแรงอยู่กับงาน driving และ execution ได้นาน ซึ่งเป็นรูปแบบงานที่มีคุณค่า ไม่ได้แปลว่า career ไปต่อไม่ได้ เพราะงั้นโพสต์นี้จะคุยเรื่อง **พฤติกรรม** มากกว่าพยายามตัดสินว่าใครควรได้ title อะไร จากที่ผมเห็น พฤติกรรมมักมาก่อน แล้ว title ค่อยตามมา — บางทีก็ที่บริษัทเดิม บางทีก็ที่บริษัทถัดไป

เรื่องที่สอง เรื่องชื่อโพสต์ ทุกก้าวที่เขียนไว้ในนี้มีราคาที่ต้องจ่าย และ career content ส่วนใหญ่ชอบข้ามส่วนนั้นไป ดังนั้นทุก section ข้างล่างจะปิดท้ายด้วยราคาของมัน บอกกันแบบเปิดเผย ไม่ได้จะขู่ให้กลัวนะครับ — ราคาพวกนี้คุ้มที่จะจ่าย และผมจะอธิบายว่าทำไมในตอนท้าย แต่คุณควรรู้ก่อนว่ากำลังเซ็นรับอะไรอยู่

อีกนิดเดียว เรื่อง scope: โพสต์นี้ตั้งใจไม่พูดเรื่อง AI agent หรือ coding workflow ครับ เรื่องนั้นผมเขียนแยกไว้ที่อื่นแล้ว skillset ในโพสต์นี้คือตัวที่ตั้ง bar อยู่ ไม่ว่าโค้ดจะถูก generate มาจากอะไรก็ตาม

อีกกลุ่มที่โพสต์นี้ยังไม่ครอบคลุม คือคนที่ไม่ได้อยู่สาย software engineering แต่รู้ domain ของตัวเองลึก และตอนนี้เขียนโค้ดจน deploy solution ที่ใช้การได้แล้วครับ ถ้าคุณอยู่กลุ่มนี้ จุดเริ่มต้นของคุณต่างออกไป คุณอาจมี judgment ใน finance, operations, design, education หรือสายงานอื่นสะสมมาหลายปี ความสามารถใหม่คือการเปลี่ยนความรู้นั้นให้กลายเป็น software ที่ทำงานได้ หัวข้อนี้ควรได้โพสต์แยกของตัวเองครับ ผมจะเขียนแยก แทนที่จะยัดไว้ใน ladder ของ engineer ทั้งที่โจทย์คนละแบบ

## What Actually Changes Between Mid and Senior

ช่วง junior ไป mid หัวใจของมันคือ **ความเข้าใจ** — คุณเข้าใจไหมว่าโค้ดของตัวเองทำอะไร debug ได้ไหม อธิบายได้ไหมว่าทำไมระบบถึง behave แบบนั้น นั่นคือทั้งหมดของโพสต์ fundamentals

แต่ช่วง mid ไป senior เป็นคนละเรื่องครับ: **ownership ต่อผลลัพธ์**

วิธีที่ผมใช้อธิบายเรื่องนี้คือ trust radius ครับ ทีมคุณอาจไม่เคยเรียกชื่อนี้ก็ได้ แต่ทุกสัปดาห์ทุกคนกำลังตอบคำถามคล้าย ๆ กันอยู่: เราส่งอะไรให้คนนี้ได้ โดยไม่ต้องตามเช็คทุกขั้น? บันไดของมันหน้าตาแบบนี้

- **Task**: มีคนตัดสินใจแล้วว่าจะสร้างอะไร ยังไง คุณลงมือทำ
- **Feature**: มีคนตัดสินใจว่าจะสร้างอะไร ส่วนวิธีทำคุณเป็นคนเลือก และส่งมอบทุกชิ้นเอง
- **Problem**: ยังไม่มีใครตัดสินใจอะไรเลย คุณได้รับอาการของปัญหาหรือเป้าหมายมาหนึ่งอย่าง แล้วต้องกลับมาพร้อม diagnosis แผน และสุดท้ายคือ fix ที่ ship แล้ว
- **Area**: ไม่มีใครส่งอะไรให้คุณอีกแล้ว คุณเฝ้าดูส่วนหนึ่งของระบบ มองเห็นเองว่ามันต้องการอะไร แล้ว drive งานด้วยตัวเอง

การเปลี่ยนช่วงแบบนี้ไม่มีใครส่ง calendar invite มาประกาศหรอกครับ คุณจะเจอในวันทำงานธรรมดามากกว่า: ticket มีอยู่สองประโยค product เอา complaint มาให้แทน requirement หรือ tech lead ถามว่า คุณคิดว่าเราควรทำยังไงดี แล้วนั่งรอฟังคำตอบจากคุณ input ที่มีโครงสร้างน้อยลง มักมาพร้อม trust ที่มากขึ้น ถึงเอกสารจะยังตามมาไม่ทันก็ตาม

![Trust radius from task to area](/assets/images/blogs/mid-level-2026/trust-radius.png)

ช่วงนี้มีเรื่องที่ต้องยอมรับหน่อยครับ: **การเขียนโค้ดให้ดีขึ้นอย่างเดียว เริ่มพาคุณไปต่อได้ไม่ไกลเท่าเดิมแล้ว** โค้ดยังสำคัญนะครับ และทุกอย่างข้างล่างก็สมมุติว่าคุณสร้างสิ่งที่ออกแบบไว้ได้ เพียงแต่งานที่ขยาย trust radius ต่อจากนี้ จะเพิ่มเรื่อง scope, ลำดับงาน, production, judgment และการคุยกับคนเข้ามาด้วย โพสต์นี้เลยเน้นเรื่องพวกนั้นมากกว่า syntax หรือ framework

**ราคาที่ต้องจ่าย:** คำสั่งชัด ๆ จะเจอน้อยลงครับ ticket ยังมีอยู่แหละ แต่งานส่วนหนึ่งเริ่มก่อน ticket จะถูกเขียนด้วยซ้ำ อิสระนี้มีประโยชน์มาก และก็มาพร้อมความรับผิดชอบร่วมกัน ถ้าทีมเลือกสร้างของผิด

## From Ticket to Problem — You Write the Tickets Now

โพสต์ fundamentals สอนวิธีอ่าน ticket: งานนี้ใหญ่แค่ไหน subproblem มีอะไรบ้าง แต่ละอันเขียนเป็นประโยคภาษาคนได้ว่ายังไง พอถึง mid-level skill ตัวนี้จะพลิกทิศครับ คุณไม่ใช่คนรับ ticket ที่ถูกเกลามาดีแล้วอีกต่อไป คุณคือคน **ผลิต** มัน — ให้ทีม ให้ junior และให้ตัวคุณเองในอนาคต

งานนี้ยากกว่าที่ฟังดูด้วยเหตุผลเฉพาะเจาะจงอันหนึ่ง: เวลาแตกงานให้ **ตัวเอง** คุณเก็บครึ่งหนึ่งของแผนไว้ในหัวได้ แต่เวลาแตกงานให้ **คนอื่น** ทุกอย่างต้องออกมาจากหัวให้หมด เส้นแบ่งระหว่าง task แต่ละตัวต้องเขียนออกมาให้เห็น interface ระหว่างชิ้นงานต้องมีชื่อเรียก และคำว่า เสร็จ ต้องนิยามเป็นรายชิ้น ในแบบที่คนทำเช็คเองได้โดยไม่ต้องเดินมาถามคุณ

สมมุติว่าเจอโจทย์หน้าตาแบบงานของแท้จาก wallet domain นะครับ product บอกมาว่า: อยากให้ user เติมเงินเข้า wallet จากบัญชีธนาคารที่ผูกไว้ — ประโยคเดียวจบ การแปลงประโยคนั้นให้กลายเป็นงานที่ทีม execute ได้ หมายถึงการผลิตอะไรประมาณนี้ออกมา:

1. **API contract สำหรับ top-up** — รูปร่างของ request/response, error code, พฤติกรรม idempotency เสร็จเมื่อ: contract ผ่านรีวิว และ stub endpoint คืน fixture data ได้
2. **Integration กับ bank provider** — ซ่อนไว้หลัง interface เริ่มจาก sandbox ก่อน เสร็จเมื่อ: ยิง charge ผ่าน sandbox สำเร็จหนึ่งเคส และ handle เคสล้มเหลวได้หนึ่งเคส เห็นทั้งคู่ใน logs
3. **State machine ของ transaction** — PENDING → COMPLETED / FAILED พร้อมกฎว่า balance ขยับได้เฉพาะตอน COMPLETED เท่านั้น เสร็จเมื่อ: การเปลี่ยน state ทุกแบบมีเทสต์ รวมถึงแบบที่ห้ามเกิดด้วย
4. **Webhook handling สำหรับ confirmation แบบ async** — เพราะ bank transfer ไม่ได้ confirm ทันที เสร็จเมื่อ: webhook ที่ถูก replay ซ้ำไม่ทำให้เงินเข้าสองรอบ
5. **Reconciliation report** — เช็ครายวันว่าผลรวม COMPLETED ฝั่งเรา match กับ settlement file ของ provider เสร็จเมื่อ: report รันได้ และ mismatch ที่เราแอบใส่เข้าไปทดสอบถูกจับได้

สังเกตไหมครับว่าเกิดอะไรขึ้น: หนึ่งประโยคจาก product กลายเป็นงานห้าชิ้น แต่ละชิ้นมีนิยามคำว่าเสร็จที่ไม่ต้องพึ่งเจ้าของงานมายืนยัน และสมมติฐานเสี่ยง ๆ (confirmation เป็น async, webhook มาซ้ำได้, ต้องมี reconciliation) โผล่ขึ้นมา **ระหว่างแตกงาน** — ไม่ใช่โผล่มาเป็นเซอร์ไพรส์ตอนสัปดาห์ที่สาม

ถ้างานใหญ่ขึ้น เครื่องมือที่ใช้ได้ดีคือ **design doc** ครับ ไม่ต้องเป็น template สิบหน้าที่ไม่มีใครอ่านก็ได้ เอาแค่หนึ่งหรือสองหน้า: ปัญหาคืออะไร มี constraint อะไร ทางเลือกสองสามแบบแลกอะไรบ้าง แล้วคุณเลือกแบบไหน หน้าที่ของ doc คือทำให้การเห็นต่างมีราคาถูก คำถามที่โผล่ใน comment วันอังคาร รับมือง่ายกว่าคำถามเดียวกันที่มาในรูป incident อีกสามสัปดาห์ถัดมาเยอะครับ

และชิ้นสุดท้าย: **การตัด scope แบบตั้งใจ** engineer ระดับ mid ที่พูดได้ว่า เรา ship ชิ้นที่ 1–4 ตอนนี้ ส่วน reconciliation ตามมา sprint หน้า และนี่คือเหตุผลว่าทำไมมันปลอดภัย — คนคนนั้นกำลังทำงานระดับ senior อยู่ครับ การ ship 70% แบบตั้งใจ ชนะการ ship 100% แบบเลท แต่มีเงื่อนไขเดียว: คุณต้องอธิบายได้ว่า 70% **ก้อนไหน** และทำไม 30% ที่หายไปถึงไม่กัดเราทีหลัง

**ราคาที่ต้องจ่าย:** ความคิดของคุณจะกลายเป็นของสาธารณะและถูกวิจารณ์ได้ แผนที่อยู่ในหัวไม่มีวันผิดออกสื่อ แต่ ticket breakdown, design doc, การตัด scope — พวกนั้นมีชื่อคุณแปะอยู่ และจะมีคนไม่เห็นด้วย การถูกมองเห็นแบบนี้แหละคือประเด็นของมัน และมันจะไม่มีวันหายอึดอัดสนิทหรอกครับ

## Milestones — The Skill Devs Are Never Taught

มี gap หนึ่งที่ผมไม่ค่อยเห็นใครคุยกันช่วงต้น career ครับ: developer หลายคนไม่ค่อยได้ฝึกสร้าง timeline เพราะมี project manager, tech lead หรือ quarterly plan ส่ง date มาให้ แล้วหน้าที่ของคุณคือ report status เทียบกับ date นั้น วิธีนี้ใช้ได้เรื่อย ๆ จนวันหนึ่งมีคนถามว่า งานนี้คุณจะแบ่งยังไง แล้วเราจะ verify ของชิ้นแรกได้เมื่อไหร่?

แล้วคนก็แปลกใจกันว่าทำไม engineer ที่โตเร็วที่สุดถึงมีความสามารถร่วมกันอยู่ตัวหนึ่ง: **การแตกงานออกเป็น milestone ที่วัดผลได้** มันไม่ใช่เรื่องบังเอิญครับ และมันก็ไม่ใช่ skill ของ PM ที่แต่งตัวมาเป็นงาน engineer ด้วย เพราะการออกแบบ milestone **คือ** งาน engineering เอง:

- ตัดสินใจว่า phase แรกอะไรควร stub อะไรควรสร้างของแท้ — นั่นคือการตัดสินใจเชิง architecture
- ตัดสินใจว่าแต่ละ checkpoint ต้อง demo อะไรได้ — ซึ่งบังคับให้คุณสร้างงานเป็น vertical slice แทนที่จะก่อทีละ layer แนวนอน
- ตัดสินใจว่าจะรับความเสี่ยงเรียงลำดับยังไง — อันนี้คือ failure-mode thinking จากโพสต์ fundamentals เป๊ะ ๆ แค่ย้ายจาก runtime มาใช้กับการส่งมอบงานแทน

### Measurable means verifiable by someone other than you

คุณสมบัติที่นิยาม milestone ของแท้: คนอื่นตรวจได้โดยไม่ต้องเชื่อคำพูดคุณ ลองเทียบกันครับ

- ~~Top-up คืบหน้าไปประมาณ 80% แล้ว~~ — อันนี้ไม่ใช่ milestone วัดอะไรไม่ได้เลย แล้วเลข 80% เนี่ย ค้างอยู่ที่เดิมได้สามสัปดาห์แบบสบาย ๆ
- Top-up วิ่งครบ end-to-end กับ sandbox ของ provider แล้ว — นี่ demo นี่ request log — อันนี้คือ milestone ครับ ใครก็ verify ได้ในห้านาที

milestone จะถูก verify ด้วยของสามอย่าง: **demo** หนึ่ง, **เทสต์ที่คนอื่นกดรันเองได้** หนึ่ง, หรือ **ตัวเลขบน dashboard** หนึ่ง ถ้า verify ด้วยสามอย่างนี้ไม่ได้ มันยังไม่ใช่ milestone ครับ มันเป็นแค่ความตั้งใจ

### Sequencing is risk management

สัญชาตญาณตั้งต้นของทุกคนคือ: ทำงานที่คุ้นมือและเข้าใจดีก่อน แล้วเก็บของน่ากลัวที่ยังไม่รู้ — external provider, data migration, infrastructure ตัวใหม่ — ไว้ท้ายสุด มันให้ความรู้สึกว่างานเดินครับ แต่มันคือลำดับที่แย่ที่สุดเท่าที่เป็นไปได้ เพราะสิ่งที่มีโอกาสระเบิด timeline ของคุณมากที่สุด คือสิ่งที่คุณได้เรียนรู้เป็นอย่างสุดท้าย

เอาสิ่งที่น่ากลัวที่สุดใส่ไว้ใน milestone **แรกสุด** ครับ ในรูปแบบที่บางที่สุดเท่าที่จะแตะของหน้างานได้ สำหรับ wallet top-up: milestone แรกไม่ใช่ data model นะครับ — มันคือ request แบบ hardcode หนึ่งตัววิ่งผ่าน sandbox ของธนาคารไปได้ และ confirmation หนึ่งตัววิ่งกลับมา spike ตัวเดียวนั้นตอบคำถามที่ชี้ชะตา design ทั้งระบบ

### Why this grows you faster than almost anything else

milestone ช่วยให้ทีมเห็นว่างานเดินไปถึงไหน โดยไม่ต้องทักมาถามทุกวันครับ สมมุติว่าคนหนึ่งมี checkpoint ที่ demo หรือรัน test ให้ดูได้เรื่อย ๆ ส่วนอีกคนตอบว่า ไปได้ดี ประมาณ 80% แล้ว มาสามสัปดาห์ คุณคงเดาได้ว่าใครจะถูกวางใจให้รับงานก้อนถัดไป เรื่องนี้ไม่ต้องมีการเมืองซับซ้อนเลยครับ คนรอบตัวแค่เห็นหลักฐานว่าของกำลังกลับมาเป็นชิ้น ๆ

**ราคาที่ต้องจ่าย:** estimate กับลำดับงานของคุณจะถูกมองเห็นครับ บางครั้งคุณจะพลาด date ที่ตัวเองเสนอ และมันอึดอัดอยู่แล้ว ข้อดีคือ miss ที่มองเห็นได้ ยังเอามาคุยและแก้เร็วได้ ส่วนคำว่า ยังทำอยู่ ให้ข้อมูลกับทีมน้อยกว่ามาก

## Depth — Own a Slice End to End

ความรู้แบบกว้างพาคุณผ่าน early career มาได้: database นิดหน่อย networking นิดหน่อย มีทุกอย่างพอให้ทำงานได้ แต่ move ของ mid-level เป็นอีกแบบครับ: เลือก slice หนึ่งของระบบ แล้วรู้มันให้ **หมดทั้งเส้น**

[โพสต์ fundamentals](/blog/fundamentals-how-much-you-need-early-career-2026-th) อธิบายเรื่องการอ่าน sequence diagram กับการไล่ flow ไว้แล้วครับ พอมาถึง mid-level คุณควรวาด flow ของ feature ที่ตัวเองดูแลได้ และใช้มันคุยต่อได้ด้วยว่าจุดไหนพังได้ จุดไหนเปลี่ยนแล้วกระทบของอื่น

สมมุติว่าคุณ own เส้น top-up ของ wallet ครับ คุณควรพาใครสักคนเดินตั้งแต่ user กดยืนยัน ไปจน balance เปลี่ยนได้: request ถูก validate ที่ไหน เช็ค idempotency ตอนไหน transaction สถานะ PENDING ถูกสร้างยังไง provider timeout เท่าไหร่ webhook กลับมาแล้วเปลี่ยน state แบบไหน จากนั้น ledger, cache และ analytics ขยับตามกันยังไง ที่สำคัญคือคุณรู้ด้วยว่าแต่ละช่วงพังแบบไหน อะไร retry อะไรไม่ retry แล้วของซ้ำหรือเงินหายแทรกเข้ามาได้ที่จุดไหน

วลีสุดท้ายนั่นแหละคือข้อสอบครับ **ในระบบที่เกี่ยวกับเงิน คำถามเชิงลึกมีข้อเดียวเสมอ: มูลค่าโผล่หรือหายไปได้ที่จุดไหนบ้าง?** ถ้าคุณตอบคำถามนี้สำหรับ slice ของตัวเองได้ คุณ own มัน ถ้ายังตอบไม่ได้ คุณแค่ทำงานอยู่ใกล้ ๆ มัน

![Owning one vertical slice through every layer](/assets/images/blogs/mid-level-2026/depth-own-a-slice.png)

ความลึกยังโผล่ในพฤติกรรมอีกสองอย่างที่ควรสร้างแบบตั้งใจ:

**อ่าน source ของ dependency ที่ตัวเองใช้** ถึงจุดหนึ่ง documentation จะพาไปต่อไม่ได้ — ORM ทำอะไรแปลก ๆ กับการ reuse connection, HTTP client แอบ retry ทั้งที่เราไม่ได้ขอ, queue library ack เร็วกว่าที่เราเข้าใจ engineer ระดับ mid จะเปิด issue แล้วรอ ส่วนคนที่กำลังโตพ้นระดับนั้นจะเปิด source ของ library แล้วหาคำตอบเอง มันแทบไม่เคยยากเท่าที่เห็นจากข้างนอกครับ และมัน compound ด้วย: ทุกครั้งที่มุดเข้าไปในไส้ของ dependency รอบถัดไปจะเร็วขึ้นเสมอ

**รู้ว่าทำไมมันถึงเป็นแบบนี้** ทุกระบบแบกการตัดสินใจที่ดูผิดจนกว่าจะรู้ประวัติของมัน — double-write แปลก ๆ ตัวนั้น, field ที่ไม่มีใครกล้า rename, retry count ที่เป็นเลข 7 เป๊ะ ๆ ในทีมจะมีคนหนึ่งที่รู้ว่าอันไหนเป็นเสารับน้ำหนัก อันไหนเป็นฟอสซิล การกลายเป็นคนคนนั้นสำหรับ slice ของคุณเป็นงานเงียบ ๆ ที่ไม่มีใครปรบมือให้: อ่าน PR เก่า ถามคนที่อยู่ในเหตุการณ์ จดสิ่งที่รู้มาเก็บไว้ แต่มันก็ทำให้คุณเป็นคนที่ถูกถามก่อนใครจะแก้อะไร — ซึ่งนั่นแหละครับคือหน้าตาของการ own area เมื่อมองจากด้านใน

**ราคาที่ต้องจ่าย:** ความลึกมันช้า และไม่จ่ายผลตอบแทนเป็นความแปลกใหม่เลย ระหว่างที่คุณนั่งอ่าน edge case ของ settlement reconciliation รอบที่สาม ใครสักคนใน feed กำลังโพสต์เรื่อง framework ตัวใหม่แวว ๆ อยู่ 😅 ความลึกเอาไป demo ใน sprint review ไม่ค่อยขึ้นหรอกครับ มันมักโผล่ในช่วงที่ไม่หวือหวา: ตอนมี incident, ตอน design review หรือบ่ายที่ทุกคนกำลังไล่ว่า retry setting ทำไมต้องเป็นเลข 7

## Designing for Change — Boundaries, Coupling, and Contracts

การรู้ flow ทั้งเส้นเป็น skill หนึ่งครับ แต่ออกแบบให้ requirement รอบถัดไปแก้อยู่แค่ไม่กี่จุด เป็นอีก skill หนึ่ง

ลองนึกถึงงานทั่ว ๆ ไปนะครับ เปลี่ยน email provider แต่ต้องแก้ signup rule ด้วย เพิ่ม export แบบ CSV แล้วต้องยัด `if` เข้า service ก้อนเดิมอีก หรือแก้ database table ทีเดียว API response เปลี่ยนตาม ถ้า requirement หนึ่งลาก controller, business rule, database, integration และ test ให้เปลี่ยนพร้อมกัน ปัญหาอาจไม่ใช่ว่าไฟล์เยอะครับ ระบบอาจยังไม่มี boundary ล้อมของที่เปลี่ยนบ่อยไว้

มีสามคำที่ช่วยคิดเรื่องนี้ครับ:

- **Coupling** ดูว่าส่วนหนึ่งต้องรู้รายละเอียดของอีกส่วนมากแค่ไหน ถ้า business logic รู้ error code ของ provider หรือชื่อ column ใน database แปลว่ารายละเอียดพวกนั้นไหลออกมาไกลเกินจำเป็นแล้ว
- **Cohesion** ดูว่ากฎที่มักเปลี่ยนพร้อมกันอยู่ด้วยกันไหม permission rule เดียวที่ copy อยู่ห้า controller จะเปลี่ยนยาก เพราะกฎหนึ่งข้อมีบ้านห้าหลัง
- **Contract** บอกว่า module รับปากอะไรกับคนเรียก โดยไม่เปิดทุกอย่างข้างในให้คนอื่นผูกตาม contract ที่ดีให้ข้อมูลเท่าที่ caller ต้องใช้ และซ่อนรายละเอียดที่ caller ไม่ควรรู้

Adapter, Strategy, Facade, dependency inversion หรือ contract test ช่วยสร้าง boundary ได้ แต่ยังไม่ต้องเริ่มจากชื่อ pattern ครับ เริ่มจากถามก่อนว่า **เราคาดว่าอะไรจะเปลี่ยนอีก?** ถ้า boundary ช่วยกัก change แบบนั้นไว้ได้ มันก็คุ้มกับ code ที่เพิ่มขึ้น ถ้าตอบไม่ได้ เราอาจกำลังเพิ่ม layer เฉย ๆ

### Practice: review the blast radius of one change

หยิบ requirement ที่เพิ่ง ship ไปสักตัว แล้วลองตอบ:

1. ต้องแก้กี่ module?
2. มีกฎข้อไหนถูกแก้ซ้ำมากกว่าหนึ่งที่?
3. มีรายละเอียดของ database หรือ provider หลุดเข้า business หรือ API code ไหม?
4. change หน้าตาแบบนี้มีโอกาสกลับมาอีกหรือเปล่า?
5. interface หรือ module boundary ที่เล็กที่สุด ซึ่งจะกัก change รอบหน้าไว้ได้ หน้าตาเป็นยังไง?

ไม่ต้องรื้อทั้งระบบครับ หา change ที่กลับมาบ่อยสักชนิด ล้อมมันไว้ แล้วดูว่ารอบถัดไปง่ายขึ้นไหม feedback จากงานรอบหน้ามีประโยชน์กว่าการเอา architecture template ไปครอบทุกที่

**ราคาที่ต้องจ่าย:** boundary ทุกชั้นเพิ่ม concept, interface และที่ให้คนเดินหา code ครับ มีน้อยไป change ก็กระจาย มีเยอะไปงานง่าย ๆ ก็กลายเป็นพิธี judgment อยู่ที่การเลือกว่าจุดไหนแพงพอจะคุ้มกับ boundary หนึ่งชั้น

## Production Ownership — Done Means Observable

ช่วง early career คำว่า เสร็จ คือ merge แล้ว deploy แล้ว แต่นิยามฝั่ง senior เข้มกว่านั้นครับ: **เสร็จ คือรันอยู่ใน production มองเห็นพฤติกรรมของมันได้ และคุณรู้ว่าจะ roll back ยังไง**

![Done means observable and rollback-ready, not just merged](/assets/images/blogs/mid-level-2026/done-means-observable.png)

### Observability that answers questions

การมี dashboard ไม่เท่ากับมี observability นะครับ ข้อสอบคือคุณตอบคำถามเฉพาะเจาะจงได้เร็วแค่ไหนต่างหาก สำหรับ wallet:

- success rate ของ top-up เปลี่ยนไปไหมหลัง deploy เมื่อวาน?
- user คนหนึ่งบอกว่า balance ของเขาผิด — คุณไล่ transaction ทุกตัวที่แตะ wallet ของเขา เรียงตามลำดับ ได้ภายในห้านาทีไหม?
- provider ช้าอยู่ ณ ตอนนี้ — ช้าแค่ไหน ตั้งแต่เมื่อไหร่ แล้วช้าทุก request หรือแค่ region เดียว?

ถ้าทุกคำถามข้างบนพาคุณไปนั่ง grep raw logs เป็นชั่วโมง แปลว่ายังมีงาน engineering เหลืออยู่ครับ เริ่มจาก structured logs ที่มี request ID ไหลครบทุก hop แล้วเก็บ metric ในระดับ operation ด้วย เช่น top-up ถูกลองกี่ครั้ง สำเร็จกี่ครั้ง ล้มเหลวกี่ครั้ง อย่านับแค่ HTTP 200 อย่างเดียว ถ้าระบบข้ามหลาย service ก็ควรมี trace ให้ตามต่อได้ โพสต์ fundamentals ให้ vocabulary ของ failure ไว้ ส่วน observability คือหลักฐานว่า failure นั้นเกิดที่ไหน

### Debugging as hypothesis reduction

มี log, metric กับ trace แล้ว งาน debug ก็ยังไม่จบเองนะครับ skill ถัดไปคือใช้หลักฐานพวกนี้โดยไม่กระโดดแก้ไปเรื่อย

1. **บอกอาการให้แคบก่อน** คำว่า API ช้า ยังช่วยไม่มาก แต่ถ้าบอกว่า request ของ account ที่มีเกิน 10,000 records ใช้เวลาเกินสองวินาที แบบนี้เริ่มทดสอบต่อได้
2. **หาขอบเขตของปัญหา** เริ่มเมื่อไหร่ กระทบ version, region, tenant, request type หรือ data shape แบบไหน แล้วอะไรยังทำงานปกติ?
3. **ตั้ง hypothesis ทีละข้อ** เขียนไปเลยว่าคิดว่าอะไรเกิด และหลักฐานแบบไหนจะหัก hypothesis ข้อนี้ได้
4. **เช็คหลักฐานที่ถูกที่สุดก่อน** เทียบ request ที่ผ่านกับที่พัง เปิด trace ดูเวลา deploy รัน `EXPLAIN` หรือใช้ `git bisect` อย่าเปลี่ยนสามอย่างพร้อมกัน ไม่งั้นตอนหายเราก็ยังไม่รู้ว่าอะไรแก้ได้
5. **เช็คว่าหายแล้ว และเก็บสิ่งที่เรียนรู้** กลับไปทดสอบอาการเดิม เฝ้าดูว่าจะกลับมาไหม แล้วจดว่า signal ตัวไหน ถ้ามีตั้งแต่แรก จะช่วยย่นเวลารอบนี้ได้

[โพสต์ fundamentals](/blog/fundamentals-how-much-you-need-early-career-2026-th) ให้เครื่องมือแยกชิ้นไว้แล้ว ตั้งแต่ networking, query plan, connection limit ไปจน failure mode ของ dependency ส่วน mid-level คือการหยิบเครื่องมือพวกนั้นมาต่อเป็นวิธีสืบปัญหา ในวันที่คำตอบยังไม่โผล่มาให้เห็นครับ

### Deploys and the rollback question

ก่อน rollout อะไรก็ตาม ถามหนึ่งคำถาม: **ถ้าอันนี้ผิด เรากลับมายังไง?** บางทีคำตอบก็ง่าย — deploy เวอร์ชันก่อนหน้ากลับไป จบ แต่พอมี data เข้ามาเกี่ยว เรื่องจะเริ่มสนุกครับ เพราะ schema change มัน un-deploy ไม่ได้ วินัยมาตรฐานของการแก้ schema ที่มีของวิ่งอยู่โดยไม่มี downtime คือ:

<MermaidDiagram
  chart={`
flowchart LR
    E["Expand\nAdd the new column/table\nOld and new coexist\nNothing reads the new one yet"] --> M["Migrate\nDual-write old + new\nBackfill history\nVerify counts match"] --> C["Contract\nSwitch reads to new\nWatch, then remove old path"]
`}
  size="medium"
/>

Expand, migrate, contract ครับ แต่ละ phase ย้อนกลับได้ด้วยตัวมันเอง ซึ่งนั่นคือประเด็นทั้งหมดเลย — ไม่มีวินาทีไหนที่คุณติดอยู่ในขั้นที่เดินถอยไม่ได้ ลอง rename column บนตาราง wallet transactions ด้วยท่า `ALTER TABLE` แบบใสซื่อดูสิครับ จะมีหนึ่ง deploy ที่โค้ดเวอร์ชันเก่ากำลังเขียนลง column ที่ไม่มีอยู่แล้ว เวอร์ชันสาม phase มันงานเยอะกว่าและน่าเบื่อสุด ๆ — แต่ความน่าเบื่อนั่นแหละคือเป้าหมาย

ก่อนบอกว่า change นี้พร้อม ลองเช็คสั้น ๆ ครับ:

- application version เก่ากับใหม่รันพร้อมกันได้ไหม?
- ทั้งสอง version อ่าน record เก่าได้หรือเปล่า?
- rollout หยุดกลางทางได้ โดย data ไม่เสียไหม?
- backfill หยุดแล้วรันต่อได้ และรันซ้ำได้ไหม?
- ใช้หลักฐานอะไรยืนยันว่า migration ครบแล้ว?
- เมื่อไหร่ถึงลบ path เก่าได้อย่างปลอดภัย?

### Incident behavior

ตอน production พัง ผมใช้ลำดับนี้ครับ: **ทำให้นิ่งก่อน ค่อยหาสาเหตุ แล้วค่อยป้องกันรอบถัดไป** ถ้า roll back, ปิด flag หรือ fail over แล้วหยุดผลกระทบได้ ก็ทำก่อน แม้ตอนนั้นจะยังตอบ root cause ไม่ครบ เพราะ outage ที่หยุดแล้วแต่ยังต้องสืบต่อ ยังดีกว่าปล่อยให้ user เจอปัญหาระหว่างที่ทีมพยายามเข้าใจทุกอย่าง หลังระบบนิ่งค่อยกลับมาเขียนว่าเราพลาด signal ไหน ขาด guardrail อะไร และจะเปลี่ยนอะไร โดยคุยที่ระบบ ไม่ไล่หาคนรับผิดครับ

**ราคาที่ต้องจ่าย:** การ own production หมายถึงคุณต้องตามเรื่องจนจบครับ บางทีก็ on-call แบบมี pager บางทีก็คือคนที่กลับมาไล่ต่อเช้าวันถัดไป พอ slice ของคุณพัง จะจบคำอธิบายแค่ว่า โค้ดฝั่งผมทำงาน ไม่ได้แล้ว คุณต้องช่วยทำให้ระบบนิ่ง เข้าใจสาเหตุ แล้วทิ้งระบบไว้ในสภาพที่ปลอดภัยกว่าเดิม ความรับผิดชอบแบบนี้เหนื่อยได้ เพราะงั้น ownership ที่ดีต้องมี rotation, documentation และทีมที่ไม่ฝากทุกอย่างไว้กับ hero คนเดียวด้วย

## Correctness Under Failure — The Senior Upgrade of Transactions

[โพสต์ fundamentals](/blog/fundamentals-how-much-you-need-early-career-2026-th) คุยเรื่อง transaction, isolation level, race condition และ delivery guarantee ของ queue ไว้แล้วครับ พอมาถึง mid-level คุณต้องเริ่มมองเห็นตอนของพวกนี้ชนกัน ACID คุ้มครองงาน **ภายใน** database ตัวเดียว แต่ failure หลายแบบเกิด **ระหว่าง** ระบบ ซึ่งไม่มี transaction ก้อนเดียวครอบทั้งหมด

ของพวกนี้คือ fact ประจำ domain ของระบบที่แตะเงินทุกระบบครับ ทำพลาดเมื่อไหร่ เสียเงินของแท้เมื่อนั้น:

**At-least-once delivery แปลว่าของซ้ำมาแน่ เสมอ** message queue ส่งซ้ำได้ webhook replay ได้ user กดปุ่มรัว ๆ ได้ network timeout **หลังจาก** operation สำเร็จไปแล้วก็ได้ — client เลย retry สิ่งที่สำเร็จไปแล้วซ้ำอีกรอบ คุณไม่มีสิทธิ์เลือกว่าจะไม่ให้ของซ้ำเข้ามานะครับ คุณเลือกได้อย่างเดียวว่าจะทำให้มันไม่มีพิษ นั่นคือ **idempotency**: operation เดิม ทำซ้ำสองครั้ง ต้องให้ผลเท่ากับทำครั้งเดียว

<MermaidDiagram
  chart={`
sequenceDiagram
    participant App as Mobile App
    participant WS as Wallet Service
    participant DB as Database

    App->>WS: POST /topups (Idempotency-Key: abc-123)
    WS->>DB: INSERT topup, key=abc-123
    DB-->>WS: created
    WS--xApp: response lost (network timeout)
    Note over App: user's app retries
    App->>WS: POST /topups (Idempotency-Key: abc-123)
    WS->>DB: key abc-123 exists?
    DB-->>WS: yes — return original result
    WS-->>App: 200 (same topup, not a second charge)
`}

/>

**Check-then-act พังเมื่อเจอ concurrency** อ่าน balance เช็คว่าพอ แล้วค่อยหัก — สอง request วิ่งมาพร้อมกัน ผ่านด่านเช็คทั้งคู่ก่อนที่ใครจะได้หักสักคน โพสต์ fundamentals ตั้งชื่อ race pattern นี้ไว้แล้ว เวอร์ชัน mid-level คือการมองมันออก **ในตอนรีวิว** ในโค้ดที่หน้าตาบริสุทธิ์สุด ๆ และรู้จักวิธีแก้เป็นชื่อ ๆ: atomic conditional update (`UPDATE ... WHERE balance >= amount`), optimistic locking ด้วย version column หรือ serializable isolation ในจุดที่คุ้มค่ากับต้นทุนของมัน

**Backfill คือ production code** สักวันคุณจะต้อง recompute แถวย้อนหลังเป็นล้าน — bug คิด fee ผิดมาสองเดือน หรือ column ใหม่ต้องมีค่าย้อนหลัง backfill ที่ไม่ **resumable** (มันจะไปตายเอาแถวที่ 700,000) และไม่ **idempotent** (มันจะถูกกดรันสองรอบแน่นอน) ก็คือ incident ที่ใส่ชุด maintenance script มาหลอกเราดี ๆ นี่เอง ต้องแบ่ง batch มี checkpoint ทำให้รันซ้ำ batch ไหนก็ไม่มีพิษ และ rate-limit ไว้ไม่ให้มันลาก database ตัวหลักล่มไปด้วย

**เงินต้องบวกลบแล้วลงตัว — แบบพิสูจน์ได้** ชั้นสุดท้ายคือ reconciliation: การเช็คอัตโนมัติเป็นกิจวัตรว่า transaction สถานะ COMPLETED ของเรา match กับ settlement record ของ provider และผลรวม ledger entry ทั้งหมดของ wallet เท่ากับ balance ของมัน ไม่ใช่เพราะเราคาดว่าจะมี bug นะครับ — แต่เพราะนี่คือกลไกที่ **เจอ** bug ที่เราไม่ได้คาด ภายในไม่กี่วัน แทนที่จะเป็นไม่กี่ไตรมาส

**ราคาที่ต้องจ่าย:** งานพวกนี้เงียบมากตอนที่มันสำเร็จครับ ไม่มีใครเห็น double-charge ที่ไม่เกิดขึ้น และประโยคว่า reconciliation ยัง match อยู่ ก็ไม่ใช่ demo ที่หวือหวา เก็บหลักฐานไว้เถอะครับ ทั้ง test, reconciliation result, incident trend และ runbook เพราะ reliability จะถูกมองเห็นและให้คุณค่าได้ง่ายขึ้น เมื่อทีมเห็นว่ามันถูกดูแลยังไง

## Judgment — The Expensive Skill

ทุกอย่างที่ผ่านมาเป็นกลไกที่ฝึกได้ครับ แต่ตัวนี้มาช้ากว่าเพื่อน: **judgment** — การตัดสินใจให้ดีในวันที่ไม่มีใครรู้คำตอบที่ถูก และทุกทางเลือกมีราคาของมัน

- **Build vs buy** ถ้าเป็น engineer สัญชาตญาณแรกมักอยากสร้างเองครับ สนุกกว่า แถมดูเหมือนฟรีเพราะเงินเดือนก็จ่ายอยู่แล้ว แต่ลองคิดต่ออีกนิด การสร้างเองแปลว่าทีมต้องดูแล maintenance, edge case, security patch และ on-call ไปตลอดอายุของระบบ payment provider มีอยู่เพราะโจทย์ย้ายเงินให้ถูกต้องลึกมาก เรา integrate เข้ามา แล้วเก็บแรงของทีมไว้สร้างส่วนที่ทำให้ product ต่างจากคนอื่นก็ได้ แต่ buy ก็ไม่ได้ถูกเสมอไปนะครับ ถ้า domain นั้นคือจุดแข็งของ product คุณก็คงไม่อยากยกมันให้คนนอกทั้งหมด judgment อยู่ที่การคิดราคาของทั้งสองทางให้ครบ

- **Boring technology** ถ้าระบบทำข้อมูลหายไม่ได้ คำถามที่ใช้การได้กว่าคือ **ทีมนี้ operate อะไรไหวตอนตีสาม?** PostgreSQL, queue กระแสหลัก หรือ cache ธรรมดา อาจฟังไม่น่าตื่นเต้น แต่ failure mode มีคนเขียนไว้เยอะ หาคนรับช่วงง่าย และปัญหาส่วนใหญ่ยังพอ search เจอ เทคโนโลยีใหม่ทุกตัวใช้ innovation budget ของทีมครับ เก็บ budget ก้อนนั้นไว้จ่ายในจุดที่ความใหม่ซื้อประโยชน์กลับมาได้ดีกว่า

- **When not to refactor** ช่วง mid-level นี่อยากรื้อของเก่าง่ายมากครับ คุณเริ่มมองเห็นแล้วว่าโค้ดส่วนไหนรก แต่ยังไม่รู้ครบว่าความรกนั้นซ่อน edge case อะไรไว้บ้าง รื้อหมดแล้วคุณอาจได้เรียน edge case ชุดเดิมใหม่ผ่าน incident ทีละตัว ผมเขียนเรื่องนี้ไว้ทั้งโพสต์แล้ว — [The Rewrite Mindset](/blog/the-rewrite-mindset-when-its-worth-it-and-when-its-not-th) — ช่วงนี้ขอเหลือหลักเดียว: refactor เมื่อความรกทำให้งานตอนนี้ช้าลงแบบวัดได้ ถ้ามันแค่ไม่ถูกใจ ปล่อยไว้ก่อนได้ครับ

- **Recognizing good enough** โค้ดบางส่วนต้องแน่นมาก เช่น balance update กับ idempotency path แต่ report หลังบ้านบางตัวอาจต้องแค่พร้อมใช้วันพฤหัส ถ้าเราใช้มาตรฐานหนักเท่ากันทุกจุด correctness budget จะหมดกับงานที่ไม่ได้คืนประโยชน์เท่ากันครับ judgment คือการรู้ว่า **จุดไหน** ต้องใช้มาตรฐาน **ระดับไหน**

สัญญาณหนึ่งที่บอกว่า judgment กำลังพัฒนา คือ design ของคุณมักเรียบง่ายลงครับ และ ship พร้อม list สั้น ๆ ว่าตั้งใจ **ไม่ทำ** อะไร เพราะอะไร มันอาจดูน่าตื่นเต้นน้อยกว่า architecture diagram ใหญ่ ๆ แต่ทีมเอาไป ship และ operate ต่อได้ง่ายกว่ามาก

**ราคาที่ต้องจ่าย:** judgment แปลว่าคุณต้องรับผิดชอบเส้นทางที่ไม่ได้เลือกด้วยครับ ถ้า execute ตามการตัดสินใจของคนอื่น ผลของการตัดสินใจเป็นของเขา แต่พอคุณเป็นคนเคาะเอง — buy แทน build, เลื่อน refactor ออกไป — แล้ววันหนึ่งมันกลายเป็นทางที่ผิด จะไม่มี ticket ให้ชี้นิ้วใส่ ประโยคว่า ผมตัดสินใจเอง และนี่คือเหตุผล เป็นประโยคที่คุณต้องพูดได้ในวันที่แย่ที่สุด ไม่ใช่เอาไว้พูดเฉพาะวันที่ผลออกมาสวย

## Code Review as a Multiplier

ช่วง early career การรีวิวคือสิ่งที่คนอื่นทำ **กับ** โค้ดของคุณ พอถึง mid-level มันกลายเป็นหนึ่งในกิจกรรมที่ leverage สูงที่สุดของคุณทั้งสองขา — และระดับความสูงของสิ่งที่คุณมองหาจะเปลี่ยนไป

อีกฝั่งก็สำคัญครับ: **ส่งงานที่คนอื่นรีวิวไหว** PR สองพันบรรทัดมักได้ LGTM เพราะคนอ่านไม่ไหว ไม่ได้แปลว่าทุกอย่างดีหมด ลองแยกให้หนึ่ง PR มี concern หลักเรื่องเดียว เขียน description ว่าทำไมถึงเลือกทางนี้ และบอกไปเลยว่าจุดไหนยังไม่มั่นใจ คนรีวิวจะช่วยคุณได้ง่ายขึ้นเยอะ

### Test the risk, not the method

เลือก test ให้เข้ากับ risk ครับ:

- pricing หรือ permission rule ที่เป็น pure logic มักใช้ unit test ก็พอ
- query, constraint หรือ transaction boundary ควรรันกับ database engine ที่ใช้ในงาน
- external integration ต้องมี contract fixture, mock server, sandbox check หรือ consumer-driven contract สักแบบ
- user journey ที่สำคัญอาจคุ้มกับ E2E test เล็ก ๆ แต่ไม่ต้องลาก browser มาทดสอบทุก branch
- ordering, duplicate delivery และ race condition ต้องลองรันซ้ำหรือรันพร้อมกัน happy path รอบเดียวบอกอะไรได้น้อยมาก
- legacy code ที่ยังไม่กล้าแก้ ใช้ characterization test ล็อก behavior เดิมไว้ก่อนได้

ใน PR หนึ่งตัว ลองหาก่อนว่า failure ไหนแพงสุด มันเกิดที่ boundary ไหน dependency ตัวไหนต้องใช้ของแท้ และตัวไหนแทนได้อย่างปลอดภัย วิธีนี้ให้ testing strategy ที่มีเหตุผลกว่า mock ไปเรื่อยจน coverage เป็นสีเขียวครับ

ส่วนปี 2026 มีของเพิ่มมาอีกอย่างครับ: โค้ดใน PR จำนวนมากเริ่มจาก AI รวมถึง PR ของคุณเองด้วย skill การรีวิวยังใช้ชุดเดิม แต่ **ปริมาณ** โค้ดโตเร็วขึ้น และโค้ดที่ดูน่าเชื่อก็ยังพาเราไปผิดทางได้ เพราะงั้นคำถาม level 2 ถึง 4 ยิ่งสำคัญ ใครเขียน draft แรกไม่สำคัญเท่ากับว่า engineer ที่กด approve เข้าใจผลของมันแค่ไหน

**ราคาที่ต้องจ่าย:** งานรีวิวที่ดีมันช้า และมันไม่ผลิต commit ที่มีชื่อคุณเลย หนึ่งไตรมาสของการรีวิว level 4 อย่างตั้งใจ จะไม่โผล่ใน metric ส่วนตัวของคุณสักตัว — มันโผล่แค่ในกราฟ incident ของทีมที่ค่อย ๆ ลดลง คุณต้องเชื่อในบัญชีล่องหนเล่มนั้นครับ เพราะผลตอบแทนมันมีอยู่ของแท้ แค่ไม่มีป้ายชื่อแปะ

## Making Your Work Legible — The People Layer

มีเรื่องหนึ่งในที่ทำงานที่ฟังแล้วอาจขัดใจครับ: **คนอื่นเอางานที่เขามองไม่เห็นหรือไม่เข้าใจ ไปใช้ต่อไม่ได้** manager วางแผนต่อไม่ได้ engineer คนถัดไปต่อยอดไม่ได้ และคนเข้า on-call รอบหน้าก็ operate ไม่ได้ การทำให้งานอ่านออกจึงไม่เท่ากับการอวยตัวเองครับ มันคือการสื่อสารในงาน engineering ผ่าน artifact ชุดเดียวกับที่โพสต์นี้สะสมมาตลอดทาง:

![The artifacts that make engineering work legible](/assets/images/blogs/mid-level-2026/legibility-artifacts.png)

- **เขียนของลงกระดาษ** decision ที่คุยกันใน call เมื่อวาน อีกสัปดาห์ก็เริ่มจำกันคนละแบบแล้วครับ เขียนสั้น ๆ ก็ยังดีว่าเลือกอะไร ไม่เลือกอะไร และเพราะอะไร ถ้า debug กันทั้งวัน ก็เก็บเป็น runbook ให้รอบหน้าใช้สิบนาทีแทน ส่วนคำถามที่ตอบใน channel ได้ ก็ตอบให้ทีมเห็นพร้อมกัน ดีกว่าเก็บคำตอบไว้ใน DM คนเดียว

- **เห็นต่างโดยไม่มีดราม่า** บาง design ที่คุณไม่เห็นด้วยก็จะเดินหน้าต่อครับ ตอนคุยกัน ให้ยก concern เป็น path ที่เช็คได้ เช่น ถ้า webhook มาก่อน transaction commit จะเติมเงินซ้ำไหม แล้วเสนอทางเลือกพร้อม cost ของมัน พอทีมตัดสินแล้วก็ค่อยเดินไปกับ decision นั้นเต็มที่ คุณไม่จำเป็นต้องชนะทุก discussion ถึงจะสร้าง trust ได้

- **บอกข่าวร้ายให้เร็ว** ถ้า milestone คือวันพฤหัส แต่วันจันทร์คุณเห็นแล้วว่าไม่ทัน บอกตอนนั้นเลยครับ update อาจสั้นแค่ว่า provider sandbox ยัง block end-to-end test ถ้าบ่ายวันอังคารยังไม่คลาย เราควรขยับ demo หรือโชว์ stubbed path แทน ข้อความแบบนี้ส่งยาก แต่ทีมยังมีเวลาเลือก ถ้ารอถึงวันพฤหัส ทางเลือกก็แทบไม่เหลือแล้ว

- **ปลดบล็อกให้ junior — ด้วยคำถาม** สิบห้านาทีของคุณอาจช่วยเขาได้ทั้งวันครับ ถ้ายื่นคำตอบให้ ปัญหาวันนี้ก็จบ แต่ถ้าพาไล่ว่าควรเช็คอะไรก่อน เพราะอะไร และ error กำลังบอกอะไร รอบหน้าเขาจะไปต่อเองได้มากขึ้น ความ senior เลยไม่ได้อยู่ที่คุณทำอะไรคนเดียวได้บ้างอย่างเดียว มันอยู่ที่คุณช่วยให้เรื่องยากของคนรอบตัวเบาลงได้แค่ไหนด้วย

**ราคาที่ต้องจ่าย:** ชั่วโมงที่ไม่ผลิตโค้ดเลยครับ doc, comment ในรีวิว, สิบห้านาทีที่ว่า, runbook — สัดส่วนที่โตขึ้นเรื่อย ๆ ของงานที่ดีที่สุดของคุณ จะเลิกวัดเป็น commit ได้ ช่วงแรกมันให้ความรู้สึกเหมือนกำลังถอยหลังในฐานะ engineer แต่ไม่ใช่ครับ นี่แหละคือหน้าตาของการเป็น multiplier เมื่อมองจากด้านใน

## Business Context — Which Technical Win Does Not Matter

ชั้นสุดท้ายของ senior mindset และเป็นชั้นที่ engineer ต่อต้านนานที่สุด: การรู้ว่างานของตัวเองต่อสายเข้ากับเงินยังไง เบื้องหลังทุกการตัดสินใจทางเทคนิค มีคำถามเดียวที่ทำให้ทุกอย่างเข้าที่: **ถ้าของชิ้นนี้ผิด เลท หรือล่ม บริษัทเสียอะไร?**

ลองเทียบสอง path ใน wallet ครับ balance update ต้องมี idempotency, reconciliation และ review แบบละเอียด เพราะพลาดแล้วเงินหาย ส่วนหน้า admin settings ใช้ test กับ review ตามปกติก็พอ เพราะเคสแย่สุดอาจเป็นความรำคาญ มาตรฐานสองแบบนี้ใช้ได้ทั้งคู่ **เมื่ออยู่ถูกที่** ถ้าเอาความเข้มระดับ payment ไปลงทุกหน้า ทีมจะหมดเวลา ก่อนถึง path ที่ต้องการความเข้มนั้นมากกว่า

เลนส์ตัวนี้ต้องการความรู้ที่ไม่ได้อยู่ใน codebase: บริษัทของคุณหาเงินยังไงกันแน่ ระบบไหนอยู่บนเส้นทางรายได้ หนึ่งชั่วโมงที่ checkout ล่มราคาเท่าไหร่ เทียบกับหนึ่งชั่วโมงที่ analytics pipeline ข้อมูลค้าง ทั้งหมดนี้ไม่มีสักอย่างที่เป็นความรู้ engineering — แต่ทุกอย่างเปลี่ยนการตัดสินใจ engineering ครับ

วิธีคิดนี้ช่วยตอนขอเวลาทำงานเทคนิคด้วยครับ ถ้าบอกแค่ว่า transaction module รก เราอยาก refactor งานก็มักแพ้ feature แต่ถ้าบอกว่า change ใน module นี้ใช้เวลานานขึ้นสามเท่า และเกี่ยวกับ incident สองในห้าตัวล่าสุด งาน cleanup สองสัปดาห์จะซื้ออะไรกลับมาได้บ้าง คนฟังก็ตัดสินใจง่ายขึ้น งานเหมือนเดิมครับ ต่างกันที่คุณอธิบาย cost กับผลตอบแทนออกมาแล้ว

**ราคาที่ต้องจ่าย:** งานเทคนิคสวย ๆ บางชิ้น คุณจะเลือกไม่ทำมันครับ — redesign สุดหรูที่ไม่มีปัญหาของใคร justify มันได้, optimization บนเส้นที่ไม่มีใครจ่ายเงินให้ การปล่อยมืองานที่ถูกในเชิงเทคนิคแต่ผิดในเชิงเศรษฐศาสตร์ มันเจ็บทุกครั้งแหละครับ แต่การทำแบบนั้นได้ซ้ำ ๆ นี่แหละ ที่ทำให้ judgment ของคุณถูก trust กับการตัดสินใจที่ใหญ่ขึ้น

## Where AI Fits — Deliberately Short

โพสต์เกี่ยวกับ engineering ปี 2026 ติดหนี้คุณหนึ่งย่อหน้าที่ต้องพูดเรื่องนี้แบบเปิดเผยครับ เอาล่ะ

AI ช่วย draft ticket, สรุป log, ร่าง test หรือช่วยท้าทาย design ได้ครับ ผมเองก็ใช้ tool ในจุดที่มันช่วยงาน แต่การ scope ปัญหา เลือกว่า risk ตัวไหนควรถูกทดสอบก่อน รับผิดชอบผลใน production และอธิบาย trade-off ให้ทีมเข้าใจ ยังต้องมี engineer ที่รับผิดชอบการตัดสินใจอยู่ดี พอ code ผลิตได้ถูกลง งานรอบโค้ดเหล่านี้ก็ยิ่งกินสัดส่วนของงาน และของ bar มากขึ้น

ทีมที่พึ่ง mid-level ในงาน execution เยอะต้องระวังจุดนี้ครับ coding agent ทำให้คนหนึ่งคนสร้าง change ได้ใหญ่และเร็วขึ้น แต่ความเข้าใจระบบไม่ได้โตตามทุกบรรทัด ทางออกคงไม่ใช่ปิด tool และคงบังคับให้ทุกคนกลายเป็น senior พรุ่งนี้ไม่ได้ สิ่งที่ทำได้คือจัดทางเข้า production ให้คุมได้มากขึ้น:

- ระบุ owner ของ change ให้ชัด
- กำหนด review boundary กับ permission เท่าที่จำเป็น
- ใช้ automated check และเก็บ trace ว่า agent ทำอะไร
- rollout เป็นช่วง พร้อมทางย้อนกลับ

บริษัทใหญ่มี governance เพื่อให้คนจำนวนมาก ship ร่วมกันอยู่แล้วครับ coding agent แค่ทำให้เรื่องนี้เร่งด่วนขึ้น เพราะ change ที่คนหนึ่งคนสร้างได้โตขึ้นมาก

ฝั่ง workflow ผมเขียนแยกไว้แล้ว — [วัฒนธรรม engineering ที่ขยับไปทาง orchestration](/blog/the-shift-from-code-to-orchestration-building-ai-native-engineering-culture-th) กับ [ภาพรวมของ career landscape](/blog/the-future-of-software-engineering-career-th) — เลยจะไม่เล่าซ้ำที่นี่ สำหรับโพสต์นี้ ประโยคเดียวแบกทุกอย่างไว้ได้: **งานส่วนที่บรรยายไว้ข้างบนไม่เคยเป็นเรื่องของการพิมพ์โค้ดตั้งแต่แรก และนั่นแหละคือเหตุผลว่าทำไมมันยังเป็นส่วนที่ตั้ง bar อยู่**

## The North Star — The Bar Is Portable

ขอคุยเรื่องตลาดสักหน่อยครับ เพราะต่อให้ mindset ดีแค่ไหน ความกังวลเรื่องงานก็ยังอยู่

ใช่ครับ ตลาดแข่งขันสูง hiring bar ขยับขึ้น ทีมถูกจัดให้ lean ลง และ mid-level คือส่วนที่กว้างที่สุดของพีระมิด — level ที่มีคนเยอะที่สุดและ skill ทับซ้อนกันมากที่สุด ส่วน senior เองก็เป็นช่วงกว้างเหมือนกัน แค่ title ยังบอกไม่ได้ว่าระบบใหญ่แค่ไหน ต้อง own การตัดสินใจระดับไหน หรือมี support รอบตัวเท่าไหร่ เพราะงั้นการเทียบ title ข้าม job listing อาจพาเข้าใจผิดได้ครับ ลองเทียบ behavior, scope และบริบทที่ต้อง operate แทน นั่นคือสถานการณ์ตามที่มันเป็น และไม่มี mindset post ไหนเปลี่ยนมันได้

แต่นี่คือจุดที่ความกังวลมองพลาด และมันสำคัญมาก: **bar ที่บรรยายไว้ในโพสต์นี้อยู่ในตัวคุณ ไม่ได้อยู่ในป้ายชื่อบริษัท** ทุก skill ข้างบน — การแตกความคลุมเครือ, milestone thinking, production ownership, correctness under failure, judgment, legibility — ติดตัวไปได้หมด ไม่มีสักตัวที่ต้องมีนายจ้างชื่อดังถึงจะฝึกได้ และไม่มีสักตัวที่ถูกยึดไว้ตอนคุณลาออก

คุณไม่จำเป็นต้องเข้าบริษัทใหญ่ถึงจะได้ทำงานระดับ senior ครับ บริษัทท้องถิ่นที่ทำ payroll ให้ SME ไทยก็มีโจทย์ correctness under failure ส่วน startup 12 คนอาจส่ง **area** ให้คุณดูแล เร็วกว่าที่องค์กรใหญ่จะส่ง **feature** ให้หลายปี agency, consultancy หรือของที่คุณสร้างเองก็นับ บริษัท tech ใหญ่เป็นทางเลือกหนึ่งที่ค่าตอบแทนดีและมี trade-off ของตัวเอง เพียงแต่มันไม่ใช่ประตูบังคับของเส้นทางนี้

![Many paths, one north star](/assets/images/blogs/mid-level-2026/north-star.png)

ดังนั้น แทนที่จะ optimize เพื่อรอดจาก list ของใครสักคน optimize เพื่อเป็น **คนที่รับ problem ไปแล้ววางใจได้** ดีกว่าครับ เพราะคนแบบนั้นมีทางเลือกในทุกสภาพตลาด ทุกขนาดบริษัท ทุกประเทศ ความต้องการคนที่ own ผลลัพธ์ได้ ไม่เคยลดลงเลยนะครับ ที่ลดลงมีอย่างเดียว คือความต้องการคนที่ทำได้แค่ปิด task

และขีดจำกัดที่ต้องยอมรับของ blog post: ผมตัดสินอนาคตของคุณจากหน้าจอนี้ไม่ได้ และต่อให้ได้ก็ไม่อยากทำครับ สิ่งที่โพสต์หนึ่งทำได้ คือปักดาวเหนือไว้ดวงหนึ่ง แล้วบรรยายภูมิประเทศระหว่างทางให้แม่นที่สุดเท่าที่มุมมองของ engineer หนึ่งคนจะทำได้ ความหวังของผมคือข้อที่เดาได้ไม่ยาก — ขอให้คุณเดินเลยดาวดวงนี้ไป แล้วสักวันเวอร์ชันอนาคตของโพสต์นี้ เป็นโพสต์ที่คุณเขียนได้ดีกว่าผม

## A Realistic Path — Start With Work You Already Own

เวอร์ชันก่อนของ section นี้ให้ลองเขียน postmortem กับ migration plan แต่พอกลับมาอ่านอีกที มันขอเยอะไปหน่อยครับ หลายคนไม่มี incident ให้หยิบมาเขียน ไม่มีสิทธิ์วาง schema migration หรือในทีมอาจไม่มีธรรมเนียมเขียน design doc ด้วยซ้ำ ถ้าต้องรอโอกาสพวกนั้น path นี้ก็เริ่มยากเกินไป

เอาแบบที่เริ่มได้ในแทบทุกบริษัทดีกว่าครับ เลือก system, service หรือ feature สักตัวที่คุณแตะอยู่แล้ว แล้ววนสี่ขั้นนี้:

![A practical ownership loop: describe, break down, prioritize, and adjust](/assets/images/blogs/mid-level-2026/ownership-practice-loop.png)

1. **อธิบายของที่คุณดูแลอยู่** เลือก slice มาหนึ่งตัว แล้วเขียนเป็นภาษาคนว่า flow เริ่มจากไหน ผ่าน component อะไร data เปลี่ยนที่จุดไหน พังได้ยังไง เราจะรู้ได้ยังไง และถ้าพังจะพากลับมายังไง จะเป็นหนึ่งหน้า diagram หรือ README ก็ได้ครับ format ไม่สำคัญเท่ากับคนอื่นอ่านแล้วตาม flow ได้
2. **แตกงานก้อนใหญ่หนึ่งก้อน** หยิบ feature หรือ problem ถัดไปที่ใหญ่เกิน ticket เดียว แล้วแบ่งเป็นชิ้นที่มีขอบเขตชัด แต่ละชิ้นต้องบอกได้ว่าเสร็จเมื่อไหร่ ถ้ายังอธิบายทีละชิ้นไม่ได้ ก็ยังไม่ต้องรีบลงโค้ดครับ แตกต่ออีกหน่อย
3. **จัดลำดับชิ้นงาน** ลองถามว่าชิ้นไหนยังไม่รู้อะไรเยอะสุด ชิ้นไหน block ตัวอื่น ชิ้นไหนส่ง value ให้ user และชิ้นไหนย้อนกลับง่าย แล้วเสนอ order พร้อมเหตุผล คุณไม่ต้องมีอำนาจจัดทั้ง roadmap หรอกครับ แค่จัดลำดับ slice ของตัวเองให้ดี ก็ได้ฝึกเยอะแล้ว
4. **แชร์ แล้วค่อยปรับ** เอา breakdown กับ order ไปไว้ในที่ที่ทีมเห็นได้ พอเรียนรู้อะไรเพิ่มก็ update ว่าอะไรเปลี่ยน อะไรติด และตอนนี้ต้องการ decision เรื่องไหน เป้าหมายไม่ใช่เดาทุกอย่างให้ถูกตั้งแต่วันแรกนะครับ เป้าหมายคือทำให้งานยังอ่านออก ระหว่างที่มันกำลังขยับ

ถ้าวันหนึ่งงานส่ง postmortem, migration หรือ design ก้อนใหญ่มาให้ ก็รับโอกาสนั้นไว้ครับ ของพวกนี้ฝึกได้ดีมาก เพียงแต่มันเป็นส่วนต่อยอด ไม่ใช่ด่านบังคับ ส่วน loop ข้างบนเริ่มได้จากงานที่อยู่บนโต๊ะคุณสัปดาห์นี้เลย

ขอวางขอบเขตของผมไว้เหมือนโพสต์ที่แล้วครับ: ผมมีประสบการณ์ 7 ปี เลยพอเขียนช่วง mid ไป senior จากสิ่งที่เจอและสิ่งที่เฝ้าดูมาได้ หลังจากนั้นเส้นทางแตกออกหลายแบบเกินกว่าที่ผมจะวาดแผนที่ให้ทุกคนได้ จากจุดนั้น self-reflection ยังเป็นเครื่องมือที่ผมไว้ใจมากที่สุดครับ

## Closing — Every Cost Is a Transfer

ถ้าย้อนดูราคาของแต่ละ section คุณจะเห็นของคล้ายกันครับ: คำสั่งที่เคยชัดเริ่มน้อยลง ความคิดต้องเอาออกมาให้คนอื่นเห็น งานบางส่วนวัดเป็น commit ไม่ได้ และบางคืนคุณต้องตาม production ต่อ สิ่งที่ได้กลับมาคือ trust ทีละนิด **เราเอาความสบายบางส่วนไปแลกกับพื้นที่รับผิดชอบที่ใหญ่ขึ้น** ส่วน title เป็นชื่อที่บริษัทค่อยเอามาเรียกสิ่งนี้อีกที

![Every cost is a transfer from comfort to trust](/assets/images/blogs/mid-level-2026/comfort-trust-transfer.png)

ข่าวดีคือหลายอย่างเริ่มเองได้ครับ คุณอธิบาย flow ที่ดูแลอยู่ แตก task ก้อนถัดไป จัดลำดับพร้อมเหตุผล แล้วแชร์ให้ทีมเห็นได้เลย ไม่ต้องรอ promotion cycle หรือ project พิเศษ

ปิดท้าย ลองเลือก decision หนึ่งตัวในระบบของคุณครับ จะเป็น schema, retry policy หรือ build-vs-buy ก็ได้ แล้วตอบให้ได้สองคำถาม: **ทำไมถึงเลือกแบบนี้ และถ้าพังจะเกิดอะไรขึ้น?** ไล่ flow ให้ครบ ดูว่ามันกำลังป้องกันอะไร แล้วเขียนไว้สักหนึ่งหน้า ระหว่างทางคุณอาจพบว่า decision เดิมถูกหรือผิดก็ได้ ประเด็นคือหลังจากนี้คุณอธิบายและดูแลมันต่อได้ นั่นคือจุดเริ่มของการถูกวางใจให้รับ decision ตัวถัดไปครับ
