## Introduction

This is the sixth post in the backend system design series — same format: start with the most naive thing that works, then let real-world constraints push it into shape.

This one is the system the last two posts kept pointing at. In the omnichannel post (#3) we said "consume a *prepared, chunked* audience — don't re-derive it here, that's a different system." In the icon-serving post (#5) we said eligibility "comes from the Audience domain — a rule engine that writes a per-user membership store, its own post, next in the series." This is that post. The Audience Platform answers one question — **who?** — and hands the answer to whoever needs it.

The interesting part is *whoever needs it*, because the two consumers we've already built want the answer in completely different shapes:

- **Bulk, chunked list** — the omnichannel backbone wants "campaign C targets these 40M users," split into ~40–60K chunks it can stream through its delivery workflow. It reads the whole audience, once, in order.
- **Per-user membership store** — the icon serving path wants a point lookup at request time: "is user `10001` eligible for campaign 5, right now?"

Same audience logic — *gold-tier users in TH who topped up in the last 30 days* — two output shapes with opposite access patterns. That contrast is the spine of this post, the same way "two systems, opposite SLAs" was the spine of the icon post. One brain, two mouths.

Here's the road. The first half is the evolution of *how we compute the audience*; the payoff at the end is *how we serve it two ways*.

**Lower scale — a single box still works:**

1. **Inline SQL** at send time — works until the query is 40M rows mid-send
2. **Precompute on a schedule** — build the segment ahead of time, read it at send
3. **A read replica** — squeeze what a single box can do before going distributed
4. **An analytical engine** — OLTP stops being enough once the workload is real analytics

**Larger scale — now we're in distributed-system territory:**

1. **Spark** — rules and data outgrow what SQL-on-one-engine can express
2. **A data lake** — give Spark something to read that isn't your production DB
3. **Near-real-time** — "nightly" isn't fresh enough for some rules
4. **A reusable segmentation service** — stop building a custom job per use case

Let's start simple.

## The naive version — a SQL query at send time

You want to send a promo to *gold-tier users in Thailand who topped up in the last 30 days*. The honest first version writes exactly that, as a query, at the moment of sending:

```sql
SELECT user_id
FROM users
WHERE tier = 'gold'
  AND country = 'TH'
  AND last_topup_at > now() - interval '30 days';
```

The sender runs it, gets a list of IDs, and starts pushing. For a few thousand users this is completely fine — ship it. The rule lives in one place, it's always fresh (the query runs against live data), and there's nothing to operate.

> **P.S.** Stop here if that's you. Under roughly 1,000 users with a rule this simple — one or two conditions, no cross-table joins — this inline query isn't a temporary shortcut, it's the right answer. Nothing in the sections below makes the system more *correct*; they make it survive scale and complexity you don't have yet. Add machinery when the cracks below actually show up, not before.

Then it grows, and three cracks open at once:

- **The query gets heavy.** At tens of millions of rows, with `last_topup_at` from a different table and `tier` derived from yet another, this becomes a multi-join full scan on your **production** database — the same database serving live traffic. One marketing send can stall the app.
- **The send blocks on it.** The delivery job can't start until the query returns. A query that takes minutes (or times out) means the campaign sits there waiting, and a retry re-runs the whole expensive thing.
- **You can't reuse it.** The next campaign wants "gold-tier in TH" too, but with a different topup window. Every campaign re-derives its audience from scratch, hammering the DB again, and nobody can answer "how big is this segment?" without running the query.

All three say the same thing: **deriving the audience at send time, on the live DB, doesn't scale.** The fix is to move the computation off the hot path and do it *ahead* of the send.

> **P.S.** Before reaching for "precompute it ahead of time," though, it's worth asking honestly: how fresh does this audience actually need to be? If a user losing gold tier at 9am can still see the promo until the next scheduled run, that lag is acceptable — precompute solves everything in the next few sections. If eligibility has to flip the moment the underlying fact changes — a fraud flag, a balance crossing a threshold — no amount of precomputing on a schedule gets you there, and you're really looking at the near-real-time path later in this post, not this one. Most audience use cases tolerate the lag, which is why precompute is the right next step for most people — but that's a decision worth making on purpose, not by default.

## Precompute on a schedule — build the segment, then read it

**The problem:** the audience query is expensive and we run it at the worst possible moment — right when we need to send. But the inputs barely change minute to minute. "Gold tier in TH, topped up in 30 days" is basically the same list at 2pm as it was at 2am.

**The fix:** run the query *ahead of time* on a schedule, store the result, and let send time just **read** it. A nightly job computes each segment's members and writes them to a `segment_members` table:

```sql
-- nightly, per segment: delete + insert in ONE transaction,
-- so a re-run or a retry rebuilds the segment instead of doubling it
BEGIN;
DELETE FROM segment_members WHERE segment_id = 'gold_th_active';
INSERT INTO segment_members (segment_id, user_id, computed_at)
SELECT 'gold_th_active', user_id, now()
FROM users
WHERE tier = 'gold' AND country = 'TH'
  AND last_topup_at > now() - interval '30 days';
COMMIT;
```

Two boring details in that snippet are doing the real work:

- The job is **idempotent** — a bare `INSERT` alone is the classic mistake here, because the first retry after a crash doubles every member; the delete-and-insert in one transaction (with a primary key on `(segment_id, user_id)` as the backstop) makes re-running always safe.
- Once the audience is precomputed it can silently go stale, so **alert on the age of `max(computed_at)`, not just on job failure** — a nightly job that dies quietly on Friday night means a three-day-stale audience by Monday, and "the job didn't page" is not the same as "the data is fresh."

Now a send is a cheap read: `SELECT user_id FROM segment_members WHERE segment_id = 'gold_th_active'`. The heavy lifting happened overnight, when nobody's waiting.

This buys a lot — the send is now a simple `segment_id` lookup instead of a multi-join scan, the expensive query runs off-peak, and a segment is now a *thing* you can size and reuse. But it trades one problem for two new ones:

- **Freshness.** The list is only as fresh as the last rebuild. Run this batch job (a cron job) once a day, and the list can be up to 24 hours stale — it doesn't matter if the job runs at midnight, 1am, or 3am, what sets the staleness is how often it runs, not the exact time. A user who topped up this morning won't be "gold-tier active" until the next rebuild. For a weekly promo that's fine; for "remind users who *just* abandoned a cart," it's useless. Keep this in mind — it's what forces streaming later.
- **It still runs on the production DB, and it's still one box.** The rebuild needs somewhere to land: a new table, `segment_members(segment_id, user_id, computed_at)`, written by a job that still hits the same Postgres serving live traffic. And that table isn't small — a user can match more than one segment, so worst case it holds **S × N** rows (S = segments in the org, N = total users). As segments multiply and rules get more complex (cross joins, windowed aggregates over a year of transactions), a single box doing both live traffic and this analytical batch starts to slow down and fail.

The second crack is the structural one. We don't just need to precompute — we need to precompute *somewhere that isn't the live transactional database*. But "somewhere else" doesn't mean a cluster yet, and this next rung is the one most write-ups skip.

## Squeeze the single box first — a read replica

Most posts jump from "one Postgres" straight to "a Spark cluster," and readers walk away believing distributed compute is step three. It isn't. There's a cheap move first, and — being honest — a lot of companies could stop here forever.

**Get the segment job off the primary — for reads, and for where it lands.** Right now production is doing two different jobs: serving live transactional traffic, and holding the `segment_members` table this batch writes into. Split them apart:

- **Reads** — point the batch's queries at a **read replica**: a live copy of the primary, kept in sync through replication, that only accepts reads and never writes. Because it's read-only, it can absorb read traffic from anywhere — the live app's own reads as well as this batch job's — while every write, from the app or from this job, still has to go through the primary. Nearly every managed database offers a replica as a checkbox, and replication lag of a few seconds is irrelevant for a marketing audience that already tolerates hours of staleness.
- **Output** — don't write `segment_members` back into that same box. Land it somewhere else entirely: a small dedicated database used only for segments, a plain file in object storage (S3/GCS) the sender reads directly, or Redis holding the segment as a set.

None of these need Spark or a warehouse yet — the computation is still the same SQL query, just moved so production only does the one job it should: live traffic. This is a cheap infrastructure change that buys years at many companies — take it *early*, the moment the batch visibly competes with live traffic.

The next section solves a different problem: the read query itself getting too heavy even for a replica, not where the result is stored.

**Before going further, a quick recap.** Two consumers, two read patterns, and neither cares which engine produced the list:

- **Omnichannel** — wants the whole audience in one **chunk read**, in order.
- **Icon serving** — wants a single user's eligibility as a **real-time read**, a point lookup right now.

> Most of applications in this world maybe stop it right here already. Please keep in mind.

## Move the analytics to an analytical engine — from OLTP to OLAP

Same move as the last rung — precompute, then read from somewhere that isn't the hot path — just pointed at something else. Instead of reading from a replica, read from an **OLAP** engine.

Here's why. Once rules become 90-day windowed aggregates and multi-way joins over hundreds of millions of rows, even a replica starts taking hours — not because the hardware is weak, but because this is real analytical work now. A replica is still an **OLTP** engine (Postgres/MySQL: built for many small point reads and writes, one row at a time). **OLAP** engines are built for the opposite — scanning a few columns across millions of rows at once. That's the fix, and it's still not a cluster:

- **Columnar analytical database** — ClickHouse, or DuckDB on one beefy machine.
- **Managed warehouse** — BigQuery, Snowflake, Redshift, Athena.

Either runs the same kind of query one or two orders of magnitude faster. "A scheduled SQL query on the warehouse, output written to a segment table" is — quietly — the mainstream audience platform in 2026. Plenty of teams run exactly that at tens of millions of users and never need more.

> **P.S.** If your team is already Java/Spring, there's a lever worth pulling before reaching for a separate OLAP store: scale the batch job itself with **Spring Batch**. Chunked reads, transactions, checkpointing, restart-from-failure, and remote partitioning to spread the work across more than one JVM if needed — it can push a read replica further than a naive query would, with no new storage engine involved. It stops being enough at the same trigger as the rest of this rung: once the workload is genuinely OLAP-shaped, or once you need Spark's built-in distributed joins across billions of rows, Spring Batch doesn't have those baked in.

Worth being precise here, so "just point at something else" doesn't undersell what changed. A read replica and an OLAP engine solve *different* problems:

- **Replica** — same OLTP engine, just a synced copy. It only separates *where reads happen* from *where writes happen*; the data itself never leaves the OLTP world, and staying in sync is native database replication.
- **OLAP engine** — a different storage engine entirely, built to hold data in a shape suited for analytics. It doesn't fill itself through replication — something has to move the data in: an initial bulk copy, a scheduled Spark job, or CDC (streaming the OLTP write log into the OLAP store continuously).

Same job, the same *kind* of query — but a different source, and populating that source is its own problem. (The exact SQL text usually needs rewriting too — Postgres, ClickHouse, BigQuery, Snowflake, and Redshift each speak their own SQL dialect.) If you're not already running an OLAP store, a replica is the simpler, honest answer. If you do reach for OLAP, that data-movement problem is exactly the ingestion story the data lake section covers in full.

**So when do you actually leave a rung?** Rough triggers, not fixed rules:

| You're on | Move up when |
|---|---|
| Inline SQL at send time | audiences pass ~1M users, campaigns multiply, or a send query first slows the app |
| Precompute on the primary | the batch visibly competes with live traffic — the replica is one config change, take it |
| Read replica | analytical SQL takes hours on a row store — the shape is wrong, not the hardware |
| Warehouse / columnar DB | jobs can't finish inside their refresh window, rules need real *code* (custom scoring, ML features) rather than SQL, or the inputs no longer live in one database |

One more dial: **every rung up adds ongoing cost and on-call work.** A replica is a checkbox; a warehouse is a bill; the Spark-plus-lake world ahead is a bill *and* a team. Climb because a trigger fired, not because the next rung looks interesting on a resume.

If you're on a managed warehouse and it's coping, skim the next two sections. The series follows the self-managed path from here — Spark, then the data lake underneath it — because that's where things break in interesting ways at the largest scale, and because past this point, efficient progress needs a data lake no matter which engine got you here.

## Rules and data outgrow one database — move to Spark

**The problem:** even the warehouse rung has a ceiling, and it's less about size than about *shape*. Real targeting looks like "users whose **average** transaction over the last 90 days is above X, who used feature Y at least 3 times, who are **not** in the suppression list, joined against a churn score from the data-science team." The windowed aggregates and anti-joins are easy work for a warehouse — but the churn score arrives as files the DS team ships, the feature events live in a stream rather than a table, and the scoring logic the business actually wants is *code*, not a `WHERE` clause. When the inputs stop living in one queryable place and the rules stop being expressible as SQL, you need an engine that treats arbitrary data plus arbitrary logic as the normal case — over hundreds of millions of users and *billions* of transaction rows.

**The fix:** move the computation to a distributed engine. This is where Spark earns its place — it's built to process billions of rows in parallel across a cluster, and (the part I care about most) it lets you write the rule once and let the engine figure out the distribution.

Going deep, because you asked — here's what the audience job actually looks like:

**The job, in DataFrame terms.** Read the inputs as DataFrames, apply predicates, join attributes, aggregate, write membership:

```scala
val users   = spark.read.parquet("s3://lake/users/dt=2026-07-01")
val txns    = spark.read.parquet("s3://lake/transactions/dt=2026-07-01")
val churn   = spark.read.parquet("s3://lake/churn_scores/dt=2026-07-01")
val suppress = spark.read.parquet("s3://lake/suppression/current")

val txn90 = txns
  .where($"event_date" >= date_sub(current_date(), 90))
  .groupBy($"user_id")
  .agg(avg($"amount").as("avg_amt"), count("*").as("txn_count"))

val audience = users
  .where($"tier" === "gold" && $"country" === "TH")
  .join(txn90, Seq("user_id"))
  .join(broadcast(churn), Seq("user_id"))           // small dim → broadcast
  .where($"avg_amt" > 5000 && $"churn_score" < 0.3)
  .join(suppress, Seq("user_id"), "left_anti")       // exclude suppressed
  .select($"user_id")
```

**Broadcast the small side.** `churn_scores` and `suppression` are small relative to `users`/`transactions`. Marking them `broadcast` ships them to every executor and turns a shuffle-heavy join into a map-side join — the single biggest, easiest win. The default `autoBroadcastJoinThreshold` catches some of this, but be explicit for the ones you know.

**Watch data skew.** Group or join on something lumpy — a few "power users" with millions of transactions, a `country` where 80% of users live — and one partition does all the work while the cluster idles. The whole job runs as slow as that one straggler task. What helps: salt the hot keys, enable Adaptive Query Execution (`spark.sql.adaptive.enabled`) so Spark can split skewed partitions at runtime, and pre-aggregate before the join (the `txn90` rollup above shrinks the big side *before* it meets `users`).

**Partition the read, prune at the source.** Notice the inputs are partitioned by `dt=` / `event_date`. Reading `dt=2026-07-01` and filtering `event_date >= today-90` lets Spark **prune** to just the relevant date partitions instead of scanning all of history. This is the difference between reading 90 days and reading five years — and it's why the storage layout (next section) matters as much as the query.

One operating note before moving on: this rung is where the audience platform stops being a small side project. A Spark cluster brings job failures, executor OOMs, and version upgrades of its own — in practice it arrives together with the first person whose actual job is running it. Budget the human, not just the cluster.

Spark fixes the *compute*. But it raised a question by reading from `s3://lake/...` instead of the production DB — **where did that data come from, and why isn't Spark reading Postgres directly?** That's the data lake.

## Give Spark something to read — the data lake

**The problem:** Spark needs to read users, transactions, churn scores, feature events — and those live in a dozen different production systems (the payments DB, the auth service, the data-science team's outputs, the event stream). Pointing Spark at each live production database is exactly the load problem we've been running from, multiplied: now an analytical cluster is hammering every transactional system at once. And these jobs need *historical* data ("transactions over 90 days") that the hot OLTP databases prune or archive away.

**The fix:** land a copy of everything in one place built for analytical reads — a **data lake** — and let Spark read only from there. The production systems are never touched by a Spark job again. Going deep on the pieces, since this is where the real engineering lives:

**Columnar file format — Parquet.** Store the lake in a columnar format, not row-oriented CSV/JSON. An audience query touches a few columns (`tier`, `country`, `amount`) out of dozens. Parquet stores each column together, so Spark reads *only those columns* off disk and skips the rest. It also carries per-row-group min/max statistics, so the reader can skip whole chunks that can't match a predicate (**predicate pushdown**). On wide tables this is a 10×+ I/O difference versus row formats.

![Row storage reads every column off disk; columnar Parquet reads only the columns the query needs](/assets/images/blogs/design-audience-platform/parquet-columnar.png)

**Table format — Iceberg / Delta Lake.** Raw Parquet files in a folder get you 80% there but have no notion of a transaction — a reader can catch a half-written table mid-update, schema changes are a manual nightmare, and "what did this table look like yesterday?" is unanswerable. A table format (Apache Iceberg, Delta Lake) layers a metadata log over the Parquet files to give you **ACID snapshots** (readers always see a consistent version), **schema evolution** (add a column without rewriting history), and **time travel** (query `as of` a past snapshot — valuable for reproducing "why was this user in the audience last Tuesday?"). For an audience platform whose outputs drive money-spending campaigns, that reproducibility is worth the extra infrastructure to run.

**Partitioning strategy.** Lay the data out so the common queries prune well. Partition fact tables by **event date** (`dt=YYYY-MM-DD`) so a 90-day window reads 90 directories, not the whole history. Resist partitioning by something high-cardinality like `user_id` — you'll get millions of tiny files and the metadata overhead will be far bigger than the data itself (the classic **small-files problem**). When small files do accumulate from incremental writes, run **compaction** to rewrite them into fewer, larger files.

**Getting data in — ingestion.** Two patterns, usually both:
- **Batch snapshots** — a daily dump of a source table into the lake (`dt=2026-07-01/`). Simple, but a day stale and heavy for big tables.
- **CDC (change data capture)** — stream the source database's write-ahead log (via Debezium → Kafka) into the lake continuously, so the lake trails production by minutes, not a day. More moving parts, far fresher. This is also the starting point for the streaming approach in the next step.

**Orchestration.** The lake isn't one job — it's ingest → transform → build-segments → publish, with dependencies between them. An orchestrator (Airflow, Argo Workflows, Dagster) runs that DAG on schedule, handles retries, and won't start the segment build until its inputs for the day have landed.

CDC + compaction is fresh but is real infrastructure to run — Debezium, Kafka connectors, compaction jobs, and the orchestrator each want patching, monitoring, and someone to page. This is the rung where the platform acquires a team, which is exactly why the replica-and-warehouse section told you to make sure a trigger fired first. Most platforms land in the middle and tune per table: profile data daily is fine, transactions you want within the hour. This leads directly to the limitation that breaks "nightly" entirely.

## Nightly isn't fresh enough — refresh every 1–2 hours

**The problem:** a day stale is too stale for a lot of rules. "User topped up this morning → they should be eligible for the topup-bonus banner by lunch, not tomorrow." Marketing reacts within a day, not within a month. Nightly means every segment is, on average, 12 hours behind reality — and for anything tied to recent behavior, that's where the audience is wrong.

**The fix — and this is the honest middle ground, not the textbook one:** don't jump to second-level streaming. Move to **microbatch**. Refresh every segment on a **1–2 hour cadence** so its result is at most an hour or two old. At any moment you're maintaining anywhere from a few thousand to a few million segment results, each refreshed on that schedule. For most targeting rules, "fresh within the hour" might as well be real-time to the business — and it's *vastly* cheaper and simpler to run than a true streaming stack.

Worth being clear about what microbatch *is*, because it's an approach, not a specific technology. You can implement the same 1–2 hour cadence two ways: as **scheduled batch jobs** (an orchestrator fires the Spark job every hour) or as a **streaming engine running in micro-batch mode** (Spark Structured Streaming with a processing-time trigger, say). Both land at the same place. The batch-vs-streaming argument matters far less than people make it — **the end goal is getting eligibility right within your freshness target.** Pick whichever your team operates well; don't let the label drive the design.

Making a 1–2 hour cadence actually work, going deep:

**Incremental, not full-rebuild.** Recomputing every segment from five years of history every hour would overload the cluster and never finish — the refresh-window test again: an hourly job that takes 90 minutes isn't an hourly job. Lean on the table format from the last step: read only what *changed* since the last run (Iceberg/Delta incremental reads, or the CDC feed), update the affected users, and leave the rest. The hourly job touches deltas; a heavier full rebuild still runs nightly to correct any drift.

**Idempotent, snapshot-style publishes.** Each run produces a fresh, complete segment result and publishes it as a versioned snapshot the consumers switch to atomically — no half-updated segment ever visible. Same discipline as the icon post's "overwrite, never delete": a failed run leaves the previous good snapshot in place rather than blanking the segment.

**Tier rules by freshness.** Most segments are happy on the 1–2 hour cadence. A few ("abandoned a cart 10 minutes ago") genuinely need seconds — and *those*, and only those, justify a true low-latency streaming path (Flink, or Structured Streaming with a tight trigger, stateful, with event-time watermarks for late events). Treat real-time as the rare opt-in for the handful of rules that earn it, not the default for all of them. Paying streaming costs for "gold tier in TH" would be all cost, no benefit.

Now the audience is computed at scale and fresh within the hour. The last step is to stop building all of this by hand for every new campaign — and to serve those segment results in the two shapes our consumers need.

## Stop rebuilding per campaign — a reusable segmentation service

**The problem:** everything so far is real, but each segment is a hand-written Spark job somebody on the data team ships and maintains. Marketing wants a new audience and files a ticket; an engineer writes Scala, tests it, schedules it, wires its output to the membership store. That's a bottleneck and it doesn't scale past a handful of segments. The campaign platform (#7) can't self-serve audiences if every one needs an engineer.

**The fix:** turn the pile of hand-written jobs into a **platform**. Rules become *configuration*, not code; an engine compiles them to the batch/stream jobs; and the results are materialized into whatever shape the consumer needs.

- **Rules as a DSL / config.** A segment is defined as data, not code — predicates, time windows, joins against named attributes, a freshness tier (batch vs. real-time). Something like `tier = gold AND country = TH AND avg_txn_90d > 5000 AND NOT suppressed`, stored as data. Non-engineers (or the #7 UI) compose segments; the platform validates and estimates size before anything runs.
- **One engine, generated jobs.** The platform translates a rule into the actual Spark plan and, if the rule is tagged real-time, the streaming topology too. The author never touches Scala. New attributes get registered once in a catalog and become available to every rule.
- **Reusability.** Segments compose — "gold_th_active" can be referenced by ten campaigns and computed once. The platform dedupes the work instead of re-deriving the same audience per campaign, which is exactly the v1 mistake, finally fixed.
- **Guardrails before the money moves.** Estimating size before a rule runs is half the job; *checking it after* is the half that saves you. A rule edit that flips an `AND` to an `OR` can quietly turn 2M users into 40M, and now a campaign is about to spend real budget on 38M people who shouldn't get it. So the platform holds any segment whose size jumps outside a sane band run-over-run (say ±30%) until a human confirms it, before any consumer — delivery or serving — picks it up. It's a one-line anomaly check guarding the single most expensive mistake this system can make.

And here's the payoff the whole post has been building to. The ETL's native output is the **segment result** — for each segment/campaign, the list of users in it: `campaign → [users]`. It's written to **object storage** (S3/GCS) as a versioned snapshot every 1–2 hours; that snapshot is the shared artifact everything else reads. From that *one* artifact, the two consumers read in opposite directions:

<MermaidDiagram chart={`
flowchart TD
    ENG["Segmentation ETL"] --> SEG[("Segment results in object storage\ncampaign -> [users]")]
    SEG -->|"read + chunk"| OMNI["#3 Omnichannel\nstream the chunks out"]
    SEG -->|"invert (add/remove)"| MEM[("Membership index\nuser -> {campaigns}")]
    MEM --> ICON["#5 Icon serving\npoint lookup per request"]
`} />

**Shape 1 — delivery reads the segment result and chunks it (for #3 Omnichannel).** The omnichannel backbone reads an entire audience once, in order, to fan a campaign out. The segment result *is* what it consumes: it reads `campaign → [users]` and splits it into ~1K-user chunks to stream through its delivery workflow. For a 40–60M audience that's the ~40–60K chunks the omnichannel post sized its observability around. The audience platform produces the list; delivery does the chunking. One big sequential read — no random access needed.

**Shape 2 — invert the segment result into a per-user membership index (for #5 Icon serving).** The icon path asks the opposite question at request time: given *one* user, which campaigns are they in — in under a millisecond, thousands of times a second. Reading `campaign → [users]` to answer that would mean scanning every campaign's list per request, which is hopeless. So we build the **inverted index**: flip `campaign → [users]` into `user → {campaigns}` and store it for point lookup — a Redis set per user, `10001 -> {1, 2, 3, 4, 5}`. Back-of-envelope before you trust it: 100M users, each holding a set of a few small campaign IDs, lands in the tens of GB — a modest Redis cluster, nothing exotic. The inversion is a step in the same ETL: as each segment-result snapshot lands in object storage, fan its users out into their per-user sets. This is exactly the membership store the icon-serving post read from — now you can see how it's built: it's the segment results, turned inside out.

**Membership churns — and removals are the trap.** Each 1–2 hour refresh isn't the same list as last time: some users newly **match** a segment, some **drop out** (their last topup aged past 30 days, they hit the suppression list). Adds are easy — `SADD` the campaign to the user's set. Removals are the one people forget: if user `10001` no longer qualifies for campaign 5, you must `SREM` it from their set, or they keep an eligibility they've lost and the icon keeps showing. So the inversion can't be append-only. Two ways to keep it honest:

- **Diff and apply** — compare the new snapshot against the previous one per segment, compute the *added* and *removed* user sets, and apply both (`SADD` the adds, `SREM` the removes). Cheap when churn is small, which it usually is — but it leans on having the prior snapshot and getting the diff right. One way to make that diff itself cheap at scale: if user IDs are dense integers, load both snapshots as **Roaring Bitmaps** and let bitmap subtraction do the set math — `added = new.andNot(old)`, `removed = old.andNot(new)` — exact, and fast even at tens of millions of IDs, because the values stay compressed instead of sitting in a full array or hash set.
- **Rebuild and swap** — recompute every user's full set from the new snapshots and atomically swap it in (write to a shadow keyspace, flip a pointer). Dead simple and self-correcting — no drift can accumulate — at the cost of rewriting everything each cycle. The 1–2 hour cadence makes this affordable, which is a good reason not to go real-time. The sizing detail people miss: during the swap the shadow and live keyspaces coexist, so peak memory is roughly **double** steady state — size the cluster for the peak, or swap shard by shard.

Either way, the segment result in object storage is the source of truth; the Redis index is a derived, disposable view you can always rebuild from it. And one cheap safety net on top of whichever strategy you pick: give every per-user set a TTL slightly longer than the refresh cadence. Then any drift from a missed removal ages out on its own within a cycle or two, instead of persisting until someone notices the wrong banner in production.

**A word on Bloom filters — the false-positive trap.** It's tempting to shrink the membership check with a Bloom filter: instead of storing every user's full set, keep a compact probabilistic structure that answers "is user X in campaign Y?" in a fraction of the space. The catch is what *probabilistic* means here. A Bloom filter never gives a false negative, but it *will* return false positives at some rate — and for eligibility, a false positive is a user being told they qualify for something they don't: the wrong banner, a benefit they shouldn't get, a campaign that shouldn't reach them. The reason this matters: even a low false-positive rate becomes real business errors at volume. Eligibility is a correctness domain, not a "probably" one, so the membership store stays exact. Bloom filters earn their keep where a false positive is *harmless* — a pre-check in front of an expensive lookup you'll verify anyway — not as the final answer to "is this user eligible?"

## Wrapping up

We started with a `WHERE` clause typed at send time and ended with a platform that computes audiences over a data lake and serves them two ways. The path, in order:

1. **Inline SQL at send time** — fine until the query is 40M rows on the production DB, mid-send.
2. **Precompute on a schedule** — build the segment ahead (idempotently, with a staleness alarm), read it cheap at send; but it still pounds the OLTP DB.
3. **A read replica** — the cheap rung most write-ups skip; split reads and the segment's output off the primary, no new engine yet.
4. **An analytical engine** — once the workload is real analytics (OLTP vs. OLAP), a columnar DB or warehouse runs the same SQL far faster; plenty of teams rightly stop here.
5. **Spark** — for when the rules become code and the inputs stop living in one queryable place.
6. **A data lake** — Parquet + a table format, partitioned and ingested, so Spark never touches production.
7. **Refresh every 1–2 hours** — incremental ETL keeps thousands-to-millions of segment results fresh within the hour; true streaming stays the rare opt-in.
8. **A reusable segmentation service** — rules as config, one engine, guardrails on segment size, one segment result read two ways: chunked for delivery, inverted into a per-user index for serving.

The meta-point, same as the rest of the series: **nobody starts here.** A few thousand users and a simple rule? Type the `WHERE` clause and move on. Every layer above earned its place — scale forced precomputation, load forced the replica, real analytics forced the warehouse, code-shaped rules forced Spark, Spark forced the lake, recency forced microbatch, and repetition forced the platform. And each climb had a trigger you can measure — the batch that visibly slows live traffic, the job that can't finish inside its refresh window — not a technology that looked interesting.

And if there's one thing to carry over: the audience platform's job isn't just *computing* the right set of users — it's serving that set in the shape each consumer actually reads. Omnichannel streams it; icon serving point-looks-it-up. Get the computation right and pick the wrong serving shape, and you've built a brilliant brain that can't talk to either mouth.
